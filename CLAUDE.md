# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What is CaveWhere

CaveWhere is a Qt 6 / C++ desktop application for cave mapping and 3D visualization. It features 2D-to-3D survey morphing, loop closure detection, Git-based project sync, and LiDAR note processing.

## Build Commands

From a fresh checkout:
```bash
git submodule update --init --recursive
conan profile detect --force
conan install . -o "&:system_qt=False" --build=missing -of conan_deps
cmake --preset conan-release -DCMAKE_TOOLCHAIN_FILE=conan_deps/conan_toolchain.cmake
cmake --build build/<preset> --target all
```

The build directory name (`build/<preset>`) is generated by CMake from the Qt version and platform, e.g. `build/Qt_6_11_0_for_macOS-Debug`. Use `ls build/` to find the correct name.

Run the application: `./build/<preset>/CaveWhere`

Build and run tests:
```bash
cmake --build build/<preset> --target cavewhere-test cavewhere-qml-test
./build/<preset>/cavewhere-test -d yes 2>&1 | tee /tmp/cavewhere-test.log
./build/<preset>/cavewhere-qml-test --platform offscreen 2>&1 | tee /tmp/cavewhere-qml-test.log
```

Always redirect test output to a log file so failures can be inspected without re-running. Run a single C++ test with Catch2 tags (`cavewhere-test "[TagName]"`); run a single QML test with `cavewhere-qml-test --platform offscreen -input test-qml/tst_YourTest.qml`. Always run both suites before pushing.

## Architecture Overview

### Entry Point and Root Singleton

`main.cpp` sets up the QML engine and exposes `cwRootData` as a QML singleton — the central hub that owns and exposes all major subsystems to QML: `cwProject`, `cwCavingRegion`, `cwScrapManager`, `cwLinePlotManager`, `cwSurveyImportManager`, `cwSurveyExportManager`, `cwRemoteServices`, `cwPageSelectionModel`, `cwSettings`, and more.

### Data Model Hierarchy

```
cwCavingRegion (root)
└── cwCave (multiple)
    └── cwTrip (multiple)
        └── cwSurveyChunk (shots)
            ├── cwShot (from/to station, distance/bearing/inclination)
            └── cwNote (2D scanned image)
                └── cwScrap (digitized outlines)
```

### Project Persistence (`cwProject`)

Handles all file I/O and Git integration. Three formats: legacy `.cw` (JSON + SQLite, auto-converted on load), bundled `.cw` (zip archive, new default for single-file saves), and `.cwproj` directory (Git-backed for collaboration). Serialization uses Protocol Buffers (`cavewhere.proto`). Save/load operations are queued through `cwSaveLoad` to avoid blocking the UI thread.

### Rendering

Uses Qt's RHI (Rendering Hardware Interface). Key classes: `cwRegionSceneManager`, `cwRhiScene`, `cwRhiItemRenderer`. Line plots are handled by `cwLinePlotManager`. 3D math is in the `QMath3d` submodule.

### Survey Import/Export

`cwSurveyImportManager` handles Compass, Survex, and Walls formats via the `dewalls` submodule. `cwSurveyExportManager` outputs Survex and Compass; Compass station names are written in uppercase (Compass is case-sensitive; CaveWhere is not). `cwSurveyNetwork` performs loop closure detection.

### Remote/Git Features

`QQuickGit` submodule wraps libgit2. `cwGitHubDeviceAuth` handles OAuth. Credentials are stored via `qtkeychain`. `cwRemoteRepositoryCloner` and `cwRemoteAccountModel` manage remote repository workflows.

### UI Structure

- **Page navigation**: `cwPageSelectionModel` manages multi-page routing
- **Keyword filtering**: `cwKeywordItemModel` / `cwKeywordFilterPipelineModel` for hierarchical search
- **Themes**: All colors go through `Theme.qml` tokens — never hardcode hex values in component QML files

## Coding Conventions

- **Comment sparingly.** Let clear names and structure carry the code. Comment only to explain non-obvious behavior (a subtle invariant, a workaround, a "why" not visible from the code), and keep it brief.

### C++
- 4-space indentation, opening braces on the next line
- `UpperCamelCase` for classes, `m_` prefix for members, `lowerCamelCase` for functions, `k` prefix for file-scope/`constexpr` constants (e.g. `kMaxGridPointsPerAxis`)
- Full `const` correctness; use Qt signal/slot patterns with `Q_OBJECT`
- Use function-pointer syntax for `connect`/`disconnect` (`connect(obj, &Class::signal, this, &Class::slot)`), never the `SIGNAL()`/`SLOT()` string macros
- Address Sanitizer is enabled in macOS Debug builds
- **No magic numbers.** Extract numeric/color literals (z-values, radii, offsets, font sizes, RGB tuples) to named `constexpr`/`const` constants, typically in an anonymous namespace at the top of the `.cpp`. Fix adjacent raw literals too rather than matching a bad sibling.
- **`AsyncFuture::waitForFinished` is test-only.** It spins a nested `QEventLoop`, causing re-entrancy bugs (use-after-free, iterator invalidation). In production use `AsyncFuture::observe(future).context(this, callback)` (auto-disconnects on `this` destruction; destructors just call `future.cancel()`). Existing `waitForFinish()`/`waitToFinish()` helpers on managers exist solely for tests.
- **Use `.at(i)` or a `const` reference for read-only access to `QList`/`QVector`, never non-const `operator[]`.** `operator[]` calls `detach()`, which reallocates when refcount > 1 — a data race from worker threads. This includes access through smart pointers: write `ptr->at(i)`, not `(*ptr)[i]`.
- **Do not wrap `QList`/`QVector` in `std::shared_ptr`.** They are already implicitly-shared value types. To snapshot for a worker thread, `auto snap = m_container;` and capture by value; workers must stick to const access.
- **Do not use Qt bindable properties** (`Q_OBJECT_BINDABLE_PROPERTY*`, `QBindable`, `QProperty`, `setBinding`) in new code. Use plain members with explicit setters following `if (m_x == x) return; m_x = x; emit xChanged(); /* side effects */` and declare `Q_PROPERTY` with `NOTIFY` (no `BINDABLE`). Some older classes still use the macros; don't add more or refactor unless touching the file anyway.
- **Use `QStringLiteral(...)` for compile-time `QString` constants.** It bakes the UTF-16 payload in at compile time; `QString::fromLatin1(...)` and `QString(const char*)` heap-allocate on every call — fine for one-shot init, wasteful in per-frame hot paths (e.g. `buildPipelineKey`, which runs once per frame per RHI object). Don't wrap a `constexpr const char*` in `QString::fromLatin1` for DRY's sake; repeat the `QStringLiteral` instead — it's free at runtime and compilers fold duplicate instances.
- **Keep production public API free of test scaffolding.** No `*ForTesting` methods/members or `friend class FooTestAccess` on production classes. Do not add member data that exists only for tests to observe — e.g. call counters, "times invoked" tallies, or last-value caches. Tests verify through the real surface: disk-state checks (file existence, byte content, mtime), existing public signals (`objectPathReady`, `saveFlushCompleted`, `discardCompleted`), and standard wait helpers (`waitSaveToFinish`, `waitLoadToFinish`, `waitForDiscardToFinish`). Use `SignalSpy`/`SignalSpyChecker` to count emissions instead of a member counter. `cwSaveLoad::waitForFinished()` and `cwProject::waitSaveToFinish()` are pre-existing exceptions; don't add new ones.
- **Avoid mutable `static` and `thread_local` state.** No function-local `static`/`thread_local` mutable variables, no namespace-scope mutable globals (immutable `const`/`constexpr` constants in an anonymous namespace are fine). They never clear on project/palette change, are invisible to the user, and don't compose with the codebase's concurrency. A warn-once/dedup cache is the classic trap — route diagnostics through the real channel (the validator → `cwErrorModel`) instead. If you genuinely need mutable shared state, prefer a member on the owning object (or an injected dependency) so its lifetime and threading are explicit. **If you do add a mutable `static`/`thread_local`, call it out explicitly so it can be reviewed** — don't slip it in.
- **Never use `goto`** — no exceptions; the codebase has zero usage. To escape nested loops, use a `[&]()` lambda (capturing by reference) that uses `return` to exit. Name the lambda when the body is more than ~10 lines or you'd otherwise comment its intent; an inline IIFE is fine for short, self-documenting bodies. Both inline to the same machine code as `goto` and read top-down. Avoid the `bool flag; if (flag) break;` pattern.
  ```cpp
  const auto seedConnected = [&]() {
      for (cwTrip* trip : cave->trips())
          for (cwSurveyChunk* chunk : trip->chunks())
              for (int i = 0; i < chunk->stationCount(); ++i)
                  if (isValid(chunk->station(i))) {
                      insertChunkToConnected(chunk, connected, edges);
                      return;                  // escapes all three loops
                  }
  };
  seedConnected();
  ```

### QML
- Strict typed properties (no `property var`); lowerCamelCase property names; Capitalized component filenames
- Use `Theme.qml` for all colors and font tokens — never hardcode hex values or font sizes
- Prefer `id` selectors over `objectName`; keep bindings simple; avoid deep nesting
- Register modules via `qt_add_qml_module` in CMakeLists.txt
- Use `Loader` for deferred/conditional loading; avoid expensive operations in bindings
- Use `QCanvasPainterItem` (C++) for custom painting; do not use QML `Canvas` or `Shapes`
- Prefer `TapHandler` over `MouseArea` in new components (existing `MouseArea` is legacy — don't refactor unless touching the file)
- Import `QtQuick` as `QQ` (never `Q`), so it's easy to search for
- **Item body ordering** (blank line between groups): `id`, `objectName`, property declarations, property assignments/aliases, signal declarations, functions, property bindings, signal handlers (`on*`), child objects, `states`/`transitions` (last)

#### C++ testing
- Tests must be safe to run as **multiple concurrent processes** (nightly CI runs several in parallel). Never use hardcoded temp paths — include `QCoreApplication::applicationPid()` in the name or use `QTemporaryFile`/`QTemporaryDir`.

#### QML testing
- Prefer `tryVerify()`/`tryCompare()` over fixed `wait()`. `QQ.Transition` with `QQ.ScriptAction` fires next frame, so `tryVerify` the side-effect rather than assuming the state change is enough.
- Collapse note overlays (`setNoteOverlaysCollapsed(true)`) before clicking the note image in tests — the NoteTransformEditor overlay (z=2) can intercept clicks.

#### Text and fonts
- Use **`QC.Label`** (`import QtQuick.Controls as QC`) for all text — it inherits `font.family` and tracks the palette for dark mode. Never use bare `Text {}` / `QQ.Text {}` (they render black in dark mode).
- Use **`BodyText`** for paragraph/body copy (wraps `QC.Label` with `Theme.fontFamilyBody`).
- Prefer `Theme.qml` font tokens over hardcoded sizes: `fontSizeCaption` (11), `fontSizeSmall` (12), `fontSizeBody` (14), `fontSizeUI` (16), `fontSizeMedium` (18), `fontSizeTitle` (20), `fontSizeLarge` (24), `fontSizeXLarge` (30).

### Commits
- Short, capitalized, imperative subjects ("Fix", "Add", "Implement")
- Body: what changed, why, build/test commands run
- Attach screenshots for QML UI changes; link related issues; note submodule/dependency changes explicitly
