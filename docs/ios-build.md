# Building CaveWhere for iOS

## Prerequisites

- macOS with Xcode installed (Xcode 17+ recommended)
- Qt 6.11+ for iOS (installed via Qt Online Installer)
- Conan 2.x package manager
- Ninja build system (`brew install ninja`)
- CMake < 4.0 (pinned via Conan profile)

## 1. Create the iOS Conan Profile

Create `~/.conan2/profiles/ios`:

```ini
[settings]
arch=armv8
build_type=Release
compiler=apple-clang
compiler.cppstd=gnu20
compiler.libcxx=libc++
compiler.version=17
os=iOS
os.version=16.0
os.sdk=iphoneos

[conf]
tools.cmake.cmaketoolchain:generator=Ninja
tools.apple:sdk_path=/Applications/Xcode.app/Contents/Developer/Platforms/iPhoneOS.platform/Developer/SDKs/iPhoneOS.sdk

[tool_requires]
!cmake/*: cmake/[>=3 <4]
```

Update `tools.apple:sdk_path` to match your Xcode SDK. Find it with:

```bash
xcrun --sdk iphoneos --show-sdk-path
```

Update `compiler.version` to match your Apple Clang version (`clang --version`).

## 2. Install Conan Dependencies

Conan dependencies must be built with the Ninja generator (not Xcode) to avoid
bundle identifier errors on iOS executable targets like bzip2.

Install for both Debug and Release so the Xcode multi-config generator works:

```bash
conan install . -o "&:system_qt=True" -o "&:mobile=True" --build=missing \
    -pr:h ios -pr:b default \
    -of build/Qt_6_11_0_for_iOS/conan_deps \
    -s build_type=Release

conan install . -o "&:system_qt=True" -o "&:mobile=True" --build=missing \
    -pr:h ios -pr:b default \
    -of build/Qt_6_11_0_for_iOS/conan_deps \
    -s build_type=Debug
```

Adjust the output path (`-of`) to match your build directory name. The directory
name is generated by Qt Creator from the Qt version, e.g. `Qt_6_11_0_for_iOS`.

Key options:
- `system_qt=True` - use the Qt installation from Qt Online Installer (not Conan Qt)
- `mobile=True` - skips desktop-only dependencies (wxwidgets, gdal, survex, etc.)
- `-pr:h ios` - host profile targets iOS arm64
- `-pr:b default` - build profile uses native macOS tools (for protoc, etc.)

## 3. Configure with CMake (Qt Creator)

In Qt Creator, select the **Qt for iOS** kit and add these CMake arguments:

```
-DQT_CREATOR_SKIP_CONAN_SETUP=ON
-DQT_ADDITIONAL_PACKAGES_PREFIX_PATH=/path/to/build/Qt_6_11_0_for_iOS/conan_deps
```

- `QT_CREATOR_SKIP_CONAN_SETUP` prevents Qt Creator from running its own Conan
  install (which fails because it uses the Xcode generator).
- `QT_ADDITIONAL_PACKAGES_PREFIX_PATH` tells Qt's toolchain where to find the
  Conan-installed packages.

Then run CMake configure and build.

### Configure from Command Line

```bash
cmake -S . -B build/Qt_6_11_0_for_iOS \
    -G Xcode \
    -DCMAKE_OSX_ARCHITECTURES=arm64 \
    -DCMAKE_OSX_DEPLOYMENT_TARGET=16.0 \
    -DCMAKE_OSX_SYSROOT=iphoneos \
    -DCMAKE_PREFIX_PATH=/Users/$USER/Qt/6.11.0/ios \
    -DCMAKE_TOOLCHAIN_FILE=/Users/$USER/Qt/6.11.0/ios/lib/cmake/Qt6/qt.toolchain.cmake \
    -DQT_ADDITIONAL_PACKAGES_PREFIX_PATH=$(pwd)/build/Qt_6_11_0_for_iOS/conan_deps \
    -DQT_CREATOR_SKIP_CONAN_SETUP=ON \
    -DQT_QMAKE_EXECUTABLE=/Users/$USER/Qt/6.11.0/ios/bin/qmake \
    -DCMAKE_XCODE_ATTRIBUTE_DEVELOPMENT_TEAM=YOUR_TEAM_ID
```

## 4. Build and Deploy

Build from Qt Creator or Xcode. The app requires a valid Apple Developer
signing identity to deploy to a physical device.

## How It Works

### Why Ninja for Conan, Xcode for the App?

The Xcode generator requires `PRODUCT_BUNDLE_IDENTIFIER` for all executable
targets. Some Conan recipes (e.g. bzip2) create CLI executable targets without
bundle identifiers, causing build failures. Using Ninja for Conan dependency
builds avoids this, while the main project still uses Xcode for iOS deployment.

### Cross-Compilation Nuances

When building for iOS with the Xcode generator on macOS:
- `CMAKE_CROSSCOMPILING` is `FALSE` (Xcode handles multi-arch natively)
- `CMAKE_SYSTEM_NAME` is `Darwin` (not `iOS`)
- The project detects iOS via `CMAKE_OSX_SYSROOT` matching `iphoneos`

The top-level `CMakeLists.txt` handles this by:
1. Parsing `conanbuildenv-*.sh` to find the native macOS `protoc` binary
2. Setting `<package>_DIR` cache entries for all Conan packages so
   `find_package` uses the correct Conan-generated configs
3. Adding `conan_deps` to `CMAKE_MODULE_PATH` and `CMAKE_PREFIX_PATH`

### Static Linking

All Conan dependencies are built as static libraries for iOS (protobuf, abseil,
openssl, libssh2, etc.). This is required because iOS does not support loading
shared libraries from arbitrary paths at runtime.

## Troubleshooting

### "Bundle identifier is missing" during Conan install

The Conan profile is using the Xcode generator. Make sure the iOS profile has:
```ini
tools.cmake.cmaketoolchain:generator=Ninja
```

### protoc not found / protobuf::protoc target missing

The native macOS protoc must be on PATH during CMake configure. The project
parses `conanbuildenv-*.sh` automatically, but if this fails, you can manually
set:
```
-DCMAKE_PROGRAM_PATH=/path/to/.conan2/p/b/protoXXX/p/bin
```

### App dimensions look wrong / keyboard looks old on device

Make sure the `LaunchScreen.storyboard` is included in the build. Without it,
iOS runs the app in legacy compatibility mode. The `QT_IOS_LAUNCH_SCREEN`
property on the target and `UILaunchStoryboardName` in Info.plist handle this.

### "Could NOT find Protobuf (missing: Protobuf_INCLUDE_DIR)"

Delete `CMakeCache.txt` and reconfigure. Stale cache entries from previous
configure runs can point to wrong package locations.
