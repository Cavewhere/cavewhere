CaveWhere · Implementation Plan

Embed the User Manual into CaveWhere

A File → Docs viewer that renders the manual's Markdown natively, turns every article into a real navigable page, and is searchable — with no web-engine dependency and no break for the mobile build.

Branch manual (app code) Source of truth docs/manual/*.md Renderer Qt TextFormat.MarkdownText Nav cwPageSelectionModel, page per article Status CP1–CP4 + CP6 done · CP5 mostly implemented (styling pass + themed links + tests, uncommitted) · remaining: dark-mode visual pass + drawer test gap

Goal

Let a user open the manual inside CaveWhere (File → Docs) and read it the way they read any other part of the app: the same breadcrumb, the same back/forward, a search box that finds a topic, and cross-reference links that navigate in place. The Markdown files under docs/manual/ stay the single source of truth — the same source that already builds the web page for cavewhere.com.

The decision that drives everything: no web view

CaveWhere ships no web-rendering module today — there is no QtWebEngine or QtWebView anywhere in conanfile.py (the Qt modules requested are only qtshadertools, qtdeclarative, qtsvg, qttools, qttranslations, qtimageformats), in the submodules, or in CMake. Rendering the built manual.html would mean adding one:

  • QtWebEngine — Chromium; hundreds of MB, and it does not ship on iOS.
  • QtWebView — lighter (system webview), but still a new module with per-platform quirks.

Both bring their own navigation history and their own find-in-page — exactly what we don't want, since the goal is to reach into the content with CaveWhere's back/forward and search. A web view fights both goals and costs the mobile build.

Decision — confirmed

Render the Markdown natively with Qt 6's TextFormat.MarkdownText (zero new dependencies; keeps iOS/mobile alive), and make each article its own registered page so the existing history stack carries back/forward and the breadcrumb. We rebuild the chrome (sidebar, search, header) natively — which is what makes the integration work — and theme the reading column through Theme.qml.
Update (CP5): the original "honest cost is fidelity — article bodies use Qt's default Markdown styling" is now largely bought back. cwDocumentStyler post-processes the parsed QTextDocument to match manual.html's scale (heading sizes, block spacing, line-height, responsive images) and recolors links to the theme — so the render is close to the web manual, not Qt's raw defaults.

Architecture

File → Docs ──▶ gotoPageByName(null,"Docs") │ ┌───────────────┴────────────────┐ │ DocsPage.qml (shared) │ ← root: StandardPage │ ┌──────────┬──────────────────┐│ │ │ sidebar/ │ reading column ││ ← Flickable + Label(MarkdownText), │ │ TOC + │ renders slug's ││ images → qrc: absolute (rewritten in C++) │ │ search │ .md body ││ │ └──────────┴──────────────────┘│ └─────────────────────────────────┘ │ registerPage(Docs, "Enter Survey Data", docsPageComponent, {slug:…}) # × ~48 │ back() / forward() / breadcrumb ◀── all free via cwPageSelectionModel

Pieces

  • DocsPage.qml — root StandardPage (same trivial wrapper AboutPage.qml uses). One shared component with a slug property; a themed reading column (Flickable + a read-only QC.TextArea with textFormat: MarkdownText, since CP6 — was a QC.Label in CP1–CP5), post-processed by cwDocumentStyler, plus a native sidebar/TOC and search box as chrome.
  • Registration loop in MainContent.qml's Component.onCompleted (alongside the existing About/Settings/Cavern registrations at :200–212): register a "Docs" landing page, then loop the llms.txt manifest calling registerPage(docsRoot, title, docsPageComponent, {slug}) for each article.
  • Link interceptor — articles link to each other by relative path (../survey-data/enter-survey-data.md#anchor). On onLinkActivated: map a .md path → its slug and gotoPageByName (in-app, enters history); http(s) → Qt.openUrlExternally.
  • Search index + filter model — a small model backed by cwManualIndex's in-memory title/summary/keywords (front matter it already parsed). Typing filters; selecting jumps to that article's page.
  • File-menu item — copy the About item (FileMenu.qml:130–135): text: "Docs…"; onTriggered: RootData.pageSelectionModel.gotoPageByName(null, "Docs").
  • Content staging — CMake's qt_add_resources globs the .md + images + llms.txt into the binary (:/manual/…); the runtime cwManualIndex C++ class rewrites each body's relative image links to absolute qrc: URLs, so no Markdown base URL is needed (Qt's Text has none anyway).

Content pipeline — no build-time codegen

docs/manual/**/*.md ─┐ docs/manual/images/ ─┼─ CMake file(GLOB) + qt_add_resources ─▶ :/manual/… compiled into the app docs/manual/llms.txt ─┘ │ (at runtime) ▼ cwManualIndex (C++) ── parses llms.txt → ordered (title, slug); chapter from the path strips front matter + rewrites image links → qrc: absolute (front-matter summary/keywords for search — deferred to CP4) │ ┌──────────────┴───────────────┐ ▼ ▼ RootData.manualIndex.articles RootData.manualIndex.body(slug) ──▶ registration loop + TOC + search ──▶ DocsPage reading column

No Python, no pandoc, no generated manifest.json/search-index.json. The app renders Markdown natively, so the HTML conversion pandoc did for the website is never needed here — the only work is embedding the raw .md + images and parsing a little front matter. Embedding is CMake's existing qt_add_resources mechanism (already used at cavewherelib/CMakeLists.txt:217); the parsing lives in a small cwManualIndex C++ class covered by cavewhere-test. llms.txt stays the single ordered index — the same file the website build reads — so there is no second manifest to keep in sync. scripts/build-manual-html.py is unaffected: Python stays a docs-website tool, never a CaveWhere build dependency.

Decisions — resolved

1 · How to ship the content

Compile into the app, via CMake's qt_add_resources. Single binary, offline, simple — and no build script (see Decision 6). If the PNGs push the binary too far, the fallback is down-sampling the in-app image copies before ever reaching for a data dir.

✓ qt_add_resources (no codegen)

2 · The sketch app

Ignore it entirely — it's a dead prototype. Docs is registered only in the main app (MainContent.qml). Do not touch sketch/qml/Main.qml.

✓ Main app only

3 · Landing page

Mirror index.md's chapter grouping as the TOC, search box on top. It already reads as the table of contents.

✓ Chapter-grouped TOC

4 · Intra-page anchors

Cross-page navigation first; land intra-page heading scroll in CP3 if the anchor→y glue is cheap, otherwise fast-follow. Not a blocker for shipping the viewer.

✓ Cross-page first, anchors fast-follow

Decision 5 — the CP2 content seam (added after the CP1 review)

Register each article by a slug, not by pushing a Markdown string. CP1 shipped a placeholder markdownText string to prove the render path; CP2 replaces it with a slug property that DocsPage resolves — via markdownText: RootData.manualIndex.body(slug) — into the article's Markdown body with its image links already rewritten to absolute qrc: URLs. A raw pushed string couldn't carry the per-article image base, and Qt's Text/Label has no settable Markdown base URL, so the C++ index resolves images to absolute resource paths rather than relying on a baseUrl that doesn't exist.

Use registerPage, not registerSubPage. registerSubPage is a little funny — it implicitly reuses the parent page's QQmlComponent and merges the new properties over the parent's (cwPageSelectionModel.cpp:111-122) — so we avoid it. Register each article with an explicit component and explicit properties instead: registerPage(docsRoot, title, docsPageComponent, {slug}) (the CavePage idiom, CavePage.qml:55). DocsPage stays one slug-driven component reused for every article — the "Docs" landing page is that same component with the slug unset (it shows the chapter-grouped TOC), and each article passes its own {slug}. Passing the component explicitly keeps the registration readable and free of the parent-property inheritance registerSubPage would splice in.

Decision 6 — build the resource with CMake, parse at runtime in C++ (no Python in the app build)

CaveWhere must build without Python — it is a moving target across build platforms, and the app build already has CMake. So CP2 does no build-time codegen: CMake's qt_add_resources globs the manual's .md + images + llms.txt into the binary, and a runtime cwManualIndex C++ class does the parsing (order/titles from llms.txt, chapter grouping from the file path, front-matter stripping, and image-link rewriting). Front-matter summary/keywords for search are read in CP4, not CP2.

This is better than the earlier build-manual-resource.py idea on every axis that matters: zero new build dependency; the parsing is unit-tested in cavewhere-test instead of an un-tested script or fragile CMake regex; there is no generated manifest.json/search-index.json to check in or regenerate; and llms.txt stays the single ordered index shared with the website build. Editing a page is just a rebuild (rcc re-embeds). Python remains fine for the website HTML build, which is not part of building CaveWhere. The pandoc/Markdown→HTML conversion the website needs never applied here — the app renders Markdown natively.

CP4 · Sidebar TOC + search

Find a topic and jump to it

✓ Done — 83 [cwManualIndex]+[cwManualSearchModel] assertions + 13 tst_DocsPage cases green; app + both test binaries link clean.

Added a persistent left sidebar to DocsPage: a search TextField over a chapter-grouped ListView TOC (ListView.section keyed on the article's chapter, so the model's index order groups for free). The TOC is driven by a new cwManualSearchModel (QAbstractListModel, QML_NAMED_ELEMENT): an empty query lists every article in order; a non-empty one keeps rows whose haystack contains every whitespace term (AND). The reading column takes the full width in compact mode.

Parser (the A1/R1 "summaries for free" piece): cwManualIndex now captures the llms.txt trailing summary (the text after ):) into Article.summary at construction — zero new I/O — and adds searchText(slug), a lowercased title + summary + keywords haystack. Keywords are read from each article's front matter lazily and cached on first search, so app startup still touches only llms.txt.

Plan drift — "sections → chapters" is stale

The A1/R1 follow-up proposed deriving chapters from llms.txt's ## sections. That doesn't hold: llms.txt's only sections are Start here / Chapters / Meta (every article sits under one ## Chapters), and the website build (build-manual-html.py) itself derives the sidebar chapter from the directory path (chapter_title(relpath)), not from those sections. So C++ chapterForPath is already a faithful port of the Python, and the two can't be collapsed through llms.txt. The genuinely duplicated chapter table is left as-is (correct, tiny, curated); only the summary half of A1/R1 was single-sourced from llms.txt, which is where the real CP4 payoff was.

CP5 · Reading polish & tests

Reading typography, themed links, dark-mode verification, tests

◑ Mostly implemented (not yet committed). The scope grew past "dark-mode verification": a user observed the native Markdown render was too tightly spaced, too small, over-large images, and a hard-to-read body font. That is now handled by a new C++ post-processor, cwDocumentStyler (DocumentStyler, QML_NAMED_ELEMENT), which restyles the parsed QTextDocument to match the web manual: heading sizes (2.6 / 1.9 / 1.45 rem over the body), per-block margins and proportional line-height, a muted lead paragraph and image captions, and — the one thing no stylesheet can do for Markdown — images clamped to the reading width so figures shrink on a narrow window instead of overflowing. Wired in DocsPage.qml as a DocumentStyler bound to the body's textDocument.

Reading font. The body was rendering in the app's default UI font (Yanone Kaffeesatz, a condensed display face) which is tiring as body copy. Added two Theme.qml tokens for a fixed manual pairing independent of the user's chrome font: fontFamilyReading ("Fira Sans", the readable body) and fontFamilyHeading ("Yanone Kaffeesatz", the display heads); the styler sets headings to fontFamilyHeading while the body uses fontFamilyReading. Both are bundled/registered fonts, mirroring the existing fontFamilyMono token.

Themed link color — done, via the post-processing route this checkpoint predicted. cwDocumentStyler gained a linkColor property (Theme.textLink) and recolors anchor char-formats in its fragment pass, so links track the theme (fixing dark-mode link readability) instead of QTextDocument's default anchor color. This confirms the "post-processing anchor char-formats" option over setDefaultStyleSheet (which is HTML-only).

CP6 · Find on page + intra-page anchors

Search within the current article, and jump to headings

✓ Done — 23 tst_DocsPage cases green (25 QtTest incl. init/cleanup; 11 new: find open/close, find-button-opens-find, enter/shift-enter walk, compact-bar-shares-hamburger-row, reserves-space-above-document, highlight+count, next/previous, no-match "None", close-clears-selection, #anchor scroll, navigation resets find); app + both test binaries link clean. The reading column is now a read-only QC.TextArea (MarkdownText) inside a plain QQ.Flickable; a new cwDocumentFinder (DocumentFinder, QML_NAMED_ELEMENT) runs QTextDocument::find() over the body's QQuickTextDocument for the find bar and resolves GitHub-style heading slugs (matching build-manual-html.py) for #anchor scroll. Four twbs-icons (search, chevron-up/-down, x) were added to cwResources.qrc. The find bar is a full-width toolbar docked in a reserved band above the reading column (in wide mode the column always reserves this band — it holds the search button when find is closed and the bar when open, so opening find never shifts or covers the text), sitting flush above the document with only a 1px separator line — per user request, neither an overlay nor a content-hugging pill. In compact mode the bar instead shares the drawer-toggle (hamburger) row, starting just right of it, so it costs no extra vertical space (anchors align to the toggle's row — no LayoutItemProxy needed). The query field stretches; the count and prev/next/close sit right. A top-left search.svg RoundButton (hidden while the bar is open; sits right of the drawer toggle in compact mode) opens find and carries a tooltip with the platform shortcut via Shortcut.nativeText (⌘F on macOS). Match navigation uses the standard keys — StandardKey.FindNext (F3 / ⌘G) and FindPrevious (Shift+F3 / ⌘⇧G), plus Enter / Shift+Enter in the field — and the next/prev buttons carry those shortcuts in their tooltips.

Two conscious deviations from the scope below. (1) Highlight-all is deferred: QML's TextEdit exposes no extra-selection API, and true simultaneous highlight would mean mutating the document's char formats (invasive for a read-only viewer). The current match is highlighted via the selection and the bar shows a running "3/12" count — the standard find affordance (ship-simplest-first). (2) Themed link color is lost in the swap: TextEdit has no linkColor (only Label did), so links rendered in QTextDocument's default anchor color. Resolved in CP5: cwDocumentStyler now recolors anchor char-formats to Theme.textLink (the "another approach" — setDefaultStyleSheet is HTML-only), so links track the theme in both modes.

The CP4 search box finds which article to open (it filters the TOC across the manual). This checkpoint adds the other axis: find-on-page — a Ctrl+F bar that locates, highlights, and scrolls to text matches inside the article you're reading — and lands the same-page #anchor scroll that CP3 deferred (decision 4), since both need the same foundation.

Why it's not a toggle. Before CP6 the reading column was a QC.Label with textFormat: MarkdownText, and Qt's Text/Label has no built-in find or highlight and exposes no document handle. The move (done in CP6) was to render the body through a document-backed read-only text view (a read-only QC.TextArea, or a Label whose QQuickTextDocument we reach) so we can call QTextDocument::find() to locate a match, select it, and scroll it into view. That same QTextDocument is what anchor→y resolution needs, so #anchor links (landing-page TOC jumps and heading cross-refs) fall out of the same plumbing — the CP3 onLinkActivated early-return on # becomes a real scroll.

Scope: swap the reading column to the document-backed view (keep the qrc image rewriting and Theme styling); a small find bar (text field + next/prev + match count + highlight-all + Esc-to-close), opened by Ctrl+F; wire #anchor clicks to scroll to the heading's document position. Not in scope: cross-article "find in all pages" (that's what CP4's TOC search already covers a different way).

Risks & nuances

  • Markdown fidelity. Qt renders GitHub-flavored Markdown including tables, but with default styling and no per-element CSS. Captions (our img+em pattern) render as plain italic under the image — acceptable. Anything that leaned on the HTML's CSS won't carry; the text must (and, per the authoring rules, already does) stand on its own.
  • Front-matter stripping at load. The renderer must drop the leading --- … --- block, same as build-manual-html.py does.
  • Intra-page anchors need anchor→y-position glue (decision 4) — now folded into CP6, which brings in the document-backed body that makes both anchor-scroll and find-on-page possible.
  • Binary size from compiled images (decision 1).
  • Two renderers, one source. Keep the web (manual.html) and in-app (native) paths as separate renderers over the same .md — don't try to make one artifact serve both.

Follow-ups (from the CP2 & CP3 reviews — not blocking)

Deferred improvements surfaced reviewing CP2 and CP3. None is a correctness bug against today's manual; each is worth doing before or alongside CP4, where the payoffs land.

  • A1/R1 · Make llms.txt the one source; stop duplicating the website build. cwManualIndex::slugForPath/chapterForPath (cwManualIndex.cpp:136,156) re-implement scripts/build-manual-html.py's slug scheme and chapter table as a second hardcoded source of truth — add a chapter directory and you must edit both, or C++ silently falls back to titleCase() (view-3d → "View 3d", not "3D View"). Blast radius: the parser only. Best done at the start of CP4.
    Partially resolved in CP4 — closed. The summary half shipped: the parser reads the llms.txt trailing text into Article.summary and CP4 search uses it. The chapters half was dropped as unworkable — llms.txt has no per-chapter sections (only Start here / Chapters / Meta) and the website build derives chapters from the directory anyway, so chapterForPath is already the single faithful port (see the CP4 plan-drift note).
  • R1 (from the CP6 review) · Unify the heading-slug scheme; stop hand-porting it a third time. CP6 added cwDocumentFinder::headingSlug (cwDocumentFinder.cpp:16) — a third, independent slug implementation (after slugForPath and the Python), used to resolve same-page #anchor links to a heading's document position. Its comment now honestly notes it is not byte-identical to the website: scripts/build-manual-html.py slugs headings with pandoc's GFM auto-identifiers plus a per-page --id-prefix key--, a different algorithm and namespace. The two coincide only for the plain-ASCII headings the manual currently uses; they can diverge on numbers-leading headings, &/unicode, and the page prefix — and nothing tests the two against each other. Every in-app #anchor now rides on this regex matching whatever the author wrote. This extends the A1/R1 debt (which covered only path slugs) to heading slugs: the real fix is a shared slug contract (one function both the app and the build call) or generating an anchor table from the build. Blast radius: cwDocumentFinder + a cross-check test. Not a live bug against today's ASCII-only headings; do it before headings gain punctuation/unicode.
  • R1 (from the CP5 styling review) · Shared QQuickTextDocument-observer base for the two doc wrappers. cwDocumentFinder and cwDocumentStyler are the only two QQuickTextDocument wrappers in the tree and now duplicate an identical block verbatim: the QPointer<QQuickTextDocument> m_document (with the same word-for-word "Held by QPointer… auto-nulls" comment), the Q_PROPERTY(QQuickTextDocument* document …), the identical textDocument() helper, and the setDocument guard shape. A small cwTextDocumentObserver base (holds m_document, textDocument(), setDocument() + a virtual onDocumentChanged()) would remove the copy. Blast radius: both classes + their tests. Borderline (only two consumers today) — do it if a third consumer appears or when either class is next touched. Also record the invariant it should enforce: a document post-processor must only change formats, never insert/remove text, or cwDocumentFinder's cached character positions corrupt (the styler is format-only today; a comment on apply() notes this).
  • A3 · Move the "empty slug = landing" rule into C++. The routing lives in QML today (DocsPage.qml:19: slug === "" ? landingBody() : body(slug)). Folding it into body("") returning the landing keeps the QML seam a single dumb call and puts the routing next to the data. Minor; do it if CP3/CP4 touch the seam anyway.
  • A2 · cwManualIndex as a QML_SINGLETON — considered, rejected. It's a stateless read-only service (a textbook singleton, and RootData is already a god-object hub), but every subsystem is reached through RootData.* and this plan deliberately chose RootData.manualIndex. Consistency wins; keep it on RootData. Recorded so it isn't re-litigated.
  • Body/render memo (nice-to-have). renderMarkdown (cwManualIndex.cpp:193) re-reads the resource and re-runs the regex on every body() call. A trivial QHash<slug,QString> memo if it ever matters; negligible for a viewer today.

From the CP3 review — three 🟢 coupling notes, none a live bug, all latent given today's curated manual. A1 below naturally pairs with CP4 (it builds the TOC/search over the same index).

  • R1 · Navigate by page handle, not a reconstructed address string. DocsPage.qml:19,70 hardcodes docsRootName: "Docs" and builds "Docs/" + title, restating both the registration name at MainContent.qml:224 and cwPageSelectionModel::seperator() ("/"). The model already exposes gotoPageByName(parentPage, subPage), which navigates a child by name with no string-building or separator knowledge in QML. Needs the Docs parent cwPage* threaded into DocsPage (or a slug-nav helper on the model). Tiny/harmless today; the string form works.
  • R2 · slugForLink's empty return conflates three outcomes. cwManualIndex.cpp:141,167,171 return QString() for same-page anchor, external link, and not-a-registered-article. The caller can't tell them apart, so DocsPage.qml:64's startsWith("#") guard is load-bearing (removing it sends anchors to openUrlExternally), and an in-manual relative link to a non-article page would also be handed to the browser as a bare relative path. A small result type or split isExternal/isAnchor predicates would make the contract precise. Not reachable from the link-checked manual today.
  • A1 · In-app doc nav is keyed by human title, not the stable slug. Articles register under article.title as the page name (MainContent.qml:228) and addresses resolve by name, so CP3 round-trips slug → title(slug) → "Docs/<title>" → page-by-name. Correctness depends on titles staying unique and never reworded; the stable id (slug) never appears in the address space. Viewer polish — open (surfaced 2026-07-20)

  • Pointing-hand cursor over document links. Links in the rendered Markdown body (DocsPage.qml's bodyId QC.TextArea) don't show a Qt.PointingHandCursor on hover, so they don't read as clickable. Add hover-cursor feedback over anchor spans. Note the body is a read-only TextArea rendering MarkdownText; hoveredLink / a HoverHandler gated on the link region is the likely hook. Blast radius: DocsPage.qml only.
  • Reset the scroll position to the top on page switch. Switching articles keeps the reading column (scrollId QQ.Flickable) at its previous contentY, which lands the new page mid-scroll and causes a rendering glitch. On slug change (alongside the existing onSlugChanged that closes find), reset scrollId.contentY = 0. Watch for ordering vs. the document re-layout — the reset must apply after the new body's height is known, or it can clamp against the old contentHeight. Blast radius: DocsPage.qml only.
