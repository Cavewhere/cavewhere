Label Placement Scaling Plan

cwCaptureLabelPlacer · export hang on large surveys · 2026-07-02

TL;DR. Station-label placement during export is O(n²). On a 150-mile cave (Fisher Ridge, tens of thousands of stations) it never finishes — the UI thread hangs inside cwCaptureLabelPlacer::placeLabel. A new benchmark (testcases/test_cwCaptureLabelPlacerBenchmark.cpp, run with ./cavewhere-test "[benchmark]") measures the curve. The fix has layers, cheapest first: (1) cull off-viewport stations, (2) cap the failed-search spiral, (3) replace the three brute-force linear scans with a spatial-hash grid (memory-proportional, because cave plots are sparse curves on a mostly-empty page), and (4) move the whole pass off the main thread with cwConcurrent + AsyncFuture. A separate scaling wall — the distance-transform grid and tile images cost O(page area), which explodes on a huge single-surface export — is addressed by tile-local placement with obstacle halos (§5).

1. The benchmark & baseline

The benchmark drives cwCaptureLabelPlacer directly (no camera / RHI needed) with a synthetic serpentine cave, seeding obstacles exactly the way cwCaptureViewport::placeLabelsAfterTiles does: station dots as hard obstacles before finalize(), legs as soft obstacles after, one label placed per station.

Doubling the station count roughly quadruples the time, and per-station cost grows linearly (156 → 2720 µs) — the signature of an O(n²) algorithm. And this is the easy case where every label finds a home; the real file adds failed-spiral blow-ups and lead-leader obstacles on top.

2. How placement works today

After all tiles are grabbed, placeLabelsAfterTiles() (main thread) builds a distance transform of passage ink, seeds obstacles, and then calls placeStationLabels(), which invokes placeLabel() once per station.

Figure 2. The export label pipeline. Everything runs on the main thread; the red box is where the hang lives.

Inside placeLabel(): spiral search + brute-force scans

For one station, the placer spirals outward through DT cells looking for the first cell with enough clearance to seat the label, collects a few more candidates, scores them, and commits the winner. The problem is what happens at every candidate cell.

Figure 3. Left: the spiral visits candidate cells ring by ring. Right: at each candidate it linearly scans all hard obstacles, all placed labels, and (when scoring) all soft centerline legs. Nothing is indexed.

3. Where the O(n²) actually comes from

Four independent multipliers stack up. The first two are the dominant quadratics; the last two turn a bad case into a catastrophic one on real data.

SourceCodePer labelOver n labels
Placed-label rescan cwCaptureLabelPlacer.cpp:405 O(labels placed so far) O(n²)
Soft-obstacle scoring :449 (per candidate × every leg) O(n) legs O(n²)
Hard line-obstacle scan :385 (the hang frame) O(L) × cells visited O(n·L·cells)
Unbounded failed spiral :425, maxRadius = maskW+maskH scans the whole grid when no cell fits amplifier
The failed-spiral trap. windowExhausted() only starts limiting the search after the first successful candidate (firstHitRadius >= 0). When a label can't be placed — routine in a dense plot — the spiral runs all the way to maxRadius, and every clear cell it touches pays the full three-scan cost above. Plus there is no viewport cull: placeStationLabels() loops all stations, including ones entirely off the exported page, each running a full max-radius spiral that finds nothing.
The code even documents the broken assumption at cwCaptureLabelPlacer.cpp:401: "Brute-force scan — N is small (~30 labels per export)." Fisher Ridge has on the order of 50,000 stations.

4. Which spatial index? (research)

The three hot scans reduce to two query shapes over a mostly-static set: rect-vs-rect overlap (placed labels) and segment-vs-region (hard + soft obstacles). Queries are always local — a leader is a short segment, a label is a small rect — so we never need a global search, only "what's near here." The second decisive fact is the shape of the data.

Cave plots are sparse. A centerline is a 1-D curve (thin passages) winding through a large, mostly-empty 2-D page — and a whole-cave export at 300 DPI makes that page enormous. So the deciding axis is not raw query speed (all indexes localize well) but how memory scales when the page is huge and empty. That is exactly where a dense uniform grid falls down.
Figure 4. The workload that actually matters: a sparse passage curve on a giant page. A dense grid pays for every empty cell (left); a spatial-hash grid stores only the cells the curve touches (right).
StructureMemory on a huge sparse pageLocal queryDynamic insertVerdict
Dense uniform grid (array of buckets) O((page/cell)²) — empty cells still cost headers O(1)+k trivial fails ~1.8M cells → 40+ MB of empty headers on a 47k×31k page; grows with page²
Spatial-hash grid (hash of occupied cells) O(occupied cells + memberships) ≈ O(n) O(1)+k trivial best keeps every grid win; memory follows the data, not the page
Quadtree O(nodes), adaptive — big voids stay one node O(log n + k) node splits runner-up good memory, but pointer-chasing, ~10–11 levels deep on this page, harder insert
kd-tree (existing cwCollisionRectKdTree) O(n) O(log n + k) typ., O(n) worst unbalanced ⇒ degrades tension balanced=static (bad for growing labels); the existing one degrades on our top-to-bottom insert order. Rect-only.
R-tree / BVH O(n) O(log n + k) bulk-load overkill extra dependency / 3D tool; no local-query edge over a hash grid
The correction from the first draft: the right move is dense-array → spatial hash, not grid → quadtree. The hash variant delivers the memory-proportionality you'd want a quadtree for, without giving up O(1) insert, contiguous bucket scans, or implementation simplicity. A quadtree only pays off if page density varied over many orders of magnitude and we needed the tree for other queries (nearest-neighbour, ray march) — we don't.

Recommendation: cwSpatialHashGrid2D

Bucket 2D space into square cells sized to a few label-diagonals, keyed by hash(cellX, cellY) → bucket so only non-empty cells exist. Each bucket holds indices of items whose bbox overlaps it; a query rasterizes the query region to cells and tests only those buckets. Every O(n) scan becomes O(k) local, and memory is O(n) regardless of page size. Three instances:

  • Hard obstacles (lead leaders) — inserted as registered; segment-bbox buckets.
  • Soft obstacles (centerline legs) — built once after finalize(). Biggest set, clearest win.
  • Placed labels — dynamic; insert on commit. Non-overlap bounds per-bucket count, so lookups stay O(1) with cell ≈ label size.

Keep the existing cwCollisionRectKdTree in mind only as a lower-effort drop-in for the placed-labels rect set if we want to touch less code first — but it doesn't help the two segment scans and degrades on sorted inserts, so it isn't the primary choice.

Figure 5. The soft-obstacle score for one candidate. Today every leader is tested against every leg (left). With a grid, the leader only visits the buckets it actually passes through and tests the few segments there (right).

5. The other scaling wall: page-area memory & tiling

A fast index fixes the O(n²) time. But a whole-cave single-surface export at 300 DPI hits a second wall that has nothing to do with the label index: several data structures cost O(page area), and the page is gigantic.

The distance-transform grid already blows up. placeLabelsAfterTiles unites every tile into one parentBounds and builds a single QVector<float> m_dt at maskW × maskH. On a ~47k×31k-px plot at 2-px cells that is ~376 M cells × 4 B ≈ 1.5 GB — before a single label is placed. The summed tile QImages are gigapixels on top. So "whole huge cave on one page" is memory-infeasible in the current architecture regardless of which label index we choose. The O(n²) time is the hang you hit; this page² memory is the wall right behind it.

Tile-local placement (invisible seams)

The export already tiles for the GPU grab (Rows × Columns of grabToImage). The regression is that placement then re-assembles everything into one global DT and one global pass. The fix is to keep placement per-tile, streaming: build the DT and the spatial grids for one tile (plus a halo), place that tile's labels, emit the overlay, release the memory, move on. Target artifact is one giant image/PDF, so seams must be invisible — which sets the requirements below.

Figure 6. Tile-local placement. Each tile places within its bounds but reads a halo of neighbouring obstacles and border labels so seams are invisible in the final single surface.

Why tiling

  • Bounds memory for the DT, tile images, and the index — the only way a huge single-surface export fits in RAM at all.
  • Parallelism — tiles place concurrently (cwConcurrent::mapped), folding into Fix 5.
  • Streaming — never hold the whole page's obstacle mask at once.

Costs / caveats

  • Seam handling (halo + border ledger + leader clamp) is the real complexity — required because the target is one seamless surface.
  • Asymptotics unchanged: with a spatial-hash grid we're already ≈O(n·k); tiling buys memory + parallelism + feasibility, not a better big-O.
  • Bigger structural change than Fixes 1–3, so it lands after them.

6. Proposed fixes

Fix 1 · Cull off-viewport stations cheap

Before spiraling, skip any station whose anchor lies outside clampRect (plus a label-diagonal margin). Off-page stations can never seat a label anyway, and each one currently burns a full max-radius spiral. Biggest win for the least code.

Fix 2 · Bound the failed spiral cheap

Cap maxRadius at a sane multiple of the label size (a label that has to travel a third of the page isn't useful). Make the search-window logic apply to the no-hit-yet case too, so a hopeless anchor gives up early instead of scanning the whole grid.

Fix 3 · Spatial grid for the three scans the real fix

Introduce cwSpatialGrid2D and route the three hot loops through it. Each turns from O(n) into O(k) local. This is what flips the benchmark curve from red to green.

Figure 7. The three algorithmic fixes composed: off-page anchors (red) never enter the loop, each remaining spiral is radius-capped, and all obstacle/label queries hit only local grid buckets.

7. Fix 5 · Get it off the main thread last

Even at O(n·k), a huge export takes real time; it must not freeze the UI. Today the whole placement pass runs synchronously inside the final grabToImage ready callback on the main thread. The port mirrors the Compass-importer pattern (commit c76ee6de):

  1. Extract a pure worker. Placement input is already value-typed (tile QImages, station positions, leg segments). cwCaptureLabelPlacer touches no QObject/GUI state, so it runs safely off-thread. Split "compute placements" (worker) from "own & paint the placed rects" (main-thread items).
  2. Dispatch with cwConcurrent::run([in = std::move(in)](QPromise<Result>& p){…}) → returns QFuture<Result>. The QPromise& overload unlocks setProgressValue (one step per label) and cooperative isCanceled().
  3. Deliver back on the main thread with AsyncFuture::observe(future).context(this, […]{ buildOverlayItems(future.result()); emit finishedCapture(); }) — auto-disconnects if the viewport dies; the destructor just calls future.cancel().
  4. Progress bar for free by handing the future to cwFutureManagerToken::addJob(...).
Ordering matters. Do the algorithmic fixes (1–3) first — they make it fast; the benchmark proves each step. Threading only stops the UI from freezing; layering it on an O(n²) core would just hang a background thread instead.

Reading:
  • Placement (44.5 s) is the entire bottleneck; the distance transform (104 ms) is not — the earlier "DT dominates" note was an artifact of a tiny test cave.
  • The cost is 1.65 B + 1.2 B linear comparisons — Phase 3 must index both the placed-label and soft-obstacle scans.
  • The Phase-1 viewport cull culled 97.7% of calls in O(1); the whole 44.5 s is the 735 non-culled stations. Without the cull this would be ~30 min.
  • Caveat: this was the preview render, not a full export. The page was small (671×612), so the DT and memory were cheap. A full whole-cave single-surface export has a huge page → the DT (O(page area)) and its memory become significant, and even 104 ms drops frames on the GUI thread. Hence Phase 4 (tiling) is expected to matter for the full export and should be investigated right after Phase 3 — re-profile a full-cave export to confirm.
Invariant to protect: placement output must not change from the fixes — the grid is a query accelerator, not a policy change. The existing [cwCaptureLabelPlacer] tests (leader-crossing, obstacle rejection, deterministic order) must keep passing at every phase, so a before/after placement-equivalence check on a mid-size cave is worth adding in Phase 3.
