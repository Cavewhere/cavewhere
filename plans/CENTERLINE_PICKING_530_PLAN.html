Pickable Survey Centerline

Make the 3D line plot hittable by the pick BVH with a screen-space proximity test, and expose enough on the hit to clamp a pick to a station. Foundation for the 3D measurement tool (#487), which is tracked and built independently.

Scope. This plan covers GitHub #530 only, delivered as two commits: Commit 1 — centerline pickable and Commit 2 — clamp-to-station-ready hit contract. The measurement tool itself (two-point readout: distance / azimuth / clino / vertical / horizontal) is #487 and out of scope here — but Commit 2 is shaped precisely so #487 can consume it without re-touching the BVH.

2. Root cause

The picker is a two-level BVH (cwGeometryItersecter). Every geometry kind is enumerated into primitives, packed into a per-object model-space sub-BVH, and tested per-pick. Lines are the one kind that contributes zero primitives, so the centerline never enters the structure:

  • countNodePrimitives() (cwGeometryItersecter.cpp:317) has no Lines case → falls through default: return 0.
  • addLines() (:621) still uses the legacy flat-BVH shape — one Node per segment — but launchBuildJob() drops every node where countNodePrimitives() <= 0. The debug line literally says "Lines are currently not picker-visible — the BVH skips them".
  • cwRenderLinePlot::setGeometry() (:46) already registers the centerline as Type::Lines with a synthetic std::iota index list, awaiting this fix.
Figure 1. The drop-out path. Lines are registered but enumerate to zero primitives, so they are filtered out before the BVH is ever built.

3. Where lines plug into the BVH

Lines fit the existing two-level architecture the same way points do — a point is a 0-D proximity primitive with a radius; a line segment is a 1-D proximity primitive with a radius. We reuse the entire Phase-A/Phase-B build pipeline (enumerate → median split → sub-BVH) and add one new Primitive::Kind plus one new branch in the per-primitive test.

Figure 2. Lines become a third primitive kind alongside triangles (scraps) and points (LAZ/LiDAR). Once present, all four measurable surfaces in #487 are pickable — scraps and clouds already were.

4. Screen-space proximity model

A ray almost never exactly intersects a 1-D line, so picking is a proximity test: accept when the closest distance between the ray and the segment is within a pick radius. Per the design decision, that radius is screen-space — a constant number of pixels regardless of zoom — so it must grow with distance from the camera. The BVH is deliberately camera-agnostic (tests feed it bare rays), so we pass it a tiny evaluator instead of a camera.

// new — cwPickQuery.h (camera-agnostic, trivially testable)
struct cwPickTolerance {
    // Allowed world-space pick radius at ray-depth t (t = projectedDistance
    // along the ray). Perspective is linear in t; ortho is constant. A
    // linear model  radius(t) = constant + slope*t  is exact for both.
    double slope    = 0.0;   // perspective: world radius per unit depth
    double constant = 0.0;   // ortho: fixed world radius (also a near floor)

    double radiusAt(double t) const { return constant + slope * t; }
    bool   enabled()          const { return slope > 0.0 || constant > 0.0; }
};

The picker fills it from the camera (cwCoordinatePicker already holds m_camera). No fov accessor is needed — the perspective slope falls straight out of the projection matrix:

// cwCoordinatePicker — derive a kPixelRadius-pixel tolerance for this pick
constexpr double kPickPixelRadius = 6.0;
const QRect vp = camera->viewport();
cwPickTolerance tol;
if (camera->projection().type() == cwProjection::Ortho) {
    // pixelsPerMeter is exact and valid only for ortho.
    tol.constant = kPickPixelRadius / camera->pixelsPerMeter();
} else {
    // Perspective + PerspectiveFrustum: tan(fovY/2) == 1 / P(1,1);
    // worldPerPixel(t) = 2*tan(fovY/2)/heightPx * t.
    const double p11 = camera->projectionMatrix()(1, 1);
    tol.slope = kPickPixelRadius * 2.0 / (p11 * vp.height());
}
Central-ray approximation. radiusAt keys off ray-depth t (projectedDistance), while true world-per-pixel keys off view-space z. They differ by 1/cos(angle-from-forward) — up to ~10–15 % at the frame edge. For a 6 px pick tolerance that is sub-pixel slop, accepted deliberately rather than carrying the cosine. Exact only for the central ray.
Figure 3. The accept radius r(t)=constant+slope·t is a cone in perspective (constant pixels on screen) and a cylinder in ortho. The fine test compares the true ray–segment distance d against r at the hit's depth.

Ray–segment closest approach

The substantive new math: the minimal distance between the (semi-infinite) pick ray and the finite segment AB, plus the closest point on the segment (which becomes pointWorld). Computed in double precision — at world-magnitude coordinates (~10⁴) the float cancellation problem that already bit the sphere test bites here too. The clamp logic is the standard segment/segment solver (Ericson) with the ray's parameter clamped only at its lower bound and the segment's clamped to [0,1].

// returns squared distance + closest point on the segment (model space)
struct RaySegmentHit { double dSq; QVector3D segPoint; };

RaySegmentHit raySegmentClosest(const QRay3D& ray,
                                const QVector3D& A, const QVector3D& B);
Two degenerate cases the helper must name. (1) Zero-length segment (a from==to shot — no longer filtered out now that addLines builds one node; see §8): treat e=|B−A|²≈0 as a point and fall back to ray-vs-point. (2) Ray parallel to the segment (denom≈0): the determinant solve is unstable, so pin the ray parameter to 0 and let the segment clamp pick the nearest endpoint — correct distance for the picking use, no NaN.
Identity-matrix radius caveat. Like the existing point/sphere path, the distance test compares a model-space dSq against a world-space radius. That is exact for the identity model matrix the line plot uses (cwRenderLinePlot passes a default QMatrix4x4). Documented inline, identical to the assumption the sphere branch makes.

5. Hit priority — runtime kind filter

Should a station near the cursor win over a scrap surface behind it? For the measurement tool, yes; for ordinary coordinate readout, no — nearest wins. This must be switchable per pick.

Why priority can't be a depth bias. The BVH prunes nodes whose box is farther than best.tWorld(). Letting a far line "outrank" a near surface would force traversal to stop pruning surfaces beyond the line's depth — breaking the hot path for every pick. Priority therefore lives in the caller, expressed as a cheap kind filter, not as a ranking tweak inside traversal.

Bundle the tolerance and an optional kind filter into one options struct. Default {} = all kinds, disabled tolerance → identical to today.

// cwPickQuery.h
struct cwPickQuery {
    cwPickTolerance tolerance;                        // §4
    enum class Kind : quint8 {                        // bitset over primitive kinds
        Triangles = 0x1, Lines = 0x2, Points = 0x4
    };
    Q_DECLARE_FLAGS(Kinds, Kind)
    static constexpr Kinds All = Kinds(Kind::Triangles) | Kind::Lines | Kind::Points;
    Kinds kinds = All;
};
Q_DECLARE_OPERATORS_FOR_FLAGS(cwPickQuery::Kinds)

cwRayHit intersectsDetailed(const QRay3D&, const cwPickQuery& = {}) const;

A QFlags bitset (not a 3-way enum) so a caller ORs exactly the kinds a pass should hit. Filtering is one hoisted check per top-level leaf, right beside the existing isPickable() guard (we already hold sub.object.geometry().type() there): map the object's type to its Kind bit and skip when !(query.kinds & bit). Pruning is untouched; the hot path pays nothing.

Figure 4. Priority is a caller choice. Equal priority is one All pick by depth. Station priority is a cheap Kind::Lines pre-pick (single tiny sub-BVH) with a Kind::Triangles | Kind::Points fallback — two click-time traversals, the BVH stays generic.

Alternative considered: a single traversal tracking bestLine and bestSurface separately, then a policy enum. Rejected for now — it needs dual pruning thresholds and a relaxed top-level cull, i.e. more machinery for no measurable gain at these sizes.

6. Broad-phase bounding

Because the radius grows with depth, a fixed AABB pad baked at build time can't bound it. Instead the per-pick traversal derives one conservative scalar — the radius at the farthest scene depth — and pads the box tests with it; the fine test still uses the exact radiusAt(hitDepth), so accuracy is preserved while no near-grazing pick is rejected early.

// intersectsDetailed, once per pick, only when query.tolerance.enabled():
const double farDepth = ray.projectedDistance(farthestCorner(rootWorldBox));
const float  linePad  = float(query.tolerance.radiusAt(farDepth)); // conservative bound
// fold into the existing top/sub box pads (max with the point-tube pad):
const QVector3D pad(std::max(tubeDist, linePad), ...);

Line objects are few (one centerline, a few thousand segments), so the conservative over-test costs a handful of extra leaf visits. The leaf AABBs themselves stay tight (no baked pad), so when the tolerance is disabled the structure and traversal cost are byte-for-byte what they are today.

One honest cost. The pad is global, so when a tolerance is active every top-level box inflates — including a giant LAZ object's. That means a few extra point sub-BVH descents per coordinate pick (which now always carries a tolerance). Picks are click-time, not per-frame, so this is acceptable; if a huge-cloud project ever shows it in a profile, the fix is to scope the pad to line-bearing top-level subtrees. Noted, not pre-optimized.
Figure 5. A ray that grazes within r(t) of a segment would miss the tight AABB (left). Inflating the box test by the conservative linePad (right) lets traversal reach the leaf, where the exact distance test decides the hit.

7. The cwRayHit contract

A line hit adds no fields to cwRayHit and repurposes none gratuitously. It slots into the existing per-kind contract: firstIndex already means "first index of the hit primitive," which is exactly a segment id, and pointWorld is simply snapped onto the line. A consumer needs only those two; everything else follows the conventions points already set.

The whole line-specific contract is two facts: pointWorld lies on the line, and firstIndex identifies the segment. u/v stay NaN exactly like points. A single fillLineHit() helper writes these — the analogue of the existing fillPointHit().
The firstIndex == from-vertex identity depends on the iota index list that setGeometry synthesizes. Commit 2 leans on it. Guard it with an assert / comment at the segmentEndpoints seam so a future switch to a truly indexed line buffer fails loudly instead of mis-mapping stations.

9. Commit 2 — clamp-to-station-ready hit contract

Stations are centerline vertices: shot i owns vertices [2i, 2i+1] = its from/to solved station positions (confirmed in cwLinePlotGeometry). So clamping a pick to a station is a screen-space snap to the nearer endpoint of the hit segment — no new geometry data and no new hit state (Commit 1 already fills firstIndex + pointWorld per §7). This commit adds the two consumer hooks and proves the snap end-to-end.

FileChange
cwRenderLinePlot.h/.cpp edit Add std::optional<QPair<QVector3D,QVector3D>> segmentEndpoints(int firstIndex) const — reads points[firstIndex] / points[firstIndex+1] from the live vertex buffer (bounds-checked; the §7 assert anchors the iota index == vertex identity). Keeping the accessor on the render object keeps the BVH generic.
cwLinePlotStationSnap new (free fn / small helper) Given a line cwRayHit, the cursor point, the camera, and kPickPixelRadius: fetch the segment's two endpoints, project them, and if the nearer one is within the pixel radius of the cursor, return that endpoint's world position + its vertex index (the station); else return the free point on the segment. Pure, camera-aware, no BVH coupling — exactly the seam #487 plugs into.
cwCoordinatePicker.cpp edit After a line hit, run the snap so the reported coordinate is the exact station position when the user clicks near a vertex. (A "snapped to station" Q_PROPERTY can wait for #487; not required here.)

10. Tests

Commit 1 — testcases/test_cwGeometryItersecter_linePick.cpp new

  • A known polyline at large world coordinates (the regime the brute-force test already targets).
  • With a populated cwPickQuery: rays passing within radiusAt(depth) hit; rays just outside miss; firstIndex maps to the expected segment; pointWorld lies on the segment.
  • Zoom invariance: a near ray (small slope·t) and a far ray (large slope·t) both behave per the radius model.
  • Kind filter: Kind::Lines ignores a coincident triangle/point object; Kind::Triangles | Kind::Points ignores the centerline; All returns the nearer by depth.
  • Parity guard: default query {} → line objects never hit, and a mixed scene's triangle/point hits are identical to a triangles/points-only run.

11. Out of scope / handoff to #487

  • Measurement UI — two-point pick, distance / azimuth / clino / vertical / horizontal readout. Built independently on #487; it consumes intersectsDetailed(ray, query) with the §5 priority recipe and the §9 snap helper for stations.
  • Station names on the hit. The vertex→name map was discarded during geometry generation; surfacing "from A4 to B12" needs per-segment name metadata added to cwLinePlotGeometry::Result and plumbed through setGeometry. Purely additive; the measurement math needs only positions, which Commit 2 already provides.
  • Per-trip visibility filtering of picks (skip hidden trips during traversal) — a separate follow-up noted in #530; the per-vertex visibility buffer already exists to drive it.
  • Adopt an explicit cwPickQuery in the three other BVH callers. Today cwLeadView::isLeadVisible (cwLeadView.cpp:574), cwBaseTurnTableInteraction::unProject (cwBaseTurnTableInteraction.cpp:1121) and the pivot intersects(ray) (cwBaseTurnTableInteraction.cpp:135) all call with the default query — kinds = All, tolerance disabled. That keeps lines off (behavior unchanged), but these are occlusion / pivot queries that should never clamp to the centerline, and relying on the implicit default couples their correctness to it: a future global default flip (or a maintainer enabling a default tolerance) would silently change occlusion and pivot behavior. Make the intent explicit — cwPickQuery{ .kinds = Triangles | Points } — and give each its own coverage. A no-op-today change spanning two unrelated subsystems, so it's deferred out of Commit 1; pair it with #487.

12. Decisions locked in

  • Screen-space radius, ~6 px, via the camera-agnostic cwPickTolerance linear model — exact for the central ray in both perspective and ortho.
  • Lines live in the BVH unconditionally; the per-pick tolerance (not a baked radius) gates whether they can be hit, so picking-disabled cost is unchanged.
  • Priority is a runtime kind filter chosen by the caller (cwPickQuery::kinds), never a depth bias inside traversal — the BVH stays generic and its pruning untouched.
  • cwRayHit gains no fields; lines reuse the existing per-kind contract (pointWorld on the line, firstIndex = segment), u/v stay NaN.
  • Conservative single-scalar broad-phase pad from the far-depth radius; exact fine test.
  • Clamp lives one layer above the BVH (render-object endpoint accessor + camera-aware snap), keeping the intersecter generic.
  • Two commits: (1) pickable centerline, (2) clamp-to-station hooks. #487 proceeds independently on top.
