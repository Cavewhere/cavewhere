Placement-rule params — interpreting the bytes

Design note for the step before the Offset stroke rule (see CEILING_CHANNEL_DESIGN.html). Goal: let a placement rule read its own parameters, so a layer can say "space ticks every 3 mm" or "offset this stroke by −1.2 mm" instead of every rule running from a hard-coded constant.

What already exists (and what doesn't)

The storage and round-trip are already built. A rule is persisted as a name plus an opaque parameter blob, and unknown rules survive a load/save untouched:

// cavewhere_symbology.proto
message PlacementRule {
    string name   = 1;   // registry key: "Uniform spacing", "Offset stroke", …
    bytes  params = 2;   // proto-encoded per-rule struct  ← already round-trips; NOT yet decoded
}
// cwPlacementRuleData mirrors it: { QString name; QByteArray params; }

What's missing is only interpretation: today the layout looks up name → cwPlacementRule* and throws the bytes away; every rule runs from a documented constant (kDefaultStampSpacingMm = 2.0, etc.). So this is not "add a parameter system" — it's "stop discarding the bytes, and give each rule a typed view of them." The persistence layer doesn't change at all.

The three decisions that matter

1 · Encoding: a per-rule proto message recommended

The proto comment already commits to "proto-encoded per-rule struct." Honour it: each rule that takes parameters gets a small message, serialized into params.

// cavewhere_symbology.proto  (new messages, near PlacementRule)
message UniformSpacingParams { optional double spacingMm = 1; }   // unset → 2.0 mm
message OffsetStrokeParams   { optional double offsetMm  = 1; }   // unset → 0 (on the line); sign = side
 Per-rule message recGeneric key/value map

The only real downside of per-rule messages — a proto edit per rule — is small and exactly the kind of change proto is good at. Recommend per-rule messages.

2 · Plumbing: the bytes ride in a per-rule context recommended

Rules are stateless singletons, so a rule's params must arrive at call time. We already follow the convention "read-only inputs go in a context struct; the mutable in/out stays a separate argument." Params are a read-only input, so they belong in the context — the layout just rebuilds the (tiny) context per rule:

struct cwPlacementContext {
    const cwStrokePath &strokePath;
    const cwDecorationLayer &layer;
    double worldPerPaperMm = 0.0;
    const QByteArray &ruleParams;     // ← NEW: this rule invocation's params (may be empty)
    cwSceneContext *scene = nullptr;
};

// layout: keep (rule, params) paired, rebuild the context per rule
for (const auto &step : resolvedStack) {                 // step = { const cwPlacementRule *rule; QByteArray params; }
    const cwPlacementContext ctx{ strokePath, layer, worldPerPaperMm, step.params, nullptr };
    step.rule->apply(positions, ctx);
}

Rebuilding the context per rule is free (a handful of references + a double). The alternative — a third apply(positions, context, params) argument — works but splits the read-only inputs across two places, against the convention. Recommend params-in-context.

Scope note — terminals. stampPath() takes cwStampGeometry, not the context, so a terminal that needs a param (e.g. a future bending step size) would read it from cwStampGeometry instead. Our two first consumers don't need that: Uniform spacing is a Generate apply() and Offset stroke is a stroke-transform hook — both see the context. Adding a param to cwStampGeometry is a later, parallel extension; flagged, not built.

3 · Presence & defaults: optional fields + fall back to the documented constant

proto3 scalar defaults are 0, which is indistinguishable from "deliberately set to 0" — and for spacingMm, 0 is a degenerate value, not a default. Use optional fields (explicit presence, exactly as DecorationLayer.offsetCurveWidthMm already does) so a rule can tell "unset → use my constant" from "set to 0 → honour it (and guard)":

// cwUniformSpacingRule::apply
double spacingMm = kDefaultStampSpacingMm;                  // documented default survives as the fallback
if (!context.ruleParams.isEmpty()) {
    UniformSpacingParams p;
    if (p.ParseFromArray(context.ruleParams.constData(), context.ruleParams.size()) && p.has_spacingmm()) {
        spacingMm = p.spacingmm();
    }
}
const double spacingWorld = spacingMm * context.worldPerPaperMm;
if (spacingWorld <= 0.0) { return; }                        // still guards a bad/zero value

Net effect: a layer with empty params behaves byte-identically to today; the hard-coded constants become the fallbacks, not dead code. A malformed blob also falls back rather than crashing.

Aside: why not just a void*?

The tempting shortcut is to drop the bytes-and-proto entirely and hang a void* (or a cwPlacementRuleParams*) off cwPlacementRuleData, then static_cast it back inside each rule. It fails on five hard requirements this data actually has:

  1. It has to persist. Params are written to a .cw/.cwproj and synced over git to other people and machines. A void* is an address in this process's heap — meaningless the moment it's saved or reopened elsewhere. The bytes are the durable form; the typed message is only a decode of them. A pointer has no serialized form at all.
  2. cwLineBrush is an implicitly-shared value type. It copies, compares with operator==, and round-trips by value. A void* member breaks all three: copying duplicates the pointer (aliasing, double-free), equality compares addresses instead of contents, and a copied brush handed to a worker thread shares mutable state with the original. A QByteArray copies, compares, and shares safely.
  3. Type safety. void* discards the type; every rule casts blind, and a wrong cast (a rule rename, a struct-layout drift between builds) is undefined behaviour, not a parse error. The proto decode fails cleanly and falls back to the documented default.
  4. Ownership & lifetime. Who news and deletes the struct, and when? Rules are const singletons shared across threads; the layer data that owns the params is copied freely. A void* has no answer; the owned QByteArray does.
  5. Unknown-rule round-trip & evolution. The bytes let a rule this build doesn't recognise survive a load/save intact, and let a param field added by a newer build be preserved by an older reader (proto unknown fields). A void* can represent neither — there's no struct to point at for an unknown rule, and no schema to evolve.

In short, a void* conflates "typed in-memory access" with "the durable, shareable parameter data." We already hold the durable form — the bytes; the per-rule proto message is just the typed lens onto it, which is exactly the part a void* would throw away.

A one-line encodeParams(msg) / decodeParams<T>(bytes) helper pair removes the boilerplate and keeps encode/decode symmetric.

Things the design deliberately keeps out

  • The registry stays dumb. It maps name → rule and knows nothing about param types. Decode is each rule's private business — no generic "param schema" framework, no registry-level validation. (If we ever want an editor to introspect a rule's params, that's a deliberate later addition.)
  • No IO change. The persistence layer already treats params as opaque bytes; interpretation lives entirely at the seed (encode) and layout (decode) edges.
  • The offsetCurve* fields are not rule params. They're per-layer line styling for the trace terminal and stay where they are for now (a later commit may fold offsetCurveOffsetMm into the trace rule's params — out of scope here).
  • No editor. Authoring is C++ (seed/tests) until commit 8.

Forward-compatibility

proto3 C++ preserves unknown fields through a parse + re-serialize, so a newer palette that adds a param field survives an older reader that only knows the old fields — provided the reader decodes and re-encodes the message. (If a reader never decodes a rule's params, the bytes round-trip trivially as the opaque blob they already are.) Renaming a rule is breaking (already documented); removing a param field uses reserved, same as the rest of the schema.

Proposed scope & sequence

  1. Param mechanism + first consumer. Add ruleParams to the context; make the layout keep (rule, params) paired through the stage sort; add the encode/decode helpers and the UniformSpacingParams message; make Uniform spacing read spacingMm (fallback 2.0 mm). Tests: param drives spacing; empty params reproduces today; malformed blob falls back.
  2. Offset stroke rule (next doc). Now a rule can read a signed offsetMm, so the channel's +d/−d sides are two instances of one rule with different params — the sign problem dissolves. Adds the offset-polyline helper + stroke-transform stage + ceiling-channel demo; absorbs commit 4.5.
  3. DAG (later). Brush references; the reference's transform ultimately carries the offset, retiring the per-layer carrier entirely.

Open questions for you

  • Helper shape: a free encodeParams/decodeParams<T> pair, or a tiny method on each rule (T paramsOr(defaults, bytes))? (Lean: free helpers — rules stay thin.)
  • Decode timing: decode inline in apply() every layout (simplest; layout is an edit-time pass, not per-frame), or decode once when the stack is resolved and carry a typed value? (Lean: inline — re-layout is cheap and rare.)
  • Message granularity: one message per rule (recommended), or shared messages for rules with identical param shapes (e.g. several rules that only need a single distanceMm)? (Lean: per-rule — clarity over DRY for tiny messages.)
  • First consumer: confirm Uniform spacing / spacingMm is the right proving ground, vs. picking a different existing constant.
