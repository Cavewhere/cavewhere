Centerline Per-Shot Visibility — Redesign Plan

Replace the per-trip tripId+R8-texture visibility with per-shot de-sharing and per-vertex visibility, as the foundation for arbitrary keyword / named-region filtering of the 3D centerline.

Status / baseline. The current per-trip implementation is committed as 9f79c241 ("Add keyword visibility for the survey centerline") on branch dev_cavewhere. This plan supersedes its internal representation only — the user-facing feature (keyword + Type = Line Plot filtering) is unchanged. Evolve the working tree in place on top of that commit; squash before merge if a single clean commit is wanted. Do not stash-and-restart — the keyword/manager half is reused almost verbatim.

2. Why redesign

The committed implementation keys visibility by trip: vertices are de-shared per trip, each carries a tripId, and an R8 texture (one texel per trip) holds visible/hidden, sampled in the vertex shader. That is the right shape for a coarse, unbounded, single-membership axis (trips). It does not extend cleanly to what we actually want next:

  • Arbitrary keyword groups on a series of shots — e.g. "this zone is a named level," which can span multiple trips and overlap other groups. A segment then has multiple, overlapping memberships, and visibility becomes an AND of them. One tripId texel can't represent that.
  • At cave scale (typical 1–2k stations; the longest cave on Earth ~100k) the texture indirection buys nothing measurable — its only edge was being smaller than per-vertex storage when groups ≪ vertices, which stops being true once groups get fine-grained.
  • Re-uploading an R8 texture is the same class of GPU transfer as re-uploading a small buffer, so the indirection is pure machinery (texture, sampler, SRB binding, recreate-on-width, texelFetch) with no payoff at these sizes.

The per-shot model below is simpler and strictly more flexible, and it is the natural foundation for the named-region feature.

3. The core idea

De-share at the shot, not the trip. A station is not a renderable thing — a shot is. The centerline is a set of segments, so make the shot the atomic, independent unit: each shot owns its own two endpoint vertices. A station reused by consecutive shots is duplicated, once per shot.

Figure 1. The complete de-share. No vertex is shared between two shots, so collapsing a shot's vertices affects exactly that shot — no special masking logic, no dedup hash.

Visibility is per-shot, stored per-vertex. Each vertex carries a 1-byte visibility value (both vertices of a shot share it) in a separate dynamic buffer parallel to positions. The vertex shader reads it directly and collapses hidden vertices off-clip. No R8 texture, no sampler, no SRB texture binding, no texelFetch, no index buffer (shots are consecutive vertex pairs → non-indexed line list).

The CPU resolves every grouping. Trip, level, named region, even a single shot are all just "a set of shots." The manager maps any active keyword filter to per-shot visibility (the AND of every group a shot belongs to) and writes the bytes. One mechanism covers every axis, including overlapping ones, with no group-count cap.

5. Geometry — per-shot de-share

cwLinePlotGeometry::generate becomes simpler:

  • For each cave → trip → chunk → shot: resolve the from/to station world positions. If both exist, append two vertices (from, to) as a segment. No QHash dedup — every shot is independent.
  • Lay vertices out grouped by trip (and contiguous per trip), so a trip is a single [vertexStart, vertexCount) range. Emit that trip → range table (indexed by the same running id as the index → UUID table the manager already resolves).
  • Keep the bug #435 bootstrap: a shot whose station has no solved position is skipped (its segment simply isn't emitted), exactly as today's vertexFor returns the −1 sentinel.
  • Depth/length accumulation is unchanged (read positions; respect isDistanceIncluded()).

Result struct: points (2 per emitted shot), a per-trip vertexRanges table, the tripUuids (index→UUID) table. The per-vertex tripIds array is gone. (No index array — the renderer draws a non-indexed line list.)

6. Render / RHI — per-vertex visibility

  • Positions buffer — per-shot vertex pairs, uploaded once per solve, never touched on toggle.
  • Visibility buffer — a separate dynamic buffer, one entry per vertex (both vertices of a shot share the value), cwTracked independently of positions so a toggle re-uploads only this buffer. Format an impl detail (RHI vertex attributes generally want 4-byte alignment, so likely a uint or normalized UByte4 read as .r — note this is the same ~4 bytes/vertex the deleted tripId attribute used).
  • No index buffer — draw as a non-indexed line list; segment i = vertices [2i, 2i+1].
  • Shader — if (visibility < 0.5) { gl_Position = vec4(2,2,2,1); return; }, reading the per-vertex attribute. No texture sample.
  • Render-object API: replace setTripVisible(id,bool) with a per-vertex / per-range write (e.g. setVisibility(QVector<quint8>) or setRangeVisible(start, count, bool)); keep setGeometry(points, vertexRanges) resetting visibility to all-visible.
Keep the constants. Reuse the kTripVisible = 0xFF / kTripHidden = 0x00 sentinels (rename to kVisible/kHidden); the 0.5 shader threshold is unchanged.

7. Manager — CPU resolution

The manager keeps its keyword-item-per-group machinery (committed). What changes is the push:

  1. It holds, per group, the set of shots/vertex-ranges that group covers. For trips this is the trip → range table from geometry. For future named regions it is the region's shot set.
  2. On a toggle (proxy setVisible) or after a solve, it writes per-vertex visibility = AND of every group's state over that vertex's shot, then pushes the (small) buffer.
  3. setGeometry resets all-visible; reconcile re-applies hidden groups by writing their ranges — same reset-then-seed flow as today, but writing buffer ranges instead of texels.

For the current trips-only feature, each shot belongs to exactly one group (its trip), so the AND is trivial and writes are contiguous per-trip ranges.

8. Combination semantics (overlap / AND)

A shot is visible iff every group it belongs to is visible. Because the combination is resolved on the CPU into a single per-vertex byte, this handles arbitrary overlap and any number of groups with no cap — the reason per-vertex visibility beats a GPU bitmask (which would cap groups at the mask width and bloat the vertex attribute). On any single group toggle, only that group's shots need recomputing/re-uploading.

Modestly more memory (positions ~2×, but the index buffer and texture go away), immaterial at cave scale — typical caves are 1–2k stations (low KB); the largest on Earth ~100k stations (~MBs). Toggle re-uploads are sub-millisecond and only on user action. Net: simpler GPU path, more flexibility, trivial cost.

10. Implementation steps

  1. Geometry — rewrite generate for per-shot pairs; emit trip → vertex-range + tripUuids; drop tripVertexLookup and the per-vertex tripIds; keep #435 handling. Update cwLinePlotTask pass-throughs. Rewrite the geometry test.
  2. Render / RHI — positions buffer = pairs; add the dynamic per-vertex visibility buffer (separate cwTracked); drop the index buffer (non-indexed line list); delete the R8 texture / sampler / SRB-texture / recreate path; update the shader to read the attribute. Run the line-plot tests.
  3. Manager — store trip → range; replace the texel push with per-range visibility writes (AND for overlap); keep reconcile/Type/UUID identity; re-seed hidden ranges after solve.
  4. Tests — geometry per-shot duplication; reuse the [LinePlotManager][keyword] cases (registration, Type, toggle, survive-re-solve, removal) adjusted for the new visibility query.
  5. Verify — build cavewhere-test + cavewhere-qml-test; run the full C++ suite and the QML suite (offscreen), teeing to /tmp/*.log. Then manually confirm in the running app that filtering hides/shows centerline segments.
Per-sketch-plan workflow: for staged commits run only the new tests at each step; run the full C++ + QML suites once at the end before pushing.

11. Tests

  • Geometry (test_cwLinePlotGeometry.cpp): a shared station appears once per shot (e.g. A1→A2→A3 ⇒ 4 vertices, A2 twice); vertex count = 2 × emitted shots; per-trip vertex-range table is correct; UUID table maps by identity; #435 shots with no position are skipped.
  • Manager ([LinePlotManager][keyword]): one item per trip; each publishes Type = Line Plot + the trip's keywords; default all-visible; hiding a proxy hides exactly that trip's shots; hidden state survives a re-solve (UUID identity); removing a trip drops its item.

12. Future / out of scope

  • Named regions / arbitrary keyword groups — the payoff. A region = a user-selected set of stations/shots (possibly spanning trips/caves) owning its own cwKeywordModel; the manager registers a keyword item per region whose shots AND into the per-vertex visibility. Needs a data-model entity + persistence (protobuf) + selection UI. Per-shot visibility is the enabling foundation.
  • Splays — currently stored on stations (cwStation::m_splays) but not rendered. Per-shot naturally accommodates them (each splay = one shot = 2 vertices) if/when rendered.
  • Picking — the centerline is not pickable today (the BVH skips Lines); tracked in #530. Per-shot helps: shotId = vertexId / 2 maps a hit straight back to a shot.
  • Related cleanup issues from the committed work: #531 (visibility-proxy base class), #532 (cwTracked::setValue rvalue overload).

13. Decisions locked in (from discussion)

  • De-share per shot — each shot owns its 2 vertices; shared stations duplicated per shot. Shot is the atomic, leak-free unit of visibility.
  • Per-vertex visibility, no texture — visibility stored per-vertex in a separate dynamic buffer; the R8-texture indirection is dropped (no benefit once visibility is shot-grained; texture re-upload ≈ buffer re-upload).
  • Drop the index buffer — non-indexed line list.
  • CPU resolves combination — per-shot visibility = AND of all its groups; no GPU bitmask, no group-count cap, native overlap. (Bitmask rejected: it caps groups and only wins on zero-re-upload toggles, which don't matter at cave scale.)
  • Groups can overlap; a shot can be in several at once (trip + level + region).
  • No nesting of groups assumed.
  • Named regions defined by a station/vertex selection (volume of stations), per the user.
  • Keep the manager/keyword layer from commit 9f79c241 — it is representation-independent.
