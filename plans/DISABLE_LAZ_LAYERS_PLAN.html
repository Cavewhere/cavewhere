Disabling LAZ Layers

A per-layer enabled property: kept in the project, synced via Git, with disabled layers excluded from rendering, processing, and the async LAZ load.

Context

This is the “commented-out code” gesture for LAZ layers. A user disables a layer when:

  • Re-survey. The old scan should stay in the project as a record, but should not be loaded, rendered, or fed into algorithms.
  • Post-clip cleanup. Commit 5cd906b made the LAZ clip lossy — only point locations survive; intensity / classification / RGB / GPS time are dropped. Deleting the original is not acceptable (future algorithms may need the dropped attributes), but leaving both the original and the clip visible is confusing. Disable the original; the file stays on disk, the clip becomes the only visible layer.

This is the project-level visibility mechanism. It persists, it syncs through Git like every other project edit, and it lives on the object itself — the same way a layer’s name and source path do.

A separate, lighter-weight session-level hide gesture (right-click “Hide” in the scene, viewport chip for unhide) is intentionally deferred to a follow-up plan. See the bottom of this document.

Status snapshot

The save/load substrate landed in a separate series (plans/LAZ_LAYER_FILE_PLAN.html commits 1–6) while this plan was paused. As a result, persistence now goes through a per-layer <basename>.cwlaz sidecar instead of the LazLayerState proto overlay this plan originally proposed. The enabled property itself, the cancellation side effects, and the reload() short-circuit are also already in place. What remains is the visibility wire-up, the QML toggle, and the post-clip auto-disable hook.

UX

Data section toggle

cavewherelib/qml/GeospatialLayerPage.qml is the existing Data-section page that lists LAZ layers (RootData.region.lazLayers). Each row gains an Enable/Disable toggle (eye icon or labeled checkbox). Disabled rows render dimmed and tagged “Disabled” so they’re still discoverable. Re-enabling triggers the normal async load.

Post-clip auto-disable

After a successful clip, every source layer that contributed is automatically set to enabled = false. No prompt, no setting. The user’s recovery path is the Data section toggle — they can re-enable any layer there at any time. The new clip layer remains visible.

This matches the “clip means the original should step aside” intent without adding an inline-prompt surface that would be the same answer almost every time. Disabling is reversible and the Data section makes the change discoverable.

Effective visibility

Render-side visible iff layer.enabled AND (keyword filter accepts). Both inputs are ANDed in cwLazLayersSceneNode. If a layer is disabled, the keyword filter cannot bring it back — the user must re-enable it from the Data section.

Architecture

1. enabled on cwLazLayer done

Already shipped: 

Q_PROPERTY(bool enabled READ enabled WRITE setEnabled NOTIFY enabledChanged)

bool enabled() const { return m_enabled; }
void setEnabled(bool enabled);

signals:
    void enabledChanged();

private:
    bool m_enabled = true;   // default: enabled

The setter follows the project pattern (if (m_enabled == enabled) return; m_enabled = enabled; emit enabledChanged(); /* side effects */;). On true → false: cancels the in-flight load (m_loadRestarter.future().cancel()), drops m_geometry back to default, clears the bbox / spacing / error message, and resets m_loadStatus to Idle. On false → true: calls reload().

2. Persistence done

Per-layer enabled (plus a stable QUuid identity) is persisted in a sidecar <basename>.cwlaz next to each .laz in GIS Layers/. Discovery is by folder scan; cwLazLayerModel::rescan pairs each .laz with its .cwlaz sibling and calls cwLazLayer::setData(...) before announcing the row, so the final enabled bit is in place by the time observers (the scene node, the keyword filter pipeline, cwSaveLoad) see rowsInserted. The full design lives in plans/LAZ_LAYER_FILE_PLAN.html. The .cwproj file no longer carries a LAZ-layer overlay (field 7 on ProjectMetadata is reserved).

For Git sync, the .cwlaz sits inside the same GIS Layers/ directory the .laz already lives in, so toggling enabled produces a single-file diff on the .cwlaz. The merge-handler shape for LAZ layers is itself an out-of-scope follow-up tracked in LAZ_LAYER_FILE_PLAN.html.

3. Scene-node materialization todo

A disabled layer has no geometry to draw — cwLazLayer::setEnabled(false) already drops m_geometry and resets m_loadStatus to Idle (cwLazLayer.cpp:174), and reload() short-circuits on !m_enabled (cwLazLayer.cpp:240). Leaving an empty cwRenderPointCloud and a cwKeywordItem in the scene for it is pure overhead and surfaces the layer as a chip in the keyword filter UI even though it has nothing to draw.

The fix: don’t materialize the scene-node state for disabled layers at all. Treat enabledChanged as the canonical add/remove signal for the scene node.

  • cwLazLayersSceneNode::addLayer (called from rowsInserted) always connects cwLazLayer::enabledChanged to a new handler — even if the layer is currently disabled — so a later toggle is observed. If layer->enabled() is true, proceed with the current body (create the render object, hook loadStatusChanged → syncLayerGeometry, add the keyword item). If false, return after the enabledChanged connect.
  • The new enabledChanged handler is a single dispatch:
    • false → true: run the materialization body (factor the current addLayer body into a private materialize(cwLazLayer*) helper that both call sites share). setEnabled(true) in cwLazLayer already calls reload() after emitting enabledChanged, so syncLayerGeometry will push the geometry as soon as the async load lands on Loaded.
    • true → false: run the tear-down body (factor the current removeLayer’s render-object / keyword-item disposal into dematerialize(cwLazLayer*); the model-disconnect that removeLayer does last must stay in removeLayer, not in dematerialize). setEnabled emits enabledChanged before it calls setLoadStatus(Idle), so the tear-down disconnect runs first and the subsequent loadStatusChanged finds nothing to sync — no warning, no dangling lookup.
  • cwLazLayersSceneNode::removeLayer (called from rowsAboutToBeRemoved) calls dematerialize if the render object exists, then disconnects everything from the layer (covers both the enabledChanged and the loadStatusChanged hookups). No new state to track.
  • rebuild() / clear() are unchanged — clear() walks m_pointClouds and m_keywordItems, both of which are now narrower (disabled layers simply aren’t in either container).

Side effects of this approach:

  • setVisibleForAll still iterates m_pointClouds for capture-mode screenshots; disabled layers are correctly excluded.
  • visibleLayers() currently iterates m_model->layers() and filters by m_pointClouds[layer.id()].isVisible(); disabled layers will be absent from m_pointClouds and so dropped silently. That’s the right answer for the post-clip auto-disable path (Commit 3) — a disabled layer is not a valid clip source.
  • pointCloudForLayer(layer) returns nullptr for a disabled layer. The only callers are tests; the new test fixture asserts this explicitly so the contract is documented.
  • The keyword filter chip list now only contains enabled layers, which matches the “commented out” mental model — a disabled layer is not in the scene, period.

No new state, no recursion guard, no visibility composition — the existing add/remove machinery does the work and enabledChanged is its trigger.

4. Load skipping done

cwLazLayer::reload() short-circuits on !m_enabled at cwLazLayer.cpp:240. setSourcePath still records the size/mtime fingerprint (the cwLazLayerModel::rescan in-place-overwrite detector relies on it), but does not kick the restarter while the layer is disabled. Re-enabling calls reload(), which starts the load.

cwLazLayerModel::rescan needs no change — newly-discovered layers default to enabled = true, and any disabling has already been applied by cwLazLayer::setData(...) from the paired .cwlaz before rowsInserted fires.

5. Data section toggle todo

cavewherelib/qml/GeospatialLayerPage.qml has both a wideDelegateComponent (around line 90) and a narrowDelegateComponent (around line 156). Both bind to model.name, model.sourceCS, model.pointCount. The model currently exposes those four roles plus LayerRole, SourcePathRole, LoadStatusRole (cwLazLayerModel.h:37); there is no EnabledRole yet.

  • cwLazLayerModel.{h,cpp} — add EnabledRole to the Roles enum, populate it in data(), accept it in setData() (route through layer->setEnabled(value.toBool())), and register the string "enabled" in roleNames(). Connect cwLazLayer::enabledChanged in connectLayer (already at cwLazLayerModel.cpp:537) so the model emits dataChanged(index, index, {EnabledRole}) when the property changes from outside the model (e.g. the post-clip auto-disable).
  • GeospatialLayerPage.qml — add a per-row QC.CheckBox (or eye-icon button) bound to model.enabled; dim the row when disabled (opacity: model.enabled ? 1.0 : 0.5); show a small “Disabled” chip via QC.Label + Theme.fontSizeCaption. Apply consistently in both delegates.

6. Post-clip auto-disable todo

cavewherelib/src/cwLazClipInteraction.cpp:174 resolves contributing layers via m_sceneNode->visibleLayers(). Capture their QUuids into a list at that point. Today, the success branch of the AsyncFuture callback (around lines 233–253) calls modelGuard->rescan() to surface the new clip layer. After that rescan(), iterate the captured UUIDs, look each one up in cwLazLayerModel (a small cwLazLayer* layerById(QUuid) helper on the model, or a linear search through layers()), and call layer->setEnabled(false) on any pointer that’s still alive.

UUIDs (not QPointer<cwLazLayer>) because identity is what matters: a layer that was removed and re-paired during the rescan still represents the same source artifact and should still get disabled. Stale UUIDs (layer truly gone) are silently skipped.

Failure / cancellation branches do nothing — the user’s state is untouched on a no-op clip.

Out of scope & follow-ups

  • Session-level hide — a transient right-click “Hide” gesture with a viewport chip for unhide, never persisted, never synced. Conceptually paired with this plan but deferred to its own document. Will use a new cwHiddenObjects manager keyed by QUuid, ANDed into the same visibility composition: enabled AND (not in session-hidden) AND (keyword filter accepts).
  • Disabling other object types (scraps, trips, traces). The same enabled pattern generalizes; each type adds the property, persists it, and ANDs into its own scene visibility path.
  • Replace original destructive mode for clipping. Not in this plan; clipping always preserves the source file.
  • Multi-select disable from the Data section. Trivial extension once the per-row toggle works.
  • Persistence of keyword filter accept/reject. Still not persisted; out of scope.
  • Sync-merge handler for LAZ layers. Tracked in LAZ_LAYER_FILE_PLAN.html’s “Out of scope” section. The .cwlaz shape (UUID + enabled bit) is the prerequisite; a real per-layer merge UI is a separate effort.

Decisions made

  • Property name: enabled. Matches the “comment-out code” analogy directly — users disable a layer the way they comment out a block.
  • Disabling a mid-load layer cancels the in-flight load. cwLazLoader already honors cancellation via QPromise::isCanceled() between point chunks. The asyncfuture Restarter propagates the outer cancel down to the inner worker, so cancellation is a single m_loadRestarter.future().cancel() call. The observe-chain receives the canceled future; the m_enabled guard in the applyResult callback drops any stale delivery.
  • Persistence lives on the .cwlaz sidecar, not the .cwproj. Decided during the LAZ_LAYER_FILE_PLAN.html series. The .cwlaz next to each .laz means a disable/enable toggle is a single small file diff that Git handles cleanly, and adding new per-layer fields (point size, source-CS override, …) later doesn’t require touching project metadata.
  • Disabled layers aren’t materialized in the scene at all. Because cwLazLayer::setEnabled(false) already drops m_geometry and reload() short-circuits, a disabled layer’s render object would be permanently empty. Rather than carry an empty cwRenderPointCloud + cwKeywordItem through the scene and filter chip list, cwLazLayersSceneNode simply doesn’t create them. enabledChanged is the add/remove trigger. This also keeps the keyword filter UI consistent with the “commented out” mental model — a disabled layer is not present in the scene, period. The trade-off is that pointCloudForLayer(layer) can return nullptr for a layer that’s in the model; the test fixture documents the contract.
  • Post-clip behavior is silent and unconditional. No prompt, no setting. The Data-section toggle is the recovery path. An inline-prompt surface that asks every time would be the same answer almost always.
