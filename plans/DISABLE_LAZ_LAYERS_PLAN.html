Disabling LAZ Layers

A per-layer enabled property: kept in the project, synced via Git, with disabled layers excluded from rendering, processing, and the async LAZ load.

Context

This is the “commented-out code” gesture for LAZ layers. A user disables a layer when:

  • Re-survey. The old scan should stay in the project as a record, but should not be loaded, rendered, or fed into algorithms.
  • Post-clip cleanup. Commit 5cd906b made the LAZ clip lossy — only point locations survive; intensity / classification / RGB / GPS time are dropped. Deleting the original is not acceptable (future algorithms may need the dropped attributes), but leaving both the original and the clip visible is confusing. Disable the original; the file stays on disk, the clip becomes the only visible layer.

This is the project-level visibility mechanism. It persists, it syncs through Git like every other project edit, and it lives on the object itself — the same way a layer’s name and source path do.

A separate, lighter-weight session-level hide gesture (right-click “Hide” in the scene, viewport chip for unhide) is intentionally deferred to a follow-up plan. See the bottom of this document.

UX

Data section toggle

cavewherelib/qml/GeospatialLayerPage.qml is the existing Data-section page that lists LAZ layers (RootData.region.lazLayers). Each row gains an Enable/Disable toggle (eye icon or labeled checkbox). Disabled rows render dimmed and tagged “Disabled” so they’re still discoverable. Re-enabling triggers the normal async load.

Post-clip auto-disable

After a successful clip, every source layer that contributed is automatically set to enabled = false. No prompt, no setting. The user’s recovery path is the Data section toggle — they can re-enable any layer there at any time. The new clip layer remains visible.

This matches the “clip means the original should step aside” intent without adding an inline-prompt surface that would be the same answer almost every time. Disabling is reversible and the Data section makes the change discoverable.

Effective visibility

Render-side visible iff layer.enabled AND (keyword filter accepts). Both inputs are ANDed in cwLazLayersSceneNode. If a layer is disabled, the keyword filter cannot bring it back — the user must re-enable it from the Data section.

Architecture

1. enabled on cwLazLayer

New Q_PROPERTY on cavewherelib/src/cwLazLayer.{h,cpp}: 

Q_PROPERTY(bool enabled READ enabled WRITE setEnabled NOTIFY enabledChanged)

bool enabled() const { return m_enabled; }
void setEnabled(bool enabled);

signals:
    void enabledChanged();

private:
    bool m_enabled = true;   // default: enabled

setEnabled follows the project pattern (if (m_enabled == enabled) return; m_enabled = enabled; emit enabledChanged(); /* side effects */;). Side effects:

  • If transitioning true → false: cancel any in-flight load (m_loadRestarter.cancel()), drop m_geometry back to a default-constructed cwGeometry, set m_loadStatus = Idle, emit the relevant property-changed signals so the model UI updates. The render object becomes invisible via the scene-node wire-up below.
  • If transitioning false → true: call reload() to kick off a fresh async load.

2. Persistence

LAZ layers are currently discovered via folder-scan of GIS Layers/ and intentionally not serialized in the project protobuf (see the reserved 6 comment on ProjectMetadata at cavewherelib/src/cavewhere.proto:25). The folder scan stays authoritative for discovery; we only need to persist per-layer user state on top.

Add a new repeated message keyed by relative file path: 

// cavewhere.proto
message LazLayerState {
    optional string relativePath = 1;  // path relative to GIS Layers/
    optional bool enabled = 2;         // absence == true (default)
}

message ProjectMetadata {
    ...
    repeated LazLayerState lazLayerStates = 7;  // new slot
}

On save, write a LazLayerState entry only for layers whose enabled differs from the default (keeps the file lean and the diff churn-free). On load, after the folder scan populates the model, walk the state list and apply setEnabled(false) to any layer whose relative path matches an entry with enabled = false. State entries for paths that no longer exist on disk are silently dropped (or trimmed on next save).

3. Visibility wire-up

cavewherelib/src/cwLazLayersSceneNode.{h,cpp} ANDs the existing keyword-filter pipeline state with layer->enabled() when setting each per-layer cwRenderPointCloud’s visibility. A visibleChanged listener on the render object re-applies the override if the filter pipeline tries to flip a disabled layer back to visible (recursion-guarded by bool m_applyingDisable). Listens to cwLazLayer::enabledChanged to re-evaluate on toggle.

4. Load skipping

cavewherelib/src/cwLazLayer.cpp: reload() short-circuits if !m_enabled — sets status to Idle, leaves m_geometry empty, does not kick the restarter. setSourcePath still captures the size/mtime fingerprint and stores the path, so the layer is identifiable in the model without owning the bytes.

cavewherelib/src/cwLazLayerModel.cpp: rescan() needs no change to its current diff-based behavior — newly-discovered layers are created with the default enabled = true, and disabling happens after-the-fact via the persisted state during load. The existing (size, mtime) fingerprint check still handles in-place overwrites; a layer that was disabled and whose file is later overwritten will reload only when the user re-enables it.

5. Data section toggle

cavewherelib/qml/GeospatialLayerPage.qml currently renders a delegate per layer for RootData.region.lazLayers (around line 70). Add a per-row toggle bound to model.enabled; dim the row content (and any preview/status) when disabled; show a “Disabled” chip or italic label.

6. Post-clip auto-disable

cavewherelib/src/cwLazClipInteraction.{h,cpp}: at commit() near line 174, capture the contributing layers’ QUuids (already stable per layer via cwLazLayer::id()). On success in the AsyncFuture callback, after rescan(), iterate the captured UUIDs, look up each in cwLazLayerModel, and call layer->setEnabled(false) on whichever pointers are still alive. No signal to QML, no settings branch — the disable is silent and unconditional.

Failure / cancellation branches do nothing — the user’s state is untouched on a no-op clip.

Out of scope & follow-ups

  • Session-level hide — a transient right-click “Hide” gesture with a viewport chip for unhide, never persisted, never synced. Conceptually paired with this plan but deferred to its own document. Will use a new cwHiddenObjects manager keyed by QUuid, ANDed into the same visibility composition: enabled AND (not in session-hidden) AND (keyword filter accepts).
  • Disabling other object types (scraps, trips, traces). The same enabled pattern generalizes; each type adds the property, persists it, and ANDs into its own scene visibility path.
  • Replace original destructive mode for clipping. Not in this plan; clipping always preserves the source file.
  • Multi-select disable from the Data section. Trivial extension once the per-row toggle works.
  • Persistence of keyword filter accept/reject. Still not persisted; out of scope.

Decisions made

  • Property name: enabled. Matches the “comment-out code” analogy directly — users disable a layer the way they comment out a block.
  • Disabling a mid-load layer cancels the in-flight load. cwLazLoader already honors cancellation via QPromise::isCanceled() between point chunks (cwLazLoader.h:53; checks at cwLazLoader.cpp:340 and :359). m_loadRestarter.cancel() propagates through to that promise, so cancellation is a single call — no extra plumbing needed.
  • State entries for missing paths on save. Not worth worrying about either way — the simpler implementation path (write the state of layers currently in the model) is fine. Stale entries that survive are a few bytes per entry; lost entries on file removal are recreated the next time a user disables the layer.
