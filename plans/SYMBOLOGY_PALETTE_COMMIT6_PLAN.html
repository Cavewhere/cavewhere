Commit 6 — Palette selection

The payoff commit of the symbology-palette series. Commits 1–5 built the data model, persistence, the decoration-layout engine, and the live render path — but the seed's only decorated brush (floor-step) is unreachable from the UI, and cwSketch::paletteSnapshot() still returns a fixed built-in seed regardless of project. Commit 6 makes the active palette selectable per sketch (with region and app-wide defaults), recomputes the snapshot from the resolved palette and re-emits on change, and replaces the three hard-coded toolbar radio buttons with a real palette picker and brush picker. After this commit a cartographer can pick the floor-step brush and watch its ticks render on the canvas through normal use. See SYMBOLOGY_PALETTE_PLAN.html (Decision 6 §730, UI integration §1868), the SYMBOLOGY_PALETTE_RENDER_PATH_PLAN.html follow-ups (§303, §460, §506), and SYMBOLOGY_PALETTE_4X_ROADMAP.md.

Bottom line. Three layers, one commit. (1) Model: a first-hit-wins resolver chain (region → settings → shipped default) on cwSketch — no per-sketch palette (§10.E) — making paletteSnapshot() recompute from the resolved cwSymbologyPalette, with the paletteSnapshotChanged() signal the render path has been waiting on (render-path plan §303). (2) Persistence: round-trip CavingRegion.defaultPaletteId through proto + save/load, plus cwSketchSettings.defaultPaletteId in QSettings. (3) UI: expose cwSymbologyPaletteManager to QML via cwRootData and replace SketchToolbar.qml's radio row with a palette picker and a category-grouped brush picker (both backed by dedicated QAbstractListModels), with missing-brush and missing-palette banners. No proto-version bump (the symbology format is unreleased on fall2026). Stays on the GUI thread; no async, no MutateScene collision. CaveWhere follows the OS color scheme — no in-app override (resolved §10.A).

3. Decision 6 — the resolver chain

Active palette is a project-wide (region) choice, with an app-wide default behind it. First hit wins:

1 · per-project
cwCavingRegion::defaultPaletteId
→
2 · app-wide
cwSketchSettings::defaultPaletteId
→
3 · shipped default
defaultPalette()

Each level holds a QUuid that is null/unset by default, meaning "fall through to the next level." Resolution walks the chain, takes the first non-null id that paletteManager.paletteById() actually resolves to a loaded palette, and falls back to defaultPalette() if none do (so a region that references a deleted/missing palette still renders against the shipped default rather than going blank).

No per-sketch palette (resolved §10.E). Cave maps use one consistent symbology standard, so the palette belongs to the project, not the individual sketch: the toolbar picker sets cwCavingRegion::defaultPaletteId, so one pick re-skins every sketch in the project, and a bulk re-palette is a single setter. A per-sketch override would only fight that bulk workflow; it can be added later as a cheap additive proto field if a concrete need (e.g. a one-off inset in a different style) appears. "Zen mode" (sketch with fewer symbols → switch to a detailed pack) is the same region-level switch — or, longer term, a view-level detail toggle — not a per-sketch palette.

Depends on the prep commit (SYMBOLOGY_PALETTE_DEFAULT_QRC_PLAN.html): the chain bottoms on the qrc-embedded defaultPalette(), not the old in-code seedPalette(). Land that first.

The resolver lives on cwSketch because it owns the snapshot the render path reads per frame. cwSketch reaches the region by walking up its parent chain (m_parentTrip → cave → region) and reads the app default from the cwSketchSettings singleton. The cwSymbologyPaletteManager singleton is the lookup oracle for both id levels.

Why the resolver is read-through, not cached-by-copy. The snapshot is recomputed (and paletteSnapshotChanged() emitted) on any input change — see §5 — so we store the resolved cwPaletteSnapshot in m_paletteSnapshot and refresh it, rather than re-walking the chain on every paletteSnapshot() read. paletteSnapshot() stays a cheap member return, matching what the render path already calls per frame.

5. When the snapshot re-resolves

cwSketch::resolveSnapshot() runs (and conditionally emits paletteSnapshotChanged()) on any of:

  • setParentTrip() — the sketch was attached to a region whose default differs (re-wires the region subscription too).
  • cwCavingRegion::defaultPaletteIdChanged — the project's palette changed; subscribe via the parent chain.
  • cwSketchSettings::defaultPaletteIdChanged — only consulted when the region default is unset.
  • cwSymbologyPaletteManager::palettesChanged reload (directory rescan) — a previously-missing id may now resolve, or a loaded palette's brushes changed. Snapshot-content dedup (cwPaletteSnapshot::operator==) makes both cases re-skin correctly while staying quiet on a no-op rescan.

The render path reacts for free: cwSketchCanvas's new connection pushes the fresh snapshot into cwDecoratedStrokePathSource::setSnapshot(), which rebuilds every finished batch and re-skins the canvas — exactly the "switching the active palette re-skins every stroke" behavior in §1880.

6. UI — palette picker & brush picker

Replace the radio row in SketchToolbar.qml with two pickers (Decision 6 / §1870 layout):

[Palette: UIS Standard ▾]   [Brush: Wall ▾]        (+ Erase tool, unchanged)

Palette picker

  • A QC.ComboBox populated from RootData.paletteManager.palettes() (display palette.name, value palette.id). A C++ QAbstractListModel over the manager may be cleaner than poking the QList<cwSymbologyPalette*> from QML — open question §10.
  • Selecting a palette sets the current project's cwCavingRegion::defaultPaletteId (reached via the sketch's parent chain / RootData). Because the palette is region-level, one pick re-skins every sketch in the project — the resolver + render-path connection do it automatically (§5). This is also the bulk re-palette path: a single setter.
  • Strokes whose brushName doesn't resolve in the new palette trigger the missing-brush banner (below); they fall back to the no-ink missing-brush style from the snapshot resolver (already handled in cwPaletteSnapshot / the layout's findBrush miss path).

Brush picker

  • A picker over the active palette's lineBrushes(), showing displayName grouped by category. Per §1881: categories sort by the lowest zOrder among their members (wall-ish groups float to the top); within a category, brushes sort alphabetically by displayName (locale-aware). Stable across palette reloads regardless of file order.
  • Selecting a brush sets the canvas's current drawing brush name, which is what the next beginStroke() uses. This replaces the toolbar's hard-coded property string brushName — the value now comes from the picker's selected brush name rather than a literal.
  • Repopulates whenever activePaletteId changes (palette switch) or the manager reloads.
  • Keyboard shortcuts preserved: W → wall, F → feature stay as muscle-memory accelerators (§1893), implemented as shortcuts that set the picker's selection if those brush names resolve in the active palette (no-op otherwise).

Missing-resolution banners

Two small inline warnings (using Theme.qml tokens, QC.Label / BodyText), both pointing the user at the palette picker:

  • Missing brush. The current sketch has strokes whose brushName doesn't resolve in the active palette — e.g. after switching to a palette that lacks wall. This is the §1880 "missing-brush banner" surface.
  • Missing palette review fold-in. The region's defaultPaletteId (or the app-wide default) points at a palette the manager doesn't have — e.g. a project saved with a region defaultPaletteId and opened on a machine where that palette isn't installed. The resolver silently falls through to the shipped default (correct behavior — the sketch still renders), but the user is never told their chosen palette is missing and that they're seeing substituted symbology. The banner makes that visible.
    Why this is here: surfaced by the Phase 1 qt-cpp-review (error-handling agent). The cwSymbologyPaletteManager routes every load-time palette problem (duplicate id, forward-incompatible, load failure) to its cwErrorModel, but a dangling reference from a project/QSettings is invisible. This is a UI-level detection, not a resolver change: the picker has both the sketch's activePaletteId and the manager's palettes(), so it can tell whether the id resolves. Especially important for the region-level id, which is persisted and meant to be portable/collaborative.
QML conventions to honor: import QtQuick as QQ; strict typed properties; QC.Label/BodyText for all text (dark-mode safe); all colors and font sizes via Theme.qml tokens; prefer TapHandler in any new components; register any new QML component via qt_add_qml_module. The existing radio MouseArea/legacy bits are replaced wholesale, so the new pickers start clean.

7. Color scheme — follow the OS

No in-app color-scheme override (resolved §10.A). CaveWhere follows the system light/dark appearance: Theme.qml already reads Qt.application.styleHints.colorScheme, and the painter / model (Decision D's onColorSchemeChanged in cwDecoratedStrokePathSource) consult the same QStyleHints source.   • ProjectMetadata.defaultPaletteId persists in the project file as UUID text (null = unset = omitted on write). Sketches carry no palette id.

10. Decisions (resolved in review)

A. No color-scheme override; defaultPaletteId → top-level cwSettings

Dropped colorSchemeOverride. CaveWhere follows the OS light/dark setting (§7) — the user changes appearance via the system and the app respects it. This removes the enum, its QSettings round-trip, and the segmented settings control. cwSettings.defaultPaletteId lives at top level (app-wide, not sketch-canvas-specific).

B. Palette/brush enumeration → dedicated QAbstractListModels

Add two models: a palette list over the manager, and a brush list over the active palette with the §1881 category/zOrder sort baked in. The sort rule lives in C++ where it gets a unit test, and ComboBox/grouped views bind cleanly.

C. Current drawing brush → cwSketchCanvas::currentBrushName (C++)

Promote the toolbar's property string brushName to a currentBrushName property on cwSketchCanvas, so the keyboard shortcuts, beginStroke(), and the picker share one testable source of truth. The brush picker binds to it.

D. Granularity → one commit

Land model + persistence + both pickers as a single commit, matching the "commit 6" payoff framing.

E. Palette scope → region-primary, no per-sketch override

Dropped cwSketch::activePaletteId. A cave map uses one consistent symbology standard, so the palette is a project (region) choice: the chain is region → settings → shipped default and the toolbar picker writes cwCavingRegion::defaultPaletteId. This makes the use case that actually matters — re-palette a whole project, incl. a "zen → detailed" pack swap — a single setter that re-skins every sketch; a per-sketch override would only fight that bulk workflow. "Zen mode" is the same region-level switch (or, later, a view-level detail toggle), not per-sketch state. A per-sketch override can be re-added later as a cheap additive proto field (no version bump) if a concrete need appears. Palette inheritance (palette-extends-palette) is out: brush-name substitution across flat palettes already gives "same names, different rendering"; structural inheritance is an authoring optimization to revisit only if packs start duplicating heavily. Reverses the earlier "all three levels" lean after thinking through zen + bulk re-palette.

12. Phase 3 — execution breakdown (UI: pickers & models)

Phases 1–2 are landed (90df72cb resolver, f32129fe persistence). The remaining work — the undone rows of §4, all of §6, and the UI rows of §9 — is the UI layer. It spans C++ models, a render-path edit, new QML, and three test files, so it is built as four gated sub-phases 3a–3d, bottom-up: each must build clean and pass its targeted tests before the next, and each stops at a phase boundary for review (per the standing pause-between-phases discipline). Nothing is committed unless you ask.

Why this order. 3a and 3b are pure C++ with no QML dependency, so they land and get unit-tested first; the QML pickers (3c) bind to the models and properties they expose; the banners (3d) are the smallest surface and reuse the paletteById probe, so they go last. Each sub-phase is independently green — if review stops the series midway, what landed still compiles and tests.

Two integration unknowns, resolved against the code before 3a (so the model's shape is fixed up front):

i. Brush model's data source → the resolved cwSymbologyPalette*, exposed by cwSketch

cwPaletteSnapshot is lookup-only (findBrush(name) / brushCount(); the brush hash is private) — it cannot enumerate brushes, so the picker can't source from it. The resolver chain is private inside cwSketch::resolveSnapshot() (cwSketch.cpp ~323-367). The only enumeration is cwSymbologyPalette::lineBrushes() → QVector<cwLineBrush>. Resolution: expose the palette cwSketch already resolves — Q_PROPERTY(cwSymbologyPalette* resolvedPalette READ resolvedPalette NOTIFY paletteSnapshotChanged) (cache the pointer resolveSnapshot() already derives). The brush model takes a settable cwSymbologyPalette* palette and copies lineBrushes() by value on set; QML binds model.palette: sketch.resolvedPalette. Keeps one resolver + one change-subscription hub (cwSketch already listens to region / settings / manager and emits a single deduped paletteSnapshotChanged); the model stays decoupled and unit-testable in isolation; the by-value copy avoids the dangling-pointer window when the manager reloads palettes. Rejected: re-running the resolver inside the model (would duplicate cwSketch's three-way subscription bookkeeping — wrong altitude).

ii. Category grouping → flat model, sorted + boundary-flagged in C++; grouped popup in QML

The codebase has no ListView.section / DelegateModel / grouped-ComboBox precedent; every existing combo (MapOptions, CSComboBox, ComboBoxWithInfo) is flat textRole. The one grouping precedent (cwKeywordGroupByKeyModel → KeywordTab.qml) is a heavier nested model built for a two-level filter, not a picker. Resolution: keep cwLineBrushListModel flat, pre-sorted in C++ by the §1881 rule (category by lowest member zOrder, then displayName locale-aware), with roles name / displayName / category / isFirstInCategory (a C++-computed group-boundary flag). The sort and the boundaries become the single C++-testable artifact (what §10.B intends). Since a bare QC.ComboBox can't render non-selectable section headers, the brush picker is a small grouped popup component (BrushPicker.qml) that shows a category header where isFirstInCategory is true; the palette picker stays a flat QC.ComboBox (no grouping needed). Rejected: the nested grouped-model pattern (buries the sort, complicates the unit test, overkill for a picker).
