Commit 5 — Render path

Putting the 4.x decoration engine onto the actual sketch canvas. Until now every placement rule has been verified through cwSketchDecorationLayout's unit tests and a PNG reference panel; nothing the cartographer draws is decorated on screen. Commit 5 wires the layout engine into the live render path so a finished stroke renders as its brush's full symbology — traced lines and stamped glyphs — at the sketch's map scale, in light or dark mode. Part of the symbology-palette series; see SYMBOLOGY_PALETTE_4X_ROADMAP.md, SYMBOLOGY_PALETTE_PLAN.html (§8 "Rendering pipeline", Decision 11, Decision 12), and the 4.x satellite plans it builds on (TRACE_RULE_UNIFICATION_PLAN.html, CEILING_CHANNEL_DESIGN.html, SYMBOLOGY_PALETTE_ORIENTED_POINT_PLAN.html).

Bottom line. Replace the prototype path model — which width-tessellates the raw centerline and knows nothing about glyphs — with a path source that runs cwSketchDecorationLayout per stroke and flattens cwBrushDecorationGeometry into the painter's draw list. The engine, the cache, and the geometry types already exist and are unit-tested; commit 5 is plumbing and a small painter extension, not new symbology compute. It stays synchronous on the GUI thread (the cache thread-safety guard rides with the later async work), defers scene-level collision (MutateScene, iter 2), and ships the first thing a user actually sees from all of 4.x.

The path model (cwSketchPainterPathModel.{h,cpp}) is a commit-2 prototype. It:

  • renders only the raw centerline — a pressure-tessellated ribbon (cwSketchStrokeGeometry::buildPath); no glyphs, no decoration layers;
  • uses a single uniform width kSketchStrokeRenderWidth for every stroke;
  • picks color from a two-bucket wall / non-wall split (m_snapshot.producesScrapOutline(brushName) → wallStrokeColor vs nonWallStrokeColor, set from Theme.qml in SketchItem.qml);
  • self-seeds its own cwSymbologyPaletteSeed::create().snapshot() in its ctor — a second, independent palette copy from the one cwSketch already owns.

Meanwhile cwSketchDecorationLayout / cwGlyphTessellationCache / cwBrushDecorationGeometry are fully built and unit-tested but referenced nowhere outside testcases/. Commit 5 closes that gap: the path source becomes the layout engine's first real consumer.

A note on the master plan's stale names

SYMBOLOGY_PALETTE_PLAN.html (§8) predates the 4.x rule collapse. It calls the new class cwOffsetCurveBatches and frames it as batching "OffsetCurve layers". After 4.5/4.6 there is no "OffsetCurve mode": offset became the Offset stroke stroke transform, and a layer's geometry is decided by its Terminal rule (a Trace line/polygon or a Stamps set). So the class commit 5 actually builds renders both traces and stamps via the layout engine, not just offset curves. This plan supersedes the §8 naming; see Decision B.

Concretely, five moving parts:

  1. A new path source that owns a cwGlyphTessellationCache + a cwSketchDecorationLayout, runs the layout for each finished stroke, and flattens the resulting cwGlyphSubPath ink (trace regions + stamp sub-paths) into draw items.
  2. Single palette source of truth: the model stops self-seeding and takes its snapshot from cwSketch::paletteSnapshot(); the same snapshot feeds the tessellation cache.
  3. Map-scale reactivity: the model re-runs the layout when cwSketch::mapScale() changes (paper-mm → world-m conversion lives inside the layout), so flipping 1:250 ↔ 1:1000 re-renders symbology at the new density (plan §6 "dynamic scale switching").
  4. A small cwSketchPathSource::Path extension so a draw item can carry the style a brush demands: a paper-fixed width, cap/join/dash, fill-vs-stroke, a z value — and a single, already-resolved color (Decision D resolves light/dark at build time, so the low-level item stays lean — Decision E).
  5. A painter extension: sort by z before drawing and honour the per-item style. Grid/line-plot draw items are unaffected (they use the existing defaults).

3. The draw-item contract: extend cwSketchPathSource::Path

Today a Path is {painterPath, strokeColor, strokeWidth, z} and the painter hardcodes RoundCap/RoundJoin, applies strokePenScale to every width, and ignores z. A brush needs more: its own cap/join/dash, a paper-fixed width, and a fill pass for closed glyphs. The cleanest move — the painter and its backends (cwSketchDraw) are small and value-style — is to widen Path minimally and teach the painter the new fields, leaving every existing producer (grid, line-plot, debug) on the defaults.

Kept lean on purpose (Decisions D + E). The public item carries one resolved strokeColor, not a light/dark pair — the source pair lives in the model's internal batch record (§4), and a theme toggle re-resolves each batch's color without a layout rebuild (D). So the extension adds only two small enums and a bool to the existing struct:

struct Path {
    QPainterPath painterPath;
    QColor       strokeColor;                   // single, already light/dark-resolved (Decision D)
    double       strokeWidth = 1.0;             // <= 0 still flags a fill pass

    // Width-unit flag (Decision E). false = screen pixels (grid/line-plot/legacy,
    // keep the existing strokePenScale path). true = world metres: paper-mm baked
    // to world-m in the model, drawn straight under worldToItem with no penScale.
    bool             widthInWorldMetres = false;

    Qt::PenCapStyle  cap  = Qt::RoundCap;
    Qt::PenJoinStyle join = Qt::RoundJoin;

    double           z = 0.0;                   // paint order (Decision 12)
};

Dash dropped (5a). An earlier draft carried Qt::PenStyle dash (mirroring cwDecorationLayer::lineDash). But the live RHI canvas (cwSketchDrawCanvas / QCanvasPainter) has no dash support, so on-screen it would always render solid — and no producer in 5a–5c emits a non-solid dash. Rather than plumb a field only the QPainter export backend could honour, dash is omitted; the brush's lineDash stays in the data model and a dash arg can be re-added to setStrokePen when an export backend actually renders dashed strokes.

Fill passes (closed/filled glyphs, strokeWidth <= 0) reuse strokeColor as the fill color — the convention the painter uses today. A filled glyph emits two items: a fill pass and a pen pass, mirroring cwGlyphSubPath's Polygon (pen + fill).

Memory: batch sub-paths by resolved style (Decision E)

The struct above is small, but the real memory lever is how many exist. A naïve "one Path per sub-path" emits, say, a floor-step stroke's 50 ticks as 50 items with identical style — and at 100k strokes that multiplies badly. So the model keeps the prototype's batching: merge every sub-path that shares a resolved (strokeColor, width, widthInWorldMetres, cap, join, dash, z) tuple into a single QPainterPath (one moveTo per disjoint piece). The style fields then live once per batch (a handful of items), not once per stamp, and the painter issues far fewer draw calls. Fill and pen passes batch separately (different draw op); equal-z batches keep emission order. The only retained geometry is the batched ink itself — no per-stroke cwBrushDecorationGeometry intermediate is kept (it is re-derived from the layout on change), which is what keeps the doubled glyph colors (cwGlyphSubPath carries 4 QColors each) out of resident memory.

Why widen Path rather than add a parallel struct

The painter already iterates one Path list through one drawPaths() loop across two backends. A second item type would fork that loop, the snapshot copy in synchronize(), and both cwSketchDraw implementations. Widening keeps one code path; the new fields are inert for existing producers. This is the "right altitude" choice — generalise the one draw item, don't special-case a symbology lane beside it.

4. The new path source

Rename and rewrite cwSketchPainterPathModel → cwDecoratedStrokePathSource (Decision B). It keeps the existing reactive shape — it is a QObject + cwSketchPathSource, connects to cwSketch's stroke signals, and emits pathsChanged() — but its per-stroke build changes from "tessellate the centerline" to "run the layout and flatten it".

class cwDecoratedStrokePathSource : public QObject, public cwSketchPathSource {
    // owns the engine
    cwGlyphTessellationCache m_cache;
    cwSketchDecorationLayout m_layout{&m_cache};   // const transform, reads m_cache

    cwPaletteSnapshot m_snapshot;     // from cwSketch::paletteSnapshot(), NOT self-seeded
    double m_mapScaleRatio = 1.0/250; // from cwSketch::mapScale()->scale()
    int    m_activeStrokeIndex = -1;

    // Internal batch: the public Path the painter draws, plus the source color
    // pair so a theme toggle re-resolves without re-running layout (Decision D).
    struct Batch { Path path; QColor srcLight, srcDark; };

    QList<Batch> m_finished;          // finished strokes, batched by resolved style
    Path         m_activePreview;     // plain neutral centerline (Decision C)
public:
    void setSnapshot(const cwPaletteSnapshot &snap);   // -> m_snapshot + m_cache.setSnapshot(); schedule rebuild
    void setMapScaleRatio(double ratio);               // schedule rebuild (geometry changes)
    QList<Path> paths() const override;                // active preview + finished, z-sorted

private:
    void scheduleRebuild();           // the single seam (Decision A) — sync today, async-swappable
    void reResolveColors();          // theme toggle: flip each batch.path.strokeColor, no layout
};

Every trigger routes through scheduleRebuild() — the seam Decision A calls for, so the async follow-up swaps only this one method's body. A pure color-scheme change takes the cheaper reResolveColors() path instead (no layout).

Per finished stroke the build is:

  1. resolve the brush: m_snapshot.findBrush(stroke.brushName) — missing ⇒ no ink;
  2. extract the world-metre centerline from stroke.points (drop pressure, see Decision F);
  3. const auto geom = m_layout.layout(brush, world, m_mapScaleRatio);
  4. for each geom.layers[i] (declaration order == paint order within the brush), fold each cwGlyphSubPath (a Trace region, or a stamp's transformed sub-paths) into the matching style batch (§3): Stroke ⇒ a pen batch; Polygon ⇒ a fill batch + a pen batch; penWidthMm → world-m via cwGlyphTessellationCache::paperMmToWorldM with widthInWorldMetres = true; stash the source penColor{Light,Dark} / fillColor{Light,Dark} on the batch and set the resolved path.strokeColor for the current scheme (Decision D); set z = brush.zOrder (per-layer order is the stable-sort tiebreak — Decision 12, "no per-glyph z").

Glyph stamps already arrive transformed onto the stroke (cwResolvedStamp::subPaths are world-metre, placed), and trace regions are world-metre polylines/polygons — so the only work is the color pick and appending into the right batch's QPainterPath; no geometry math in the model.

Rebuild triggers (all via scheduleRebuild()). A new finished stroke folds into batches incrementally (cheap — one stroke's layout). A color-scheme change takes the reResolveColors() fast path — flip each batch's resolved color, no layout (Decision D). setMapScaleRatio (geometry truly changes) and setSnapshot rebuild all finished batches from the layout.

Known cost at scale (accepted for commit 5). Two cases re-run the full placement pipeline for every finished stroke synchronously: a scale switch (unavoidable — paper-mm→world-m means geometry changes) and a stroke erase (inherited from the prototype, which rebuilds all finished batches on remove because batches don't track their contributing strokes). At large sketch sizes both can stall the UI. Commit 5 accepts the prototype's behaviour rather than gold-plating; the fixes — async layout (Decision A) for the scale case, incremental batch-contributor tracking for erase — are follow-ups, taken when a measurement shows they're needed.

6. Painter changes (cwSketchPainter::paint / drawPaths)

Color is already resolved in the model (Decision D), so the painter does not touch the color scheme — it only gains z ordering and per-item style:

  • z sort. Sort the Path list by z (stable, so equal-z keeps emission order = per-layer order) before drawing. Applies to the strokes layer only; grid/line-plot keep their current fixed ordering.
  • Per-item style. Use the item's cap/join instead of the hardcoded round caps; honour widthInWorldMetres — pixel widths keep width × strokePenScale, world widths draw as-is under worldToItem (the paper-mm→world-m bake already happened in the model, so no per-element scale juggling — exactly the contract in plan §7). (Dash dropped — see §3.)
Both cwSketchDraw backends already accept cap/join in setStrokePen, so per-item cap/join needs no interface change. Dash is not added: cwSketchDrawCanvas (RHI, via QCanvasPainter) cannot dash and would render solid regardless, so a setStrokePen dash arg is deferred until an export backend renders dashed strokes (see §3).

7. Decisions for review

A. Threading: sync now, async as a follow-up — but design the seam now decided

Commit 5 keeps the layout synchronous on the GUI thread — exactly where the prototype model already builds paths today. The cache is only ever touched from that one thread, so it is safe as-is, and the thread-safety work rides with the async follow-up where it is actually needed and actually testable.

Why sync-first is easier to get correct, not just smaller:

  • It separates two bug classes. The synchronous pipeline is the correctness reference — deterministic tests, an eyeballable PNG, a proof that layout → flatten → batch → paint is right. Add concurrency afterward and any discrepancy is isolated to the threading layer, instead of "is it the layout or the threads?"
  • The sync version is not throwaway. Async only changes where layout() runs. The reactive model, the flatten/batch step, paths(), and the painter all stay. The data flow is already async-ready by design — cwPaletteSnapshot is passed by value with no lifetime entanglement, layout() is a const transform, returns are by value — so cwConcurrent::run is additive, not a rewrite.
  • We don't yet know we need it. Drawing new strokes is already incremental and cheap (only that one stroke's layout runs). The heavy passes are erase / scale-switch (§4), unmeasured at real sketch sizes. Async carries real cost — the cache-safety mechanism, cancellation on stroke change, partial-result rendering, test complexity — best paid when a measurement demands it.

The one thing to do now (the seam): funnel every rebuild trigger (setSnapshot, setMapScaleRatio, stroke change, theme change) through a single "schedule rebuild" entry point, and make paths() tolerate a partially-built result. Then the async follow-up is a purely internal swap inside the model (dispatch the rebuild to a worker, update batches + emit pathsChanged() on completion) with no change to the cwSketchPathSource contract.

How to guard the cache later is open — not QReadWriteLock by default. The header note says "QReadWriteLock, it's read-mostly", but that's a shaky fit: tessellate() is a lazy-insert cache (a miss mutates), and QReadWriteLock has no atomic read→write upgrade, so it forces a release-reacquire double-check dance; its critical sections are nanosecond-scale QHash ops where a plain mutex is usually faster; and the expensive glyph tessellation must run outside any lock regardless. Better candidates, all decided with the async work: per-worker cache (zero locking — the CLAUDE.md-blessed "give each worker its own cache"; a palette's glyph working set is small, so per-worker re-tessellation is cheap); warm-then-freeze (pre-populate single-threaded — the snapshot is immutable and the map scale is fixed per pass — then the parallel section is pure reads against an immutable cache, no lock at all); or a plain QMutex with compute-outside-lock + double-check if one shared cache is wanted. TBD with the async follow-up.

B. Class name: cwDecoratedStrokePathSource decided

The master plan's cwOffsetCurveBatches lies post-collapse (no OffsetCurve mode; it renders stamps too). cwDecoratedStrokePathSource names what it is and emphasises the cwSketchPathSource interface it implements.

C. Active (in-progress) stroke: plain centerline preview decided

Running the full layout on every pen-move would re-tessellate glyphs and re-place stamps mid-stroke — costly and visually noisy (ticks popping along a growing line). The active stroke renders as a plain centerline preview: the prototype's existing cheap path — cwSketchStrokeGeometry::buildPath at the current kSketchStrokeRenderWidth (a screen-pixel width, widthInWorldMetres = false), in one neutral Theme token. The full brush decoration runs only when the stroke is committed and becomes finished. (So kSketchStrokeRenderWidth survives as the preview width even though the decorated finished render no longer uses it.)

D. Light/dark: carry the pair per batch, re-resolve on toggle (no re-layout) decided

The public draw item (cwSketchPathSource::Path) stays lean — one resolved strokeColor, which is all the painter ever reads (it stays color-scheme-agnostic). But the model keeps each batch's source (light, dark) pair in its internal batch record. On QStyleHints::colorSchemeChanged the model walks its handful of batches and flips each strokeColor = dark ? srcDark : srcLight — a trivial loop, no re-layout, no geometry rebuild — then emits pathsChanged(). A theme toggle is instant even on a 100k-stroke map.

Why this, over the two alternatives (full comparison was reviewed; memory is a tie because batching collapses item count, so it is not the deciding axis):

  • vs. "resolve at build, re-layout on toggle" (the earlier draft of D): that re-runs the full placement pipeline for every finished stroke on a toggle — a multi-second UI freeze at scale, made worse by the synchronous Decision A. It paid that to save ~16 bytes per batch — savings batching already made irrelevant. Wrong trade once E is in the design.
  • vs. "painter picks at draw time": that keeps the model simplest but makes the painter query the color scheme — and the painter runs on the render thread, so reading QGuiApplication::styleHints()->colorScheme() there is a cross-thread QObject access; doing it correctly means capturing the scheme in synchronize() and threading it through PaintContext. More render-path churn, and it couples the painter to the global scheme. The chosen option keeps the scheme query on the GUI thread (where the model lives) and the painter a pure geometry→draw transform.

Cost of the chosen option: a mild denormalization — each internal batch stores its resolved color next to the source pair, kept in sync by the one re-resolve loop. Contained, and the public interface stays minimal (the pair never leaks onto Path, so grid/line-plot producers are untouched). No per-stroke cwBrushDecorationGeometry intermediate is retained, so the four-QColor per-sub-path geometry never stays resident.

E. Pen-width unit + lean low-level item decided

Symbology widths are paper-mm baked to world-m and ride worldToItem directly (plan §7: "does not use strokePenScale"); grid/line-plot widths are screen-pixels scaled by strokePenScale. A one-byte widthInWorldMetres flag lets both coexist in one draw list without the model needing the renderer's DPI/scale to pre-convert (pre-baking to pixels would need strokePenScale = DPI × mapScaleRatio at GUI-thread build time, which the model doesn't cleanly have).
Minimise the low-level item (your steer): dash is Qt::PenStyle (a 4-byte enum, the layer's own lineDash type), not a heap-allocating QVector<double> custom pattern; color is the single resolved QColor from D; cap/join are small enums. And the model batches sub-paths by resolved style (§3) so these fields live once per batch, not once per stamp — the painter holds a handful of items, not one per glyph. That batching, plus not retaining the per-stroke intermediate, is what actually bounds memory at scale.

F. Trace-line pressure: uniform lineWidthMm, taper deferred decided

The prototype renders a pressure-tapered ribbon; the layout signature (layout(brush, QVector<QPointF>, mapScale)) drops pressure, and Decision 2 already removed cwPenStroke::width in favour of brush-defined width. So a finished stroke's traced line renders at the brush's uniform lineWidthMm. Accepted regression: pen pressure isn't standard across the input devices tested, so dropping the taper is fine; the layer's lineAcceptsPressure flag keeps the door open, but threading pressure through cwStrokePath / the layout signature is a deliberate follow-up, not commit 5.

9. Seeing it in the app

For how the work is split into commits, see §12 Commit sequence (5a groundwork → 5b rename → 5c wire-in). This section is only about what's visible on the canvas afterward.

Is a test palette wired into SketchItem.qml in scope? no

Important to be clear-eyed here, because it governs what you actually see after commit 5. Two facts about the current app:

  • The shipped seed is post-collapse-correct: wall / feature / scrap-outline carry an explicit {Trace} terminal, and floor-step already carries a full stamp stack (Uniform spacing → Align to tangent → Rigid stamp). So no blank-stroke regression — existing strokes still render — and the seed does contain a stamped brush.
  • But the sketch toolbar (SketchToolbar.qml) only exposes wall, feature, and scrap-outline — all Trace-only lines. floor-step exists in the seed but is unreachable from the UI; brush selection is commit 6 work.

So wiring the render path, by itself, changes the three toolbar brushes from the wall/non-wall two-color split to per-brush styled trace lines — and shows no stamps through the normal UI, not because the engine can't, but because the stamped brush isn't selectable yet. The richer 4.x demos (oriented stream arrows, ceiling channel, filled glyphs) likewise live only in the tests' demoPalette().

Three ways to bridge that, and the recommendation:

  • (a) Verify headless only — the render check (below) drives demoPalette() through the real painter and proves stamps/traces/fills all render. In-app look is ~unchanged. Honest, but anticlimactic.
  • (b) Inject a test palette into SketchItem.qml / the canvas — throwaway scaffolding commit 6 immediately rips out, and it risks shipping a demo palette. Keep out of scope.
  • (c) Add one or two real decoration brushes to the seed — gives a visible in-app result, but what those symbols are and look like is a cartography/product decision that deserves its own change, not a rider on the render-mechanism commit.
Decided: option (a). No seed change and no test palette in SketchItem.qml — both out of commit 5's scope. Commit 5 ships the mechanism and reads the seed snapshot (the single-source-of-truth wiring); the committed proof is the headless render check (driving demoPalette() through the real painter). Visible in-app decoration waits for commit 6, which brings palette selection (activePaletteId + a brush picker). To eyeball stamps on the canvas while developing commit 5, the cheapest hack is a temporary floor-step toolbar button (the brush is already in the seed) — or point the snapshot at demoPalette() for the richer demos; either as an uncommitted local hack, reverted before committing.

10. Tests

  • testcases/test_cwDecoratedStrokePathSource.cpp (new file, per the one-feature-one-file convention; tag [cwDecoratedStrokePathSource]):
    • a wall stroke (Trace terminal) ⇒ one pen Path, widthInWorldMetres, width == lineWidthMm→world, color resolved for the active scheme, polyline == centerline;
    • a floor-step stroke ⇒ the traced edge plus N stamp items at the expected spacing, all flattened (this is the headline "stamps now render" assertion);
    • a filled-glyph brush ⇒ a fill item + a pen item per Polygon sub-path;
    • scrap-outline (no decoration ink) ⇒ no finished items;
    • brush missing from the snapshot ⇒ no items, no crash;
    • theme toggle re-resolves, doesn't re-lay-out (Decision D): simulate a colorScheme change ⇒ each batch's strokeColor flips to its dark/light source, while the painterPath geometry is bit-for-bit unchanged (proves the fast path, not a rebuild);
    • setMapScaleRatio change re-runs layout (stamp spacing in world-m changes), and setSnapshot swaps the resolved brush;
    • z ordering: items emitted sorted by (zOrder, layerIndex).
    Palette source: the seed (cwSymbologyPaletteSeed::create().snapshot()) covers wall (trace), floor-step (stamps), and scrap-outline (no ink). The filled-glyph case has no seed brush, so either add a synthetic filled brush in the test or extract demoPalette() from test_cwSketchDecorationLayout.cpp into testlib/ for reuse. Pure compute, no temp files — safe under concurrent test processes.
  • Painter render check — extend the existing offscreen-QPainter approach (cwSketchDrawQPainter, --platform offscreen) with a panel that paints a decorated stroke in both color schemes and asserts non-blank + that light vs dark differ. Must use the same Y-flipped worldToItem the live canvas uses (determinant() < 0) — not a Y-down identity — so glyph chirality (Align-to-tangent's +Y = left normal, stamp orientation) is validated against the real render, not a mirror of it. The 4.x reference PNGs don't necessarily exercise that flip. Follows the "visualize headless output" convention; PNG path overridable by env var.
  • QML — a tst_ smoke test that the canvas paints a decorated stroke without error (offscreen), if a lightweight hook exists; otherwise rely on the C++ render check.
