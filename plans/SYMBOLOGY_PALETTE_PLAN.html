Symbology Palette: Custom Lines, Areas, and Points

Context

CaveWhere's sketch system today carries a prototyping cwPenStroke::Kind enum (Wall, Feature, ScrapOutline, Eraser) with per-kind defaults and special-case behaviour baked into C++. Kind was scaffolding to get sketching working end-to-end; this plan replaces it.

The central design principle is declarative symbology: every line, area, and point is an instance of a brush whose look and behaviour are described in data. Anything the renderer or the rest of the application currently special-cases on Kind becomes a flag or a capability declared on the brush. Whether a stroke contributes to a scrap outline, whether it accepts pressure variation, what its decorations are, what scale it shows at — all data, all editable, all shippable in a palette.

Brushes can be edited, shared, and synced from a public git repository.

Every line is a centerline plus decorations

A line on a cave map is rarely a single stroke — it is a composite of a centerline and a series of decoration layers placed along the centerline at authored spacing. The centerline carries the geometry; decorations carry the symbology.

What "custom line" means in cave cartography

Every cave-map symbol below is built from the same vocabulary: a centerline + zero or more decoration layers. The brushes differ only in their decoration layers' parameters.

Symbol non-overlap is the central constraint

Centerlines may intersect each other (a floor step crossing a pit edge is legitimate). Decorations may not overlap each other or step on unrelated centerlines/decorations. Resolutions, in preference order:

  1. Shift along arclength up to ±½ spacing (snap to a free slot).
  2. Shorten the decoration (e.g. 10 px → 4 px).
  3. Halo with a background-coloured buffer so a near-miss reads cleanly.
  4. Hide the offending instance entirely.

Each decoration declares its own policy. A floor-step hatch can be shortened or hidden; a ceiling-step T must stay full-size but allows a halo.

Scale is part of the symbol

A symbol that reads clearly at 1:250 may be illegibly dense at 1:1000 and a black smear at 1:5000. Every brush and every decoration layer inside a brush carries a scale visibility window. At export the active cwSketch::mapScale resolves visibility. On screen, the cartographer flips cwSketch::mapScale directly from the toolbar and everything downstream re-renders (see Decision 7's dynamic-scale-switching discussion). This is the "vector mipmap" — different decoration layers light up at different scales, no raster step.

Scope

In scope (this plan)

  • A data-driven brush definition (cwLineBrush) covering centerline style + ordered decoration layers + collision policy + scale windows.
  • A palette model (cwSymbologyPalette) — a versioned collection of brushes with its own proto schema, identity, and optional git remote.
  • An on-disk palette directory plus a built-in seed palette mirroring today's symbols so nothing regresses.
  • Replacement of cwPenStroke::Kind with a brushName reference; capabilities like "this is a scrap outline" become brush flags.
  • Rendering: cwOffsetCurveBatches (the renamed cwSketchPainterPathModel) batches OffsetCurve layers by (brushName, offset, width, color, dash, cap, join) and emits per-decoration draw work; cwSketchPainter orchestrates a placement pass that resolves collisions.
  • A minimal in-app palette picker and per-stroke brush picker.

Explicit non-goals

  • Areas and points are deferred to a later iteration; the data model is shaped so they can be added later as cwAreaBrush / cwPointBrush peer types sharing the decoration-layer vocabulary.
  • In-app brush editor UI is deferred — iteration 1 ships a palette picker; authoring new brushes happens by editing palette files directly.
  • Therion .th2 import/export is out of scope. The brush vocabulary is chosen with .th2's linetype/pointtype taxonomy in mind so a mapping can be added later without restructuring.
  • Cross-sketch / region-wide collision is out of scope. The placement pass operates within a single cwSketch.

Key design decisions

1. Brushes are data, not classes

A brush is a value type serialised to proto/JSON. There is no per-brush C++ subclass. Adding a new brush means appending a row to a palette file — no rebuild required. 

struct cwLineBrush {
    // Identity — see Decision 2 for the name vs displayName split.
    QString name;                 // machine identifier — kebab-case, unique within palette,
                                  //   ASCII only, immutable once published. Renaming `name`
                                  //   is a breaking change for every sketch that references it.
    QString displayName;          // human-readable, localizable — shown in pickers and logs.
    QString category;             // display-only grouping in pickers — "Walls", "Floor features", …

    // Paint order. Higher z paints later (on top). Wall ships at z=100,
    // feature at z=10, etc. See Decision 12 for the full sort key.
    int zOrder = 0;

    // Topology flag — closed loops of this brush feed the chain-based
    // scrap-outline detector and become cwScrap outlines. Independent
    // of rendering: a `scrap-outline` brush has `scrapOutline = true`
    // and zero decoration layers (the user-drawn polyline is the
    // topological boundary but no ink hits the page).
    bool scrapOutline = false;

    // Decoration layers, drawn in order. The brush has no built-in
    // "centerline" — the visible line on a normal wall is just an
    // OffsetCurve layer with offset = 0 (Decision 10). N parallel
    // visible lines = N OffsetCurve layers at the offsets the
    // cartographer wants. Floor step = one OffsetCurve(offset=0) layer
    // + one RigidStamp tick layer. Empty = no ink (scrap-outline).
    QVector<cwDecorationLayer> decorations;

    // Scale window (iteration 3). 0 = unbounded.
    double minPaperScale = 0.0;   // e.g. 1.0/1000 (smaller = zoomed out)
    double maxPaperScale = 0.0;
};

Centerline is not a special concept on the brush. An earlier draft of this plan carried six fields on cwLineBrush for "the centerline" (centerlineColorLight/Dark, centerlineWidthMm, cap, join, centerlineEnabled). They duplicated what an OffsetCurve decoration layer with offset = 0 already does. Removed. Consequences:

  • A brush is a list of rendering layers, full stop. No centerline-vs-decoration split.
  • Multiple parallel "primary" lines fall out for free — a triple-line symbol is three OffsetCurve layers at {-2, 0, +2} mm, none privileged.
  • The centerlineEnabled = false case (scrap-outline) collapses to "zero decoration layers" — the brush draws no ink, but the cartographer's drawn polyline still feeds scrap detection because scrapOutline = true is a topology flag operating on cwPenStroke::points (Decision 8).
  • cap / join / acceptsPressure move to cwDecorationLayer — per-layer, so a brush's shadow lines can square-cap while the main line round-caps if a cartographer wants. See Decision 10.

No per-stroke overrides. A stroke's look is defined entirely by the brush its brushName resolves to. To vary a stroke, the cartographer clones the brush in the palette UI (wall → wall-emphasized) and edits the clone. Every variation is named, lives in the palette, and is bulk-editable. This is the considered alternative to a per-stroke QHash<QString, QVariant> override bag, which would produce a proliferation of unnamed, untyped, non-bulk-editable variants. If real iter 1 usage shows palette bloat from clone-and-tweak, brush inheritance (a parentBrushName field with sparse field overrides) lands as a non-breaking schema addition in a later iteration.

2. brushName replaces Kind on cwPenStroke

struct cwPenStroke {
    QString brushName;           // resolves via active palette; defines look entirely
    QVector<cwPenPoint> points;  // points stay in world metres (no change)
    QUuid id;
};

Why name, not UUID. The whole point of the palette system is that swapping palettes re-skins the sketch: UIS-Standard's wall → MyPersonal's wall substitutes cleanly because both palettes call the brush "wall". A UUID can't do this — the two palettes' wall brushes would have different UUIDs, so swapping palettes would orphan every stroke. Identifying by stable string name is the only way cross-palette substitution works, and it matches Therion's linetype / pointtype taxonomy. See Decision 1 for the name vs displayName split: name is the machine identifier (kebab-case, immutable, unique within palette); displayName is the localizable label shown in UI.

Flat namespace. Brush names live in a single flat namespace per palette — no "floors/sand" path syntax. Category is a display-only field for grouping in pickers. Therion is flat; the seed palette is small; community palettes converging on the same names ("wall", "floor-step", "sand-floor") is the point of cross-palette re-skinning.

Kind was prototyping; the existing sketches are also prototyping data. There is no backward-compatible load path — the field is removed from the proto schema in the same commit that adds brushName, and any pre-existing .cw sketches authored before that commit simply fail to load (or load with empty stroke lists). This trades a one-shot migration loader for a clean schema and zero deferred cleanup; the benefit dwarfs the cost since none of the prototype sketches are load-bearing.

No per-stroke overrides — clone the brush instead. The prototype's cwPenStroke::width and cwPenStroke::color fields go away. A stroke's look is defined entirely by its brushName; if a cartographer wants a thicker wall in one passage, they clone the wall brush in the palette UI to wall-emphasized, edit the clone, and reassign the affected strokes' brushName. Every variation is a first-class palette entity — named, discoverable, bulk-editable. Three reasons this is preferable to a per-stroke override bag (QHash<QString, QVariant> or std::optional<T> per field):

  1. Bulk editing is trivial. "Make all emphasized walls darker" is one edit on wall-emphasized. Override-bag variants don't have a name, can't be searched, and silently diverge from any base.
  2. Collision rules stay tractable. The MutateScene stage (Decision 11) reasons about glyph buffer sizes and centerline widths from brush data. Per-stroke overrides would force the placement pipeline to re-read every stroke's local override before answering "does this glyph fit?", which the brush-only contract sidesteps entirely.
  3. Palettes stay portable. Sharing a sketch via git sync (Decision 15) sends the brushes the sketch references. With overrides, two sketches using the same brush wouldn't necessarily render the same way.

If iter 1 usage shows clone-and-tweak producing palette bloat, brush inheritance lands later as a non-breaking schema addition (parentBrushName + per-field "overridden" flags) — but it's explicitly deferred so we ship the simpler design first.

Eraser is not a brush. Eraser becomes a tool mode (sketch state machine), not a stroke type. The prototype's Kind::Eraser row falls away with the rest of the enum.

Capabilities, not subclasses. Anything that read stroke.kind == Wall becomes brush(stroke).scrapOutline. Anything that read stroke.kind == Feature becomes default behaviour (!scrapOutline). When a new capability is needed, a new boolean (or enum field, if mutually exclusive states ever arise) is added to cwLineBrush and to the palette schema. The whole point: no central switch on a kind taxonomy.

3. Palette is a versioned bag of brushes with its own schema

cwSymbologyPalette is a free-standing entity with its own proto file (cavewhere_symbology.proto), not embedded in cavewhere.proto. Two reasons:

  1. Independent versioning. A new brush field shouldn't bump the .cw project file version. Palette versions evolve on their own clock.
  2. Distribution. A palette is a repository. Bundling its schema with the project schema would force third-party consumers to pull in all of cavewhere.proto to read a palette.
// cavewhere_symbology.proto
syntax = "proto3";
package CavewhereSymbologyProto;
import "QtProto.proto";

message Palette {
    string id          = 1;       // UUID — see Decision 6 for why palettes keep UUIDs
    string name        = 2;       // human-readable display name
    string description = 3;
    string author      = 4;
    string version     = 5;       // semver-ish, author's choice
    repeated LineBrush lineBrushes = 6;
    repeated Glyph     glyphs      = 7;     // see Decision 9
    // repeated AreaBrush  areaBrushes  = 8;  // reserved (iter 5)
    // repeated PointBrush pointBrushes = 9;  // reserved (iter 5)
}

message LineBrush {
    // Identity — Decision 2.
    string name        = 1;                // machine identifier, kebab-case, unique within palette
    string displayName = 2;                // localizable, mutable
    string category    = 3;                // display-only grouping in pickers
    bool   scrapOutline = 4;               // topology flag — feeds chain-based scrap detector
    int32  zOrder       = 5;               // paint order (Decision 12)
    repeated DecorationLayer decorations = 6;  // empty = no ink (scrap-outline case)
    double minPaperScale = 7;
    double maxPaperScale = 8;
    // No centerline fields. The visible line on a normal wall is an
    // OffsetCurve decoration layer with offset = 0. See Decision 1.
}

message DecorationLayer {
    // A layer is a rule stack (Decision 11b — no mode). The Terminal rule
    // produces the layer's geometry; its OutputKind names what (stamps vs.
    // a traced polyline).
    string glyphName = 1;                  // ref into Palette.glyphs by name (Decision 9);
                                           //   empty for a polyline-tracing layer
    reserved 2; reserved "mode";           // was int32 mode (Decision 11b removed modes)
    // Ordered rule pipeline: Generate / MutatePerLayer / Terminal / MutateScene
    // rules, sorted by stage at execution time. Empty = no rules = no ink
    // (there is no per-mode default stack; the seed/editor write explicit
    // stacks). MutateScene rules land in iter 2.
    repeated PlacementRule rules = 3;
    double bufferMm           = 4;         // collision keep-clear, default 1.0 mm.
                                           //   Serialized from iter 1 for forward-compat
                                           //   but DORMANT until iter 2 (only MutateScene
                                           //   rules read it). The project-wide default
                                           //   (cwSketchSettings.defaultBufferMm) also
                                           //   lands in iter 2.
    int32  collisionPriority  = 5;         // see Decision 11 (MutateScene rules, iter 2)
    double minPaperScale      = 6;
    double maxPaperScale      = 7;

    // OffsetCurve fields. Iter 1 ships these with offsetCurveOffsetMm
    // = 0 only (the centerline-replacement path); iter 2 generalizes
    // to non-zero offsets (parallel filled lines). Colors are hex
    // strings ("#RRGGBB" / "#RRGGBBAA") matching cavewhere.proto's
    // existing convention. Ignored when mode != OffsetCurve.
    string offsetCurveColorLight = 8;
    string offsetCurveColorDark  = 9;
    double offsetCurveWidthMm    = 10;
    double offsetCurveOffsetMm   = 11;     // sign = side; 0 = on the centerline
    int32  offsetCurveDash       = 12;     // Qt::PenStyle
    int32  offsetCurveCap        = 13;     // Qt::PenCapStyle  — per-layer (Decision 1)
    int32  offsetCurveJoin       = 14;     // Qt::PenJoinStyle — per-layer (Decision 1)
    bool   offsetCurveAcceptsPressure = 15; // width modulates with stylus pressure

    // NOTE: no halo fields. See Decision 10 — halo is intentionally
    // dropped from iter 1+2 schema; if a real use case shows up later,
    // halo lands as additive fields without breaking existing palettes.
}

message PlacementRule {
    // Rule kind keyed by `name` ("uniform-spacing", "skip-sharp-bends",
    // …). The rule's own params live in `params` so the registry stays
    // forward-compatible: a palette referencing an unknown rule name
    // round-trips intact and is ignored at render time with a one-shot
    // warning. Renaming a registered rule is a breaking change for
    // existing palette files — treat rule names as part of the
    // palette schema's stable surface.
    string name   = 1;
    bytes  params = 2;                     // proto-encoded per-rule struct
}

message Glyph {
    string name        = 1;                // machine identifier, kebab-case, unique within palette
    string displayName = 2;                // localizable label
    // Strokes drawn in glyph-local paper-mm coordinates. Origin (0,0)
    // is the anchor point; +X is along the centerline tangent at stamp
    // time. Each stroke is a regular PenStroke — same schema, same
    // brushName-based rendering. Glyphs can recursively reference brushes
    // whose decorations reference other glyphs (Decision 9).
    repeated PenStroke strokes = 3;
    QtProto.QRectF authoredBounds = 4;     // paper-mm extent, cached
}

QtProto.QColor is intentionally not used — it doesn't exist in qt.proto today and adding it would force every field to choose between binary QColor and the existing colorHex string convention in cavewhere.proto. The string form ("#RRGGBB" or "#RRGGBBAA") diffs cleanly in JSON, parses via QColor::fromString, and matches what's already on disk.

JSON projection is handled by protobuf's JsonPrintOptions — same codegen, same fidelity, useful for palette authoring in a text editor.

A palette is a directory; its identity is the Palette.id inside, not the directory name (so a clone retains identity regardless of path). .cwpalette holds brushes plus glyph metadata (id, displayName, authoredBounds); each glyph's strokes live in its own glyphs/<glyph-id>.cwglyph file. One file per glyph gives clean git diffs when a single hatch is edited (instead of churning a multi-MB palette.json), and lets the glyph editor save each glyph independently. palette.json and per-glyph .json projections are written alongside the binaries for human-readable diffs.

No palette inside .cw projects. Projects reference brushes by brushName only, resolved through the sketch's active palette. A project that references a brush from an uninstalled palette renders with a fallback brush. In the unified model there is no separate "centerline" to fall back to — the cartographer's polyline still exists, only its brush is missing — so the fallback is a synthetic in-memory brush with a single OffsetCurve(offset=0) layer in a flagged debug colour (no glyph layers), scrapOutline = false (a missing brush must not silently inject or remove scrap topology), and zOrder = 0. findBrush returning nullopt is the trigger; consumers substitute this fallback rather than skipping the stroke. The UI surfaces a warning prompting the user to install the missing palette. Same coupling strategy as fonts in a word processor.

The seed palette's brush names ("wall", "scrap-outline", "feature", "floor-step") and glyph name ("floor-step-tick") are stable ASCII strings hard-coded in cwSymbologyPaletteSeed.cpp — no UUIDs to manage. brushNames embedded in any project file stay stable across rebuilds because they're just strings. The seed palette is read-only in the UI (isWritable = false); users who want to modify it fork it. floor-step earns its place in the seed because it's the one brush that proves the headline feature (glyph stamping) ships demonstrated rather than as an empty capability; everything past it — the full UIS / NSS standard symbol set — ships as a separate installable palette so it can version on its own clock.

6. Active palette is per-sketch, with a project default

Resolution order, first hit wins:

  1. cwSketch::activePaletteId (per-sketch override)
  2. cwCavingRegion::defaultPaletteId (project default)
  3. cwSettings::defaultPaletteId (app-wide)
  4. Built-in cavewhere-seed

Per-sketch granularity matters because a cartographer authoring a 1:250 area map and a 1:1000 overview map of the same region wants different decoration densities. Per-region is the practical day-to-day choice for most cavers; per-app catches the "I always use the UIS palette" preference.

The glyph editor (Decision 9) saves into one writable palette at a time, picked from the same palette list — the seed and any read-only palettes are excluded from the destination dropdown. A brush whose decoration layer references a glyph is constrained to glyphs in the same palette (Decision 9's same-palette rule); the brush editor's glyph picker filters accordingly.

7. Coordinate systems and brush units

CaveWhere has three coordinate systems today, all wired through cwWorldToScreenMatrix and cwSketchViewState:

Decision: every dimensional field on a brush and on a decoration layer is in paper millimetres. Stroke geometry stays in world metres. Conversion to world units happens once per (brush, map scale) pair at decoration-generation time: 

// Inside cwSketchDecorationLayout:
const double mapScale = sketch->mapScale()->scale();    // e.g. 1.0/250
const double mmPerWorldMeter   = mapScale * 1000.0;
const double worldMetersPerMm  = 1.0 / mmPerWorldMeter; // paper-mm → cave-m

const double hatchLengthWorld  = layer.sizeMm    * worldMetersPerMm;
const double hatchSpacingWorld = layer.spacingMm * worldMetersPerMm;
// Decoration polygons are emitted in world metres, ride the same
// worldToItem transform as the centerline, render correctly at all
// zooms and on every backend.

What this buys: a single authored brush renders identically (in paper dimensions) at 1:250 and 1:1000; on screen at any zoom; and on PDF export. The "vector mipmap" of Decision 14 reduces to "did the active paper scale fall inside this layer's [minPaperScale, maxPaperScale] window?" — no other scale arithmetic needed.

Field renames. Everywhere a previous draft of this plan said ...Px, the field becomes ...Mm (paper millimetres). The prototype's cwPenStroke::width override is removed entirely (Decision 2) — width lives on the decoration layer's offsetCurveWidthMm field (flat for now; it migrates into Trace offset polyline's rule params in commit 9, Decision 11b), full stop. The strokePenScale field on cwSketchPainter::PaintContext still exists for the prototype path-batching code, but the new symbology pipeline does not use it — it bakes paper-mm → world-m once and draws via the standard worldToItem transform with no per-element scale juggling.

No legibility floor at low zoom. An earlier draft of this plan introduced a minScreenWidthPx per-brush floor so thin centerlines wouldn't vanish when zoomed out. Dropped — the paper-fixed contract is honest: zooming out is supposed to make the map small, and a 0.3 mm line genuinely is too thin to read at a quarter of its authored scale. Users who want the line back zoom in or edit the brush. A silent "minimum thickness" floor would just lie about what they're looking at.

Dynamic scale switching is a first-class workflow. Cartographers routinely flip the same sketch between scales (e.g. 1:500 ↔ 1:1000) — strokes are cave-fixed but symbology should re-render for the new scale. When cwSketch::mapScale changes:

  1. Stroke geometry doesn't move (it's in world metres).
  2. The paper-mm → world-m conversion in the decoration generators re-runs against the new scale, so hatch spacing, glyph size, halo, and buffer all change in world coordinates.
  3. Decoration-layer scale-visibility windows (Decision 14) re-evaluate — some layers drop out entirely at smaller scales.
  4. The placement pass re-runs against the new world-coordinate glyph geometry, so decorations that no longer fit get shortened or hidden by their per-layer collision policy. At 1:1000 you naturally see less of the same cave: fewer hatches, less detail, some symbols gone — exactly what cartographers want.

The cost of a scale switch is one decoration-regeneration pass plus one placement pass — both already invalidation-aware (Decision 8), both already part of every dirty-region repaint. Switching scales is "as expensive as moving the active stroke around"; no special optimisation needed.

8. Rendering pipeline — split offset-curves from glyphs

The prototype's cwSketchPainterPathModel does one job: batches strokes by (kind, width, color) into QPainterPaths. The decoration + placement work is much heavier and has different invalidation rules (OffsetCurves invalidate per-stroke; glyph stamp placements invalidate by spatial region). Piling both into one class is the kind of thing that turns into a maintenance millstone two iterations later.

Splitting now while the responsibilities are still small:

All four feed cwSketchPainter's PaintContext. cwSketchPainterPathModel (the prototype class) is renamed to cwOffsetCurveBatches in iteration 1 — same batching code as the prototype's centerline batcher, but parametric in offset / width / color / dash so it covers both the offset=0 centerline-replacement case (iter 1) and the general parallel-line case (iter 2). The decoration siblings land alongside it in iter 1.

Brush resolution via cwPaletteSnapshot, not a pointer cache. Decoration generation, the placement pass, and centerline batching all need brushName → cwLineBrush lookups dozens of times per repaint. Earlier drafts of this plan cached a cwLineBrush const* on cwPenStroke; that's a dangling-pointer hazard the moment the palette mutates while a worker is using the stroke (CLAUDE.md's "don't pull pointers out of QList/QVector" rule generalises here). Instead: 

class cwPaletteSnapshot {
public:
    // Value type. Internal containers (QHash / QList / cwLineBrush
    // itself) are implicitly shared — constructing this is a refcount
    // bump, not a deep copy.
    std::optional<cwLineBrush>      findBrush(QStringView name) const;  // nullopt if missing
    std::optional<cwSymbologyGlyph> findGlyph(QStringView name) const;  // nullopt if missing

private:
    QHash<QString, cwLineBrush>       m_brushes;
    QHash<QString, cwSymbologyGlyph>  m_glyphs;
};

cwSketchPainter::paint grabs a snapshot from the active palette at the start of each repaint and threads it through PaintContext to every downstream consumer. The snapshot is cheap (refcount bumps), and findBrush returns a cwLineBrush by value rather than a pointer: cwLineBrush is an implicitly-shared value type (its QString fields and QVector<cwDecorationLayer> are all COW), so the copy is a handful of atomic refcount bumps — no deep copy, no heap allocation. Returning by value costs essentially the same as a pointer but eliminates the lifetime hazard entirely: the caller holds a self-contained brush that stays valid no matter what happens to the snapshot or the live palette afterward. (An earlier draft returned cwLineBrush const* into the snapshot's QHash; that invited auto b = snap.findBrush(x); snap = other; use(b); use-after-free. By-value dissolves the trap.) The nullopt result carries the "brush missing" signal that nullptr used to. This is also the abstraction we want when decoration generation moves to an AsyncFuture job later — the worker gets a snapshot by value with no lifetime entanglement to the live palette.

Iter 1 generates decorations synchronously on the UI thread. Async generation via cwConcurrent::run is a follow-up once real-sketch perf measurements identify a need. Don't pre-optimise the threading model; do design the data flow so it can move later (which the snapshot already does).

No built-in decoration types. The closed Kind enum is gone — there is no TickHatches/TBars/Dots taxonomy in code. Decorations are glyphs (Decision 9) stamped along a centerline by a registry-driven pipeline of placement rules (Decision 11). Adding a new symbol type is a palette change, not a C++ change. Adding a new positioning behaviour is one new cwPlacementRule + one registration line.

Iteration 1 ships cwOffsetCurveBatches (offset=0 only), cwGlyphTessellationCache, cwSketchDecorationLayout, cwPaletteSnapshot, the three glyph stampers + the offset=0 OffsetCurve mode, the Generate/MutatePerLayer/Stamp placement-rule registry, the glyph editor, and the palette/brush pickers. Non-zero OffsetCurve offsets (parallel filled lines that curve with the centerline), MutateScene rules, and cwSceneContext land in iter 2; scale visibility windows in iter 3.

9. Glyphs are user-drawn primitives in the palette iter 1

A glyph is a small vector drawing, authored by the cartographer in a glyph editor and stored in the palette. Glyphs are the only decoration primitive. There is no built-in TickHatches/TBars/Dots/etc. taxonomy — the cartographer draws those, registers them in the palette, and the brush references them by id. 

struct cwSymbologyGlyph {
    QString name;                  // machine identifier — kebab-case, unique within palette
    QString displayName;           // localizable label shown in pickers and logs

    // Glyph drawn in glyph-local paper-mm coordinates.
    // Origin (0,0) = anchor point on the centerline at stamp time.
    // +X axis = along the local centerline tangent at the stamp point.
    // Each stroke is a regular cwPenStroke — same brushName-based
    // resolution as any sketch stroke, which is what enables the
    // recursive composition described below.
    QVector<cwPenStroke> strokes;
    QRectF authoredBounds;         // paper-mm extent, cached
};

The glyph editor. A small mode of cwSketchCanvas with editor chrome: a fixed paper-mm authoring canvas (default 10 × 10 mm, zoomable), an origin marker (where the centerline will attach), and a tangent indicator (the +X direction). The cartographer draws strokes using the active palette's brushes — the glyph is literally a tiny sketch. Save into the palette as a new .cwglyph file. Iter 1 (the iter-1+iter-2 merge) ships the editor with stamping wired up — saved glyphs are immediately usable in brush decoration layers.

Recursive composition (DAG). Glyph strokes carry a brushName; that brush has decoration layers; those layers reference glyphs. So a "tick-with-crossbar" glyph doesn't need a hand-drawn crossbar — it can be a one-stroke tick whose brush has a decoration layer stamping a small crossbar glyph at the tick's tip. Composition out of simpler primitives.

The recursion has three constraints:

  1. Same-palette references only. Glyph A in palette P can only reference glyphs in P (transitively). Cross-palette references would introduce versioning headaches; if you want to compose across palettes, copy the glyph you need.
  2. No cycles. At palette load, DFS the glyph → brush → glyph dependency graph keyed by name. Any back-edge fails the load with a clear error pointing at the cycle (e.g., tick → tick-brush → tick).
  3. Soft depth cap. Tessellation cap at depth 4. Beyond that the inner glyph renders as a single bbox marker and a one-shot warning logs. Catches pathological palettes without breaking them.

Sub-stroke brush flags are ignored during tessellation. A glyph's strokes carry a brushName only so that the brush's OffsetCurve decoration layers (color, width, dash, cap/join) define the glyph's ink. The brush's topology and ordering flags do not apply: when a glyph is tessellated, sub-stroke scrapOutline and zOrder are ignored — only the sub-stroke brush's OffsetCurve decoration layers contribute ink. A glyph is a flattened drawing primitive, not topology, so it never feeds scrap-outline detection, and it never sorts against its parent stroke's z (the whole glyph paints within the parent layer's slot). Glyph-mode decoration layers (RigidStamp / BendingStamp / PointStamp) nested inside a glyph's brushes still drive recursive composition; it's purely the scrapOutline / zOrder fields that drop out.

Glyph tessellation cache. cwGlyphTessellationCache (Decision 8) materialises each glyph into a single QPainterPath in world metres, keyed by (glyphName, mapScale). Recursion through glyph → brush → decorations → glyph runs once per cache entry, not once per stamp. A glyph stamped 200 times along a stroke costs 200 transform-and-emit operations, not 200 tessellations — and a per-stamp scale (from the Taper scale rule) is just part of that draw-time transform, so it never multiplies the cache. Cache entries invalidate when their glyph definition changes, when any brush in the palette changes (since the cache is computed against a cwPaletteSnapshot), or when cwScale::scaleChanged fires on the sketch's map scale.

10. Decoration layers run in one of four modes iter 1 iter 2

Superseded by Decision 11b — modes are gone. The four modes below were collapsed into the rule stack: a layer no longer has a mode; what it draws is decided by its rules, with the Terminal rule's OutputKind standing in for the stamps-vs-offset-curve split. The four behaviours described here are still correct as behaviours — read "RigidStamp" as "a Rigid stamp terminal", "OffsetCurve" as "a Trace offset polyline terminal", etc.

struct cwDecorationLayer {
    QString glyphName;            // glyph reference, used by all modes EXCEPT OffsetCurve

    enum Mode {
        RigidStamp,    // glyph drawn straight at anchor + tangent rotation
        BendingStamp,  // glyph re-parameterised along centerline arclength
        PointStamp,    // single glyph placement at an explicit anchor; no centerline
        OffsetCurve,   // continuous polyline traced at perpendicular offset from
                       // the centerline — no glyph, no tiling. Iter 2.
    };
    Mode mode = RigidStamp;

    // Ordered placement-rule pipeline (Decision 11). Empty = use the
    // mode's default rule stack at render time.
    QVector<cwPlacementRulePtr> rules;

    // -------- OffsetCurve params. Iter 1 ships these with
    // offsetCurveOffsetMm = 0 (the centerline-replacement path); iter 2
    // generalizes to non-zero offsets (parallel filled lines that
    // curve with the centerline). Ignored for the three glyph-based
    // modes; they read width/color from the referenced glyph's own
    // strokes. Colors per Decision 1's light/dark pair.
    QColor           offsetCurveColorLight;
    QColor           offsetCurveColorDark;
    double           offsetCurveWidthMm   = 0.4;
    double           offsetCurveOffsetMm  = 0.0;   // perpendicular distance; sign = side; 0 = centerline
    Qt::PenStyle     offsetCurveDash      = Qt::SolidLine;
    Qt::PenCapStyle  offsetCurveCap       = Qt::RoundCap;
    Qt::PenJoinStyle offsetCurveJoin      = Qt::RoundJoin;
    bool             offsetCurveAcceptsPressure = true;  // width modulates with stylus pressure

    // Collision buffer (iter 2) — keep-clear margin used by MutateScene
    // rule queries. For glyph modes, inflates the glyph's ink bbox. For
    // OffsetCurve, inflates the swept rectangle around the offset
    // polyline. 1 mm gives visible breathing room on the printed page.
    // DORMANT in iter 1: the field is serialized for forward-compat, but
    // nothing reads it until the iter-2 scene rules exist. The project-wide
    // default (cwSketchSettings.defaultBufferMm) also arrives in iter 2.
    double bufferMm = 1.0;

    // Tiebreak hint for MutateScene rules — copied into each stamp at
    // Stamp time so scene rules can sort by priority. Distinct from the
    // brush's zOrder paint-order key (Decision 12). Higher wins.
    int collisionPriority = 0;

    // Scale window (iter 3)
    double minPaperScale = 0.0;
    double maxPaperScale = 0.0;
};

Why OffsetCurve is a peer of the stamping modes, not a stamping mode. Stamping repeats a glyph along arclength; OffsetCurve traces the cartographer-drawn polyline once at perpendicular offset and emits a single continuous polyline. The two operations share nothing mechanically — no glyph reference, no spacing, no per-stamp position. They share the decoration layer envelope (z-order, buffer, scale window, collision priority, placement-rule pipeline, cap/join, color, width) because that's the unit of cartographer control. The enum is named Mode rather than Stamper for that reason.

OffsetCurve covers two cases under one mechanism.

  1. offset = 0 (iter 1) — traces directly along the cartographer's polyline. This is what the prototype called "centerline rendering"; in the unified model there's no separate centerline concept, just an OffsetCurve layer with offset 0. Every seed brush that draws a visible line (wall, feature) uses exactly this.
  2. offset ≠ 0 (iter 2) — traces parallel to the polyline at signed perpendicular distance. This is what unlocks overhang lines (a thin parallel line above a wall), double walls in wide passages, tube-passage ceiling-height shadows, offset pit-edge runs, and triple- or N-line symbology. N parallel filled lines = N OffsetCurve decoration layers on the same brush, each with its own color / width / dash / cap-join / offset.

Generation. Sample the polyline at fine resolution, compute the perpendicular at each sample, emit a polyline at signed perpendicular distance offsetCurveOffsetMm. For offset = 0 the output equals the input (no perpendicular math needed — fast path). For non-zero offsets, iter 2 also handles miter behavior at sharp corners and detects offset-curve self-intersection at tight bends (clip or split as needed). Paper-mm → world-m baking happens the same way stamp positions do. The generator runs at the Generate + Stamp stages of the rule pipeline just like the stamping modes (emits one "stamp" representing the whole traced polyline), so scene rules in iter 2 can hide it as a single unit. Cartographers who want overlap permitted drop the scene rules from the layer.

No halo fields. An earlier draft of this plan carried haloRadiusMm + HaloColorSource (paper / scrap fill) on every decoration layer, plus a Halo if still colliding scene rule. Dropped. Reasons:

  • None of the seed brushes need it. Iter 1's first end-to-end target (floor step) doesn't need halo. Iter 1's second target (ceiling step) can rely on zOrder alone — the ceiling-step T paints over whatever is under it.
  • The "halo color when a glyph straddles two scraps" question doesn't have a clean answer. Dropping the feature dodges it.
  • If a cartographer eventually needs a halo'd glyph, they can achieve it by composing brushes (a paper-colored underlay stroke + an ink stroke on top, or a decoration layer drawn in the paper color below the inked layer). Workable; just heavier to author than a flag.
  • If real iter 1+2 use shows halo is genuinely needed (most likely for text/label backings in a later iteration), halo lands as additive fields without breaking existing palettes.

bufferMm stays — that's geometry, not visuals. It controls the keep-clear region scene rules arbitrate against, and is independent of any drawn buffer.

The three stamping modes cover dashes, ticks, and points with one mechanism; OffsetCurve covers the continuous parallel case:

11. Placement rules — a four-stage registry iter 1 iter 2

Updated by Decision 11b. The four stages stand, but the third is renamed Stamp → Terminal (it can produce a traced polyline, not just stamps) and reads its OutputKind; there are no per-mode default stacks (defaultRules is removed) and no compatibleModes() — validation is stack well-formedness, not mode compatibility.

A decoration layer's rules field is one ordered pipeline spanning four stages. The first three run per-layer; the fourth runs globally, against the union of every layer's emitted stamps plus every stroke's centerline keep-clear band:

All four stages register through the same cwPlacementRule registry, keyed by displayName — adding a new behaviour at any stage is one new class + one registration line. Treat rule names as part of the palette schema's stable surface: renaming a registered rule breaks every palette file that references it. UI translation happens via the regular Qt tr() lookup, not by changing the registered name. 

struct cwStampPosition {
    // Ordered widest-first so the layout packs without internal padding holes.
    QRectF  bufferedBboxWorld;    // ink bbox inflated by layer.bufferMm; populated at Terminal time
    QPointF anchorWorld;          // where glyph (0,0) sits
    QString glyphName;            // usually layer.glyphName; rules may swap it
    double  rotationRad = 0.0;    // 0 = +X along world +X
    double  scale       = 1.0;    // 1.0 = glyph's authored size; the Taper
                                  //   scale rule may vary it. Applied as a
                                  //   draw-time transform on the cached path,
                                  //   NOT part of the tessellation cache key.

// Read-only inputs every per-layer rule runs against, bundled so new
// inputs don't churn each rule's signature (the implemented shape).
struct cwPlacementContext {
    const cwStrokePath&      strokePath;      // arclength sampler over this stroke
    const cwDecorationLayer& layer;
    double                   worldPerPaperMm = 0.0;
    cwSceneContext*          scene = nullptr; // MutateScene state; nullptr pre-iter-2
};

// What a Stamp rule needs to turn one placed position into ink. The glyph
// is tessellated once by cwGlyphTessellationCache and passed in, so the
// rule never reaches the cache and stays cheap to re-invoke.
struct cwStampGeometry {
    const QPainterPath& glyphPath;            // unplaced world-metre glyph ink
    const cwStrokePath& strokePath;
};

class cwPlacementRule {
public:
    virtual ~cwPlacementRule() = default;
    virtual QString displayName() const = 0;  // also the registry key
    enum Stage { Generate, MutatePerLayer, Terminal, MutateScene };
    virtual Stage stage() const = 0;

    // What the layer's Terminal rule produces (Decision 11b). Non-terminal
    // rules return None; a stamp Terminal returns Stamps; the polyline
    // Terminal returns Polylines. cwSketchDecorationLayout reads this off the
    // stack's Terminal rule instead of switching on a (removed) layer mode.
    enum class OutputKind { None, Stamps, Polylines };
    virtual OutputKind outputKind() const { return OutputKind::None; }

    // Placement. Per-layer rules read `context`, read/write `positions`;
    // scene-stage rules ignore `positions` and pull from context.scene.
    virtual void apply(QVector<cwStampPosition>& positions,
                       const cwPlacementContext& context) const = 0;

    // Materialisation — the Terminal rule owns its shape (Decision 11a). A
    // Stamps Terminal overrides stampPath() (rigid transform or arclength
    // warp); a Polylines Terminal overrides tracePolylines(). The defaults
    // are empty, so Generate / MutatePerLayer / Scene rules ignore them.
    // measureBbox() (iter 2) is the cheap counterpart the MutateScene stage
    // re-runs for each trial shift/scale so only collision survivors pay for
    // a full QPainterPath.
    virtual QPainterPath stampPath  (const cwStampPosition& p,
                                     const cwStampGeometry& geo) const { return {}; }
    virtual QVector<QPolygonF> tracePolylines(const QVector<QPointF>& strokeWorld,
                                     const cwPlacementContext& context) const { return {}; }
    virtual QRectF       measureBbox(const cwStampPosition& p,
                                     const cwStampGeometry& geo) const { return {}; }
};

Decision 11a — the rule owns its shape. Turning a placed cwStampPosition into world-metre ink is the Terminal rule's job, not cwSketchDecorationLayout's. The layout orchestrates and owns resources — it tessellates the glyph through cwGlyphTessellationCache, runs the pipeline, then calls the resolved Terminal rule's stampPath() (or tracePolylines()) — but the per-terminal transform (rigid place vs. arclength warp) lives in cwRigidStampRule / cwBendingStampRule, reached polymorphically rather than through a removed-mode switch in the layout. Adding a new stamp style is then one rule class, with no edit to the layout, and the rule's name finally carries its behaviour. The geometry methods take their inputs (glyph path, stroke path) by parameter so they are pure and re-callable: in iter 2 the MutateScene stage, which shifts and scales stamps to resolve collisions, re-runs measureBbox() for each trial position (cheap), and only the survivors are materialised into a full QPainterPath once collision settles. Performance is a wash versus doing the transform in the layout — the geometry math is identical; what this buys is altitude, not speed.

Decision 11b — modes collapsed; a layer is a rule stack. This supersedes the four-mode framing of Decisions 10 & 11 below (their prose still reads in mode terms; the model is now mode-free). The repeated investigation that "the mode never carries behaviour — the rule stack does" reached its conclusion: cwDecorationLayer::Mode is removed entirely (no enum, no stored field, no preset id; the proto mode field is reserved). Every distinction the four modes encoded lives in the stack instead:

  • Repeated vs. single was a Generate-stage choice all along — Uniform spacing (every n mm) vs. Explicit point (one anchor). "Single" is just the degenerate count; a future By count rule (commit 9) covers "exactly N".
  • Rigid vs. jointed vs. bending is a Terminal-rule choice — Rigid stamp (rigid transform), Jointed stamp (warp the vertices, straight chords), Bending stamp (subdivide so the glyph follows the curve), differing only in stampPath() (the jointed/bending split is commit 4.1).
  • Stamps vs. offset curve is the Terminal rule's OutputKind (Stamps / Polylines); cwSketchDecorationLayout reads it off the stack's Terminal rule instead of switching on a layer mode. The old RigidStamp+PointStamp duplication dissolves — both end with Rigid stamp, differing only in their Generate rule.

Consequences: the Stamp stage is renamed Terminal; cwPlacementRuleRegistry::defaultRules(mode) is removed (a layer with no rules draws nothing — there is no per-mode fallback, so the seed and the future editor write explicit stacks); and compatibleModes() + its QFlags apparatus are deleted, since there are no modes to validate against. The load-time check (commit 4.1) becomes stack well-formedness — exactly one Terminal, stages ordered, Generate rules agreeing with the Terminal's input needs — not mode compatibility. Line styling (offsetCurve*) stays as flat cwDecorationLayer fields for now; it migrates into Trace offset polyline's rule params in commit 9 (params interpretation + brush editor), where terminal-specific data belongs.

cwSceneContext. The shared object that MutateScene rules use to inspect and modify the unified stamp set. Owns the R-tree internally (lazily built on first query) and a read-only list of every stroke's keep-clear band (inflated by the widest offsetCurveWidthMm/2 across the stroke's brush's OffsetCurve layers plus max(layers.bufferMm), so glyphs colliding with another brush's traced line count as collisions). Method shape: 

class cwSceneContext {
public:
    // Spatial queries
    QVector<StampHandle> queryStampsInBbox(const QRectF &worldBbox) const;
    bool                 collidesWithCenterline(const QRectF &worldBbox) const;

    // Per-stamp inspection / mutation
    const cwStampPosition& stamp(StampHandle h) const;
    void mutateStamp(StampHandle h, std::function<void(cwStampPosition&)> m);
    void hideStamp(StampHandle h);

    // Centerline keep-clear bands (read only)
    const QVector<QPolygonF>& keepClearBands() const;
};

Scene rules can shift, scale, or hide individual stamps; they cannot add new stamps (that's a per-layer concern) and they cannot move stamps across layers. Mutations rebuild the affected R-tree entries lazily between rules in the chain, so a subsequent rule sees the previous rule's effects.

Iter 1 + iter 2 ship these rules as code. Until the in-app brush editor lands (commit 9 / iter 4+), cartographers compose them by editing palette files directly; the editor then offers them through a rule picker that keeps the stack well-formed (see the validation note above). Iter 1 ships seven rules — Uniform spacing, Explicit point, Align to tangent, Rigid stamp, Jointed stamp, Bending stamp, Trace offset polyline (the jointed/bending split landed in commit 4.1); the rest are additive follow-ups:

No per-mode default stacks (Decision 11b). With modes gone, an empty layer.rules draws nothing — there is no fallback. The seed and the future editor write explicit stacks; the four old "modes" are now just four different stacks. The seed's brushes use:

In iter 1 decorations are drawn at their candidate positions with no collision resolution — overlap is allowed. Iter 2's scene-rule chain is opt-out (appended to the seed stacks above). No enum, no hard-coded policy — composition all the way down.

Execution model. cwSketchDecorationLayout sorts every layer's rule list into stage order: Generate → MutatePerLayer → Terminal → (barrier) → MutateScene. Per-layer stages can run concurrently across layers and strokes; scene rules run serially after the barrier against the unified stamp set. The cartographer can drag rules around in the UI without worrying about stage order — the framework sorts them. Exactly one Terminal-stage rule must terminate the per-layer portion of each layer's pipeline.

A layer is a rule stack; validation is stack well-formedness, not mode compatibility (Decision 11b). With compatibleModes() gone there is no (rule × mode) matrix to check. The load-time check (commit 4.1) instead verifies the stack is well-formed: exactly one Terminal rule, stages in order, and the Generate rules agreeing with the Terminal's needs — a Stamps Terminal needs a stamp-seeding Generate (Uniform spacing / Explicit point); a Polylines Terminal needs none.

The primary defense is the UX. When the in-app brush editor lands (commit 9 / iter 4+), its rule picker only offers rules that keep the stack well-formed and previews the result live, so a cartographer can't compose a broken stack in the first place — heading off essentially all of these errors at the source.

Load-time validation is the backstop for palette files edited outside the app or authored against a different CaveWhere version. cwSymbologyPaletteIO validates every layer's resolved rule stack on load: a layer that is malformed (no Terminal, two Terminals, a Stamps Terminal with no seeding Generate) has its offending rules dropped — or the whole layer dropped if it has no usable Terminal — and a one-shot warning logs a specific message. The rest of the palette loads normally — same skip-and-warn philosophy as unknown rule names (below): a malformed stack degrades that one layer rather than taking down the whole palette. An empty stack is valid and simply draws nothing (there is no default-stack fallback). MutateScene rules are exempt — they're stage-agnostic and operate on the unified stamp set after the barrier.

Forward compatibility. Unknown rule names in the palette round-trip via the params bytes field and log a one-shot warning at render time. Renaming an existing rule is a breaking change for palette files; adding new rule names is not.

12. Z order and paint order iter 1

The prototype paints strokes in creation order — fine for one-off sketching, wrong for symbology. A wall stroke drawn before a feature stroke gets covered by the feature, even though cave maps universally want the wall on top. We need explicit paint-order control.

cwLineBrush::zOrder (integer, default 0). Strokes paint in (brush.zOrder ascending, strokeCreationOrder ascending) order — higher z paints last, i.e. on top. The seed palette ships with:

Seed brushzOrder
feature10
wall100

Wall always paints over feature regardless of who was drawn when. The cartographer can edit z per brush; the brush picker shows z so it's discoverable. Large gaps between defaults (10 vs 100) leave room for community palettes to slot in without renumbering.

Within a single brush, decoration layers paint in declared order. The brush has no special "centerline first" rule — the visible line on a normal wall is just an OffsetCurve(offset=0) decoration layer, paint-order-equal to any tick or T-bar layer on the same brush. To put a tick layer above the brush's main line, list it after the OffsetCurve in the brush's decoration array. To sandwich something between two parallel lines, list the layers in that order: outer offset, glyph layer, inner offset.

If a brush needs weirder internal layering than "layers top-to-bottom in declared order," that's a sign the symbology should be two separate brushes at different zOrders — keeps the data model simple and the editor UI obvious.

Halos within a single stamp aren't z-controlled — the stamp operation always draws halo first, then ink, as a single atomic unit (Decision 10). The halo z question is "halo of which glyph, relative to other glyphs," which is the decoration-layer-ordering question above.

Distinct from collisionPriority. Z is paint order; collisionPriority on a decoration layer (Decision 10) is placement-pass arbitration — who wins a contested spot. A low-z brush can have a layer with high collisionPriority: it gets placed first and other brushes' decorations route around it, but when the placement is settled, paint order is still z-driven so the low-z brush draws under the higher-z ones. The two knobs are orthogonal.

Implementation. cwOffsetCurveBatches groups by brush; iter 1 sorts batches by brush.zOrder before emission. The scene-rule chain (iter 2) produces a flat list of resolved stamps via cwSceneContext; cwSketchPainter::paint sorts that list by (brush.zOrder, decorationLayerIndex) before stamping. No per-glyph z field needed; brush z + declaration order covers every case the seed and UIS palettes have come up against.

13. Collision avoidance is the default scene-rule chain iter 2

"Non-overlap" in symbology is stronger than "bboxes don't intersect." A hatch tick whose tip sits one pixel away from another tick reads as a cluttered blob even though they technically don't touch. The MutateScene rules in Decision 11 arbitrate against buffered bounding boxes — the ink stays its real size, but the keep-clear region around each stamp is inflated by layer.bufferMm (converted to world metres) and stored on the stamp as bufferedBboxWorld. Centerlines are seeded into cwSceneContext as keep-clear bands the same way, so a glyph that collides with another brush's centerline counts as a collision even though no other stamp is involved.

Implementation sketch for the iter-2 rules (each is a ~50-line cwPlacementRule subclass at Stage = MutateScene): 

// All values in world metres at this point.
// cwSceneContext owns an R-tree of cwStampPosition handles plus a
// list of centerline keep-clear bands.

// Try shift to fit (one scene rule, called for each visible stamp):
for each stamp s in scene.stampsSortedByPriorityDesc():
    if scene.collidesWith(s.bufferedBboxWorld):
        for delta in {+¼ spacing, -¼ spacing, +½ spacing, -½ spacing}:
            shifted = s.shiftAlongArclength(delta)
            if !scene.collidesWith(shifted.bufferedBboxWorld):
                scene.mutateStamp(s, shifted)
                break

// Try shorten to fit:                                  (next rule in chain)
// Hide if still colliding:                             (last rule)

Incremental invalidation. When a stroke is added or modified, cwSketchDecorationLayout tracks a dirty-rect. cwSceneContext rebuilds the affected R-tree region and re-runs the scene-rule chain over stamps inside (plus a small skirt covering the buffered bbox of any stamp whose keep-clear region might extend into the dirty area). Untouched regions keep their cached resolutions.

Performance target. The scene-rule pass is the most likely place to lose interactivity. Iter 2 lands single-threaded; the target is "under 16 ms for a 5 000-stroke sketch after a single-stroke edit." If we miss it, the AsyncFuture lift (via cwConcurrent::run per CLAUDE.md) is the next move — the cwPaletteSnapshot flow from Decision 8 already supports that without lifetime fixes.

Iter 1 has its own target, separate from the scene pass. Even before MutateScene exists, iter 1 runs decoration generation, glyph tessellation, and OffsetCurve batching synchronously on the UI thread. So that a later scene-pass regression isn't blamed on work that was already slow: iter 1 must repaint a 1 000-stroke sketch in under 16 ms after a single-stroke edit, measured with the seed palette plus the floor-step demo brush. Glyph tessellation is cached per (glyphName, mapScale) (Decision 8), so steady-state edits should pay only stamp transform-and-emit cost, not re-tessellation. Establish this number as a benchmark in iter 1 so iter 2's 5 000-stroke scene-pass target is measured against a known baseline rather than a moving one.

14. Scale visibility — driven by cwSketch::mapScale directly iter 3

With dynamic scale switching as a first-class workflow (Decision 7), the active paper scale is just cwSketch::mapScale — the cartographer flips it from the sketch toolbar and everything downstream re-renders. No separate "preview at scale" view-state override is needed; the scale itself is the dial.

Implementation note: cwSketch::mapScale is a Q_PROPERTY(... CONSTANT) — the pointer to the cwScale object is constant for the sketch's lifetime, but the value inside emits cwScale::scaleChanged. Decoration-cache and placement invalidation must listen on that signal, not on a (non-existent) cwSketch::mapScaleChanged notify.

Per-layer scale-visibility windows (cwDecorationLayer::minPaperScale, maxPaperScale) and per-brush windows on cwLineBrush are evaluated against cwSketch::mapScale during decoration generation:

  • If the layer's window excludes the current scale, the generator returns no candidates → the layer drops out entirely. This is how hatches disappear at 1:5000 while the centerline survives.
  • If the brush's window excludes the current scale, every decoration on that brush plus the centerline drop out — the brush is invisible at this scale. Useful for "drip line only at 1:250 or coarser detail."

Combined with the scene-rule chain's shorten / hide rules (Decision 13), this gives the cartographer two coarse controls (brush-level + layer-level visibility windows) and one fine control (the layer's collision policy, which decides what to do with marginal layouts at scales the layer nominally supports). At small scales they compose into the "some symbols don't draw, others get shortened" behaviour cave maps need.

Export and screen render the same way. cwSketchExporter reads cwSketch::mapScale; the canvas reads cwSketch::mapScale; the only difference is whether user zoom is composed in (canvas yes, export no). Scale visibility is therefore identical on screen and on paper — no separate "preview" plumbing to keep in sync.

15. Git sync reuses QQuickGit iter 4

A palette directory becomes a clone:

  • ~/.../palettes/uis-standard/.git points at a public remote (HTTPS by default; SSH for authoring).
  • "Add palette from URL" triggers QQuickGit::clone(url, paletteDir).
  • "Update palettes" runs pull against each non-seed palette.
  • Authoring palettes is a separate writable repo; pushing reuses the GitHub OAuth credentials already wired for project sync (cwGitHubDeviceAuth).
  • A palette's identity is Palette.id inside the file, not the remote URL — so a fork retains identity until the author bumps it.

No palette-registry server. Discovery is "share the git URL" until we have a reason to centralise. Until iteration 4 ships, palettes can be installed by dropping a directory into the palette location.

  • Tangent indicator (the +X direction; tells the cartographer how the glyph will be oriented at stamp time).

Stamping is wired up in iter 1 (the iter-1+iter-2 merge — see the roadmap table). Iteration 2 — scene-rule chain + non-zero OffsetCurve

  • cwSceneContext — owns the R-tree of buffered stamp bboxes plus the read-only list of every stroke's cartographer-drawn-polyline keep-clear band. Built lazily on first query, invalidated by dirty rect.
  • Ship the four MutateScene rules listed in Decision 11: Try shift to fit, Try shorten to fit, Hide if still colliding, Sort by priority desc. Each is a small cwPlacementRule subclass registered the same way as iter-1 rules.
  • Append the scene-rule chain to the seed brushes' stacks (the seed-stack table in Decision 11). Existing iter-1 brushes inherit the chain transparently; cartographers who want overlap drop the scene rules from a layer.
  • Non-zero Trace offset polyline offsets. Iter 1 shipped the trace with offset=0 (the centerline-replacement path). Honouring offsetCurveOffsetMm — miter handling at sharp corners, self-intersection on tight bends — is pulled forward to commit 4.5 (SYMBOLOGY_PALETTE_OFFSET_CURVE_PLAN.md), since it's self-contained layout compute. End-to-end targets: overhang line on a wall = one Trace offset polyline layer with offset 1 mm, width 0.3 mm; double-wall passage = two such layers at offsets ±2 mm; triple line = three layers at {-2, 0, +2}.

Iteration 4+ sketches

  • Git sync via QQuickGit clone/pull; palette manager UI with "add from URL", "update all", "remove" rows; reuse of cwGitHubDeviceAuth for push credentials when authoring.
  • Area + point brushes. Point brushes are a degenerate case of the existing model — a brush with a single decoration layer stacked Explicit point → Rigid stamp (and no Trace offset polyline layer); the cartographer places one anchor and the glyph stamps there. Area brushes add a fill component (a glyph stamped in a grid inside a closed polygon) plus a perimeter stroking with a line brush. No new schema concepts — same glyphs, same placement rules.
  • Brush inheritance (if iter 1 use shows palette bloat from clone-and-tweak). cwLineBrush::parentBrushName with per-field overridden flags. Resolution walks the parent chain; cycle detection at palette load (same machinery as glyphs). Non-breaking addition.
  • Per-stamp overrides. The sketch carries an optional QVector<cwStampOverride> alongside its strokes. Each override targets (strokeId, decorationLayerIndex, stampIndex) and supplies a parametric mutation ("move forward 1 mm", "hide", "swap to glyph G2"). Applied as a final mutator pass after the brush's rule pipeline. Hand-edits survive brush edits because they're parametric, not absolute.
  • In-app brush editor: pick a base brush, add / remove decoration layers, reorder placement rules, tweak rule params with live preview. The rule picker only offers rules that keep the stack well-formed (exactly one Terminal, stages ordered) so a broken stack can't be authored — the primary defense behind the load-time skip-and-warn backstop (Decision 11b). This is commit 9, which also migrates the flat offsetCurve* styling fields into Trace offset polyline's rule params.
  • Halo (if needed): revisited when label/text symbology or genuine ceiling-step legibility issues prove z-order isn't enough. Lands as additive fields on cwDecorationLayer; existing palettes are unaffected.
  • .th2 interop: brushes get a therionLinetype field and glyphs get a therionPointtype field; importer consults palette lookup tables to translate. The placement-rule pipeline maps Therion's "auto-place along line" + style modifiers cleanly.
