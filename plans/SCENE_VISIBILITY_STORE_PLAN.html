1 · Proposed fix — publish once, read snapshots

Introduce a scene-owned visibility store, cwSceneVisibility, that owners publish into and all consumers read from versioned immutable snapshots. The authoring copy stays with the owner (so pre-attach seeding and range-flip math keep working); the store is the single published truth that replaces every consumer-side twin.

1.1 · Addressing model

One key shape covers all three granularities:

(cwRenderObjectId, subId)  →  { bool visible; QVector<quint8> mask; quint64 entryVersion; }
 cwRenderObjectId alone    →  { bool objectVisible; }
  • Whole object — cwRenderObject::setVisible → object-level entry. Covers cwRenderPointCloud, cwRenderGLTF, whole-plot/whole-scrap-set toggles.
  • Per sub-item — cwRenderTexturedItems::setItemVisible(id, bool) → (ownerId, id) entry with empty mask. One entry per hidden scrap/LiDAR quad.
  • Per vertex — cwRenderLinePlot::setRangeVisible → (ownerId, 0) entry whose mask shares the owner's already-immutable buffer (same discipline as today's setVisibilityMask: write-to-copy-and-replace, never mutate in place). Empty mask = all-visible fast path.

Keying on cwRenderObjectId (the stable monotonic id from the cwScene pending-map refactor) rather than cwRenderObject* means snapshots contain no pointers — safe for the BVH build worker, offscreen render jobs, and any future off-thread pick (the sink-scan broad-phase). This is the same identity choice that fixed #512-class address-reuse bugs in the pending map, and it is the store-shaped version of the intersecter plan's follow-up 3.

Effective visibility is an AND across levels: objectVisible && subVisible && maskByte. Absent entries mean visible, so the store is sparse — it only holds what is hidden or masked.

1.2 · The store and its snapshot

// Owned by cwScene. All mutation is main-thread-only.
class cwSceneVisibility {
public:
    // Owner-facing publish API (called by the render-object facades)
    void setObjectVisible(cwRenderObjectId id, bool visible);
    void setSubVisible(cwRenderObjectId id, uint32_t subId, bool visible);
    void setMask(cwRenderObjectId id, uint32_t subId, QVector<quint8> mask); // shares buffer; empty = all-visible
    void removeObject(cwRenderObjectId id);          // scrub on removeItem / direct delete
    void removeSub(cwRenderObjectId id, uint32_t subId);

    // Consumer-facing read API
    cwVisibilitySnapshot snapshot() const;           // materializes lazily if dirty (copy-on-read)
    quint64 version() const;

signals:
    void changed();                                  // coalesced; consumers re-read at their own cadence
};

// Immutable, cheap to copy (shared_ptr<const Data>), safe to hand to any thread.
class cwVisibilitySnapshot {
public:
    quint64 version() const;
    bool objectVisible(cwRenderObjectId id) const;                       // absent = true
    bool subVisible(cwRenderObjectId id, uint32_t subId) const;          // ANDs object level
    const QVector<quint8>* mask(cwRenderObjectId id, uint32_t subId) const; // null = all-visible
    quint64 entryVersion(cwRenderObjectId id, uint32_t subId) const;     // gates GPU re-uploads
};
Why copy-on-read, not copy-on-write-per-mutation
Keyword toggles arrive in bursts — a filter change re-resolves every row of the accepted/rejected models, flipping tens-to-hundreds of items back-to-back. Publishing a fresh snapshot per mutation would be O(n) per write, O(n²) per burst. Instead the store mutates live (non-shared) state, bumps the version, and materializes a snapshot only when a consumer asks and the state is dirty. The RHI reads once per frame at the sync barrier; the intersecter re-reads on changed(); each burst costs one materialization per consumer, not one per toggle.

Per-entry entryVersion exists for one consumer: cwRHILinePlot must not re-upload its visibility vertex buffer because an unrelated scrap was hidden. It compares the entry version for its own key against the last-uploaded version — replacing today's cwTracked dirty bit with the same cost profile.

1.3 · Who writes, who reads

Producers are untouched. The keyword pipeline, cwKeywordVisibility, and all proxy classes keep calling the same owner APIs they call today (setVisible, setItemVisible(id, bool), setRangeVisible). Those owner methods become facades: keep the local authoring copy (unchanged), then publish to the store instead of running today's twin protocols. This confines the churn to the render objects + consumers and keeps the producer pipeline, its tests, and QML untouched.

Seeding becomes structural. Two bug classes from the #575 review die by construction:

  • Visibility seeded before scene wiring is silently dropped — the owner's authoring copy is the pre-attach buffer; cwScene::addItem asks the object to publish its current visibility state into the store (one virtual, publishVisibility(), on cwRenderObject). One seed path, at one seam, instead of per-owner conventions.
  • Empty→non-empty geometry cycle resurrects a hidden scrap — store entries are identity state keyed by (id, subId); geometry registration no longer resets them (the intersecter's addObject-resets-visibility footgun disappears because the intersecter no longer stores visibility at all). The "re-publish after registration" comment-convention and its belt-and-braces re-seeds (cwRenderTexturedItems.cpp:161, :209) are deleted, not preserved.

1.4 · Threading contract

  • Store mutation: main thread only (same rule as today's setVisible/setVisibilityMask on the intersecter).
  • snapshot() returns a shared_ptr<const Data>; any thread may hold and read it indefinitely. Buffers inside it (masks) follow the existing immutable-once-published discipline — owners write to a copy and replace.
  • The RHI reads at the scene-graph sync barrier (GUI blocked — same safety argument as the pending-map drain, but now also correct without the barrier, since it holds a snapshot).
  • The intersecter does no install-time visibility stamping at all (Phase 2 superseded an earlier draft of this bullet, which planned a re-stamp from the current snapshot at install): every query pairs the installed BVH with a store snapshot captured at query entry, so a toggle that races a build is simply read fresh on the next pick — the "#575 toggle can't race the worker" discipline holds with no stamping step. A removal racing a build is a narrower, pre-existing window — see the review follow-ups in §3.

Blast radius: cwRHIObject.h, registerRenderObject, 1–2 subclasses. Phase 4 sweep. (Landed with Phase 4 — turned out to be all five m_frame-carrying subclasses, eleven backfill blocks.)
  • Two snapshot read paths, by design — gather() reads GatherContext::visibility (per-gather, hence per-job-correct once §5's overlays exist), while cwRHILinePlot's mask upload reads the frame-global snapshot (updateResources has no gather context, and the GPU mask buffer is a shared resource). Coherent today because both see the same frame snapshot. Note for the §5 sink-inference work: the per-vertex mask is per-frame by construction — per-job vertex hiding is out of design (the per-job overlay hides whole objects only). (Phase 4 stamped this note at cwRHILinePlot::updateVisibilityBuffer.)

4 · Alternatives considered

AlternativeWhy not
Keep the twins, document the conventions The #575 review already showed the conventions leak (silent pre-wiring drops, resurrect-on-geometry-cycle). Every new consumer pays the full tax again; the per-job render-state work is already queued as the next payer.
Make the intersecter the source of truth (it already unifies all three layers) Wrong altitude: the intersecter is about pick geometry; not everything rendered registers pick geometry, and gating the renderer on the picker's state inverts the dependency. Its visibility layer is also the newest code in the file and exists only because no store existed.
Producers write the store directly, retiring the proxies Attractive endgame, but it drags the keyword pipeline, its ownership registry, and QML into the blast radius now. The proxies are thin and correct; facades keep them out of scope. Revisit after Phase 4 (open question Q2).
Push-per-mutation snapshots (store publishes on every write) O(n²) under keyword bursts; copy-on-read with a version bump gives the same immutability for one materialization per consumer per burst.
Fold whole-object visibility into the generic PendingChange protocol The pending map is about lifecycle (add/update/delete); visibility is state. Overloading queue membership as state is exactly the conflation the pending-map refactor just removed.

5 · Relation to queued work

  • Per-job render state (offscreen sink inference) — the motivating "next consumer". With the store, a render job captures {visibility snapshot, per-job overlay}; cwSceneGatherOptions::hiddenObjectIds is already the overlay half. No new protocol needed — that is the payoff this plan is buying.
  • Sink-scan BVH broad-phase (P4) — off-thread box queries want pick results that honor visibility without touching live objects; the snapshot pair (QuerySnapshot + cwVisibilitySnapshot) is exactly the pointer-free capture that unblocks it.
  • Intersecter follow-up 4 (authoring/built split) — shrinks: visibility leaves the class, so the split covers geometry/matrix/caches only.
  • Future: interactive hide tool (select in the 3D view, hide selection) — the pick result and the store share identity ((renderObjectId, subId) + primitive index), so a hit is already a store address, and hidden things become unpickable/unframed by construction. What it exposes is a two-producer conflict this plan deliberately ignores: today every key has one writer (the keyword pipeline), so the store is last-writer-wins — but a manual hide would be clobbered the next time a filter change re-resolves the accepted model and calls setVisible(true) on everything accepted. Two resolutions, choose by granularity:
    • Entity-granular hides (scrap, note, trip, cloud): model the hide as data — a hidden keyword on the entity, rejected by the existing pipeline. Pipeline stays the sole producer, no store change, and the hide persists in the project and is visible/undoable in the keyword UI.
    • Sub-entity hides (a single shot, a piece): add per-source layers to the store (keyword + manual), effective = AND across layers (masks AND bytewise); "show all hidden" clears the manual layer without touching keyword state. This is a write-side-only change — setter signatures grow a layer tag; snapshots, the read API, and all consumers are untouched — which is why it is not front-loaded into this plan.
    Line plots additionally need a reverse mapping (picked vertex → shot/trip) to choose the span to hide; cwLinePlotManager already owns the forward mapping and is the natural home.

6 · Risks & mitigations

RiskMitigation
Behavioral drift in edge cases (pre-attach seeding, geometry cycles, burst toggles) Phase 0 pins them first; Phase 1's dual-publish equivalence assertions catch divergence before any consumer flips.
RHI reads a stale or torn snapshot Snapshot is immutable by construction; the sync barrier additionally serializes the read. Per-frame single capture means one coherent view per frame — an improvement over today's three independently-timed channels.
Per-slot store lookups slow the pick traversal Lookups happen once per sub-BVH entered, not per node; hash of a POD key over a sparse table. If profiling disagrees, derive a per-slot array from the snapshot at capture (mechanical, one function — still not a hand-maintained twin).
Line-plot mask re-uploads on unrelated toggles entryVersion gating. The load-bearing property — an entry's version does not move on unrelated churn — is pinned at store level in test_cwSceneVisibility.cpp ("entryVersion tracks its own entry" + recreated-entry freshness). The originally-sketched GPU-level upload-count assert was dropped as infeasible under repo rules: updateVisibilityBuffer needs a real QRhiResourceUpdateBatch (beyond the CPU stub harness) and test-only upload counters are banned.
Wide test churn masks a real regression Phases 2 and 3 must not modify the pick-visibility regression suites' assertions — they assert behavior and are the gate. (Setup-only edits are allowed where a suite called a retired API directly: Phase 2 re-plumbed one line-plot test's setVisibilityMask call to scene.visibility()->setMask() with every assertion byte-identical.)
