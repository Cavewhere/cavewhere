0. Is this the right direction?

Short answer: yes, with the framing that this is "finish the symbology authoring surface", not "polish a one-glyph editor".

The doubt is fair — raw glyph placement already works, so a standalone single-glyph editor buys little. But a glyph's value is not drawing it once; it is being a named, reusable unit that the decoration pipeline stamps along a line (the floor-step ticks every 2 mm, dashed ceiling steps, etc.). The brush editor (commit 9) references glyphs by name; the glyph library is the surface that produces and curates those names. So the list-on-the-left / editor-on-the-right shape turns the page from "edit one glyph" into a glyph library manager, which is the correct companion to the brush editor and the natural home for add / select / remove.

The master-detail layout itself is the right UX shape and low-risk. The genuine design risk is two decisions, both pinned below: unsaved-changes handling when switching rows (§5) and name rows vs. rendered thumbnails (§6). "See all the glyphs" leans toward thumbnails, which is the bigger piece — so the plan phases it.

2. New type — cwGlyphListModel C++

cwSymbologyGlyph is not a Q_GADGET (confirmed: plain struct + Q_DECLARE_METATYPE), so QML cannot enumerate palette.glyphs() directly (this is exactly why commit 8 added loadGlyphNamed). A list needs a model exposing one row per glyph. Mirror cwLineBrushListModel's shape (roles, value-copy) — but not its refresh wiring (see the corrected bullet below):

⚠ Refresh on glyphChanged, NOT dataChanged. cwLineBrushListModel connects to the palette's dataChanged() (cwLineBrushListModel.cpp:35) — which fires only on setData() (a manager reload). But the glyph library mutates the palette object-first: setGlyph/ removeGlyph emit only name-scoped glyphChanged(QString) and never dataChanged() (cwSymbologyPalette.cpp:61–72). A verbatim mirror would therefore never refresh on add / remove / edit. cwGlyphListModel must connect to glyphChanged (a full beginResetModel/endResetModel is fine). (Brush-side works today only because brushes still change via reload — the brush editor isn't built yet; commit 9 will hit the same correction.)
class CAVEWHERE_LIB_EXPORT cwGlyphListModel : public QAbstractListModel {
    Q_OBJECT
    QML_NAMED_ELEMENT(GlyphListModel)
    Q_PROPERTY(cwSymbologyPalette* palette READ palette WRITE setPalette NOTIFY paletteChanged)

    enum Role {
        NameRole = Qt::UserRole + 1,   // kebab-case machine id (the in-palette key)
        DisplayNameRole,               // human label
        StrokeCountRole                // cheap, for an empty-glyph hint
        // (PreviewRole / thumbnail — see §6, phased)
    };
    // rowCount / data / roleNames / setPalette(copy glyphs by value) / refresh()
};
  • Binds palette to the page's writable palette; copies palette.glyphs() by value on set, on glyphChanged (object-first edits), and on dataChanged() (reload) — same lifetime-safe value pattern as cwLineBrushListModel (never dereferences the pointer again, safe across a manager reload that the reconciling reload (8.1) keeps stable anyway), but with the extra glyphChanged connection the brush model lacks.
  • Sort: alphabetical by displayName (locale-aware), or by name. No category grouping (glyphs have no category field today), so simpler than the brush model.
  • refresh() does a full beginResetModel/endResetModel — glyph add/remove/save all flow object-first through saveGlyph/ removeGlyph → palette->setGlyph/removeGlyph → glyphChanged(name) → refresh (no reload()). Good enough; per-row signals are a later optimization (already tracked as a follow-up).
  • Static sortedForList(QVector<cwSymbologyGlyph>) free function so ordering is unit-testable without standing up a model.

3. GlyphEditorPage.qml restructure

A RowLayout (or SplitView) inside the existing StandardPage:

  • Left — glyph list. ListView over GlyphListModel { palette: page.palette }. Each delegate shows the thumbnail (§6) + displayName + dimmed name. A TapHandler selects (no MouseArea); duplicate and remove are right-click (§4), matching the rest of CaveWhere. Header row has a [+] Add. Add (and the whole left rail's mutation) is disabled when !palette.writable.
  • Right — the editor. SketchGlyphCanvas + SketchInputArea + overlays + name/display fields + BrushPicker + Undo/Redo/Clear. No Save and no Cancel — edits auto-persist (prerequisite plan). Selecting a row calls glyphCanvasId.loadGlyphNamed(name) and fills the name fields.
  • Page chrome. A single Done pops the page. Because everything is auto-saved, leaving the editor is also the natural boundary for the coarse project-wide re-skin (§5).

5. Decision A — re-skin / rebuild policy DECIDED — A2 (already wired by §7); residual = preview refresh + perf watch

Auto-save reframes this. With async project-level save, switching rows no longer risks losing edits — the current glyph is already persisted — so the old "Save / Discard / Cancel on dirty" question mostly dissolves (undo remains the in-session safety net). What replaces it is a performance question: when does a glyph change force dependent geometry to rebuild?

Resolved against shipped code. §7 Tier 1/2 already wire the project-wide path: an object-first glyph edit emits glyphChanged → cwSketch::resolveSnapshot (only for sketches whose resolved palette is the one edited) → paletteSnapshotChanged → cwScrapManager re-derives that sketch's scraps, coalesced by TriangulateRestarter. That is A2 (debounced live), and it's free — no editor-close hook to build. Two residual items, both small:
  • Editor-UI preview refresh — the list's mini-canvases (§6) and cwSketchGlyphCanvas re-skin on palette dataChanged(), which object-first edits don't emit. They need to also react to glyphChanged(name) (re-skin if name matches the loaded glyph). See §6.
  • Perf watch — if live project-wide re-derive proves heavy while actively drawing in the library, gate Tier 2 during glyph editing (e.g. suspend the scrap re-derive until Done). Premature to build now; the TriangulateRestarter coalescing + snapshot-value dedup likely make it a non-issue. Measure first.

The A1/A2/A3 table below is retained for the record; A2 won by virtue of the §7 wiring rather than a fresh decision.

The key move is to separate persistence from re-skin:

  • Persistence — cheap, async, debounced; happens continuously as you edit (prerequisite plan).
  • Editor's own canvas — re-skins live; it's drawing the one glyph, trivial cost.
  • Project-wide re-skin — expensive: invalidate the glyph tessellation cache and re-run the decoration layout for every other sketch whose brushes stamp this glyph. This is what must NOT run on every pen sample.

So the open question is the granularity and timing of the project-wide rebuild, not whether to prompt:

OptionBehaviorTrade-off
A1 — Rebuild on editor close (leaning this way) While editing, only the editor canvas updates. On Done, invalidate + re-skin the project once. Cleanest perf; other open sketches show stale stamps until you leave the editor (usually fine — you're not looking at them).
A2 — Debounced live rebuild After N ms of edit idle, invalidateGlyph(name) + relayout affected sketches. Live feedback project-wide, but repeated relayout cost; needs the "which sketches use this glyph" dependency to be cheap to compute.
A3 — Manual "apply" An explicit refresh action triggers the project-wide rebuild. Reintroduces a button we're trying to remove; rejected unless perf forces it.

Outcome: A2, already shipped. The earlier recommendation was A1 (rebuild on editor close), deferred until the prerequisite settled signal propagation. The prerequisite settled it as live + coalesced (§7), so A2 is what exists with no extra work. cwGlyphTessellationCache::invalidateGlyph(name) remains an available per-cache scoping refinement if profiling wants it, but the project-wide relayout already happens via resolveSnapshot → paletteSnapshotChanged → TriangulateRestarter.

6. Decision B — list preview DECIDED: B3 — live mini-canvas per row

Each row is a small live SketchGlyphCanvas with the glyph's name below it. No image-provider/cache plumbing and no name-only interim — the preview is the real renderer, so it always matches the editor exactly (same path source, palette snapshot, color scheme, dark mode) and needs no invalidation logic.

  • The delegate hosts a non-interactive SketchGlyphCanvas (palette: page.palette, no SketchInputArea), sized to the glyph's paper extent, that loadGlyphNamed(row.name)s its glyph; the displayName (with dimmed name) sits below.
  • Required fix — re-skin previews on glyphChanged. cwSketchGlyphCanvas currently re-skins only on palette dataChanged() (cwSketchGlyphCanvas.cpp:34–57), which an object-first glyph edit does not emit — so after you edit a glyph, its preview row would go stale. Either (a) have cwSketchGlyphCanvas also connect glyphChanged(name) and re-skin when name matches its loaded glyph (cleanest, also benefits the brush editor), or (b) have the delegate re-call loadGlyphNamed on the model's row update. Prefer (a). Fold this into the §6 preview commit.
  • Performance watch-item. One RHI-backed canvas per row is the cost concern with many glyphs. Mitigations to validate: a fixed/auto-fit view (no per-row pan/zoom state), and ListView delegate recycling so only visible rows hold a canvas (cacheBuffer kept tight). If a large library ever makes this janky, the image-provider approach (a rendered QImage thumbnail) is the documented fallback — but start with the live canvas per your call.
  • Bonus: gives the brush editor (commit 9) the same live glyph preview for free.

7. Read-only palettes & validation

  • Read-only: the page can still open on a read-only palette to browse, but Add / Duplicate / Remove / draw are disabled; the toolbar's "Duplicate to edit…" remains the way to get an editable copy. Reuse the existing palette.writable gate.
  • Name validation: a glyph is only materialized (auto-saved) once its name is valid kebab-case (cwSymbology::isValidName); an invalid/empty name keeps it a draft. Inline field validation (red hint) is the friendly version.
    ✅ DONE (commit 2): the kebab-case rule is now exposed to QML as a SymbologyName QML_SINGLETON (in cwSymbologyName.h, delegating to the same cwSymbology::isValidName free function the IO layer enforces) — one validation surface for the glyph editor and the future brush editor, not a per-canvas wrapper. The page gates auto-save on SymbologyName.isValidName(...) and flags the name field with an ErrorHelpBox (the "red hint") while a drawn draft is unnamed.
