Add Trip UX — split button + unified survey-file dialog

Kills both from-a-file forks on CavePage. Discussion 2026-07-20: one-click Add Trip via a split button (A); a single "from survey file" entry that opens a dialog where the user picks the file first and then chooses Attach vs Import with explanations, Attach recommended and listed first (C); an attach/import entry on the empty trip page (B); and a zero-trip HelpArea empty state on the cave page (D). Drag-and-drop is deliberately excluded: drop support should land app-wide in one release (Fall 2026 milestone), not page-by-page — a GitHub issue gets filed when this plan's implementation finishes (see §9).

Executes the Phase-2 §15 follow-up "Attach entry point for existing native trips" (EXTERNAL_FILE_PHASE2.html). Related but not executed here: §15 A1 (create-on-success) and the Escape-cancel entry.

1. Problem

  • CavePage.qml:258 — [+ Add Trip] pops a two-item menu ("Native trip" / "External centerline…"). The 95% case costs an extra click, and "Native trip" is jargon a new user can't resolve.
  • CavePage.qml:235 and :813 — a separate "Import Survex" button sits next to the Add Trip bar. Import-a-copy vs attach-a-link are both "start from a survey file", presented as abstract verbs in different surfaces. Users can't tell which one to click before picking a file; the choice only makes sense once a concrete file is in hand, with the tradeoff explained.
  • There is no way to attach or import into a trip that already exists.

2. Target UX

ActionBeforeAfter
Add a normal trip Click Add Trip → menu → "Native trip" (2 clicks + a decision) Click the button body of [+ Add Trip |▾] (1 click, no decision)
Start a trip from a survey file (either mode) Two disjoint entries: "Import Survex" button vs Add-Trip-menu "External centerline…" Chevron menu → "Add trip from survey file…" → pick file → choose Attach (recommended) or Import a copy in the dialog
Attach/import into an existing empty trip Impossible Empty trip page shows "Add From Survey File…" next to "Add Survey Data"; same dialog, targets the current trip
Cave with no trips Empty table, nothing points anywhere HelpArea: "No trips yet — click Add Trip, or add one from a survey file via the ▾ menu."

  • Attach first, with a bold "Recommended" tag (Theme accent/bold label). Counters the reflex to pick the familiar word "Import"; attach is the mode that preserves the user's existing workflow.
  • Format-aware enablement: Import only handles Survex today. For .dat/.mak/.srv/.wpj, the Import button is disabled (not hidden) with a muted one-liner: "Import supports Survex files only." The dialog teaching the capability matrix beats a menu hiding it.
  • Both action buttons stay disabled until the scan preview is valid — same gate the Attach button has today.
  • Escape/click-outside behavior unchanged (NoAutoClose; the §15 Escape-cancel follow-up applies here too, unchanged).
Naming: user-facing strings are "Add trip from survey file" (menu item + CavePage dialog title), "Add From Survey File…" (empty-state button; the TripPage host retitles the dialog "Add survey from file" since no trip is being created), "Attach" and "Import a copy" (choice buttons). "External" and "centerline" disappear from all user-facing text — they survive only in internal names (C++ types, objectNames), which tests keep using unchanged.

7. Decisions for review

  • D1 — Attach listed first with a bold "Recommended" tag (decided 2026-07-20): counters the familiar-word pull of "Import"; attach preserves the user's existing survey workflow and is the mode the external-file feature exists for.
  • D2 — Entry points offered only on empty trips. The orchestrator allows attaching over existing chunks (Phase-2 §15 noted a warning would be needed). Recommendation: gate the empty-state affordance on chunkCount === 0 and skip the warning dialog entirely for now. This also matches the importer exactly — importSurvexToTrip hard-rejects non-empty trips.
  • D3 — Keep the CavePage orphan flow. Create-up-front + delete-on-dismiss is built and tested, and now serves Import too. Create-on-success (§15 A1) stays scheduled for revisit alongside Phase 3's cave-level attach.
  • D4 — No rename on the trip-page path (see commit 3). One rule — "adding from a file never renames an existing trip" — even if the name is still the default "Trip N".

8. Not in this plan

  • ExportImportButtons.qml:233 has its own region-level "Import Survex" menu item (surveyImportManager.importSurvex()). Different scope (region-wide, own file picker); the overlap with the unified dialog is real but pre-existing. Revisit if the import surfaces get consolidated.
  • Import support for Compass/Walls files in the unified dialog (the disabled Import button's "Survex only" note is the placeholder until the importer grows formats).
  • §15 A1 create-on-success, Escape-cancels-the-dialog, and the shared externallyBacked predicate (R2) stay where they are in EXTERNAL_FILE_PHASE2.html.
  • Drag-and-drop, entirely (Fall 2026 milestone). Drop interaction should land all at once across the app in one release — survey files onto the cave page, plus whatever other surfaces make sense (.cw projects, note images) — instead of one page quietly supporting it. Filed as a GitHub issue when this plan's implementation completes (§9); the natural design hook is the same unified dialog, opened pre-filled from the dropped path.
