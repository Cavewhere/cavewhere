Centerline Keyword Visibility

Implementation plan — make the 3D survey centerline hide/show through keyword filtering, at the trip level (date, team, cave, trip). GitHub follow-up to the Cave keyword work (#488).

Decisions locked in (from design discussion):
  • De-share vertices per trip — each vertex belongs to exactly one trip, so per-trip masking is leak-free.
  • Trip-id attribute + R8 visibility texture (option B2) — geometry stays static on toggle; flipping a keyword updates one texel, not the geometry buffers.
  • Identity (UUID) keying, not position — the index→trip mapping is carried by cwTripData::id, never by list position, so it can't silently drift between the geometry path and the manager.
  • No live pointers cross the worker boundary — only value-type UUIDs travel async; the manager resolves UUID→live cwTrip* at use time, eliminating dangling-pointer risk.
  • Render object owns visibility, proxy pokes it by id — exactly like the scrap path (cwRenderTexturedItems::setVisible(id, bool)). A toggle flips one byte of the render object's tracked vector; the manager keeps no flag array, no running-id map, and needs no re-entrancy guard.
Picking is not in scope here — the centerline isn't pickable today (the BVH skips Lines). That's tracked separately in issue #530 (Summer 2026); see §10.

1. The problem

The Keyword tab can hide scraps, notes and LiDAR by metadata, but the survey centerline never turns off. We want it keyworded at the trip level so that filtering by date, team, cave or trip hides the matching legs of the line plot — using the same keyword machinery everything else already uses.

2. How it works today

The keyword visibility system is per object: each filterable thing is wrapped in a cwKeywordItem whose keyword model is fed from the source's model, and cwKeywordVisibility calls setVisible(bool) on a small proxy when filters change. The centerline, by contrast, is one merged geometry buffer for the entire region.

Figure 1. The solver flattens every cave/trip/shot into a single vertex+index buffer. There is no per-trip handle to toggle.

Key detail from cwLinePlotGeometry::generate(): station positions are deduplicated per cave (one vertex per unique station name), while line segments are emitted per shot, and each shot is iterated under exactly one trip.

3. Why the centerline can't hide

  1. The centerline render object is never registered with the keyword system.
  2. Even if it were, it's a single object — toggling it hides the entire region's centerline, not a trip.
  3. The visibility unit we want is the trip, but the geometry has no notion of which vertex belongs to which trip.

4. De-sharing per trip

Because stations are deduped per cave, a tie-in station (where trip B starts where trip A ended) is a single shared vertex. A line segment needs both endpoints, so a per-vertex flag can't independently express two trips meeting at one station — hiding a shared vertex would leak into the neighbouring trip.

The fix: in the geometry builder, give each trip its own copy of the stations it uses (a trip-local name→vertex lookup). Junction stations get duplicated — a tiny increase — and now every vertex carries exactly one tripId.

Figure 2. Duplicating the tie-in station so each trip owns its vertices makes per-trip masking exact. Both endpoints of any hidden segment share the same trip, so they collapse together.
Two layers of id, deliberately. The per-vertex attribute is a dense running index (0…N−1), assigned to every trip in iteration order — small enough for a vertex attribute and a texture column. Separately, the geometry emits an index→UUID table (cwTripData::id, which it already has), so the running index is anchored to a stable identity, not a list position. The dense index is a per-snapshot label regenerated each solve; the UUID is how the manager re-attaches it to a live cwTrip. This is what makes the mapping robust instead of fragile — see §7½.

5. Masking in the vertex shader

Each vertex gets a second attribute uint vTripId. A one-row R8 texture (width = trip count) holds one visibility byte per trip. The vertex shader fetches that byte and, if the trip is hidden, pushes the vertex outside the clip volume so the GPU discards the line.

Figure 3. The shader indexes the visibility texture by the vertex's trip id. Hidden trips' vertices are moved to NDC (2,2,2), outside the [-1,1] clip box, so their line segments never rasterize. There is no "discard a vertex" in a vertex shader — collapsing the primitive is the standard technique.

The two GPU inputs update independently: geometry (positions, tripIds, indices) re-uploads only on a solve; the visibility texture re-uploads on a keyword toggle. This is done with two separate cwTracked members on the render object — the same pattern cwRHIPointCloud uses for geometry vs. render state.

6. Keyword wiring

Each trip is registered exactly like a scrap: a cwKeywordItem whose model addExtension(trip->keywordModel()) — which already aggregates Trip / Year / Date, the Cave keyword (from #488) and the team's Caver keywords. The item also publishes Type = Line Plot on its own model (the same Type keyword scraps/LiDAR notes use), so filtering the Type keyword toggles the whole centerline at once. Type lives on the item, not the trip, so scraps/notes under the trip don't inherit it. The item's "object" is a tiny visibility proxy. cwLinePlotManager owns the map and pokes the render object per trip.

Figure 4. The proven scrap pattern, applied to trips. The only new class is the small cwLinePlotTripVisibility proxy. A keyword toggle pokes the render object directly by trip id (green dashed path); the manager only wires items and re-binds/seeds proxies after each solve.

7. Toggle & rebuild sequences

A user toggles a keyword filter

1cwKeywordVisibility resolves the filter and calls setVisible(false) on the trip's proxy.

2The proxy flips its slot directly: cwRenderLinePlot::setTripVisible(tripId, false) — an O(1) byte change on the render object's tracked visibility vector. The manager is not involved.

3The render object marks only its visibility cwTracked dirty → the RHI object re-uploads the whole R8 texture (a few KB even for thousands of trips, so no dirty-rect tracking) in one QRhiTextureUploadDescription. Geometry buffers are untouched. The QRhiTexture is only recreated when the trip count changes on a re-solve; a plain toggle reuses it.

The survey re-solves (any edit)

1updateLinePlot() receives new geometry + a per-vertex tripIds array + an index→UUID table — all value types, no live pointers carried across the worker boundary.

2setGeometry(points, indices, tripIds, tripCount) uploads the new geometry and resets the render object's visibility vector to all-visible, sized to the new trip count.

3Reconcile resolves each index's UUID against a QHash<QUuid, cwTrip*> built from the current live region (unresolved UUIDs — trip deleted mid-solve — are skipped, no dangling deref). For each present trip it creates the keyword item if new, re-binds the proxy to the trip's freshly renumbered running id (setTarget), and removes items for trips no longer present.

4Because step 2 reset everything to visible, reconcile only re-pushes the hidden trips: setTripVisible(id, false) for each proxy that is currently hidden — so filter state survives the re-solve even though the dense ids were renumbered. No bulk flag, no full-array rebuild.

7½. Robustness — why the boring parts are handled

The architecture above is straightforward; the failure modes live in the seams (cross-file mapping, async lifetime, re-entrancy). These three measures close them.

① Identity, not position

The earlier draft aligned the per-vertex running index with an ordered QVector<cwTrip*> built by iterating RegionDataPtrs in "the same order" as the geometry. That's an unenforced positional contract across two files — sort or filter one path and visibility silently maps to the wrong trips, with no crash and no failing test.

Fix: cwLinePlotGeometry already sees cwTripData::id (a QUuid), so it emits an index → UUID table directly. The manager resolves UUID → live cwTrip* by identity. Order is irrelevant; the mapping can't drift. This also deletes the fragile positional tripsByIndex step from the task.

② No live pointers cross the async boundary

The solve runs on a worker and the result arrives later; a trip can be deleted in between. Carrying raw cwTrip* would dangle (the existing code needs validateResultsData() for exactly this reason).

Fix: only QUuid values travel async. Live pointers are obtained on the main thread, at use time, from the current region — a deleted trip just fails to resolve. The proxy stores a QPointer<cwTrip>, and each trip's destroyed() is connected to prompt entry removal (a trip delete also triggers a re-solve, so reconcile is the backstop).

③ Incremental visibility instead of a re-entrancy guard

An earlier draft routed every toggle up through the manager, which rebuilt a whole QVector<quint8> flag array and re-pushed it. That made addItem() → resolveVisibility() → proxy setVisible() trigger a full rebuild per item during reconcile, forcing a m_bulkUpdating flag to suppress the storm — a guard that only existed to paper over the design.

Fix: visibility lives on the render object (like cwRenderTexturedItems for scraps). Each toggle is an O(1) setTripVisible(id, bool) that flips one byte; Qt RHI coalesces the resulting update() calls into one upload per frame. No flag array is rebuilt, so there is nothing to storm and no guard to add. After a solve, setGeometry resets to all-visible and reconcile re-pushes only the hidden trips.

Dense trip ids change every solve — that's expected. Identity (UUID) is the stable key; the dense index is just a compact per-snapshot encoding for the vertex attribute and texture column.

9. Tests

All C++ (testcases/), since RHI rendering isn't exercised offscreen. We verify the data and wiring that drive the GPU, not pixels.

  • Geometry de-sharing — a station shared by two trips yields two vertices with distinct trip ids; a station reused within one trip yields a single vertex; tripIds.size() == points.size(); ids cover [0, tripCount).
  • Identity mapping (guards #1) — for a known dataset, the vertex carrying a given trip's station has the trip id whose tripUuids[id] equals that trip's UUID; reorder the caves/trips and re-solve → the same trip still resolves to the same live cwTrip*. This is the regression test for the positional-drift bug.
  • Async lifetime (guards #2) — delete a trip before the solve result is applied → its UUID fails to resolve and is skipped; no dangling deref (clean under ASan), no leftover keyword item.
  • Bulk push (guards #3) — adding many trips on one re-solve results in exactly one visibility push, not one per trip. Assert via a SignalSpy on the render object's visibility-changed signal (no test-only counters).
  • Manager registration — one keyword item per trip in the model; each item's model extends trip->keywordModel() and publishes Type = Line Plot; items removed on trip delete.
  • Visibility flow — toggling a trip's proxy setVisible(false) sets that trip's byte to 0 in cwRenderLinePlot::tripVisibility(); state is preserved after a re-solve.

Note: the GPU masking itself (vertex texture fetch, off-clip collapse, the new uint attribute binding) can't be unit-tested offscreen — it stays manual verification. The data tests above pin everything that feeds it, including the previously-fragile mapping.

Build & run both suites per CLAUDE.md, teeing output to /tmp logs. New tests first; full suite once at the end.

10. Known follow-ups (out of scope)

  • Picking — the centerline is not pickable at all today: the CPU BVH (cwGeometryItersecter) skips Lines primitives (countNodePrimitives() returns 0). Making the centerline pickable (ray–segment proximity test with a pick radius) is tracked separately in issue #530 (Summer 2026). When that lands, per-trip pick filtering reuses this plan's per-segment trip id + visibility flags so hidden trips' centerline can't be picked. Vertex duplication from de-sharing is harmless to the BVH.
  • Trip-count ceiling — none; the R8 texture sizes dynamically to the trip count.
