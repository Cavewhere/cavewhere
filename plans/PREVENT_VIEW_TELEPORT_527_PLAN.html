Issue #527 — Prevent the view from teleporting on a missed intersection

CaveWhere · 3D turntable interaction · pivot/orbit-center selection
Plan — not yet implemented

1. The problem

When the user starts a rotate or pan in the 3D view, the code casts a ray from the cursor into the scene and uses the hit point as the new orbit/pan center. If the cave geometry is small and far from the world origin (the common case — surveys live in real-world UTM-ish coordinates), clicking on or near empty space produces a bad center and the view teleports to somewhere off-screen with nothing in it. The user is forced to hit Reset.

The whole bug lives in one function:

cavewherelib/src/cwBaseTurnTableInteraction.cpp · cwBaseTurnTableInteraction::unProject(QPoint) (lines 99–131)

QVector3D cwBaseTurnTableInteraction::unProject(QPoint point) {
    auto ray = Camera->frustrumRay(point);

    double t = scene()->geometryItersecter()->intersects(ray);   // NaN on miss
    const bool geometryHit = !std::isnan(t);

    if(std::isnan(t)) {
        t = m_gridPlane.intersection(ray);     // infinite plane — almost always hits
        if(std::isnan(t)) {
            return QVector3D();                 // ← teleport bug #1: returns (0,0,0)
        }
    }
    return ray.point(t);                        // ← teleport bug #2: unbounded grid hit
}

Two distinct teleport triggers

#TriggerWhat happens
1 Ray misses geometry and grid plane (ray parallel to plane / pointed at sky) unProject returns (0,0,0). The caller does setCenter((0,0,0)); rotation now orbits the world origin — a violent jump, since the cave may be thousands of metres from origin.
2 Ray misses the thin line/point cave geometry but hits the infinite grid plane at a grazing angle The grid-plane hit can be hundreds or thousands of metres away from the actual cave. setCenter() moves the pivot there → the next rotate frame swings the camera around that far point and the cave leaves the screen. This is the "grid is far away" case from the issue title.

Where the bad center is consumed

  • startRotating() (line 215): if(!m_centerLocked) setCenter(unProject(position));
  • startPanning() (line 163): clickWorld = unProject(...) → setCenter() + m_panAnchor = clickWorld + builds PanPlane through it.
  • bindPerspectiveIntersection() (line 150): zoom-toward-cursor uses unProject(m_perspectiveMappedPos) as the zoom target.

All three currently trust whatever unProject returns, including the (0,0,0) sentinel and the unbounded grid hit.

2. The fix idea (uses the AABB already in cwGeometryItersecter)

The user's instinct is right: bound the fallback with the scene's axis-aligned bounding box. The intersecter already computes a world-space AABB — the root box of the top-level BVH (BvhData::topLevel[0].bbox), and per-object boxes are exposed via boundingBox(const Key&). We expose the union box and use it to:

  1. Reject a grid-plane hit that lies far outside the cave's footprint (kills trigger #2).
  2. Provide a sane fallback pivot instead of (0,0,0) (kills trigger #1).
  3. Tell callers a pick was a miss so they can keep the current center rather than teleporting at all.
Design principle: a click on empty space should never move the pivot to a worse place than it already is. The safest no-teleport behavior is: on a confident geometry hit, re-center; on anything else, keep the existing center (and for pan, anchor the drag to the current center's depth so it still tracks smoothly).

3. Implementation steps

1Expose a scene-wide AABB on the intersecter

cwGeometryItersecter.h / .cpp

Add a const accessor returning the world-space union box of all objects. Prefer the already-built top-level BVH root box when the BVH is ready; otherwise fold over the live Nodes list (each Node::BoundingBox transformed by its model matrix — same as the existing boundingBox(Key)).

// Union of every object's world-space box. Null box when the scene is empty.
QBox3D boundingBox() const;

This is cheap and reuses existing data — no new traversal of triangles. QBox3D already provides unite(), contains(QVector3D), center(), size(), and intersection(QRay3D), which we lean on below.

2Make unProject report success/failure instead of a sentinel

cwBaseTurnTableInteraction.cpp / .h

Change the return type so a miss is unambiguous (no more overloaded (0,0,0)):

- QVector3D unProject(QPoint point);
+ std::optional<QVector3D> unProject(QPoint point) const;

New fallback chain inside the body:

auto ray = Camera->frustrumRay(point);

// 1) Confident hit on real geometry — always trust it.
double t = scene()->geometryItersecter()->intersects(ray);
if(!std::isnan(t)) return ray.point(t);

// 2) Grid-plane fallback, but only if the hit is near the cave.
const QBox3D sceneBox = scene()->geometryItersecter()->boundingBox();
double gt = m_gridPlane.intersection(ray);
if(!std::isnan(gt)) {
    const QVector3D gridHit = ray.point(gt);
    if(acceptFallbackPoint(gridHit, sceneBox)) {   // AABB gate, see step 3
        return gridHit;
    }
}

// 3) No usable pick.
return std::nullopt;

3AABB gate for fallback points

A small helper decides whether a non-geometry point is "near enough" to the cave to be a legitimate pivot. Grow the scene box by a margin (the issue is about far hits, so a generous margin is fine — e.g. one box-diagonal, or a fixed multiple of the box size) and test containment:

namespace {
    // Fallback pivots (grid plane, etc.) are accepted only inside the scene
    // box grown by this fraction of its own diagonal. Keeps a grazing grid
    // hit kilometres away from becoming the orbit center.
    constexpr float kFallbackBoxMargin = 1.0f;
}

bool acceptFallbackPoint(const QVector3D& p, const QBox3D& sceneBox) {
    if(sceneBox.isNull() || !sceneBox.isFinite()) return false; // empty doc — never re-center far
    const float pad = sceneBox.size().length() * kFallbackBoxMargin;
    QBox3D grown(sceneBox.minimum() - QVector3D(pad, pad, pad),
                 sceneBox.maximum() + QVector3D(pad, pad, pad));
    return grown.contains(p);
}
Tune kFallbackBoxMargin during review. Start permissive (1.0 = grid hits within one scene-diagonal of the cave are fine); tighten only if teleports still slip through. Extract it as a named constexpr per the no-magic-numbers rule.

4Callers keep the current center on a miss (the actual no-teleport behavior)

cwBaseTurnTableInteraction.cpp

startRotating — only re-center on a real pick:

void cwBaseTurnTableInteraction::startRotating(QPoint position) {
    position = Camera->mapToGLViewport(position);
    if(!m_centerLocked) {
        if(auto picked = unProject(position)) {
            setCenter(*picked);
        }
        // else: keep m_center — orbit around the existing pivot, no jump.
    }
    LastMousePosition = position;
}

startPanning — on a miss, anchor the drag to the current center's depth so panning still feels attached to the cursor and nothing teleports:

const QPoint mapped = Camera->mapToGLViewport(position);
QVector3D anchor;
if(auto picked = unProject(mapped)) {
    anchor = *picked;
    if(!m_centerLocked) setCenter(anchor);
} else {
    // Fallback: intersect this ray with the plane through the current center,
    // facing the camera. Keeps the pan smooth and the pivot put.
    anchor = rayPointAtCenterDepth(mapped);   // helper using m_center + view dir
}
m_panAnchor = anchor;
// ... build PanPlane through m_panAnchor as today ...

bindPerspectiveIntersection — zoom target falls back to the current center instead of (0,0,0):

m_perspectiveIntersection.setBinding([this]() -> QVector3D {
    return unProject(m_perspectiveMappedPos).value_or(m_center);
});

5(Optional, stronger) Ray-vs-AABB fallback pivot

If reviewers prefer empty-space clicks to still re-center onto the cave rather than do nothing, replace "keep current center" with a ray-vs-box pivot: when geometry misses, intersect the ray with the scene AABB (QBox3D::intersection(ray, &tMin, &tMax)) and use the midpoint of the entry/exit span. This pulls the pivot onto the cave's bounding volume even when the precise line geometry was missed. Keep this behind the same AABB gate so a total miss still returns nullopt. Recommend landing step 4 first (minimal, obviously correct), then evaluating step 5 as a UX refinement.

5. Testing

There is already testcases/test_cwBaseTurnTableInteraction.cpp and a rich set of test_cwGeometryItersecter*.cpp files to extend.

  • Intersecter unit test: add objects far from origin; assert boundingBox() equals the union of their world boxes; empty intersecter returns a null box.
  • Teleport regression (trigger #1): scene with geometry centered far from origin; call startRotating on a screen point whose ray misses geometry and the grid plane; assert center() did not become (0,0,0) and stayed at its prior value (within the AABB).
  • Teleport regression (trigger #2): camera at a grazing angle so the ray hits the infinite grid plane far from the cave; assert the far grid hit is rejected and center() stays put.
  • Happy path unchanged: a ray that genuinely hits geometry still re-centers on the hit point.
  • Keep tests safe for concurrent processes (PID in temp names / QTemporaryDir), per project test rules. The BVH is CPU-side, so these run offscreen without RHI.

  2. Zoom in on a passage, then rotate/pan by grabbing empty space just off the passage — confirm the view orbits the passage instead of jumping away.

7. Open questions for review

  • Step 4 vs Step 5: do we want empty-space clicks to keep the current pivot (minimal) or snap onto the AABB (more "helpful" but more behavior change)? Recommend shipping step 4 and treating step 5 as a follow-up.
  • Margin value for kFallbackBoxMargin — needs a couple of real files to tune.
  • Grid plane intent: should the grid plane fallback exist at all once the AABB gate is in, or is it only meaningful for an empty document (no geometry yet)? Worth confirming the grid is still useful as a pivot source when there is geometry.
