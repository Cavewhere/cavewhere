4.3 — Flow-aligned point symbol

A single glyph dropped at one anchor and rotated to the local stroke tangent — the oriented counterpart of the existing un-oriented point stamp. The headline use case is a stream-flow arrow: the cartographer draws a short stroke in the flow direction, a single arrowhead lands at its start, pointing downstream. Commit 4.3 of the symbology-palette 4.x series; see SYMBOLOGY_PALETTE_4X_ROADMAP.md and SYMBOLOGY_PALETTE_PLAN.html Decision 11.

Bottom line: 4.3 needs no new production code. The three rules it composes (Explicit point → Align to tangent → Rigid stamp) all exist and are all registered. 4.3 is a stack composition plus its verification: one unit test and one reference-image panel. The only real decision is whether the oriented point also ships as a real seed brush or stays a test-only demo (§5).

The realization: it is the point stamp, plus one rule

The decoration engine already ships an un-oriented single-point stamp. In test_cwSketchDecorationLayout.cpp it is the pointStampLayer helper:

// A single glyph at the stroke start (no tangent rotation).
cwDecorationLayer pointStampLayer(const QString &glyphName)
{
    return ruleLayer(glyphName, {QStringLiteral("Explicit point"),
                                 QStringLiteral("Rigid stamp")});
}

That stack seeds one stamp (Explicit point, at arclength 0) and materialises it (Rigid stamp), but never sets its rotation — so the glyph always draws at its authored orientation regardless of which way the stroke runs. The reference-image suite calls this out today, in the very caption of the 45° stream panel:

"stream 45deg — line follows, but the single arrow is NOT tangent-rotated"

4.3 closes exactly that gap. Insert the Align to tangent rule (which already exists and already runs at the MutatePerLayer stage, between Generate and the Terminal) and the single stamp picks up the local tangent angle:

Generate
Explicit point
one stamp at stroke start
→
MutatePerLayer the +1 rule
Align to tangent
rotation = local tangent angle
→
Terminal · Stamps
Rigid stamp
scale·rotate·translate glyph
⇒
Output
one oriented glyph
flow-aligned point

Why this works unchanged — the verification trace

Each piece already behaves correctly for the single-position case; nothing assumes "many stamps":

RuleStageBehaviour relevant to a lone stamp
cwExplicitPointRule Generate Appends one cwStampPosition at strokePath.pointAtArcLength(0.0), glyph = layer.glyphName. (Anchor is the stroke's first point until rule params carry an explicit world anchor — fine for iter 1.)
cwAlignToTangentRule MutatePerLayer Loops over all positions (so one is fine) and sets rotationRad = atan2(t.y, t.x) where t = strokePath.tangentNearPoint(anchorWorld).
cwRigidStampRule (via cwStampRuleBase) Terminal · Stamps Materialises each visible position: scale → rotate (CCW, world +Y up) → translate to anchor. Already used by both repeated and single-glyph stacks.

The one thing worth confirming was the tangent at the stroke's start point — does tangentNearPoint degenerate there? It does not. tangentNearPoint(worldPoint) → closestSegment, which for the start point returns segment 0 with t = 0, then returns the normalised points[1] − points[0] — the direction of the first segment. So for a diagonal stroke (0,0) → (0.57,0.57) the tangent is the unit (1,1) and rotationRad = atan2(0.707, 0.707) = π/4. Exactly the desired flow orientation.

Convention note. Align to tangent is absolute alignment, not a relative spin: a glyph authored along +X ends up along the stroke; its +Y lands on the +normal (left) side. So a flow-arrow glyph should be authored pointing along +X (downstream) in glyph-local paper-mm, matching rail-tie/floor-step conventions.

Scope

Production code

no change No new rule, no registry edit, no proto change, no protoVersion() bump, no layout-engine change. The capability is emergent from the existing rule set. This plan exists to verify and document that, and to add the demo, not to add machinery.

Tests (in testcases/test_cwSketchDecorationLayout.cpp, tag [cwSketchDecorationLayout])

  1. New helper, alongside the existing pointStampLayer, making the contrast explicit: 
    // A single glyph at the stroke start, rotated to the local flow direction.
cwDecorationLayer orientedPointStampLayer(const QString &glyphName)
{
    return ruleLayer(glyphName, {QStringLiteral("Explicit point"),
                                 QStringLiteral("Align to tangent"),
                                 QStringLiteral("Rigid stamp")});
}
  2. New functional case — "Flow-aligned point orients a single stamp to the stroke tangent":
    • One layer, one visible stamp (count == 1); kind == Stamps.
    • Anchor at the stroke start (matches pointAtArcLength(0)).
    • Diagonal 45° stroke → position.rotationRad ≈ π/4 (the assertion the existing un-oriented helper would fail).
    • Horizontal stroke → rotationRad ≈ 0 (sanity: same as the un-oriented case there).
    • A reversed stroke (running −X) → rotationRad ≈ ±π, proving the arrow flips with flow direction.
    Pure compute, no temp files — safe under concurrent test processes (repo convention).
  3. Optional contrast assertion: same glyph + same diagonal stroke through pointStampLayer vs orientedPointStampLayer → first has rotationRad == 0, second has ≈ π/4. Documents the difference 4.3 introduces in one place.

The one decision: seed brush, or test-only demo?

Should a flow-aligned point ship as a real brush in cwSymbologyPaletteSeed, or stay a test-only demo like the 4.5/4.6 railroad and ceiling-channel?

  • Test-only precedent — matches the explicit 4.5/4.6 decision ("demos are test-only, not in the seed … so we don't ship a half-mirrored demo glyph"). 4.3 stays pure compute+demo; no seed churn, no committing a stream-arrow glyph we may want to redesign. The capability is fully proven by the unit test + panel.
  • Seed brush — a downstream flow arrow is a genuinely standard cave-map symbol (unlike the railroad, which was only ever a stress test). Shipping a water-flow seed brush would make 4.3 the first 4.x feature a user could actually pick. But nothing renders the seed to screen until commits 5–6, so a seed brush buys no user-visible value yet and does commit a glyph + brush to the wire format ahead of the editor. Leans toward deferring the seed entry to the render-path work.

Recommendation: test-only for 4.3 (follow the 4.5/4.6 precedent); revisit a real water-flow seed brush when the render path (5–6) makes seed brushes user-visible. Happy to ship it in the seed now if you'd rather — it's a few lines in cwSymbologyPaletteSeed.cpp plus a glyph.

Out of scope

  • Explicit world anchor for the point (place the symbol where the cartographer clicks, not at the stroke's first vertex). That needs an cwExplicitPointParams proto message + codec entry — the same rule-params recipe Commit B established for Offset stroke — and belongs with the editor (commit 9) or a small follow-up. 4.3 keeps the documented "first point" default.
  • Relative rotation offset (e.g. arrow at tangent + 90°). A future Rotate fixed MutatePerLayer rule from the iter-1 table; additive, independent of 4.3.
  • Mirror on side / glyph cycling / multi-point — separate iter-1-table rules.
  • Getting it on screen — render path, commits 5 → 6 (with the cwGlyphTessellationCache thread-safety prerequisite noted in the roadmap).
