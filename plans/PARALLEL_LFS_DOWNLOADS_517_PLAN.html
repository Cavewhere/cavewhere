Issue #517 — Parallel LFS downloads

CaveWhere · QQuickGit submodule · Git LFS object hydration after clone / reset
Stage 1 + Stage 2 + 1A committed in QQuickGit (7dcbae2, master) · cwCaptureManager leak fix committed in cavewhere (a1f1d7ab, dev_cavewhere) · main-repo gitlink bump + pushes pending · §10 follow-ups open
Scope: QQuickGit only

1. Goal

Speed up Git LFS object hydration. Today, after a clone or reset, missing LFS objects are fetched strictly one at a time, and each fetch makes its own separate Batch-API metadata round trip. For a repository with many LFS files (the common CaveWhere case — scanned notes and LiDAR are LFS-tracked), this stacks two independent latency costs and leaves the network mostly idle.

This plan delivers the speedup in two independently-shippable stages:

  • Stage 1  Single batch request — collapse N per-object Batch-API calls into one. Pure latency win, near-zero concurrency risk, minimal UI change.
  • Stage 2  Bounded-concurrency transfers — download up to K objects at once from that single batch response, with aggregate progress.

2. Where the work lives today

All orchestration is in the QQuickGit submodule. The CaveWhere app only wires up the auth provider (cavewherelib/src/cwRemoteServices.cpp:18 → LfsBatchClient::setLfsAuthProvider) and triggers hydration through the normal clone/reset paths — no app-side change is needed.

The current serial driver

GitRepository.cpp:1421–1485 — a hand-rolled recursive lambda:

auto pointers   = std::make_shared<QVector<LfsPointer>>(plan.missingPointers);
auto nextIndex  = std::make_shared<int>(0);
auto step       = std::make_shared<std::function<void()>>();

*step = [...]() mutable {
    if (*nextIndex >= pointers->size()) { deferred.complete(hydratePointerFiles(plan)); return; }
    const LfsPointer pointer = pointers->at(*nextIndex);
    (*nextIndex)++;
    auto fetchFuture = store->fetchObject(pointer, QString(), progressCb);   // batch + download, one object
    AsyncFuture::observe(fetchFuture).context(context, [...]() mutable {
        ...
        (*step)();                                                          // next object only AFTER this one finishes
    });
};
(*step)();

So N missing objects ⇒ N Batch-API POSTs and N sequential downloads. The transfer engine itself is non-blocking; only this orchestration serializes it.

3. Two distinct wins

Win A — one batch call

The Git LFS Batch API is designed to take many objects per POST. We currently send one object per POST. With 50 LFS files that is 49 wasted HTTPS+auth round trips, independent of bandwidth.

Delivered by Stage 1. Biggest win on high-RTT links and many-small-file repos.

Win B — concurrent transfers

After one batch response, fire up to K downloads at once instead of awaiting each. StreamWriter already writes isolated temp files, so concurrent writers never collide.

Delivered by Stage 2. Biggest win when per-object latency dominates throughput.

Natural cap already exists. Every request goes through one QNetworkAccessManager (LfsBatchClient::mManager), which by default limits to 6 concurrent HTTP connections per host. So "parallel" is inherently a small bounded pool — but we still add an explicit limit (Stage 2) so behaviour is intentional and not an accident of Qt defaults.

4. Stage 1  Single batch request

Replace the per-object batch round trips with one batch call covering all missing pointers, then download serially from the response (keeping today's transfer behaviour). This isolates the metadata win from the concurrency change so it can ship and be verified on its own.

4.1 New batch-aware fetch on LfsStore

fetchObject bakes in the single-spec batch call (LfsStore.cpp:449), so it cannot be reused for a multi-object batch. Add a sibling that performs only the transfer once the caller already holds a resolved download Action: 

// LfsStore.h — new public method
QFuture<Monad::ResultBase> downloadResolved(
        const LfsBatchClient::Action& action,
        const LfsPointer& pointer,
        std::function<void(qint64,qint64)> progressCallback = {}) const;

Its body mirrors the second half of fetchObject (LfsStore.cpp:446–490): build a dedicated downloadStore (never capture this across the async chain — see the existing comment at LfsStore.cpp:444) and call client->downloadObject(action, *downloadStore, pointer, progressCallback). Keep the original fetchObject for the single-object callers (e.g. the smudge-filter fallback path) so their behaviour is unchanged.

4.2 Rewrite runLfsHydrationPipeline — batch once, then serial download

  1. Build one QVector<LfsBatchClient::ObjectSpec> from plan.missingPointers.
  2. Construct a single LfsBatchClient for plan.gitDirPath and call batch("download", specs) once.
  3. From the BatchResponse, build an oid → Action map (read the actions["download"] href/headers per object; record per-object batch errors).
  4. Drive the existing recursive step loop, but call the new downloadResolved(action, pointer, …) instead of fetchObject.
  5. Finish exactly as today: deferred.complete(hydratePointerFiles(plan)).

4.3 Error & fallback semantics

Two layers, kept faithful to today's behaviour:

  • Whole-batch failure (no remote / offline / auth). Today a per-object NoRemote triggers shouldFallbackForFetchError (LfsStore.cpp:512, currently only NoRemote) and the loop continues, ultimately falling back to leaving pointer files in place via hydratePointerFiles. With a single batch, that decision moves up: if the batch call itself returns a fallback-eligible error, skip all downloads and go straight to hydratePointerFiles(plan) (offline clone still succeeds with pointer files on disk). A non-fallback batch error completes the deferred with that error, as today.
  • Per-object error inside a successful batch (e.g. a single 404 → NotFound/Transfer, see LfsStore.cpp:475). Preserve current semantics: that object is skipped (its pointer file stays), and the remaining objects continue. Surface it the same way the per-object path does today.
Decision to confirm: today a fatal (non-fallback) error on object i aborts the whole hydration (GitRepository.cpp:1478). Keep that "first fatal error wins" rule in Stage 1 for behavioural parity; revisit only if we want best-effort hydration.

5. Stage 2  Bounded-concurrency transfers

With the single batch response from Stage 1 already in hand, replace the serial download loop with a fixed-size worker pool.

5.1 Concurrency control

  • Add a file-scope constexpr int kDefaultLfsConcurrentTransfers = 8; (matches git-lfs's lfs.concurrenttransfers default) in the anonymous namespace of GitRepository.cpp.
  • Optionally read lfs.concurrenttransfers from the repo's git config to honour user intent; fall back to the constant. (Nice-to-have — can land later.)
  • Keep the limit ≤ what one QNetworkAccessManager will run per host anyway; oversubscription just queues inside Qt.

5.2 Fan-out / completion-counter pattern

Replace the recursive single-index step with a pool driver. Shared state lives in shared_ptrs captured by the per-download continuations (same lifetime approach as today):

auto queueIndex   = std::make_shared<int>(0);        // next object to start
auto inFlight     = std::make_shared<int>(0);        // active downloads
auto completed    = std::make_shared<int>(0);        // finished (ok or skipped)
auto firstError   = std::make_shared<Monad::ResultBase>();
auto pump = std::make_shared<std::function<void()>>();

*pump = [...]() mutable {
    while (*inFlight < kConcurrency && *queueIndex < total && !firstError->hasError()) {
        const int i = (*queueIndex)++;
        (*inFlight)++;
        auto f = store->downloadResolved(actions[i], pointers[i], aggregateProgressCb(i));
        AsyncFuture::observe(f).context(context, [...]() mutable {
            (*inFlight)--; (*completed)++;
            // record fatal error / honour per-object skip exactly as Stage 1
            if (*completed == total && *inFlight == 0)
                deferred.complete(firstError->hasError() ? *firstError
                                                         : hydratePointerFiles(plan));
            else
                (*pump)();                              // refill freed slot
        });
    }
};
(*pump)();
  • Objects with a per-object batch error never enter the queue (counted as skipped up front), so the counter still reaches total.
  • On a fatal error, stop seeding new downloads; let in-flight ones drain, then complete with the stored error. (Matches Stage 1's "first fatal error wins".)

5.3 Aggregate progress

Today's progress is inherently serial: currentObject is incremented before the fetch and the text reads "Downloading LFS: file (3/50) - 1.2/4.0 MB" (GitRepository.cpp:1379–1394, 1445). With unordered parallel completion, "object i of N" is meaningless. Switch to aggregate bytes:

  • Pre-compute totalBytes = Σ pointer.size over missingPointers.
  • Keep a shared_ptr<QHash<QString,qint64>> of oid → bytesReceived; each object's progressCallback updates its own entry and reports Σ entries / totalBytes plus a count of completed objects.
  • New text, e.g. "Downloading LFS objects — 12/50 files, 34.1/120.0 MB". Add a formatLfsAggregateProgressText() helper alongside the existing formatLfsDownloadProgressText; the latter stays for the serial single-object fallback path.
  • The LfsHydrationProgress signature (GitRepository.cpp:1358, (text, current, total)) is unchanged — pass current = completedFiles, total = fileCount so existing UI consumers keep working.

6. Thread-safety & lifetime review

  • Per-object isolation. Each downloadObject owns its own StreamWriter → unique temp file (LfsStore.h:67, LfsBatchClient.cpp:526–531). Concurrent writers cannot collide.
  • Single network manager. All GETs share mManager on the GUI thread; QNetworkAccessManager is designed for many concurrent replies. No new threads are introduced — this is event-loop concurrency, not data-parallelism.
  • Store registry. LfsStoreRegistry lookups are mutex-guarded (LfsStore.cpp); unaffected.
  • SSH auth cache — watch item. LfsBatchClient caches SSH-derived auth in CachedSshAuth (LfsBatchClient.h:79). With a single batch call (Stage 1) the cache is populated once before any parallel download — Stage 2 reuses it, so there is no thundering-herd of concurrent SSH auths. This is a concrete reason to keep batch-first ordering and not parallelize the batch step.
  • No this capture across async. Continue the existing discipline (LfsStore.cpp:444): capture shared_ptr stores, never the LfsStore*. downloadResolved must follow the same rule.

8. Risks & mitigations

RiskMitigation
Behavioural drift in error/fallback handling when collapsing per-object batches.Stage 1 keeps the exact fallback rule (batch-level NoRemote → pointer files; first fatal error wins). Covered by the offline and 404 tests above.
Progress UI regressions for downstream consumers reading (text, current, total).Keep the callback signature; only the meaning of current/total shifts to files-completed/file-count, which existing bars already render sanely.
Server rate-limiting / connection exhaustion under parallelism.Explicit K cap (default 8), bounded further by QNAM's per-host limit. Configurable via lfs.concurrenttransfers.
Memory / disk pressure from K simultaneous streams.Streaming writers hold only a chunk in memory; K temp files is bounded and small. No full-object buffering (verified: test_Lfs.cpp:2181 "streams large files without buffering").
Cross-platform temp-file contention.Each StreamWriter uses a unique temp path; finalize is atomic-rename style. No shared target until copy in hydratePointerFiles (post-download, serial).

9. Sequencing & recommendation

  1. 1Ship Stage 1 alone. Single batch request + serial download. Big latency win, no concurrency risk, no progress rework. Land + verify on a real many-LFS clone.
  2. 2Then Stage 2. Bounded pool + aggregate progress on top of Stage 1's batch response.
Why batch-first matters beyond latency: doing the single batch call before any download means SSH auth is resolved and cached exactly once, so the parallel phase never races on auth. Stage 1 is therefore a prerequisite for a clean Stage 2, not just an optimization.

Net new code is small and contained to QQuickGit: one new LfsStore method, a rewrite of one driver function, one progress-formatting helper, and tests. No CaveWhere app changes, no protobuf/schema changes, no submodule API breakage for existing single-object callers.

10. Follow-ups (out of scope for this change)

Surfaced by the pre-commit review after Stage 1 + Stage 2 landed. None block shipping the parallel-download fix. Items tagged approved have been signed off as planned work; sequence them after this change has baked.

1A — Extract a reusable async-iterate / bounded-concurrency helper implemented

Done. Added two function templates in the anonymous namespace of QQuickGit/src/GitRepository.cpp: asyncPumpBounded(items, maxInFlight, context, perItem) (the single driver) and asyncForEach(items, context, perItem) (a window-of-one wrapper). perItem maps one item to a GitFuture; the driver stops on the first error and otherwise completes once every item finishes — perItem alone decides whether a soft failure aborts (return error) or is swallowed (return success). All three former hand-rolled pumps now route through it: runLfsBatchRequests (serial chunks via asyncForEach, accumulator caller-side), runLfsUploadActions (serial uploads via asyncForEach), and runLfsDownloadActions (bounded-parallel via asyncPumpBounded; the old LfsDownloadRun shrinks to a small DownloadAggregate holding only the store / byte tallies / an aborted latch). The reference-cycle break now lives in one place — the driver's single queued releasePump() — so releaseDownloadRunPump() and the releaseStep local are gone. Verified: full QQuickGit suite green under ASan (134 cases / 4164 assertions), [LFS] (55 cases) and the hidden [lfs-batch-cap] boundary tests (1440 assertions) all pass.

The recursive shared_ptr<int> nextIndex + shared_ptr<std::function<void()>> step + AsyncFuture::observe(...).context(...) re-enter shape now appears three times in QQuickGit/src/GitRepository.cpp:

  • runLfsBatchRequests — serial chunk loop (added by this change)
  • runLfsUploadActions — serial per-object upload loop
  • runLfsDownloadActions / LfsDownloadRun — the new bounded-concurrency pump

The idiom is fiddly and self-referential — capture lists, the finished guard, and exactly-once completion are easy to get subtly wrong, and the three copies have already diverged (only the new pump carries a finished flag). Extract two internal helpers in an anonymous namespace: 

// serial: run each item through perItem, stop on first error, accumulate.
GitFuture asyncForEach(items, perItem);
// windowed: keep up to maxInFlight perItem calls running at once.
GitFuture asyncPumpBounded(tasks, maxInFlight, perItem);

This collapses all three call sites to a few lines each and gives one place to get cancellation / finished / lifetime semantics right — including the reference-cycle leak below, now fixed in two ad-hoc spots that should collapse into the helper's single cycle-break.

Blast radius: ~3 functions, internal linkage (anonymous namespace), no public API or header change. Fully covered by the existing [LFS] suite. Low–medium risk; do it after this fix has baked so the refactor is reviewable on its own.

Reference-cycle leak — fixed in place (was carried over from the pre-existing idiom). Both LfsDownloadRun::pump and runLfsBatchRequests::step captured their own owning shared_ptr, leaking the run state once per hydration (every pull/fetch/clone). Now broken on every terminal path: releaseDownloadRunPump() (and a releaseStep local in runLfsBatchRequests) null the self-referential std::function via a QMetaObject::invokeMethod(context, …, Qt::QueuedConnection) post — queued because you cannot destroy a std::function from inside its own running body (UB); the captured shared_ptr keeps the state alive until the reset fires. Verified: full QQuickGit suite green under ASan (134 cases / 4152 assertions), no use-after-free.

Carry-forward for 1A: when the shared asyncForEach/asyncPumpBounded helper lands, fold both reset paths into it so the cycle-break lives in one place instead of the two ad-hoc copies that exist now.

1B — Make the 100-object batch cap configurable approved

kMaxObjectsPerBatchRequest = 100 is GitHub's hard limit (it returns HTTP 413 "More than 100 objects specified"). GitLab allows far more (default 100,000), Bitbucket documents no cap — so on those hosts we issue needless extra round trips. Read a per-repo key (e.g. lfs.batch.maxObjects, or mirror git-lfs's own transfer.maxbatchsize) and thread it into runLfsBatchRequests. The config-read infrastructure already exists in LfsBatchClient (git_config_get_string_buf). Enhancement, not a fix — defaulting to 100 is correct and safe everywhere today.

Testing — wire the chunk-boundary tests into CI

The two [.][lfs-batch-cap] tests (101 objects → ≥2 batch requests, download and upload) are hidden because committing many LFS files trips the pre-existing clean-path heap-corruption race (issue #546). They therefore don't run in the default nightly suite, so the actual >100 chunk boundary — the whole point of the 413 fix — has no default coverage and could silently regress. Add a dedicated serial CI job that runs [lfs-batch-cap], gated behind / after a fix for #546. Until then the tests commit one-file-per-commit to dodge the race (do not batch — see the in-test comments).

2A — Parallelize the upload path & surface pull/merge progress (#545) approved

runLfsUploadActions still uploads serially; push of a large LFS set is the symmetric slow case this change's pump model directly applies to. Separately, #545: pull() / pullRebaseOrMerge() route hydration through runMergeHydrationForDirectory with an empty progress callback, so they show no LFS download progress. The aggregate-progress model added here (formatLfsAggregateProgressText, LfsDownloadRun) is exactly what's needed — the remaining work is threading a callback down through the merge/checkout/reset hydration call sites. Each is its own ticket with its own tests.

2B — Promote LFS orchestration into its own subsystem approved

The LFS pipeline now spans ~700+ lines across ~14 call sites inside the 5000+-line GitRepository.cpp, yet only depends on LfsBatchClient, LfsStore, LfsHydrationPlan and AsyncFuture — a cohesive module wanting its own translation unit (e.g. LfsHydrator). A pure move-and-rename would shrink the giant file, make the orchestration independently testable, and give 1A/2A a natural home. High mechanical churn, low semantic risk; best done after 2A so the upload path moves in its final form. Covered by the existing [LFS] suite.

Promote the async helpers? — audited: keep LFS-local for now

Ran a codebase audit for other hand-rolled async-iteration / self-referential shared_ptr<std::function> pumps that could adopt asyncForEach / asyncPumpBounded. Findings:

  • Only two other pumps exist, both in cavewherelib. cwSaveLoad::syncProject() (cwSaveLoad.cpp:4850) is a retry/state-machine orchestrator, not item iteration — the helpers don't apply. cwCaptureManager::capture() (cwCaptureManager.cpp:245) is a serial signal-driven pump; a partial fit at best (signal-based, no error/result model, void completion — adapting it means wrapping each capture in a QFuture<ResultBase>, real work for a cosmetic dedup).
  • Dependency direction (correcting the audit's first pass): cavewherelib links QQuickGit (cavewherelib/CMakeLists.txt:332), so cavewherelib can include QQuickGit public headers — a QQuickGit-public helper would in fact be reachable. (Lowest-common home would be monad/ or a small common async header, since the helper only needs AsyncFuture + Monad::ResultBase.) So location is not the blocker; the lack of a genuine second consumer is.

Decision: keep the helpers anonymous-namespace-local in GitRepository.cpp. Promote only when a real second consumer materializes; doing it now would drag LFS-shaped semantics into a shared header for no payoff.
