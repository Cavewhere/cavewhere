AsyncFuture shield() — one-way cancellation barrier

Written 2026-07-17, out of the externalFiles commit-5 review. Touches two repos: the asyncfuture submodule (vpicaver fork — the primitive + its standalone unit tests) and cavewhere (one boundary fix + a regression test). Independent of the External File phase plans; Phase 2 §11 references it from the deferred-refactor notes.

1. The hazard: cancel propagates upstream into shared state

AsyncFuture deliberately propagates cancel() upstream through observation chains, in two places:

  • Private::execute() (the engine under observe(f).context(…) / .subscribe(…)) watches its own chain future and, when a consumer cancels it, calls futurePtr->cancel() on the observed future (asyncfuture.h:~1277).
  • DeferredFuture::complete(QFuture) installs pushCancel — "Pushes cancel to child futures in the chain" (asyncfuture.h:~597-611, now gated on CancelPropagation::Propagate) — so a deferred completed from a future forwards its own cancellation into that future.

Usually that is what you want (cancel a chain → the QtConcurrent worker future it observes gets marked). It becomes a defect when the chain bottoms out in a shared future — one handed out to many observers and used by its owner for bookkeeping. The concrete case (pre-fix): cwSaveLoad::pendingJobsFinished() returned the raw d->m_pendingJobsDeferred.future(), the project-wide job-drain future; commit 2 now shields it (cwSaveLoad.cpp:4436). QFuture copies share one QFutureInterface, so a cancel arriving through any derived chain marks the one shared object canceled:

  • cwSaveLoadPrivate polls m_pendingJobsDeferred.future().isFinished() to decide when to re-arm the drain cycle (cwSaveLoadPrivate.cpp:663, 1429) — a canceled future reads as finished while real jobs are still queued.
  • Every sibling observer of the same drain cycle (other cwExternalCenterlineSync::reconcile() calls, future save-flush waiters) sees its chain cancel spuriously.

Exposure before the fix: any caller that canceled the future returned by cwExternalCenterlineSync::reconcile() reached the shared deferred in one hop (the [SaveLoad][Shield] regression test demonstrates the hang). The attach orchestrator dodged it by hand with a value-completed AsyncFuture::Deferred firewall (cwExternalCenterlineAttach.cpp, see the comment on the Deferred), and commit 7's updateFromSource would have needed the same incantation. Per-consumer incantations are the same bug farm shape as the pre-refactor localMutationOccurred emits — commit 2 fixed it at the source instead, so updateFromSource needs no firewall of its own.

     results flow DOWN ▾                  cancel flows UP ▴

   [ caller / dialog ]
          │  future.cancel()
          ▾
   [ chain future        ]  ...........  cb skipped (no rollback / no completion)
          │  execute() watch: futurePtr->cancel()     asyncfuture.h:~1277
          ▾
   [ reconcile()'s chain future ]
          │  same watch, one level up
          ▾
   [ pendingJobsFinished() future ]  ◄── ONE shared QFutureInterface
          │
    ┌────────┤─────────────────┐
    ▾        ▾                    ▾
 cwSaveLoad   other reconcile     any other waiter
 re-arm poll  observers           on the drain cycle
 (canceled    (their chains       (spuriously
  reads as     cancel too)         canceled)
  finished!)


 WITH shield — propagation dies at the barrier
 ====================================================================

   [ caller / dialog ]
          │  future.cancel()
          ▾
   [ shield future ]   settles canceled; consumer's observers stop cleanly
          ▏
   ══════════════════  one-way glass: results/cancel forwarded DOWN,
          ▲                 nothing pushed UP (no pushCancel watch)
          │  forwarder watch only (finished / canceled / progress)
   [ pendingJobsFinished() future ]   isCanceled() == false, untouched
          │
    everything below runs exactly as if the cancel never happened

3. Semantics — and why the "design question" dissolves

Two candidate behaviors were on the table for what happens when a consumer cancels the future shield() returned:

A — "asyncio-style" (soft shield)B — hard shield (cancel is a no-op)
Consumer's view The shield future settles as canceled. Watchers get the canceled signal, chains built on it stop, the dialog can dismiss. Cancel means "stop notifying me". Cancel does nothing; the future later delivers the real result. Cancel means nothing.
Producer's view Untouched. Work runs to completion; its result is dropped into the already-canceled shield future (a guarded no-op). Untouched (same).

"asyncio-style" refers to Python's asyncio.shield(task): when the outer await is cancelled, the awaiting coroutine receives CancelledError (so the consumer side genuinely observes the cancellation and unwinds), while the inner task keeps running to completion. Cancellation is delivered to the consumer's view only; the producer is protected.

Resolution: option B is not implementable with QFuture, so the question answers itself. QFuture::cancel() calls the non-virtual QFutureInterfaceBase::cancel(), which flips the shared canceled flag synchronously. A library cannot veto or intercept that; it can only react afterwards (that is exactly what pushCancel is — a QFutureWatcher reaction). So the shield future will always read isCanceled() == true after a consumer cancel; the only thing shield() controls is whether the library reacts by pushing the cancel upstream. Declining to react is option A. A faithful option B would require a custom non-QFuture handle type — out of scope and worse ergonomics.

Contract

  • Consumer cancels shield future → shield future is canceled (unavoidable, and useful); upstream is untouched; the upstream's eventual completion is dropped (complete-on-finished-future is already a guarded no-op).
  • Upstream completes → result forwarded to the shield future.
  • Upstream cancels (or reports an exception) → forwarded downstream, matching complete(QFuture)'s existing forwarder behavior. Shield is one-way glass, not a result filter.
  • Progress forwarded via the existing track() machinery.
  • Shield gives the producer no visibility of consumer cancellation. Producers that want a cancel window (attach's scan-stage poll) keep a Deferred and poll future().isCanceled() — shield does not replace that pattern, it protects infrastructure from consumers.

4. API

// asyncfuture.h

// Public policy enum. Propagate is the existing behavior and the
// default; Blocked is the shield semantic. Naming note: this is NOT
// "cancellation prevention" — the returned future stays cancelable,
// the policy only controls whether that cancel is pushed upstream.
enum class CancelPropagation { Propagate, Blocked };

// Private::DeferredFuture<T>: complete(QFuture) grows a policy
// parameter instead of a second method name. Both policies install
// the forwarder watch (finished -> completeByFinishedFuture,
// canceled -> cancel) and track(future) for progress; Blocked
// deliberately does NOT install the second
// watch(this->future(), ..., pushCancel, ...).
void complete(QFuture<T> future,
              CancelPropagation policy = CancelPropagation::Propagate);

// Public: Deferred<T> and Deferred<void> forward the policy.
deferred.complete(future, AsyncFuture::CancelPropagation::Blocked);

// Public free function; the common consumer form.
template <typename T>
QFuture<T> AsyncFuture::shield(QFuture<T> future) {
    Deferred<T> d;
    d.complete(future, CancelPropagation::Blocked);
    return d.future();
}

Opt-in, not a default change. complete(QFuture)'s propagation stays as-is — the fork's Restarter work ("Propagate Restarter outer cancel to the inner worker", 6a5b05b) deliberately relies on propagation being the default. shield() is the explicit opt-out for boundaries where the observed future is shared.

5. Commit sequence

Commit 1 — asyncfuture fork: the primitive (landed: fork commit 7a8b768)

CancelPropagation policy parameter on complete(QFuture) in DeferredFuture, Deferred<T>, Deferred<void>; free AsyncFuture::shield(); fork README documentation for both (shield() section + the policy parameter on the Deferred::complete(QFuture) entry). Unit tests in the standalone asyncfuture suite (build recipe: see the repo's CI / existing test project; cavewhere's superbuild does not compile them). Gates (all covered by the new ShieldTests class):

  • Result forwarding: complete the inner deferred with a value → shield future finishes with that value.
  • Upstream cancel forwarding: cancel the inner → shield future is canceled.
  • The barrier itself: cancel the shield future → inner future's isCanceled() stays false; the inner deferred can still complete afterwards without crash or warning; a separate direct observer of the inner future still fires.
  • Shared-fan-out miniature of the cavewhere hazard: one Deferred<void>, two consumers each observing their own shield() of it; cancel consumer 1's shield future → consumer 2 still completes; the shared future is not canceled.
  • Progress forwarding via track() (progressValue/range reach the shield future).
  • Thread hop: inner is a QtConcurrent::run future completed on a pool thread → shield future completes on the consumer thread per existing watch() delivery rules.
  • Parity check: shielded vs plain complete(QFuture) behave identically in every respect except the downstream-cancel push (loop both variants through the same assertions).

Commit 2 — cavewhere: shield the shared drain future at the source (landed: 57b9ba16)

  • Bump the asyncfuture submodule; note the bump in the commit message per house rules.
  • cwSaveLoad::pendingJobsFinished() returns AsyncFuture::shield(completeSaveJobs()) — no caller-derived chain can reach m_pendingJobsDeferred any more.
  • Regression test (tag [SaveLoad][Shield]): enqueue external-centerline jobs, cancel the future returned by cwExternalCenterlineSync::reconcile() mid-flight → the queue still drains, a second reconcile completes normally, and waitSaveToFinish does not hang — i.e. the drain cycle survived a consumer cancel.
  • Keep the attach orchestrator unchanged: its Deferred still provides the scan-window cancel poll and the one-way-cancel contract; its firewall comment gains a note pointing at shield() as the infrastructure-level backstop.

Commit 3 (optional, default: skip) — attach internals as a flat chain

With the drain future shielded at the source, attach's internal chain could be rewritten flat (observe(scan).context(…) returning futures) with the Deferred kept only as the outer bridge via complete(…, CancelPropagation::Blocked). Pure taste — the current code is correct; only do this if the file is being touched anyway.
