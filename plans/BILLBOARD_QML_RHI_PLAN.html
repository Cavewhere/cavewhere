Depth-Occluded QML Billboards in the RHI Scene

Implementation plan — rendering live QML Items as billboards that participate in the 3D depth buffer.

Issues: #535 (Lead visibility, keywords, overlap, z-order) · #536 (Station labels should be billboarded)

Date: 2026-06-16 · Status: research + spike complete, decisions settled — implementing Phase 0 next

1. Problem

Station labels (Label3d.qml via cwLabel3dView) and lead markers (LeadPoint.qml via cwLeadView) are plain Qt Quick Items anchored over the RHI 3D view and positioned by CPU projection (cwTransformUpdater::updatePoint, cwLabel3dView::updateGroupPositions). They composite on top of the 3D color output as ordinary 2D items, so they always draw over the geometry — a label behind a passage wall is not hidden. Both issues ask for the same thing: markers that are occluded by cave geometry via the depth buffer ("billboarded so they use the z-buffer"). The markers already track their 3D point and face the camera; the missing piece is depth occlusion + z-ordering.

Why it can't be done in the QML layer. A QQuickRhiItem exposes only its color texture to Qt Quick; the live depth-stencil is a non-sampleable QRhiRenderBuffer on the render thread. ShaderEffect is a 2D item — it cannot depth-test or write gl_FragDepth. Qt 6.10/6.11 has no public path to share a depth buffer between two offscreen renderers (a View3D/Qt3D scene and our custom QQuickRhiItem cannot cross-occlude). The only way to occlude a QML item against our geometry is to draw it inside our own RHI pass, sharing our depth buffer.

2. How Qt does it (the mechanism we reuse)

QtQuick3D renders 2D Items inline in its 3D pass — not by re-implementing 2D rendering, but by reusing Qt Quick's own batch renderer in a special mode. Traced through the Qt source (qtquick3d + qtdeclarative):

  1. Harvest the item's scene-graph subtree. QQuick3DItem2D calls refWindow() + refFromEffectItem(true) on the content item (the same call ShaderEffectSource uses) so its QSGRootNode is built without it appearing in the normal 2D overlay.
  2. Create a Qt Quick renderer in 3D mode. QSGRenderContext::createRenderer(QSGRendererInterface::RenderMode3D) → setRootNode(rootNode).
  3. Feed the 3D MVP as the renderer's "projection." mvp = clipSpaceCorr × cameraViewProjection × nodeWorldTransform × flipY, so the 2D vertices (and their depth) land in the 3D scene.
  4. Bind the 3D render target + command buffer via QSGRenderTarget(rt, rpDesc, cb), then prepareSceneInline() → renderSceneInline() — recording draws into the already-open 3D pass, sharing color and depth.
  5. Depth behaviour. In RenderMode3D the batch renderer disables its normal 2D z-allocation (useDepthBuffer()==false), draws with depthTest=true, depthWrite=false (so 3D geometry occludes the item), then runs a color-write-off depth post-pass so subsequent 3D content is occluded by the item too.

Sources: qquick3ditem2d.cpp, qssgrenderdata.cpp:683–746, qssgrenderpass.cpp:1175–1258, qsgbatchrenderer.cpp:3907–4032, qsgbatchrenderer_p.h:971–982. The private API is identical in Qt 6.11.1 (our build).

3. Spike result validated

A throwaway spike (cwBillboardTest.{h,cpp} + a BillboardTest in GLTerrainRenderer.qml) wired this path into cwRhiScene with no engine changes: the integration seam is gather(), which runs before beginPass (so prepareSceneInline()'s resource uploads are legal) and emits a Custom drawable that calls renderSceneInline() inside the open pass.

4. Phase 0 — Encapsulation layer (the public API)

The whole plan hinges on hiding the 9 private-API calls behind a small CaveWhere surface. Two classes are the only files in the codebase that include Qt private headers. Everything else uses public types (QMatrix4x4, QRhi*, QQuickItem*, QQuickWindow*).

cwQuickItemSubscene new — GUI thread

Owns the "make a QQuickItem renderable off-tree" lifecycle.

  • cwQuickItemSubscene(QQuickItem* content, QQuickWindow* window) → internally refWindow + refFromEffectItem(true).
  • ~cwQuickItemSubscene() → derefFromEffectItem(true) + derefWindow (the spike leaked these — real teardown lives here).
  • bool nodeReady() const / void forceSync() → wraps the rootNode()-null / updateDirtyNode retry.
  • An opaque quintptr rootNodeHandle() so the QSGRootNode type never leaks into a public header.

.cpp includes qquickitem_p.h, qquickwindow_p.h.

cwItem2DRenderer new — render thread

A pimpl wrapper over QSGRenderer inline rendering.

// header — no private types
class cwItem2DRenderer {
public:
    struct Frame {
        QMatrix4x4 mvp;                       // clip-corrected MVP
        QRhiRenderTarget* rt;
        QRhiRenderPassDescriptor* rpd;
        QRhiCommandBuffer* cb;
        QSize deviceRect;
        float devicePixelRatio;
        quintptr rootNodeHandle;              // from cwQuickItemSubscene
    };
    explicit cwItem2DRenderer(QQuickWindow* window);
    ~cwItem2DRenderer();
    bool prepare(const Frame& frame);          // lazy createRenderer + prepareSceneInline
    void render();                            // renderSceneInline
private:
    struct Impl; std::unique_ptr<Impl> d;     // hides QSGRenderer / QSGRenderTarget
};

.cpp includes qsgrenderer_p.h.

Tripwire. Both classes get focused smoke tests (construct → ref → prepare against an offscreen target → render → destroy). If a Qt upgrade breaks the private-API contract, these fail — not the whole app.

5. Phase 1 — Generic billboard render object

  • cwRenderBillboards new (cwRenderObject, GUI) + cwRHIBillboards new (cwRHIObject, render): render a set of billboards. The GUI object holds slots { QQuickItem* content, QVector3D worldPos, SizeMode mode, float depthBias }; synchronize() hands the list to the render side; gather() preps each slot's cwItem2DRenderer and appends a Custom draw.
  • Pass routing: draw in the existing Overlay pass (or a dedicated RenderPass::Billboard just before it). Overlay runs in the final target after the EDL composite, which writes combined scene+cloud depth — so billboards depth-test against everything, EDL or not, with no routing changes. (The spike confirmed this.)
  • One cwItem2DRenderer per slot (matches QtQuick3D; each caches its render list). Visible slot count is bounded by culling (Phases 3–4), so this stays cheap.
  • Slot identity (Phase-1 shortcut): the GUI hands the whole list to the render side via setBillboards(...), and the render thread keys m_slots ↔ m_renderers by list position. Correct while the content set is static (one billboard in the spike), but it bakes a positional contract into the API — addressed in Phase 3 (see "stable identity").
Batching — documented, not pursued yet. One QSGRenderer per billboard means the Qt scene-graph batcher merges draw calls within a billboard (mixed component types in one content item batch by material, as in the normal 2D pass) but not across billboards — each is an independent renderSceneInline(). This mirrors QtQuick3D's Item2D (also no cross-item batching). The shared QSGRenderContext still amortizes the expensive resources (glyph atlas, material shaders, pipeline + texture caches) across all of them, so the per-renderer cost is batch-building + draw submission, not re-uploading glyphs/shaders.

A batched alternative exists: one shared renderer over a root node holding every billboard subtree, each under its own QSGTransformNode carrying that billboard's camera-facing model matrix (renderer projection = bare viewProjection), letting the batcher merge across billboards (all label backgrounds → one batch, glyphs → glyph batches). Costs: per-frame transform-node updates as the camera moves, moving the billboard math out of a single per-renderer MVP, and more private-API surface to validate — and it is unverified that the batcher merges as hoped across refFromEffectItem subtrees. Decision: not worth it now — likely over-optimization, and the per-renderer count is already bounded by culling. Pull this lever only if a benchmark comparison (per-renderer vs. shared-renderer at realistic label counts) shows the per-renderer overhead is the bottleneck.

6. Phase 2 — Billboarding math

  • Screen-constant size (default for labels/leads): build the model matrix so the quad is a fixed pixel size independent of distance — unlike the spike's world-sized ~3.6 m tag. Derive scale from gl_Position.w / viewport height (inverse of the world-radius trick PointCloud.vert:36–54 already uses).
  • Depth bias: stations/leads sit on the wall surface, so a naive test self-occludes them. Offset the world position slightly toward the eye (or a polygon-offset equivalent) so a label on a wall stays visible while one behind a wall is hidden.
  • Centering/anchoring; ortho + perspective both confirmed in the spike.

7. Phase 3 — Station labels (#536)

Labels become occluded by walls/clouds — the headline ask of #536 — with no interaction needed (labels aren't clickable), so this is the right first real feature, free of the input problem.

Current architecture (what we're repointing). LinePlotLabelView (a cwLinePlotLabelView, subclass of cwLabel3dView) is a full-bleed QQuickItem overlay in GLTerrainRenderer.qml. cwLinePlotLabelView is purely a data source: it watches cwCavingRegion and builds one cwLabel3dGroup per cave, each holding cwLabel3dItems (plain data: text + world position) from the cave's station-position lookup. cwLabel3dView does the work: on every camera change (updateGroupPositions) it projects each station, clips off-frustum ones, sorts the rest nearest-first, greedy-packs screen rects into m_labelKdTree, and for each survivor acquireLabelItem() (from Label3d.qml, recycled via a per-group m_itemPool), sets its text, setVisible(true), and setPosition(projectedScreenCoord) — i.e. the QQuickItems are the labels, placed as 2D overlay children and drawn on top of everything (no depth). That last step is what Phase 3 replaces.
  • Edits land almost entirely in cwLabel3dView edit. For the billboard rendering work, cwLinePlotLabelView was unchanged — it only feeds cwLabel3dItems, and inherits the new rendering path. The view stays a QQuickItem (it still holds the camera, the groups, the KD-tree, and runs the cull) but stops positioning visible 2D children; instead it owns one cwRenderBillboards and feeds it the surviving labels. (Superseded for keyword visibility: §7 Phase 3b later restructured cwLinePlotLabelView to one group per trip.)
  • New scene property edit + qml: the billboard render object must be added to a cwScene (like any cwRenderObject), so cwLabel3dView gains a cwScene* property and GLTerrainRenderer.qml passes scene: rendererId.scene to LinePlotLabelView (it already passes camera). The GUI-thread cull uses cwCamera; the render-thread billboard uses cwRhiItemRenderer's matrices — same camera, so projected positions agree.
  • Cull output becomes a membership diff, not a reposition. The cull (unchanged — see culling note below) produces the visible set each frame. A station's worldPosition is camera-invariant, so per frame we only diff visible-set membership: newly-visible → addBillboard(id, content, worldPos, ScreenConstant, depthBias); newly-hidden/clipped → removeBillboard(id) (and return the item to m_itemPool, as today). updateBillboard(id, worldPos) only fires when a station actually moves (setLabels). The billboard count stays bounded by the KD-tree.
  • Stable identity edit cwRenderBillboards: replace Phase 1's positional index keying with a stable id (cwBillboardId, a strong enum class : uint32_t mirroring cwRenderObjectId #512, so it can't be confused with a label index). The API moves from whole-list setBillboards(...) to granular addBillboard() → id / updateBillboard(id, …) / removeBillboard(id), keying the render-side cwItem2DRenderer map by that id so a renderer's cached render list stays bound to its billboard across insert/remove. The id is generated (a monotonic counter), and cwLabel3dView holds the id for each visible label (in cwLabel3dGroup::m_billboardIds, parallel to m_labelItems): minted on cull-in, kept stable across camera frames, released on cull-out. That stability — not the QQuickItem* — is what the renderer cache and the fade depend on.
  • Pooling × id keying × subscenes. m_itemPool recycles a QQuickItem across different stations over time, so the content pointer's address is reused — exactly the "freed address reused" hazard #512 names. Keying by the generated id (not by QQuickItem*) sidesteps it: a pooled item released from label A (removeBillboard(id_A)) and reacquired for label B (addBillboard() → id_B) gets a fresh renderer under id_B. Each billboard owns one cwQuickItemSubscene keyed by its id (no content-keyed map, no manual refcount) — refFromEffectItem is itself reference-counted inside Qt, so even the (never-occurring) shared content case stays valid until the last release.
  • No setPosition, and the original 2D draw is auto-suppressed. The billboard's model matrix places the quad in 3D, so we never call setPosition again. We also don't need setVisible(false) to stop the old overlay draw: cwQuickItemSubscene references content via refFromEffectItem(true) — the hide flag — which already removes the item from its normal 2D rendering. The item stays "visible" (so its scene graph keeps syncing) while only the billboard renders it.
  • Fade-in — two real gotchas found in implementation edit qml. The visible trigger still works: the pool toggles the item's visible flag (false when pooled, true on cull-in), so Label3d.qml's onVisibleChanged → fadeInIfVisible fires once per appearance, and visible: false on a pooled item is what keeps it from drawing while idle. But the animation only renders after fixing two things:
    1. Animate a child's opacity, not the root's. refFromEffectItem(true) excludes the referenced item's own opacity/transform from the rendered subtree (Qt expects the effect to apply them — same reason ShaderEffectSource ignores sourceItem.opacity). So Label3d.qml is now a plain root Item wrapping a child QC.Label whose opacity animates — the child's opacity is in the subtree and shows. The root sizes to the child's implicit size (that's what buildRenderSlots reads for contentSize).
    2. Drive on-demand re-renders — without reaching into the QML. The 3D pass only renders when something calls update(); an opacity animation alone doesn't. The clean seam: the content item's opacityChanged is the "re-render me" signal, so cwLabel3dView connects to the item it already holds (function-pointer, no findChild/objectName into the QML internals). To make that signal fire, Label3d.qml mirrors the animated child opacity onto the root's opacity (opacity: labelText.opacity) — the root's opacity is excluded from the render (see above), so the binding is purely the change signal and has no visual effect. The window's sync phase refreshes the (dirty) child opacity node before each requested frame renders, so no manual forceSync is needed; the connection self-limits to while opacity is changing.
    Cull-out needs no fade (today's release just hides) — we removeBillboard immediately.
  • Size mode & depth bias. Labels use ScreenConstant (constant pixel size, matching today's screen-space overlay — readable at any zoom). Stations sit on the centerline/point geometry, so a small positive depthBias toward the camera keeps a label from being self-occluded by its own station point; the value is a tunable named constexpr.
  • contentSize timing. cwRenderBillboards reads content->width()/height() at synchronize; a freshly-created QC.Label may report size 0 for a frame, so the billboard appears a frame late — the same benign "materializes a frame later" behaviour as the rootNodeHandle == 0 skip. (The existing cull already tolerates this via m_averageLabelSize.)
Culling vs. depth occlusion — analysis & decision. The existing screen-space cull (cwLabel3dView::updateGroupPositions) already sorts overlapping labels nearest-first (sortVisibleEntries() orders by projected position.z() ascending) and greedily packs them into m_labelKdTree, so the closest label of an overlapping cluster wins its screen rect. That heuristic is aligned with depth occlusion (the nearest marker is least likely to be behind geometry; when it is, the depth test hiding it is exactly #536's ask) and orthogonal in purpose (depth removes "label through a wall"; the KD-tree removes "text crowding text"). So we keep the packing as-is — the nearest-first priority is already correct, nothing to add.

Known residual ("the hole"): packing runs on the GUI thread and can't see what the render thread's depth test will kill. If a nearer label wins its rect but is then occluded by geometry at its pixel, the farther label it culled — which might have been visible at a neighbouring pixel — stays hidden, so an overlap region can show no label even though one was visible. This is a new failure mode billboarding introduces (today labels draw unconditionally on top). Severity is geometry-dependent: a non-issue in open passage, occasionally visible in cluttered breakdown.

Decision: ship the simple version (accept the rare hole) and defer the fix to a future iteration — the added cost isn't justified blind, before we see real caves rendered. Future refinement (documented, not pursued): make the cull depth-aware on the GUI thread by ray-casting camera→station through cwGeometryItersecter (intersects(QRay3D) / intersectsDetailed — CPU-only, runs offscreen, no GPU readback) and skipping labels that will be occluded before packing, so reserved rects only go to labels that survive the depth test. Cost is one ray-cast per non-clipped label per camera change.

Phase 3b — Keyword visibility for station labels (#536) implemented

Goal. Station labels join the keyword filter the same way leads did (§8 Gap 2), keyed per trip and matching the line-plot geometry: filtering out a trip's centerline (its Type=Line Plot row, or a Cave/Year/Trip value) hides that trip's station labels with it. Two decisions settled before implementation (both mirror the lead choices): the keyword data lives on the entity (full analog — a trip-owned model, shared with the geometry), and labels are grouped per trip (per-trip groups), so each group maps 1:1 to one keyword item exactly like a scrap's leads.

Context (what's already there). Line-plot geometry already has per-trip keyword items: cwLinePlotManager::reconcileTripKeywordItems (cwLinePlotManager.cpp:215) registers one cwKeywordItem per trip carrying {Type=Line Plot} on the item plus addExtension(trip->keywordModel()), with a cwLinePlotTripVisibility proxy that flips the trip's centerline vertex range. cwLinePlotLabelView, by contrast, is trip-unaware: it builds one cwLabel3dGroup per cave from cave->stationPositionLookup() (a deduplicated station→position map). cwTrip already exposes keywordModel() and uniqueStations(); cwKeywordVisibility::invokeVisible (cwKeywordVisibility.cpp:39) has a generic setVisible(bool) slot path, so any QObject with that slot is a valid keyword target — no bespoke proxy needed.

1. cwTrip::linePlotKeywordModel() — the #1 analog. Add a lazily-created cwKeywordModel on cwTrip = {TypeKey, "Line Plot"} + addExtension(this->keywordModel()), owned by the trip — the exact parallel of cwScrap::leadKeywordModel(). Why a separate model and not trip->keywordModel(): scraps/notes/leads extend the trip's keyword model, so putting Type=Line Plot there would make every scrap and lead under the trip wrongly inherit “Line Plot” — the same leak that forced the scrap to own a dedicated leadKeywordModel. Then refactor cwLinePlotManager::reconcileTripKeywordItems to drop its inline add({Type, "Line Plot"}) + addExtension(trip->keywordModel()) and instead addExtension(trip->linePlotKeywordModel()) (the near-empty pass-through pattern, as cwLeadView::addKeywordItem now does). The label view's per-trip item references the same trip->linePlotKeywordModel(). Result: geometry and labels share one trip-owned “this is line plot” identity in the data layer.

Two keyword items per trip (geometry + labels), by design. The geometry item's target is the centerline vertex range; the label item's target is the label group — different render targets, so they remain separate registrations sharing the one data model. This is the proven coexistence pattern (a scrap already carries its own keyword item and its leads' item with overlapping inherited keywords). The filter UI groups by keyword value, so two items with the same keyword set don't create duplicate rows; toggling a value fires setVisible on both, hiding geometry and labels together. Verify during implementation that identical-keyword items dedup in the filter tree as expected.

2. Per-trip label groups; the group is the visibility target. Restructure cwLinePlotLabelView to enumerate trips (region→cave→trips), building one cwLabel3dGroup per trip from trip->uniqueStations() positioned via the cave's stationPositionLookup() (stations without a solved position are skipped, as today). Give cwLabel3dGroup a setVisible(bool) slot + a bool m_visible + Q_PROPERTY/visibleChanged, so the group is the setObject target — this is the “container as target” idea we rejected for leads, except cwLabel3dGroup already exists as the grouping object, so it adds zero objects. cwLabel3dView honors it: updatePositions/updateGroupPositions skip a hidden group and release its pooled items + billboards (so a keyword-hidden trip claims no KD-tree space and renders nothing). Per trip the view registers one cwKeywordItem with object = the group and keyword model = addExtension(trip->linePlotKeywordModel()), lazily (0↔1 on the trip having labels), mirroring the lead lifecycle.

3. Many-to-many stations & OR semantics (the one real difference from leads). A station at a junction belongs to several trips; the line plot draws its vertex once per trip, so the station shows iff any of its trips is visible. Per-trip groups give this for free: a shared station gets a (duplicate) label in each owning trip's group, and a label renders as long as its trip's group is visible. Duplicates don't double-render because the declutter KD-tree is shared across all groups within a frame (cwLabel3dView::updatePositions clears m_labelKdTree only after the group loop), so two co-located labels collide and the second is rejected — whichever trip is visible wins the rect. Cost is duplicate cwLabel3dItem value structs (text + QVector3D) for shared stations: cheap, and the alternative (per-cave groups + a station→trips refcount threaded into the declutter hot path) was rejected as more invasive for no rendering benefit.

Composition & offscreen testability. Labels are already on the billboard path (#536), and buildRenderSlots skips !isVisible() content; gating at the group adds a C++/QML-property flag the offscreen suite can exercise through the layers tab, exactly like the lead keyword tests ([[project_billboard_rhi_path_untested]] — the RHI path itself stays untested offscreen, but the visibility gate is property-level).

QObject accounting (per trip with labels). Geometry (unchanged shape): cwKeywordItem + its pass-through model + cwLinePlotTripVisibility. New for labels: cwKeywordItem + its pass-through model (the cwLabel3dGroup is the target, no proxy). Plus one shared trip->linePlotKeywordModel() on the trip (the #1 cost, same tradeoff accepted for leads). The per-trip groups replace the per-cave groups — more groups, but each is the grouping object the view needs anyway.

Files touched (3b): cwTrip.h/.cpp (lazy linePlotKeywordModel() = {Type=Line Plot} + extension); cwLinePlotManager.cpp (reconcileTripKeywordItems references trip->linePlotKeywordModel() via addExtension); cwLabel3dGroup.h/.cpp (setVisible(bool) slot + visible flag + Q_PROPERTY); cwLabel3dView.cpp (skip + release hidden groups in the cull); cwLinePlotLabelView.h/.cpp (per-trip enumeration; one group + one keyword item per trip; keywordItemModel Q_PROPERTY; trip add/remove/station-change lifecycle); GLTerrainRenderer.qml (keywordItemModel: RootData.keywordItemModel on LinePlotLabelView).

The per-cave→per-trip restructure of cwLinePlotLabelView landed as designed: the view now keys a QHash<cwTrip*, TripEntry> (no positional cave list) and watches cwCave::insertedTrips/beginRemoveTrips plus stationPositionPositionChanged per cave. cwLabel3dGroup gained a visible property + setVisible(bool) slot; the group is the keyword item's setObject target (no proxy), reached through cwKeywordVisibility's generic QObject dispatch. cwLabel3dView::updateGroupPositions releases a hidden group's pooled items + billboards and skips its projection, so a hidden trip claims no KD-tree space. cwTrip::linePlotKeywordModel() is the shared trip-owned model; cwLinePlotManager::reconcileTripKeywordItems now references it via addExtension instead of an inline Type=Line Plot. The §7 note that treated cwLinePlotLabelView as “unchanged, pure data source” no longer holds — superseded by this section.

Follow-up refactors (out of scope for 3b, surfaced by the pre-commit review). Both are consistency/maintainability wins, not behavior changes; each is its own commit.

  1. Reusable keyword-item lifecycle registry. The “lazily register one cwKeywordItem per entity when it first has content, unregister at zero, swap models safely, sync-delete in the destructor” pattern is now reimplemented four times: cwLeadView (per scrap), cwLinePlotLabelView (per trip, new in 3b), cwLinePlotManager::reconcileTripKeywordItems (per solved trip), and cwScrapManager (per scrap). Each hand-rolls addItem/removeItem, the m_keywordItemModel.isNull() guard, deleteLater vs sync-delete teardown, and model-swap re-registration — the exact invariants that bit us once (the destructor double-free fixed via declaration order in test_cwLinePlotLabelView). A small cwKeywordItemRegistry<Key> helper would own the map (Key*→{cwKeywordItem*, object}), the lazy 0↔1 transitions, the model pointer + swap, and the destructor contract, so the four call sites shrink to “build the model + target, ask the registry to ensure/drop.” Out of scope now: the four sites have divergent removal triggers (empty lead list / empty label group / solved-geometry delta / scrap deletion) and key types, so the right seam is worth settling against all four at once rather than guessing from 3b. The lifecycle invariants would become testable properties of one class.
  2. cwLinePlotLabelView should enumerate trips via cwRegionTreeModel, matching cwLeadView. 3b left the view hand-rolling raw cwCavingRegion + cwCave::insertedTrips/beginRemoveTrips subscriptions, while cwLeadView already consumes the tree model idiomatically (rowsInserted/rowsAboutToBeRemoved seeded by all<cwScrap*>()). Converging means the view takes a cwRegionTreeModel* and watches rowsInserted/rowsAboutToBeRemoved filtered by isCave(parent) → add/remove trips, seeded via all<cwTrip*>(). Caveat that keeps it from being a drop-in: the label view has a second trigger cwLeadView lacks — solved station-position changes (cwCave::stationPositionPositionChanged). cwRegionTreeModel is purely structural and carries no station-position dataChanged, so the view still needs a direct per-cave stationPositionPositionChanged subscription for the positional rebuild. Net: this converges the structural enumeration (caves/trips add/remove) and aligns the two billboard overlays, but the positional trigger stays label-specific. cwLinePlotManager is intentionally excluded — it is a back-end survex solver that reacts to deep data changes (shots, calibration, fix stations) and keys its keyword items by UUID off the solve output, not off structural tree signals; the tree model is a poor fit there. Out of scope for 3b: it changes the view's public QML property (region → regionModel) plus GLTerrainRenderer.qml wiring and partially rewrites the just-landed enumeration — a consistency-only gain best kept off the 3b diff.

8. Phase 4 — Leads (#535)

Readiness: 3/10 as a single unit — split into 4a/4b/4c (below). 4a is now designed to 8/10 (see “4a — implementation design” below); 4b/4c remain sketches. Unlike labels, this is not a clean greenfield view: cwLeadView is the legacy architecture and leads are interactive. The work is dominated by (1) a substantial rewrite of cwLeadView and (2) the input/occlusion model — not by "drive LeadPoint.qml through cwRenderBillboards".

Gap 1 — cwLeadView is a rewrite, not a one-liner

Today cwLeadView (~382 lines) uses the legacy repeater pattern: cwTransformUpdater + cwSelectionManager + a per-scrap QVector<QQuickItem*>, each LeadPoint.qml parented as a 2D child and positioned by CPU-projecting 3D→screen and setting the item's x/y (the //TODO: convert to cwItem3DRepeater at the top of the class). Porting to billboards means rebuilding it to mirror cwLabel3dView (billboard add/update/remove, per-lead cwBillboardId tracking parallel to the item list, window rebind via itemChange, opacity-driven on-demand re-render) while preserving the existing scrap/lead model wiring (leadsBeginInserted / leadsInserted / leadsRemoved / leadsReset / leadsDataChanged with the LeadPosition role, and selection). Position source is unchanged: cwScrap::LeadPosition (carpeted global) → the billboard's worldPosition. cwTransformUpdater is dropped from the lead path — leads mirror cwLabel3dView, which does not use the updater: the view computes one GUI-side projection per camera change (camera->qtViewportMatrix() * camera->viewProjectionMatrix(), then matrix.map(worldPosition)) and sets the invisible LeadPoint's x/y from it, while the billboard draws the marker from the same worldPosition. One projection path — the label-blessed one — instead of a legacy updater running a parallel projection. depthBias pulls the draw toward the eye along the line of sight, which does not move it in screen space (same invariant as labels).

Lead interaction — RESOLVED. The visible question-mark renders as a depth-occluded billboard in the 3D pass (like a label). The existing LeadPoint.qml item stays alive as the 2D input layer — its refFromEffectItem(hide=true) reference already hides its own 2D draw, so there's no double-drawing, but it still receives taps and still owns selection + the popup Loader. Decisions:
  • Minimal churn — keep LeadPoint as the (now-invisible) hit layer. Its existing TapHandler, LeadListener, selected state, and QuoteBox Loader are reused as-is. We only (a) feed the question-mark graphic to cwRenderBillboards for the occluded draw, and (b) add the occlusion gate inside onTapped. The item follows the camera only for positioning (the existing transform-updater projection); positioning is not occlusion.
  • Occlusion is tested ON CLICK, not per-frame. There is NO camera-change subscription. On tap, cast one ray from the eye through the lead's world position and ask cwGeometryItersecter::intersects(QRay3D) (CPU-only / works offscreen — see [[project_bvh_cpu_side]]); if the nearest hit is closer than the lead, the lead is behind geometry — ignore the tap. One ray cast per click, zero per-frame work. Rationale: "clicks must not pass through walls" is a correctness requirement for leads, but it only needs to hold at the moment of the click. The same test can later refine the label cull "hole" (§deferred).
  • The QuoteBox popup stays a normal 2D overlay — NOT billboarded. It is a callout anchored to the question mark; billboarding it would clip it into geometry and make it unreadable.
  • Popup behaves like a dialog: click-away closes it. Today the popup stays open (annoying) — it should dismiss on a press outside it. QuoteBox is a custom Shape-drawn callout, NOT a Qt Popup, so there is no built-in closePolicy; "close" = setting the lead's selected = false (which clears the Loader). Because selection is single (cwSelectionManager — at most one popup open), put a single view-level press-outside catcher at cwLeadView scope that deselects the open lead, rather than a per- LeadPoint catcher (those would fight each other). It must not swallow presses meant for another lead or for camera interaction — let those through / re-dispatch.
Considered but not chosen: an always-clickable hit layer that ignores depth (simplest, but lets users click leads through walls); a view-level click handler with no per-lead items (discards the existing LeadPoint popup/selection structure — bigger rewrite); replicating QtQuick3D subscene input delivery (ensureSubsceneDeliveryAgent + synthetic events — faithful but heaviest); billboarding the popup itself (poor UX).

4a — implementation design locked

Readiness after this expansion: 8/10. Mechanics below are decided; what remains is the normal write-test-iterate (and confirming refFromEffectItem-hidden items still receive taps, already asserted by the click-away tests).

1. Positioning — drop cwTransformUpdater; mirror cwLabel3dView. Decided: projection lives in the view (not centralized in cwRenderBillboards). cwLeadView holds a QPointer<cwCamera> m_camera (like cwLabel3dView, replacing the updater-backed camera()), connects viewMatrixChanged/projectionChanged to a updatePositions() slot, and there computes matrix = camera->qtViewportMatrix() * camera->viewProjectionMatrix() and sets each invisible LeadPoint's x/y from matrix.map(worldPosition). That invisible item remains the hit/selection/popup layer; the billboard draws the marker. One projection path, no legacy updater. (Centralizing projection in cwRenderBillboards was considered for 4c reuse and rejected for now — the view-side formula is trivial and already proven in cwLabel3dView; revisit only if clustering wants it.)

2. Billboard content target = the question-mark sub-item, not the whole LeadPoint. cwRenderBillboards::Billboard::content is one QQuickItem* with one subscene, so set content = questionImage (the QQ.Image inside LeadPoint). The QuoteBox Loader is a sibling of questionImage under pointId, so it is outside the referenced subtree and keeps drawing as a normal 2D overlay (occluded billboard for the marker; un-occluded 2D popup — exactly the resolved UX). Expose the image to C++ via a typed alias on LeadPoint, e.g. property alias billboardContent: questionImage, which cwLeadView reads with item->property("billboardContent") after creation.

3. New scene property on cwLeadView. Mirror cwLabel3dView: Q_PROPERTY(cwScene* scene READ scene WRITE setScene NOTIFY sceneChanged), plus ensureRenderBillboards() (lazy-create once window() and scene exist) and the itemChange(ItemSceneChange) window-rebind. scene serves double duty: it owns the cwRenderBillboards and provides the BVH for the occlusion gate (scene->geometryItersecter()). Wire it in GLTerrainRenderer.qml: scene: rendererId.scene on the LeadView.

4. Billboard lifecycle, parallel to the item list. Add QVector<cwBillboardId> billboardIds to ScrapEntry (index-aligned with items). On insert/updatePosition: addOrUpdateBillboard(scrap, i, questionImageOf(item)) with SizeMode::ScreenConstant and a kLeadDepthBias; on removeItems: m_renderBillboards->removeBillboard(id) before deleteLater(), keeping the vector index-aligned. Re-render hook mirrors labels: connect questionImage's opacityChanged/visibleChanged to m_renderBillboards->update() so the completed-lead fade and the selected highlight request frames in the on-demand 3D pass. 4a scope simplification: register one billboard per lead unconditionally; when questionImage.visible is false (completed, unselected) the subscene simply draws nothing. Cull/keyword-driven removal is 4b.

5. Occlusion gate API. Q_INVOKABLE bool cwLeadView::isOccluded(const QVector3D& worldPosition, const QPointF& tapViewportPosition) const. Body (CPU/main-thread, works offscreen — [[project_bvh_cpu_side]]): cast the ray through the tapped pixel and compare the cave hit to the billboard plane's depth at that pixel:

const QRay3D ray = camera()->rayFromQtViewport(tapViewportPosition);
const double quadDistance = ray.projectedDistance(worldPosition); // camera-facing quad plane depth
if(quadDistance <= 0.0) { return false; }
const cwRayHit hit = scene()->geometryItersecter()->intersectsDetailed(ray);
if(!hit.hit()) { return false; }
return hit.tWorld() < quadDistance - kLeadDepthBias;   // cave surface in front of the marker pixel

Why per-pixel, not eye→center (corrected during impl). The marker is a camera-facing quad; the GPU clips it per-pixel along the cave silhouette, so most of an icon stays visible even when its geometric center sits a hair behind a fold. A single eye→center ray (the original design) rejects that whole icon — the "I can see it but can't click it" bug. Casting through the actual tapped pixel and comparing to the quad plane's depth makes the click match exactly what is drawn (what you see is what you click). frustrumRay returns a unit-direction ray, so tWorld and projectedDistance are the same units; - kLeadDepthBias mirrors the eye-ward depthBias the billboard renders with. The tap pixel is mapped into the LeadView's (viewport) coordinate space: questionImage.mapToItem(pointId.leadView, eventPoint.position).

The BVH was not the bug. A brute-force-vs-BVH parity test (test_cwGeometryItersecter_bruteForce.cpp) proved intersectsDetailed matches a double-precision Möller–Trumbore reference exactly for double-sided geometry. The apparent "misses nearest triangle" was entirely cwGeometry::cullBackfaces (default true): the picker honours it and skips backface triangles, so on a folded surface it returns a farther front-facing fold. Scraps set cullBackfaces=false (cwTriangulateTask), so the live BVH is accurate. A second test (test_cwGeometryItersecter_billboardQuad.cpp) validates the quad pick reproduces the depth buffer's silhouette clipping — and flags a shared-diagonal degeneracy (a click dead-center on a two-triangle quad's diagonal can fall through both triangles on MT's u+v>1 boundary).

Files touched (4a): cwLeadView.h/.cpp (scene property, billboards, isOccluded, remove cwTransformUpdater and back camera with a QPointer<cwCamera> + inline updatePositions() like cwLabel3dView), LeadPoint.qml (billboardContent alias, leadView required property, occlusion-gated onTapped), GLTerrainRenderer.qml (scene: binding), tst_Leads.qml (occlusion-gate test: a lead behind geometry is un-clickable; an unobstructed lead still selects).

Two RHI-only bugs made leads unclickable in the app (invisible to the offscreen suite). The offscreen test runs have no QRhi, so the billboard render path (cwQuickItemSubscene) never executes — every QML test passed while the app was broken. Both bugs lived entirely on that path:

  1. The tap target was the billboard's own content. cwQuickItemSubscene calls refFromEffectItem(true) on the content to pull it out of the 2D scene (so it isn't double-drawn) — which also stops it receiving input. The TapHandler sat on that same questionImage, so the billboard hid its own click target. Fix: the TapHandler now lives on a separate hitTarget Item (sized/centred on the marker, visibility-bound to it) that the subscene never references. The earlier "refFromEffectItem still hit-tests" claim was only ever true offscreen.
  2. The billboard ignored content visibility. The subscene draws the content via its render node, which ignores the visible property, so completed (unselected) leads kept drawing and overlapped live leads. Fix: cwRenderBillboards::buildRenderSlots skips !content->isVisible() (regression-guarded in test_cwRenderBillboards.cpp).

The occlusion test (per-pixel gate) reuses kLeadDepthBias (1.0 m) as its slack, mirroring the billboard's draw-time depthBias. UX (intended): a lead behind a different scrap surface is un-clickable until the camera is oriented to see it ("clicks don't pass through walls"); a lead resting on its own scrap is clickable from the default framing. tst_ScrapInteraction::test_addLeadInteraction drops the old close-up camera hack; tst_Leads covers both gate directions.

Deferred (visual follow-up, not required for clickability): leads are placed by a smooth station-based warp (morphPoints) while the carpet is a faceted triangle mesh, so a lead can float a few units off the carpet where the passage curves. Approach A makes the click consistent with the drawn marker regardless, but snapping the lead's world position onto the carpet mesh would remove the visual float-off. Tracked separately.

Deferred (code-quality follow-ups, from the post-4a /simplify review). Both are real improvements held back because they restructure the billboard-lifecycle code, and the offscreen suite never exercises the RHI billboard path ([[project_billboard_rhi_path_untested]]) — so a refactor can pass every test and still break the app. Do each as its own change, verified in the running app, not folded into a feature commit:

  1. Stop the per-frame item->property("...") string lookups; fold the parallel arrays into one per-lead struct. cwLeadView::updateItemPositions() runs on every viewMatrixChanged/ projectionChanged (hot path) and, for every lead, does two by-name metaobject property lookups: property("position3D") and property("billboardContent") (the latter inside addOrUpdateBillboard). Cost = N leads × 2 string-keyed lookups × every frame. Fix: replace ScrapEntry's index-aligned QVector<QQuickItem*> items + QVector<cwBillboardId> billboardIds with a single QVector<LeadItem> where LeadItem { QQuickItem* item; QQuickItem* content; QVector3D worldPosition; cwBillboardId billboardId; }. content is immutable after creation (already resolved in createItem); worldPosition has a single writer (the updatePosition lambda that sets the position3D property). Caching both removes the per-frame lookups and also kills the two-parallel-array bookkeeping (the insert/removeAt pairs that must stay index-aligned). position3D must remain a QML property — LeadPoint reads pointId.position3D in the tap handler — so mirror it, don't drop it.
  2. Extract a shared cwBillboard3dView base for cwLeadView and cwLabel3dView. A base class would own all of that; subclasses override only the domain-specific updatePositions() layout (label placement vs. lead projection) and content selection. Weigh against over-abstraction: justified once a third consumer lands — clustering (4c) reuses the same billboard + occlusion + selection machinery, so do this before or with 4c rather than after.

Gap 2 — Keyword visibility for leads implemented

Status & design history. A first cut (v1) shipped in the working tree, then a v2 redesign was explored and mostly walked back in discussion. The shipped design keeps v1's working mechanism and adds exactly one change — point #1, moving the lead keyword data onto cwScrap. The three other v2 ideas were dropped after weighing them against the stated goal (fewest QObjects, least indirection, clean MVC):

  • No new cwLeadModel enumeration class (was “architecture (b), model half”). cwLeadModel already exists as an unrelated per-lead-row QAbstractListModel scoped to one cave (the leads table in CaveLeadPage.qml, via cwLeadsSortFilterProxyModel / cwLeadCSVExporter); reusing that name was a mistake. Splitting out a new all-caves, per-scrap enumeration class would add a QObject that merely duplicates the region-tree watching cwLeadView already does. So the view keeps its inline enumeration (scrapsAdded/scrapsRemoved on cwRegionTreeModel, the QHash<cwScrap*, ScrapEntry>). Architecture (b) is satisfied by responsibility, not by a class: the data (keywords) lives on the scrap, the view owns enumeration + registration + visibility.
  • No per-scrap container QQuickItem (was the v2 visibility target). A QQuickItem per scrap, purely to hold a child list, is heavier than the proxy it would replace (transform node, scene-graph + input plumbing vs. a bare QObject with one bool), and the per-scrap grouping already exists in C++ as ScrapEntry::items. Its claimed wins (free initial visibility, declarative popup-close) were marginal. So cwLeadVisibility stays as the setVisible target.
  • No Option Y (was: generalize cwKeywordItem to reference an external model). Once the proxy stays and #1 uses the established addExtension pass-through, Option Y's only justification (avoid the item's redundant empty model) is a micro-optimization not worth touching a shared class for. cwKeywordItem is unchanged.

The pipeline (context). An entity joins the filter by registering a cwKeywordItem that pairs a cwKeywordModel (its filterable keywords) with an object exposing setVisible(bool); cwKeywordVisibility calls that slot when filters change. For leads the object is the cwLeadVisibility proxy, whose setVisible forwards to cwLeadView::setScrapKeywordVisible(scrap, v), which toggles every LeadPoint in that scrap's ScrapEntry and closes the popup if the hidden scrap owned the selection.

#1 (implemented) — cwScrap owns the lead keyword data. The one change from v1. The filterable data for a scrap's leads now originates in the back-end, not the view:

  • cwScrap::leadKeywordModel() (cwScrap.h:145, cwScrap.cpp): a lazily-created cwKeywordModel seeded with {TypeKey, "Lead"} plus addExtension(this->keywordModel()). Lazy — only scraps whose leads are actually registered by the view allocate it; owned by the scrap (parent = this), so it rides the scrap's lifetime. The Type=Lead row is static; the inherited Type=Plan/Profile, Cave, Year, … track the scrap's own model automatically through the extension (no need to update on scrap type change).
  • cwLeadView::addKeywordItem (cwLeadView.cpp): the per-scrap cwKeywordItem now just does item->keywordModel()->addExtension(scrap->leadKeywordModel()) — the near-empty pass-through pattern cwScrapManager/cwLinePlotManager already use — instead of v1's inline add({Type, "Lead"}) + addExtension(scrap->keywordModel()). Effective keywords are identical (Type=Lead + the scrap's full chain); the difference is that the “Lead” identity is now data the scrap owns.

Keywords & composition (unchanged from v1). A scrap already inherits the full chain (scrap → note → trip → cave/team; cwNote.cpp:182/187/340, cwTrip.cpp:492), so a lead carries Type=[Lead, Plan] + inherited Cave/Year/Date/TripName/Caver: the “Lead” row toggles all leads, unchecking the scrap's type (“Plan”) hides its leads with it, and Cave/Year narrows leads as it does scraps. Visibility composes as a logical AND, gated at each LeadPoint root:

lead drawn  ⇔  scrap keyword-visible (LeadPoint.visible)  AND  (!lead.completed || selected)

questionImage's existing !lead.completed || pointId.selected binding governs within a keyword-visible scrap; cwRenderBillboards::buildRenderSlots skips !content->isVisible(), and isVisible() is effective, so a keyword-hidden LeadPoint suppresses the billboard and the hitTarget taps for free. A completed lead still exists, so it still counts toward the Type=Lead filter. This stays entirely on the QML-property path the offscreen suite exercises ([[project_billboard_rhi_path_untested]]).

QObject accounting (per scrap-with-leads, unchanged from v1). cwKeywordItem + its own (pass-through) cwKeywordModel + cwLeadVisibility proxy + N LeadPoints, plus the scrap-owned leadKeywordModel. #1 adds one model object on the scrap; in exchange the lead keyword identity lives in the data layer where it belongs. (We accept that single extra model as the price of the clean ownership; it is far cheaper than the container QQuickItem v2 would have added.)

Files touched (4b, as shipped): cwScrap.h/.cpp (lazy leadKeywordModel() = {Type=Lead} + extension — the only new public surface); cwLeadView.cpp (addKeywordItem references scrap->leadKeywordModel() via addExtension; doc comment updated). Everything else from v1 is retained as-is: cwLeadVisibility.h/.cpp (the proxy — kept), the keywordItemModel Q_PROPERTY on cwLeadView, the setScrapKeywordVisible/keywordVisible machinery, and the GLTerrainRenderer.qml keywordItemModel: RootData.keywordItemModel binding. No new cwLeadModel, no container QQuickItem, no cwKeywordItem change, no file deletions.

Testing (4b). The two offscreen cases (test_leadKeywordVisibilityTogglesLeads, test_hidingLeadKeywordClosesSelectedPopup) exercise the full keyword pipeline through the real layers tab UI and pass unchanged — no selector churn, because the QML object tree is unchanged (no container level was added). tst_Leads.qml: 17 passed, 0 failed offscreen. Run both full suites before pushing.

Gap 3 — Clustering is net-new and the expand UX is unspecified

Leads have no cull/cluster today. Direction (decided): a cluster is a state of LeadPoint, not a separate cluster-billboard class — the same component shows a count badge when it stands in for N leads. This reuses the one billboard pipeline, the occlusion gate, and the selection/popup mechanism (a cluster is occludable and clickable exactly like a single lead; its QuoteBox shows a member list instead of one lead's detail). A CPU pass per camera change projects lead world positions, groups those within a screen-space radius, picks one representative LeadPoint per group to carry the count + cluster state, hides the rest, and registers only the representative as a billboard — keeping the per-lead item allocation of 4a (no full label-style pooling rewrite). Open for 4c: (a) the recluster-vs-selection rule — if an open lead is absorbed into a cluster on zoom-out, does its popup close or transfer to the cluster (cwSelectionManager tracks the QQuickItem*); (b) the single-detail vs N-lead-list popup state on LeadPoint.
