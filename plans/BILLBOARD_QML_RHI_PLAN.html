Depth-Occluded QML Billboards in the RHI Scene

Implementation plan — rendering live QML Items as billboards that participate in the 3D depth buffer.

Issues: #535 (Lead visibility, keywords, overlap, z-order) · #536 (Station labels should be billboarded)

Date: 2026-06-16 · Status: research + spike complete, decisions settled — implementing Phase 0 next

1. Problem

Station labels (Label3d.qml via cwLabel3dView) and lead markers (LeadPoint.qml via cwLeadView) are plain Qt Quick Items anchored over the RHI 3D view and positioned by CPU projection (cwTransformUpdater::updatePoint, cwLabel3dView::updateGroupPositions). They composite on top of the 3D color output as ordinary 2D items, so they always draw over the geometry — a label behind a passage wall is not hidden. Both issues ask for the same thing: markers that are occluded by cave geometry via the depth buffer ("billboarded so they use the z-buffer"). The markers already track their 3D point and face the camera; the missing piece is depth occlusion + z-ordering.

Why it can't be done in the QML layer. A QQuickRhiItem exposes only its color texture to Qt Quick; the live depth-stencil is a non-sampleable QRhiRenderBuffer on the render thread. ShaderEffect is a 2D item — it cannot depth-test or write gl_FragDepth. Qt 6.10/6.11 has no public path to share a depth buffer between two offscreen renderers (a View3D/Qt3D scene and our custom QQuickRhiItem cannot cross-occlude). The only way to occlude a QML item against our geometry is to draw it inside our own RHI pass, sharing our depth buffer.

2. How Qt does it (the mechanism we reuse)

QtQuick3D renders 2D Items inline in its 3D pass — not by re-implementing 2D rendering, but by reusing Qt Quick's own batch renderer in a special mode. Traced through the Qt source (qtquick3d + qtdeclarative):

  1. Harvest the item's scene-graph subtree. QQuick3DItem2D calls refWindow() + refFromEffectItem(true) on the content item (the same call ShaderEffectSource uses) so its QSGRootNode is built without it appearing in the normal 2D overlay.
  2. Create a Qt Quick renderer in 3D mode. QSGRenderContext::createRenderer(QSGRendererInterface::RenderMode3D) → setRootNode(rootNode).
  3. Feed the 3D MVP as the renderer's "projection." mvp = clipSpaceCorr × cameraViewProjection × nodeWorldTransform × flipY, so the 2D vertices (and their depth) land in the 3D scene.
  4. Bind the 3D render target + command buffer via QSGRenderTarget(rt, rpDesc, cb), then prepareSceneInline() → renderSceneInline() — recording draws into the already-open 3D pass, sharing color and depth.
  5. Depth behaviour. In RenderMode3D the batch renderer disables its normal 2D z-allocation (useDepthBuffer()==false), draws with depthTest=true, depthWrite=false (so 3D geometry occludes the item), then runs a color-write-off depth post-pass so subsequent 3D content is occluded by the item too.

Sources: qquick3ditem2d.cpp, qssgrenderdata.cpp:683–746, qssgrenderpass.cpp:1175–1258, qsgbatchrenderer.cpp:3907–4032, qsgbatchrenderer_p.h:971–982. The private API is identical in Qt 6.11.1 (our build).

3. Spike result validated

A throwaway spike (cwBillboardTest.{h,cpp} + a BillboardTest in GLTerrainRenderer.qml) wired this path into cwRhiScene with no engine changes: the integration seam is gather(), which runs before beginPass (so prepareSceneInline()'s resource uploads are legal) and emits a Custom drawable that calls renderSceneInline() inside the open pass.

4. Phase 0 — Encapsulation layer (the public API)

The whole plan hinges on hiding the 9 private-API calls behind a small CaveWhere surface. Two classes are the only files in the codebase that include Qt private headers. Everything else uses public types (QMatrix4x4, QRhi*, QQuickItem*, QQuickWindow*).

cwQuickItemSubscene new — GUI thread

Owns the "make a QQuickItem renderable off-tree" lifecycle.

  • cwQuickItemSubscene(QQuickItem* content, QQuickWindow* window) → internally refWindow + refFromEffectItem(true).
  • ~cwQuickItemSubscene() → derefFromEffectItem(true) + derefWindow (the spike leaked these — real teardown lives here).
  • bool nodeReady() const / void forceSync() → wraps the rootNode()-null / updateDirtyNode retry.
  • An opaque quintptr rootNodeHandle() so the QSGRootNode type never leaks into a public header.

.cpp includes qquickitem_p.h, qquickwindow_p.h.

cwItem2DRenderer new — render thread

A pimpl wrapper over QSGRenderer inline rendering.

// header — no private types
class cwItem2DRenderer {
public:
    struct Frame {
        QMatrix4x4 mvp;                       // clip-corrected MVP
        QRhiRenderTarget* rt;
        QRhiRenderPassDescriptor* rpd;
        QRhiCommandBuffer* cb;
        QSize deviceRect;
        float devicePixelRatio;
        quintptr rootNodeHandle;              // from cwQuickItemSubscene
    };
    explicit cwItem2DRenderer(QQuickWindow* window);
    ~cwItem2DRenderer();
    bool prepare(const Frame& frame);          // lazy createRenderer + prepareSceneInline
    void render();                            // renderSceneInline
private:
    struct Impl; std::unique_ptr<Impl> d;     // hides QSGRenderer / QSGRenderTarget
};

.cpp includes qsgrenderer_p.h.

Tripwire. Both classes get focused smoke tests (construct → ref → prepare against an offscreen target → render → destroy). If a Qt upgrade breaks the private-API contract, these fail — not the whole app.

5. Phase 1 — Generic billboard render object

  • cwRenderBillboards new (cwRenderObject, GUI) + cwRHIBillboards new (cwRHIObject, render): render a set of billboards. The GUI object holds slots { QQuickItem* content, QVector3D worldPos, SizeMode mode, float depthBias }; synchronize() hands the list to the render side; gather() preps each slot's cwItem2DRenderer and appends a Custom draw.
  • Pass routing: draw in the existing Overlay pass (or a dedicated RenderPass::Billboard just before it). Overlay runs in the final target after the EDL composite, which writes combined scene+cloud depth — so billboards depth-test against everything, EDL or not, with no routing changes. (The spike confirmed this.)
  • One cwItem2DRenderer per slot (matches QtQuick3D; each caches its render list). Visible slot count is bounded by culling (Phases 3–4), so this stays cheap.
  • Slot identity (Phase-1 shortcut): the GUI hands the whole list to the render side via setBillboards(...), and the render thread keys m_slots ↔ m_renderers by list position. Correct while the content set is static (one billboard in the spike), but it bakes a positional contract into the API — addressed in Phase 3 (see "stable identity").
Batching — documented, not pursued yet. One QSGRenderer per billboard means the Qt scene-graph batcher merges draw calls within a billboard (mixed component types in one content item batch by material, as in the normal 2D pass) but not across billboards — each is an independent renderSceneInline(). This mirrors QtQuick3D's Item2D (also no cross-item batching). The shared QSGRenderContext still amortizes the expensive resources (glyph atlas, material shaders, pipeline + texture caches) across all of them, so the per-renderer cost is batch-building + draw submission, not re-uploading glyphs/shaders.

A batched alternative exists: one shared renderer over a root node holding every billboard subtree, each under its own QSGTransformNode carrying that billboard's camera-facing model matrix (renderer projection = bare viewProjection), letting the batcher merge across billboards (all label backgrounds → one batch, glyphs → glyph batches). Costs: per-frame transform-node updates as the camera moves, moving the billboard math out of a single per-renderer MVP, and more private-API surface to validate — and it is unverified that the batcher merges as hoped across refFromEffectItem subtrees. Decision: not worth it now — likely over-optimization, and the per-renderer count is already bounded by culling. Pull this lever only if a benchmark comparison (per-renderer vs. shared-renderer at realistic label counts) shows the per-renderer overhead is the bottleneck.

6. Phase 2 — Billboarding math

  • Screen-constant size (default for labels/leads): build the model matrix so the quad is a fixed pixel size independent of distance — unlike the spike's world-sized ~3.6 m tag. Derive scale from gl_Position.w / viewport height (inverse of the world-radius trick PointCloud.vert:36–54 already uses).
  • Depth bias: stations/leads sit on the wall surface, so a naive test self-occludes them. Offset the world position slightly toward the eye (or a polygon-offset equivalent) so a label on a wall stays visible while one behind a wall is hidden.
  • Centering/anchoring; ortho + perspective both confirmed in the spike.

7. Phase 3 — Station labels (#536)

  • Repoint cwLabel3dView / cwLinePlotLabelView edit to feed cwRenderBillboards instead of positioning 2D overlay items.
  • Stable identity edit cwRenderBillboards: labels are added and removed dynamically (culling, edits, view changes), so replace Phase 1's positional index keying with a stable per-billboard id. Move the API from whole-list setBillboards(...) to granular addBillboard(id) / updateBillboard(id, ...) / removeBillboard(id), and key the render-side cwItem2DRenderer map by that id so each renderer's cached render list stays bound to its content across insert/remove. This mirrors two patterns the codebase already settled: cwRhiTexturedItems' QHash<uint32_t, Item*> keying and cwRenderObjectId (issue #512 — "a freed object's address can be reused"). Index keying would otherwise silently re-bind every renderer past a removal point to the wrong label.
  • Keep the KD-tree overlap culling (it now bounds the billboard slot count) and the Label3d.qml fade-in — a live item, so its NumberAnimation keeps working inside the 3D pass.
  • Labels become occluded by walls/clouds — the headline ask of #536.
  • No interaction needed (labels aren't clickable) → this is the right first real feature, free of the input problem.

8. Phase 4 — Leads (#535)

  • Drive LeadPoint.qml through cwRenderBillboards → occluded leads.
  • Keyword visibility: gate lead visibility on scrap keywords and fold leads into the type keyword (cwKeywordItemModel / cwKeywordFilterPipelineModel).
  • Zoom-out clustering: a CPU pass projects lead world positions, groups those within a screen-space radius, and emits a single numbered cluster billboard (with a count) that expands to a lead list — mirrors the label KD-tree, feeds the same billboard set.
Lead interaction — resolved: hit-proxy. Lead popups are clickable, but an item rendered inside the 3D pass no longer receives normal 2D events. We keep a thin, invisible 2D hit-proxy at the projected screen position to receive input, while the visible billboard renders (and is occluded) in the 3D pass. Considered but not chosen: routing clicks through cwGeometryItersecter/cwCoordinatePicker, or replicating QtQuick3D's subscene input delivery (ensureSubsceneDeliveryAgent + synthetic events) — faithful but heaviest.

9. Cross-cutting concerns

  • Offscreen / export (cwScene::renderOffscreen): the spike binds the swap-chain target, so it is not offscreen-aware. Make cwItem2DRenderer::Frame carry the active target so billboards can render into either. Decision: both labels and leads are off in map exports by default — cwRHIBillboards skips its slots when rendering into an offscreen target. Finer control comes from keywords (a future per-keyword "show in export" toggle), not a hard-coded exclusion.
  • MSAA: the Overlay pass's sample count must flow into the QSGRenderTarget; the per-pass sample count is already tracked in RenderData.
  • Threading & lifetime: ref/deref strictly on the GUI thread; rootNode read + render on the render thread; on content-item destruction, tear down the slot before the QSGRenderer. QPointer guards as in the spike.
  • Private-API risk: isolated to the two Phase-0 .cpp files; a breaking Qt upgrade fails their unit tests, not the app.

10. Testing

  • C++ smoke tests for cwItem2DRenderer / cwQuickItemSubscene (the private-API tripwire).
  • QML/offscreen test: load geometry + a billboard, render offscreen from a "behind the wall" camera, read back, assert the billboard pixels are absent (occluded) and present from the front — automatable once the offscreen path is billboard-aware.
  • Run both cavewhere-test and cavewhere-qml-test per the usual rule (tee output to /tmp/*.log).

11. Suggested sequencing

  1. Phase 0 (encapsulation) — unblocks everything, smallest risk.
  2. Phase 1 + 2 (billboard object + math).
  3. Phase 3 (labels) — first user-visible win, no input complexity.
  4. Phase 4 (leads) — clustering + keywords + the input decision.
  5. Cross-cutting hardening + offscreen + tests; delete the spike files (cwBillboardTest.{h,cpp}, the BillboardTest block in GLTerrainRenderer.qml). remove
