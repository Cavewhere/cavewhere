Project-level palette + async save/load draft / problem framing

Untracked planning doc, fall2026 branch. Drafted 2026-06-18. Prerequisite for SYMBOLOGY_PALETTE_GLYPH_LIBRARY_PLAN.html (auto-save glyph library). This is a problem statement + options, not a settled design — several pieces are genuinely open and called out as such.

1. Why

The glyph library wants auto-save (no Save/Cancel — edits persist as you draw). That only makes sense if the palette is persisted the way every other CaveWhere serialized object is: owned by the project and written asynchronously through cwSaveLoad, not the current synchronous "write a file + full reload" path. Two coupled changes:

  1. Ownership moves to the project level. A palette becomes part of the project's saved state, not an app-global install.
  2. Writes go async through the save queue, like caves / trips / scraps, so editing never blocks the UI and there is no explicit Save.

2. Current state (what exists today)

  • Storage: cwSymbologyPaletteIO is directory-as-truth — <AppDataLocation>/palettes/<slug>/ with palette.json + brushes/*.cwbrush + glyphs/*.cwglyph. App-global, shared across all projects.
  • Manager: cwSymbologyPaletteManager scans that directory, owns the live cwSymbologyPalette objects, and (since commit 8.1) does a reconciling reload that keeps a palette pointer stable across rescans.
  • Writes: saveGlyph/removeGlyph/duplicatePalette write to disk synchronously on the UI thread, then call reload() — a full directory rescan. Fine for occasional edits; wrong for per-stroke auto-save.
  • Project linkage: the region stores only a defaultPaletteId (in project metadata); the palette data lives in app-data, referenced by id. A project opened on another machine that lacks that palette id can't resolve it (this is also the "missing palette" load-error follow-up, task #31).
  • Re-skin path: a content change → palette.dataChanged() → cwSketch re-resolves its snapshot → path source rebuilds. The glyph tessellation cache has a scoped invalidateGlyph(name) already.

3. Target

The project owns its palette(s). Editing a glyph mutates the project's palette and schedules an async write through cwSaveLoad (debounced), exactly like editing a station or a scrap. Opening a project restores its palettes without depending on the editing machine's app-data. The shipped read-only default stays app-global (it's a fallback/seed, not project data).

4. Open question — storage format decide first

Where does the project's palette data physically live? The symbology format is unreleased on fall2026, so we are free to choose without a proto-version bump.

OptionShapeTrade-off
P1 — Protobuf in the project A Palette message in cavewhere.proto, serialized into the bundled .cw / .cwproj alongside caves, written by the same cwSaveLoad pass. Uniform with every other object; one save pipeline; transactional with the rest of the project. But duplicates the existing JSON/dir IO, and the per-file dir layout (nice for Git diffs) is lost unless mirrored.
P2 — Project-local palettes/ dir Relocate the directory-as-truth layout inside the .cwproj (and into the bundle for .cw). Reuses cwSymbologyPaletteIO as-is; ties into the project-local palette idea already tracked (task #36). Great Git/sync story (per-glyph file diffs), reuses existing IO. But it is a second persistence channel outside cwSaveLoad's protobuf pass — async/dirty/queue integration has to be built to match, and bundled .cw (zip) needs the dir packed in.
P3 — Hybrid Edit against project-local files (P2) for Git-friendliness, but drive the writes through the cwSaveLoad queue so timing/threading matches the rest of the app. Best of both, most plumbing.

Leaning P2/P3 — the directory-as-truth format and the reconciling reload already exist and the Git sync story is a CaveWhere selling point; the work is the async/queue integration, not a new serializer. But P1 is the most "like every other object". Needs your call.

5. Async write path

  • Route palette mutations through cwSaveLoad (the existing queued save infra) so writes are off the UI thread and serialized with project saves. Use the AsyncFuture::observe(future).context(this, …) pattern (never waitForFinished outside tests), per house rules.
  • Debounce: a pen drag is many appendPoints; one write per glyph per idle window, not per sample.
  • Dirty tracking: the project already tracks unsaved state; the palette becomes part of that. Removes the bespoke synchronous reload()-after-every-write.

6. The re-skin / rebuild question shared with the library plan §5

Persistence and re-skin must be decoupled. Auto-save is cheap and frequent; rebuilding dependent render geometry (tessellation cache + decoration relayout for every sketch whose brushes stamp the changed glyph) is expensive and must not run per pen sample.

  • Editor's own canvas: re-skins live (one glyph, trivial).
  • Project-wide: scope to the changed glyph via invalidateGlyph(name), and run the dependent-sketch relayout coarsely — on editor close, or debounced (library plan Decision A).
  • This needs a clean "palette content changed" signal distinct from "palette persisted" — today both ride reload(). Splitting them is part of this work (relates to the granular-signal follow-up, task #38).
