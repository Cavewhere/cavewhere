Project-level palette + async save/load decided / design

Untracked planning doc, fall2026 branch. Drafted 2026-06-18, storage + write/rebuild decisions settled 2026-06-18, claims verified against the code 2026-06-18. Prerequisite for SYMBOLOGY_PALETTE_GLYPH_LIBRARY_PLAN.html (auto-save glyph library).

Code-verification pass (2026-06-18). Three findings change the work, not the direction:
  • The "Auto updates" flag already exists. automaticUpdate is a Q_PROPERTY on cwScrapManager, cwLinePlotManager, and cwJobSettings; both managers already short-circuit on it. We reuse it, we don't add one (§7).
  • Scrap + line-plot rebuilds are already async (each owns an AsyncFuture::Restarter + dirty set). The only sync build left is the sketch decoration / path-source rebuild — that's what "async geometry build, commit 9/10" actually means (§7 blockquote).
  • Editing is file-first today (saveGlyph writes a file then reload()s; the only palette signals are dataChanged() / writableChanged()). The auto-save model needs an object-first mutation API + granular signals before the queue/Restarter wiring means anything (§5, §7).

1. Why

The glyph library wants auto-save (no Save/Cancel — edits persist as you draw). That only makes sense if the palette is persisted the way every other CaveWhere serialized object is: owned by the project and written asynchronously through cwSaveLoad, not the current synchronous "write a file + full reload" path. Two coupled changes:

  1. Ownership moves to the project level. A forked palette becomes part of the project's saved state, not an app-global install.
  2. Writes go async through the save queue, like caves / trips / scraps, so editing never blocks the UI and there is no explicit Save.

2. Current state (what exists today)

  • Storage: cwSymbologyPaletteIO is directory-as-truth — <AppDataLocation>/palettes/<slug>/ with palette.json + brushes/*.cwbrush + glyphs/*.cwglyph. App-global, shared across all projects.
  • Manager: cwSymbologyPaletteManager scans that directory, owns the live cwSymbologyPalette objects, and (since commit 8.1) does a reconciling reload that keeps a palette pointer stable across rescans.
  • Writes are file-first. cwSymbologyPaletteManager::saveGlyph/removeGlyph/duplicatePalette write to disk synchronously on the UI thread (cwSymbologyPaletteIO::saveGlyph → JSON file), then call reload() — a full directory rescan. There is no in-memory "mutate this glyph on the live palette object" path: the file is the only way to change a glyph, and the change becomes visible only after the reload. Fine for occasional edits; wrong for per-stroke auto-save, which needs to mutate the object first and persist behind it.
  • Project linkage: the region stores only a defaultPaletteId (in project metadata); the palette data lives in app-data, referenced by id. A project opened on another machine that lacks that palette id can't resolve it (this is also the "missing palette" load-error follow-up, task #31).
  • Re-skin path (actual chain): a content change reaches a sketch only as a coarse reload today — cwSymbologyPaletteManager::palettesChanged (or cwCavingRegion::defaultPaletteIdChanged) → cwSketch::resolveSnapshot() → it diffs the rebuilt cwPaletteSnapshot by operator== and, if changed, emits cwSketch::paletteSnapshotChanged() → cwSketchCanvas::refreshPathSnapshot() → cwDecoratedStrokePathSource::setSnapshot(). There is no palette.dataChanged() hop, and setSnapshot() currently calls cwGlyphTessellationCache::setSnapshot(), which clears the entire cache. The scoped invalidateGlyph(name) exists but is not wired into this path yet — scoping the re-skin to it is part of this work, not a freebie.
  • Rebuild gating already exists. cwScrapManager and cwLinePlotManager each expose automaticUpdate (default on) and already run their rebuilds asynchronously through an AsyncFuture::Restarter (cwScrapManager also keeps a DirtyScraps set). The palette re-skin needs to hook into this existing machinery, not reinvent it.

3. Target

The project owns its forked palette(s). Editing a glyph mutates the project's palette object and schedules an async write through cwSaveLoad (debounced to "edit finished"), exactly like editing a station or a scrap. Opening a project restores its forked palettes from inside the project, with no dependency on the editing machine's app-data. CaveWhere's shipped palettes stay app-global, read-only, and serve as fork sources / fallbacks — the project references them by identity, it does not copy them unless the user forks.

4. Storage format DECIDED — project-local files (was P2)

Forked palettes are stored as a per-palette directory inside the project. The symbology format is unreleased on fall2026, so this is a free choice — no proto-version bump.

my.cwproj/
  palettes/
    <palette-slug-1>/
      palette.json          # palette metadata: id, name, source/version (see §6).
                            #   NB: `writable` is runtime-only (derived from version
                            #   support), not a persisted field.
      glyphs/
        my_glyph.cwglyph    # one file per glyph
      brushes/
        my_brush.cwbrush    # one file per brush
    <palette-slug-2>/
      glyphs/*.cwglyph
      brushes/*.cwbrush
  • Keep the nested glyphs/ + brushes/ layout DECIDED — keep nested exactly as cwSymbologyPaletteIO writes it today (constants kGlyphsSubdirName / kBrushesSubdirName, reconciled on save). This reuses the IO's directory scan + reconcile almost verbatim — only the root path moves to a project-relative location. palette.json stays as the palette-level metadata file (id, name; writable is runtime-only, not persisted — see below; plus the source/version fields from §6).
  • Extensions confirmed: .cwglyph and .cwbrush already exist and stay.
  • For the bundled single-file .cw (zip): the palettes/ tree is packed into the archive, same as the rest of the bundle. For .cwproj (Git-backed dir) it sits on disk as-is — per glyph/brush file diffs are the Git-sync win.
  • Reuses cwSymbologyPaletteIO as the serializer (relocated to a project-relative root); the new work is the async/queue integration, not a new format.

Why this over protobuf-in-project: per-file diffs are a CaveWhere sync selling point, the directory-as-truth IO and the reconciling reload already exist, and a forked palette stored as plain files survives a CaveWhere upgrade unchanged (no schema-migration of embedded blobs). The cost — a second persistence channel outside the protobuf save pass — is paid by routing it through the same cwSaveLoad queue (§5).

5. Async write path DECIDED

Object-first prerequisite. None of this works until the live cwSymbologyPalette can be mutated in memory. Today the only mutator is the file-first saveGlyph+reload(). So step one is a real in-memory API on the palette — e.g. setGlyph(name, glyph) / removeGlyph(name) that update the live object and emit the granular signals from §7 — with persistence riding behind the mutation, not in front of it.

  • Debounce = "edit finished", not per-sample. A pen drag is many appendPoints; the canvas commits the edited glyph back into the palette object on pen-up / mouse-release (one mutation per committed stroke), and the file write coalesces from there.
  • Debounce Restarter on the owner. SUPERSEDED by §10 step 3 (Option B) — kept for history. The first cut put an AsyncFuture::Restarter on the palette owner (cwSymbologyPaletteManager) to coalesce a burst of edits into one write. The as-built design drops it: cwSaveLoad::connectPalette() listens to the granular signals directly (like connectCave), and the editor already commits the palette object discretely (per Save / pen-up), so (objectId,tag) job compression coalesces — no Restarter is needed.
  • The write is a first-class cwSaveLoad job (option B). DECIDED — B As built (see §10 step 3): not a new Operation type and not a Custom lambda — each palette file is a genuine WriteFile Job on the existing file-job queue, keyed by (objectId = palette*, tag) and pushed via addExplicitFileSystemJob. This is what makes ordering correct forever: a BundlePackage operation is preceded by a SaveFlush that drains all pending file jobs, so the bundle is never zipped (cwZip::zipDirectory over the whole project dir) mid-write. cwSaveLoad learns the directory-as-truth palette paths (palette->directory()/{glyphs,brushes}/<name>) the same way it learns GIS Layers/ for LAZ. Each file is still written atomically (QSaveFile temp + rename, the discipline cwSymbologyPaletteIO and saveBundledArchiveAtomic already use) so even a crash mid-queue leaves whole files.
  • Dirty tracking folds the forked palettes into the project's existing unsaved-state. Today cwProject goes modified via cwSaveLoad::localMutationOccurred; a palette mutation must feed the same flag. This removes the bespoke synchronous reload()-after-every-write entirely. Production code uses AsyncFuture::observe(future).context(this, …), never waitForFinished (test-only), per house rules.

6. Palette sources & versioning DECIDED (a/c + default rollback), DEFERRED (b)

A palette a project can use comes from one of three places:

  1. (a) CaveWhere defaults — shipped, app-global, read-only. The user forks one to edit; the project references it by identity until then. in
  2. (b) GitHub-linked public palettes — the project stores only a link + pinned commit SHA-1 (submodule-style); the palette data is fetched, not embedded. Lets people share/curate public palettes. deferred — reserve the link/SHA fields now, build the fetch later
  3. (c) Project-local forks — the forked palettes living in palettes/ (§4). Self-contained and travel with the project. in — this plan

Drift, versioning & default rollback REQUIREMENT

Requirement: if a user upgrades CaveWhere and a project that uses a shipped default palette suddenly "looks bad," they must be able to roll the project back to a previous default palette version. So versioned defaults are not just drift-avoidance — explicit rollback to an older default version is a user-facing feature.

Two cases, both covered by the same version pin:

  • Forked palette (c) is already immune — it carries its own copy in the project, so an upgrade can't touch it.
  • Un-forked reference to a default (a) is where drift bites, and where rollback applies.

To support rollback, CaveWhere must retain multiple versions of each shipped default (not just the newest), and a project records which version it references:

  • Stamp each palette with an identity + version (in palette.json). A project that references a default records the version it expects (the pin).
  • On open, if the referenced version differs from the newest shipped default, surface it (keep the pinned version vs update to the new one) instead of silently re-skinning.
  • The user can explicitly switch the pin to any retained version — that's the rollback. The retained older version must still be resolvable, so the shipped defaults need to ship/keep their history.
  • Distribute CaveWhere defaults as local git repositories (versioned, distributed) so a project can check out a specific default version — this is now the leading implementation, because it gives the retained-history / rollback requirement for free (each version is a commit/tag). The lighter "version field only" alternative would require us to bundle and index old versions ourselves; git already does that. mechanism to confirm
  • For (b), the pinned commit SHA-1 is the version pin — same mechanism, remote source.

Net: reserve source (default / github / local-fork) and version/sha fields in palette.json now so the format doesn't churn later. Implement (a) read-only defaults + (c) forks + the version pin in this work; the retained-history store (local git repo of defaults) and rollback UI can be staged right after, since the pin field makes them additive. (b) GitHub-linked palettes stay a later follow-up.

7. The re-skin / rebuild question DECIDED — re-verified against code 2026-06-19

Persistence and re-skin are decoupled. Auto-save is cheap and frequent; rebuilding dependent render geometry (tessellation cache + decoration relayout for every sketch whose brushes stamp the changed glyph) is expensive and must not run per pen sample.

Re-verified against the code (2026-06-19) — two earlier claims were stale and are corrected here: (a) the per-element glyphChanged(name)/brushChanged(name) signals now exist (added with the edit auto-save path, commit 3a) and are no longer net-new — with auto-save handling persistence, §7 adds no new palette signals at all (the originally-planned saved() is dropped — see the signal-surface note below). (b) cwLinePlotManager does not consume palette geometry — it is the survey loop-closure solve; strike it from the dependent list. The real consumers are the sketch decoration path source (cwDecoratedStrokePathSource, which owns a cwGlyphTessellationCache) feeding (i) the live cwSketchCanvas and (ii) the offline scrap PNG render (cwSketchScrapRasterizer, driven by cwScrapManager).

The editor's own canvas re-skins live (one glyph, trivial). Everything else splits into two tiers — one nearly built, one not:

  • DONE Tier 1 — live canvas re-skin (cheap; the downstream already exists). The chain cwSketch::paletteSnapshotChanged → cwSketchCanvas::refreshPathSnapshot() → cwDecoratedStrokePathSource::setSnapshot() → cache clear + scheduleRebuild() is fully wired today. cwSketch::resolveSnapshot() already re-resolves on cwSymbologyPaletteManager::palettesChanged, cwSketchSettings::defaultPaletteIdChanged, and cwCavingRegion::defaultPaletteIdChanged, and de-dupes on snapshot value — so it already fires paletteSnapshotChanged only on a real content change. The one missing wire: a glyph/brush edit on the currently-resolved palette does not trigger a re-resolve. Connect the resolved palette's glyphChanged/brushChanged to resolveSnapshot() (re-armed inside resolveSnapshot() whenever m_resolvedPalette changes, so the connection follows the active palette). Once connected, the live canvas re-skins for free.
  • DONE Tier 2 — project-wide scrap re-render (the expensive rebuild; reuses the existing automaticUpdate flag). Sketch-derived scrap PNGs are rasterized by cwSketchScrapRasterizer from cwScrapManager::updateScrapGeometryHelper(), gated by cwScrapManager::automaticUpdate() and coalesced by its existing TriangulateRestarter. Previously scraps re-rasterized only on stroke changes (strokesReset/strokeEnded/strokeRemoved) — a palette content edit did not mark them dirty, so glyph edits didn't reach the 3D model until strokes changed or the project reopened. The fix (one wire): cwScrapManager::connectSketch() now also connects cwSketch::paletteSnapshotChanged → updateDerivedScrapsForSketch(sketch) — the same per-sketch re-derive that stroke edits use. That re-runs the outline detector against the freshly resolved sketch->paletteSnapshot() and re-derives every tracked scrap, routing through the existing automaticUpdate-gated TriangulateRestarter. No new flag or Restarter — reuse the one cwScrapManager already has.
    • Auto updates ON: the Restarter kicks cwScrapManager to re-rasterize affected sketch-derived scraps (coalesced — one rebuild per idle window, not per sample).
    • Auto updates OFF: the scraps are marked dirty but do not re-rasterize until the user re-enables auto updates (or triggers a manual rebuild) — so editing many glyphs doesn't thrash the whole project.
    caveat cwJobSettings::automaticUpdate is a separate global preference, not currently routed to the managers' own automaticUpdate; reuse cwScrapManager's own flag — don't assume cwJobSettings drives it.
  • Scoped invalidation is secondary, not a prerequisite. cwGlyphTessellationCache::invalidateGlyph(name) exists, but the live canvas reaches the cache only through setSnapshot(), which swaps the whole snapshot value and clears the entire (per-sketch) cache. Since a brush edit can change the ink of every glyph that uses it and each sketch owns a small cache, the whole-cache clear is defensible; threading the changed name through to invalidateGlyph(name) instead is a per-cache micro-optimization. Keep it as a follow-up unless profiling says otherwise.
  • Signal surface.
    • DONE (3a) cwSymbologyPalette::glyphChanged(QString name) and brushChanged(QString name) already exist and already drive the auto-save path; Tier 1 and Tier 2 subscribe to these same signals.
    • No net-new signal. The originally-planned saved() ("palette persisted") is dropped: its rationale was "replaces relying on synchronous reload() return," which evaporated once writes became async first-class cwSaveLoad jobs. Auto-save is object-first, so the UI binds to the live in-memory object and reflects edits immediately; durability is observable at the project level via cwProject's existing saveFlushCompleted / objectPathReady and the tests' waitSaveToFinish(). No concrete consumer needs per-palette write-completion today. If a collaborative "share / commit this palette" workflow later needs it, add it then under the granular-signal follow-up, task #38.
Async geometry build is a separate refactor — and narrower than first thought. Scrap triangulation and the line-plot solve are already async (each manager owns a Restarter). The build still on the UI thread is the sketch decoration / path-source rebuild (cwDecoratedStrokePathSource rebuilds in setSnapshot()) — that's the one "commit 9/10" should move off-thread. This plan does not add a gating flag (it reuses the existing automaticUpdate); it only wires the palette content change into Tier 1's snapshot re-resolve and Tier 2's existing cwScrapManager dirty/rebuild path, and benefits automatically once the decoration build goes async.
