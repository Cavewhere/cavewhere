Key Observation

The current pipeline is already largely async: cwLinePlotManager uses AsyncFuture::Restarter<LinePlotResultData> to coalesce rapid edits. The heavy lifting (export → cavern → parse → geometry) runs on cwTask::threadPool() as a single QtConcurrent operation. However, the internal structure nests cwTaskinline and synchronously on the worker thread via runTaskInPlace(), which calls task.setUsingThreadPool(false) and then task.waitToFinish(). This is safe because all the work stays on the same worker thread.

2. cwTask Base Class Behavior

Thread Management

  • Thread pool: Managed globally by cwTask::threadPool() (a static QThreadPool*). Initialized once per application.
  • Start behavior:
    • start() emits preparingToStart(), then either enqueues to the thread pool (if isUsingThreadPool() is true) or calls run() immediately on the current thread.
    • run() is the entry point — it emits started() and calls the virtual runTask() implementation.
  • Restart behavior:
    • restart() atomically stops the running task (if any) and sets NeedsRestart = true.
    • When the task finishes, it sees the restart flag and emits shouldRerun() instead of finished().
    • The task transitions through Running → Restart → Restarting (which is equivalent to Ready).
  • Cancellation: stop() sets the status to Stopped. There is no in-flight cancellation mechanism; the runTask() implementation must check isRunning() or needsRestart() to cooperatively stop.

AsyncFuture::Restarter

cwLinePlotManager does not use cwTask directly. Instead, it uses: 

AsyncFuture::Restarter<cwLinePlotTask::LinePlotResultData> m_restarter;

The Restarter pattern (from the AsyncFuture library) works as follows:

  • m_restarter.restart([=]() { ... return QFuture<Result>; }) takes a lambda that returns a future.
  • If a previous future is in-flight, it is implicitly cancelled before the new one starts.
  • The new future replaces the old one in m_restarter.future().
  • Consumers observe the result via AsyncFuture::observe(m_restarter.future()).context(this, handler).

Key difference from cwTask: The Restarter is driven by user signals (rapid edit events) and automatically cancels and replaces in-flight work. cwTask requires explicit restart() calls and manages its own lifecycle through signals.

3. State and Ownership

Input Snapshot

When the user triggers a re-solve (by editing a station, adding a trip, etc.), cwLinePlotManager::runSurvex() is called: 

auto input = cwLinePlotTask::buildInput(Region.data());
auto future = cwLinePlotTask::run(std::move(input));

cwLinePlotTask::Input contains:

  • cwCavingRegionData regionData — a deep value-type copy of the entire region structure (caves, trips, chunks, stations, shots, etc.).
  • RegionDataPtrs regionPointers — a parallel structure of raw pointers to the original UI objects (cave, trip, scrap pointers). Used only for bookkeeping to map results back to the correct UI object after the solve.

Thread safety achieved — with a caveat: The snapshot is value-copied, but cwCavingRegionData is built from Qt containers (QVector, QList, QString, …) which are implicitly shared, not deep-copied. The copy is a header copy + refcount bump; the underlying buffers are shared with the UI-side originals until either side mutates. That is safe only if the worker thread sticks to const access. Any non-const operator[], data(), begin(), or QString::replace()-style call on the snapshot will trigger detach() and race the UI thread's mutations on the same shared buffer (the exact failure mode CLAUDE.md's .at()-vs-operator[] rule guards against). Invariant for the port: the worker reads the snapshot through const references and at(i) only; if a transform needs to mutate, copy the relevant container into a local first and mutate that. Reviewers of the ported worker should audit for non-const access as the primary risk.

When the result comes back, the pointers in RegionDataPtrs are validated against the current region state (in validateResultsData()) to ensure no object has been deleted between snapshot and apply.

Output Results

cwLinePlotTask::LinePlotResultData contains:

  • QMap<cwCave*, LinePlotCaveData> Caves — for each cave that changed, the new station positions, network, and depth/length.
  • QSet<cwTrip*> Trips and QSet<cwScrap*> Scraps — which UI objects need to redraw.
  • QVector<QVector3D> StationPositions and QVector<unsigned int> LinePlotIndexData — geometry for the 3D line plot.
  • cwSurveyNetwork RegionNetwork — the solved network topology.

Consumption on UI thread: When the future completes, AsyncFuture::observe().context(this, [this, future]()) captures the future and runs the handler on the main thread. Inside the handler: 

cave->setStationPositionLookup(caveData.stationPositions());
cave->setSurveyNetwork(caveData.network());
m_linePlot->setGeometry(results.stationPositions(), results.linePlotIndexData());

These calls must run on the UI thread because cwCave, station lookups, and cwRenderLinePlot are not thread-safe for writes.

4. Cancellation and Re-entrancy

Current Behavior

When a user rapidly edits (e.g., moves a station, then immediately moves another), runSurvex() is called multiple times. The Restarter handles this: 

m_restarter.restart([this]() {
    auto input = cwLinePlotTask::buildInput(Region.data());
    auto future = cwLinePlotTask::run(std::move(input));
    // ...
    return future;
});

If future from the first call is still in-flight when a second restart() is called:

  1. The Restarter cancels the first future (via QFuture::cancel()).
  2. The lambda is immediately re-executed, creating a new snapshot and launching a new solve.
  3. The old future's result (if it was computed before cancellation took effect) is dropped and never observed.

What "cancellation" actually does today: Restarter::restart() flips the bit on the old QFuture — nothing more. The cwConcurrent::run lambda inside cwLinePlotTask::run() does not consult QFuture::isCanceled() at any point, so the in-flight solve runs to completion (export → cavernlib → parse → geometry) and its result is dropped on return. Today's "cancellation" is really "discard the result." This is safe but burns CPU and, more importantly, holds the cavernlib mutex (Q1) in front of the new solve. Cooperative cancellation is the work proposed in Q4 — see that section for the concrete plan.

waitToFinish() — current uses

cwLinePlotManager::waitToFinish() wraps AsyncFuture::waitForFinished(m_restarter.future()), which spins a nested QEventLoop. CLAUDE.md restricts this to test code (the nested loop delivers queued signals mid-destruction and causes use-after-free). However, the current destructor at cwLinePlotManager.cpp:67-70 calls waitToFinish() in production: 

cwLinePlotManager::~cwLinePlotManager() {
    m_restarter.future().cancel();
    waitToFinish();   // production path — violates the test-only rule
}

This is a latent UAF: if a .context(this, …) callback is queued when destruction starts, the cancel sets a flag, waitForFinished pumps the event loop, and the queued lambda fires against a half-destructed this. The port plan removes this in Phase 0 (see §7); the destructor only needs m_restarter.future().cancel() because every observer is bound through .context(this, …) and Qt auto-disconnects the observer when the context object is destroyed.

5. External-Centerline Pipeline Integration Target (deferred — see note)

Scope note: This section is kept as background research for the eventual Phase 3 work, but is not part of the port being executed from this document. The implementation plan in §7 covers Phase 0 (destructor cleanup), Phase 1 (drop cwTask inheritance from the lineplot sub-tasks), and Phase 2 (consolidate cwLinePlotManager on an explicit AsyncFuture chain and switch to Monad::Result). Once Phases 0-2 land, we'll write a fresh plan for the external-centerline pipeline that supersedes the chain sketched here — in particular, the reconcile step's per-job aggregation via AsyncFuture::combine() and its partial-failure policy (fail-fast vs. drain-and-report) need concrete answers before Phase 3 can start. Read the rest of this section as "where we think we'll end up," not "what we're building now."

The goal is to attach an external-centerline pipeline to the same async chain. The proposed structure is: 

scanSource → copyFromSource → scanProjectCopy → reconcile → runCavern → updateNetwork

Pipeline steps

Step Purpose Existing or new function Thread / queueing
scan Read the entry file and walk *include / .mak / .wpj closure. Returns the dependency list and any soft warnings. New: cwExternalCenterlineScanner::scan(entryFile) (free function, sync). Worker thread via QtConcurrent::run. Pure read, not project-tree mutation, so it does not go through cwSaveLoad.
reconcile Make the in-project copy match the dependency list: copy anything newer, delete anything not in the closure. Same call handles both initial-attach (empty destination) and edit-in-place (source == destination, no copies needed). New: cwExternalCenterlineSync::reconcile(scan, attachmentDir). Submits each FS mutation as a cwSaveLoad job (Action::Copy for files, Action::Remove for GC). Per-job futures are joined with AsyncFuture::combine() to produce a single QFuture<Monad::Result<void>> that completes when every enqueued job has settled. The cwSaveLoad pool is single-threaded (setMaxThreadCount(1)), so jobs execute in submission order. Open question for the Phase 3 plan: partial-failure policy — if job #3 fails, do we cancel #4..N and abort, or drain the rest and report an aggregate error? Defer to the Phase 3 plan; not specified here.
runCavern Run cavern on the driver file (unchanged). Existing: cwCavernTask wrapped as a future. Worker thread. Writes only to the scratch temp dir, not the project tree, so no queueing needed.
updateNetwork Parse .3d and apply station positions to the region (existing path). Existing: cwSurvex3DFileReader::readNetworkAndLookup followed by splitLookupByCave. Worker thread for the parse; the apply step lands on the UI thread via .context(this, …).

The earlier sketch had a separate scanProjectCopy step for a "validate the copy is good" pass — cut, because the copy is byte-identical to the source by construction and cavern is the actual ground truth for whether the file parses. A redundant scan would just slow every solve down.

AsyncFuture chain

Flat. One .context(this, …) per step. No .subscribe() — subscribe() uses the calling thread as the context and doesn't auto-disconnect when this is destroyed, so a result arriving after the owning object is gone will run a lambda against freed memory. context(this, …) binds both the thread (the context object's thread) and the lifetime (auto-disconnect on this destruction) — the only safe pattern. 

using cwExternalCenterlineScanner::ScanResult;
using Monad::Result;

QFuture<Result<ScanResult>> scanFuture =
    QtConcurrent::run([entryPath]() { return scan(entryPath); });

AsyncFuture::observe(scanFuture)
    .context(this, [this](Result<ScanResult> scan) -> QFuture<Result<void>> {
        if (scan.hasError()) return AsyncFuture::completed(scan.error());
        return m_sync->reconcile(scan.value(), m_attachmentDir);    // -> cwSaveLoad jobs
    })
    .context(this, [this](Result<void> reconciled) -> QFuture<Result<CavernResult>> {
        if (reconciled.hasError()) return AsyncFuture::completed(reconciled.error());
        return runCavern(m_driverPath);
    })
    .context(this, [this](Result<CavernResult> cavern) {
        updateNetwork(cavern);                                       // UI thread
    });

What changed vs. the earlier sketch:

  • Flat, not nested. Each .context(this, …) returns its own QFuture<T>; AsyncFuture chains them. No inner AsyncFuture::observe() calls inside lambdas.
  • .context(this, …) everywhere, never .subscribe().
  • Fully-typed lambda parameters — no auto r.
  • [this] captures — no [&].
  • Three steps, not six. Dropped the separate copyFromSource (folded into reconcile) and scanProjectCopy (redundant — cavern is the ground truth). No ReconcileResult at all; the chain only needs success/failure to gate the cavern step.
  • Uses the existing Action::Copy job in cwSaveLoad, not Custom with a hand-rolled read+write.
  • Error propagation via Monad::Result: each step short-circuits if the previous returned an error, surfacing it to updateNetwork for display in CavernOutputPage.

6. cwProject / cwSaveLoad File Queue

The Job Queue System

cwSaveLoadPrivate (the implementation detail of cwSaveLoad) contains a Job system for serialized filesystem access: 

struct Job {
    enum class Kind { File, Directory, };
    enum class Action { Move, Remove, EnsureDir, WriteFile, Copy, Custom };

    const void* objectId;  // For tracking which object owns this job
    Kind kind;
    Action action;
    std::function<void(const Monad::ResultBase&)> onDone;
    Payload payload;  // WriteFilePayload, CopyFilePayload, CustomPayload, or EmptyPayload

    QString oldPath;
    QString path;
    QString dataRoot;
    QString projectRoot;
    // ... constructors ...
};

Job Actions

  • WriteFile — serialize a protobuf message to disk.
  • Copy — copy a file from source to destination.
  • Move — move or rename a file/directory.
  • Remove — delete a file/directory.
  • EnsureDir — create a directory if it doesn't exist.
  • Custom — run an arbitrary std::function<Monad::ResultBase()> in the job queue.

Enqueueing Mechanism

From cwSaveLoad.h: 

// Enqueues filesystem deletion jobs for a conflicting project descriptor file and its
// accompanying data directory. Used to clean up the "loser" project directory left on
// disk after a rename/rename merge conflict where ours-wins resolution was applied.
void enqueueConflictingProjectCleanup(const QString& conflictingDescriptorRelPath);

void enqueueOrphanDirectoryCleanup(const QString& orphanDirRelPath);
void setOwnedTempDir(const QString& path);

And from private methods (visible in the .cpp): 

void addFileSystemJob(Job job, cwSaveLoad* context);
void addExplicitFileSystemJob(Job job, cwSaveLoad* context);
QFuture<Monad::ResultBase> enqueueOperation(cwSaveLoad* context,
                                            std::function<Monad::ResultBase()> customAction);

Why the Queue Is Necessary

Filesystem race conditions: If multiple threads write to the project tree concurrently (e.g., one thread saves a cave proto while another thread copies an external centerline file into the project), the results are undefined. The queue ensures a total order of operations. Key points:

  • The project directory tree is the single source of truth for the on-disk state.
  • All mutations (writes, copies, moves, deletes) must be linearized.
  • The queue runs in its own thread (or is carefully synchronized) to prevent interleaving.
  • When cwProject saves a cave, it enqueues a WriteFile job rather than writing directly.

Example: Adding a Custom Job

To enqueue an external-centerline copy operation, you would likely do something like: 

std::function<Monad::ResultBase()> copyAction = [sourcePath, destPath]() {
    return cwSaveLoadPrivate::Job::copyFileWithValidation(sourcePath, destPath);
};
auto future = d->enqueueOperation(this, copyAction);

The returned future completes when the job is dequeued and executed, and captures any error that occurred during the copy.

Status: The cwSaveLoad job queue exists and is actively used for serialized filesystem access. It can accommodate the external-centerline pipeline's filesystem operations via Custom actions.

7. Port Strategy Proposal

Scope

This document covers Phases 0, 1, and 2 — the lineplot pipeline port itself. Phases 3 (external-centerline pipeline) and 4 (unified solve manager) are out of scope here and will get their own plan after Phase 2 lands; the sketch in §5 is background only.

Phase 0: Fix the destructor UAF

Default the destructor; Restarter + context auto-disconnect handle cleanup

Today (cwLinePlotManager.cpp:67-70): 

cwLinePlotManager::~cwLinePlotManager() {
    m_restarter.future().cancel();
    waitToFinish();   // spins a nested QEventLoop — latent UAF
}

waitToFinish() calls AsyncFuture::waitForFinished(m_restarter.future()), which runs a nested event loop. CLAUDE.md restricts this to test code precisely because the nested loop delivers queued signals mid-destruction. If a .context(this, …) handler is queued when destruction starts, it can fire against a half-destructed this.

Plan:

  1. Default the destructor (remove the body entirely, or = default in the header). Both the explicit cancel() and the waitToFinish() are unnecessary:
    • AsyncFuture::Restarter::~Restarter() (asyncfuture.h:1861-1870) already calls activeInner.cancel() and cancels the outer deferred when the member is destroyed.
    • Restarter also wires QObject::destroyed on its context (which is this) to the same cancel path (asyncfuture.h:1847-1858), so cancellation actually fires before Restarter's destructor runs — right when the manager starts to die.
    • Every observer is bound through .context(this, …), so Qt auto-disconnects them on destruction; no queued lambda fires against the dead object.
    The cancelled future may still complete on its worker thread, but its result has no observer left and is dropped.
  2. Add a destructor test (cavewhere-test) that destroys a manager mid-solve under ASan and verifies no use-after-free.
  3. cwLinePlotManager::waitToFinish() stays in the header as test-only; the destructor just stops calling it.

Land this before Phase 1 so subsequent phases are protected by the destructor test.

Phase 1: Convert the lineplot sub-tasks to free functions

Rename and drop the cwTask base from four sub-tasks

Today LinePlotWorker::run() uses four cwTask-derived classes only for their compute logic; the inheritance buys nothing because every call site runs them synchronously via the local runTaskInPlace helper (cwLinePlotTask.cpp:52-57) which just calls setUsingThreadPool(false); start(); waitToFinish(IgnoreRestart);. The cwTask machinery (Qt signals, restart state machine, thread-pool toggle) is dead weight for these four. (cwSurvex3DFileReader is not a cwTask — no change needed for it.)

Plan:

  1. Rename and convert each task to a plain free function. Use Monad::Result<T> for value-returning steps and Monad::ResultBase for void-returning steps (Monad/Result.h has no Result<void> specialization, and adding one is out of scope; ResultBase carries the error code and message we need):
    • cwSurvexExporterRegionTask → free function exportSurvexRegion(const cwCavingRegionData&, const QString& outputPath) -> Monad::ResultBase (in a new cwSurvexExporterRegion.h/.cpp).
    • cwCavernTask → free function runCavern(const QString& svxPath, const QString& output3dPath) -> Monad::Result<CavernRunResult> where CavernRunResult { QString output3dPath; QString logText; int exitCode; }. The implementation:
      1. Builds argv as {"cavern", "--log", "--quiet", "--quiet", "--output=<output3dPath>", <svxPath>} — note the new --log (today's code at cwCavernTask.cpp:66-78 does not pass it, which is why nothing is captured today).
      2. Calls cavern_run under the static QMutex (kept inside the function).
      3. Reads QFileInfo(output3dPath).path() + "/" + QFileInfo(output3dPath).completeBaseName() + ".log" into logText. Cavern writes the log on both success and failure, so the read happens unconditionally.
      4. Removes the empty .err sidecar cavern creates next to the log.
      5. If cavern_run returned non-zero, returns Result<CavernRunResult>(logText, SolveErrorCode::CavernFailed) — the log text doubles as the error message so the CavernOutputPage shows real diagnostics rather than "exit code 1".
      6. On success returns Result<CavernRunResult>({output3dPath, logText, 0}). The success path's log carries info-level messages (e.g. "Survey has no control points. Therefore I've fixed X at (0,0,0)") which the page should still surface as warnings/info; don't discard.
      Verified against system cavern 1.4.15; see §7.5 for the reproduction.
    • cwLinePlotGeometryTask → free function generateLinePlotGeometry(…) -> Monad::Result<GeometryResult>.
    • cwFindUnconnectedSurveyChunksTask → free function findUnconnectedSurveyChunks(const cwCave&) -> Monad::Result<UnconnectedSurveyChunksResult>. The nested Result type currently lives on the task class and is referenced from cwLinePlotTask.h:91,97,108; extract it to its own header cwUnconnectedSurveyChunksResult.h (struct holding the list of unconnected-chunk records) so callers depend on the struct, not the old task class. Update cwLinePlotTask::LinePlotCaveData::UnconnectedChunksErrors to reference the extracted struct.
  2. Delete the four cwTask-derived classes and the local runTaskInPlace helper. Update LinePlotWorker::run() to call the free functions sequentially, short-circuiting on the first Result/ResultBase that hasError().
  3. Audit and update sibling call sites:
    • cwSurveyExportManager.cpp:53 — new cwSurvexExporterRegionTask() wired through cwFutureManagerToken. Replace with cwConcurrent::run([…]{ return exportSurvexRegion(…); }) and publish the resulting future to the token.
    • testcases/test_cwSurvex3DFileReader.cpp:24,62 and testcases/test_cwSurvexportTask.cpp:44 — both create a cwCavernTask directly. Update to call runCavern(…) and read the returned Result<CavernRunResult>.
    • testcases/FindUnconnectedSurveyChunksTaskTest.cpp:47 — same pattern; update to call findUnconnectedSurveyChunks(…) and read the returned Result.
    • cwLinePlotGeometryTask has no other call sites (confirmed by grep); safe to delete after Phase 1.
  4. Threading is unchanged: the whole worker still runs on one worker thread via cwConcurrent::run.
  5. Update cavewherelib/CMakeLists.txt to add the new source files and remove the deleted ones.

Outcome: four fewer QObject classes, four fewer signal/slot surfaces, no behavior change. Sets up Phase 2 to thread Result through the chain cleanly.

Phase 2: Consolidate cwLinePlotManager on an explicit AsyncFuture chain

Flatten the nested observe, commit to Monad::Result

Today the restart lambda nests an AsyncFuture::observe(future).context(this, …) inside itself and captures the future by value to read future.result(). Replace with a flat chain and switch the return type to Monad::Result<LinePlotResultData>: 

// cwLinePlotTask.h (new signature):
static QFuture<Monad::Result<LinePlotResultData>> run(Input input);

// cwLinePlotManager::runSurvex():
m_restarter.restart([this]() {
    if (Region.isNull()) {
        return QFuture<Monad::Result<LinePlotResultData>>();
    }

    auto input = cwLinePlotTask::buildInput(Region.data());

    auto computeFuture = cwConcurrent::run(
        [input = std::move(input)]() mutable {
            LinePlotWorker worker(std::move(input));
            return worker.run();   // -> Monad::Result<LinePlotResultData>
        });

    return AsyncFuture::observe(computeFuture)
        .context(this, [this](const Monad::Result<LinePlotResultData>& result) {
            if (result.hasError()) {
                handleSolveError(result);   // populates m_lastCavernOutput
                return;
            }
            updateLinePlot(result.value());
        })
        .future();
});

Plan:

  1. Keep Restarter for edit coalescing.
  2. Use cwConcurrent::run (the project's wrapper around QtConcurrent on cwTask::threadPool()), matching the current convention. Do not introduce a raw QtConcurrent::run(threadPool(), …) here.
  3. Switch cwLinePlotTask::run() to return QFuture<Monad::Result<LinePlotResultData>>. The Result's value() carries LinePlotResultData on success; on failure the Result carries the error string from whichever step failed (export / cavern / parse / geometry) plus, when cavern is the source, the contents of cavern's .log file as additional context (via a wider SolveError struct — see §7.5 below). Update m_restarter, onFutureChanged, the cwFutureManagerToken::addJob call (which takes QFuture<void> — that conversion still works from the new QFuture<Result<…>>), and updateLinePlot to consume the Result branch.
  4. Wire CavernOutputPage as part of this phase — see the sketch in §7.5. Phase 2 adds the manager-side state (last-solve error / cavern log property + change signal) and the QML page; further polish (filtering, navigation from the page tree) can be a follow-up.
  5. Test impact (real, not zero): tests that observe m_restarter.future() directly will see the type change. Tests that only consume stationPositionInCavesChanged() are unaffected. Audit and update each affected test in the same PR as the type switch.

Implementation deviation: the literal Monad::Result<LinePlotResultData> return type was abandoned during Phase 2 implementation. Monad::ResultBase carries only an error string + integer code, which can't hold cavern's .log text and netskel's .err loop-closure stats — the two payloads that motivate CavernOutputPage. Instead, SolveError lives on LinePlotResultData itself (set by the worker when any step fails), the future stays QFuture<LinePlotResultData>, and the manager branches on result.hasSolveError() in its flattened context callback — calling handleSolveError() on failure (which only updates Q_PROPERTYs, leaving the last good line plot intact) and updateLinePlot() on success. This preserves the plan's clean error-branching goal while keeping the rich SolveError payload accessible to QML.

7.4 Solve temp-file lifecycle (Phase 2 deliverable)

Each solve emits four sidecar files: the staged .svx (input to cavern), the produced .3d (input to the parser), the .log diagnostic, and the .err netskel report. The pre-refactor code used a QTemporaryFile with setAutoRemove(false) for the .svx and let cavern drop the other three next to it — nothing in cwLinePlotTask::LinePlotWorker::run() ever deleted any of them. Each successful solve leaked four files into /tmp.

Plan: own a single QTemporaryDir at the top of LinePlotWorker::run() and place all four files inside it (deterministic basenames; the unique directory guarantees no cross-solve collision): 

cwLinePlotTask::LinePlotResultData run()
{
    cwLinePlotTask::LinePlotResultData result;
    if (InputData.regionData.caves.isEmpty()) return result;

    QTemporaryDir workDir;
    if (!workDir.isValid()) {
        // populate result.solveError() and return; see §7.5
        return result;
    }
    // autoRemove(true) is the default; dir + contents disappear at scope exit

    const QString svxPath     = workDir.filePath(QStringLiteral("region.svx"));
    const QString output3dPath = workDir.filePath(QStringLiteral("region.3d"));

    // ... pass these to exportSurvex(svxPath) and runCavern(svxPath, output3dPath)
}

Knock-on effects:

  • exportSurvex() drops its QTemporaryFile machinery — it just opens svxPath for writing.
  • runCavern() takes both svxPath and output3dPath as arguments; no survexFilename + ".3d" string concatenation.
  • cwCavernRunner::run() stops calling QFile::remove(... + ".err") entirely — the QTemporaryDir destructor handles cleanup, and the .err contents have already been copied into CavernRunResult::loopClosureStats in memory before the dir goes away.
  • Per-solve directory isolation makes the function reentrant-safe even if we ever drop the cavernlib mutex (which we won't — see Q1 — but it's a free property).

7.5 CavernOutputPage sketch (Phase 2 deliverable)

Why now

Today there is no UI surface for cavernlib output. When a solve fails, the only signal is a qDebug in cwCavernTask.cpp:82; the user sees an empty line plot and no explanation. Phase 2 introduces structured error propagation via Monad::Result, which is the natural moment to add the page that surfaces it.

Where the text comes from

cavern_lib.h exposes only cavern_run(argc, argv) — no message callback. Capture strategy: pass --log to cavern so it writes the log to a sidecar file, then read that file after cavern_run returns and pack it into CavernRunResult::logText.

Smoke-tested against system cavern 1.4.15 (the same code path our vendored survex submodule runs):

  • The current invocation (cwCavernTask.cpp:66-78) passes --quiet --quiet --output=out.3d with no --log flag, so cavern writes nothing to disk besides the .3d. Add --log to the argv.
  • The log filename derives from the --output value, not the input. With --output=out.3d the log lands at out.log. runCavern should build the log path as QFileInfo(output3dPath).completeBaseName() + ".log" in the same directory.
  • Cavern also writes an out.err sidecar. This is netskel's loop-closure error report (traverse / length-error L / bearing-error B / gradient-error G per loop), populated whenever the cave has loops — see survex/src/netskel.c:449-450. The initial smoke test used an unfixed-station input that produced an empty .err, which led an earlier draft to recommend deleting the file. Don't delete it. Capture its contents into CavernRunResult::loopClosureStats alongside logText. Disk cleanup is the worker's QTemporaryDir (see §7.4 below), not runCavern.
  • On success the log contains info-level messages such as "Survey has no control points. Therefore I've fixed dakeng.1 at (0,0,0)". Keep this — it surfaces silent assumptions cavern made that the user may want to fix.
  • On failure the log contains the error text (e.g. "bad.svx:2: error: Station “outer.inner.1” not exported from survey “outer”") followed by "There were N warning(s) and M error(s) - no output files produced." The .3d is not written; runCavern detects failure via the cavern_run exit code, not the file's existence.
  • Double --quiet suppresses stdout but does not empty the log file. Keep the double-quiet for the in-process call so cavernlib doesn't print to the CaveWhere process's stdout; the log file captures everything we need.

Reproduction: 

cavern --log --quiet --quiet --output=out.3d data.svx
# On clean input: exit 0, out.3d + out.log + out.err produced
# On broken input: exit 1, out.log + out.err produced (no .3d)
cat out.log   # full diagnostic text either way

8. Open Questions / Risks

Q1: Is cavern_run thread-safe?

No, not by default. cavern_run does not spawn the cavern binary as a subprocess — it calls into cavernlib in-process (the C library we ported survex to so it could run on mobile). Cavern's internal state is a big pile of global statics (parser state, message catalog, file handles, etc.), so the implementation guards entry points with mutexes to serialise concurrent callers from the same process.

Implication for the port: two solves cannot run cavernlib concurrently. The line-plot pipeline already runs one solve at a time via AsyncFuture::Restarter (a new solve cancels the old before starting), so we're fine for the internal pipeline. If we ever want to run solves from multiple independent managers in parallel (e.g. background validation of an external file while a user-triggered solve is in flight), they will contend on the cavernlib mutex and execute serially — not a correctness problem, but worth knowing before optimising for throughput.

Q2: Is cwStationPositionLookup thread-safe for concurrent reads?

Not an issue. cwStationPositionLookup is copied by value when crossing the worker / UI-thread boundary (snapshot in, fresh value out), and all reads through cwCave happen on the main thread. The worker builds its own copy from the input snapshot, computes a new lookup value, and the UI thread swaps it in via setStationPositionLookup. No concurrent access path exists. Keep the convention: only the main thread touches cwCave::stationPositionLookup().

Q3: What if cavern crashes?

CaveWhere crashes. Because cavern_run is an in-process library call (not a subprocess), a segfault, abort, or uncaught exception inside cavernlib takes the whole CaveWhere process down. Returning Monad::Result<…> from the worker only helps for cleanly-detected errors; it does nothing for hard crashes.

Mitigations to consider (none required for the initial port, but worth tracking):

  • Treat any cavernlib crash bug as a release-blocker for that survex submodule bump (relevant to issue #519).
  • Long-term option: wrap cavernlib in a child QProcess for fault-isolation. Trades off the simplicity (and mobile-friendliness) we gained by porting to a C library. Not a v1 task.
Action for the port: still return Monad::Result<LinePlotResultData> so cleanly-detected errors land in CavernOutputPage — just don't claim it makes the pipeline crash-proof.

Q4: Cooperative cancellation needs to reach inside each step.

Today's behavior (see §4): Restarter::restart() only flips QFuture::cancel(). The worker doesn't check it, so the old solve runs to completion (export → cavernlib → parse → geometry), holds the cavernlib mutex (Q1) for the duration, and the new solve waits behind it on every rapid edit. The result is dropped on return, but the latency hit is real.

Plan (for Phase 2 of the port, or a fast follow-up): use Qt 6's promise-aware QtConcurrent::run overload — the promise is supplied as the first parameter, not captured. The QFuture returned by the run owns the cancel bit, and Qt 6 forwards QFuture::cancel() into the promise so the worker observes it via promise.isCanceled(): 

// inside cwLinePlotTask::run():
return cwConcurrent::run(
    [input = std::move(input)](QPromise<Monad::Result<LinePlotResultData>>& promise) mutable {
        LinePlotWorker worker(std::move(input), promise);
        promise.addResult(worker.run());
    });
Then thread the QPromise& through each step and bail at every progress point. Use Monad::ResultBase for void-returning steps (no Result<void> specialization exists in monad/Monad/Result.h), and define a project-local error-code enum. ResultBase's built-in codes are {NoError=0, Warning=1, Unknown=2, CustomError=1024} (Monad/Result.h:24-29); project codes start at CustomError: 
// cwLinePlotErrorCodes.h
namespace SolveErrorCode {
    enum : int {
        Cancelled       = Monad::ResultBase::CustomError,       // 1024
        ExportFailed    = Monad::ResultBase::CustomError + 1,   // 1025
        CavernFailed    = Monad::ResultBase::CustomError + 2,   // 1026
        ParseFailed     = Monad::ResultBase::CustomError + 3,   // 1027
        GeometryFailed  = Monad::ResultBase::CustomError + 4,   // 1028
        ValidationFailed = Monad::ResultBase::CustomError + 5,  // 1029
    };
}

// inside LinePlotWorker:
Monad::ResultBase exportStep() {
    if (m_promise.isCanceled()) {
        return Monad::ResultBase(QStringLiteral("cancelled"),
                                 SolveErrorCode::Cancelled);
    }
    return exportSurvexRegion(m_input.regionData, m_svxPath);
}
The SolveError::Step enum on the manager (§7.5) maps 1:1 onto these codes, so handleSolveError can route from result.errorCode() to the right SolveError::Step without parsing strings. Bail points: before invoking cavernlib (this is the big one — it relieves the mutex contention), between splitLookupByCave iterations, between geometry primitives.

Verification: cwConcurrent::run is a thin variadic forward to QtConcurrent::run(cwTask::threadPool(), …) (cwConcurrent.h:17-19), so Qt's overload-resolution picks the promise-aware path based on the lambda's first parameter type. The pattern is in use elsewhere in CaveWhere; no extra plumbing needed.

Q5: External-centerline file ownership and watcher coordination.

Out of scope here. Handled by the external-centerline plan (plans/EXTERNAL_FILE_INTEGRATION_PLAN.html) — copy layout (§5.1), reconcile-with-cwSaveLoad (§5.2), watcher ownership (§5.2), and live-link vs import-mode (§5.3). This research doc is just about porting the line-plot pipeline; the file-ownership decisions are already pinned down upstream of it.

Q6: What Restarter cancellation actually does.

Restarter::restart() flips QFuture::cancel() on the previous future and starts a new one. That's the whole mechanism — it does not interrupt the running worker. Today's cwConcurrent::run lambda inside cwLinePlotTask::run() never consults isCanceled(), so the old solve runs to completion and the result is dropped on return. The "cancel" is really "discard."

This was previously flagged "resolved — well-tested," which is misleading: the result-discard path is well-tested; true in-flight cancellation is unimplemented and is the work tracked in Q4. Don't rely on Restarter for latency-bounded cancellation until Q4 is done.

Summary

Scope of this port

Phases 0, 1, and 2 — the lineplot pipeline itself. Phase 3 (external-centerline pipeline) and Phase 4 (unified solve manager) are deferred; once Phases 0–2 land, write a fresh plan for Phase 3 rather than executing the §5 sketch as-is.

Most important findings

  • Latent UAF in the destructor. Today cwLinePlotManager::~cwLinePlotManager() calls waitToFinish(), which spins a nested QEventLoop in violation of CLAUDE.md. Phase 0 defaults the destructor entirely — Restarter::~Restarter() already cancels the in-flight future, and Restarter also hooks QObject::destroyed on its context so cancellation fires before destruction completes.
  • Four lineplot sub-tasks don't need cwTask: cwSurvexExporterRegionTask, cwCavernTask, cwLinePlotGeometryTask, and cwFindUnconnectedSurveyChunksTask. Every call site runs them synchronously via runTaskInPlace. Phase 1 renames them to plain free functions returning Monad::Result<T> (or ResultBase for void), extracts cwFindUnconnectedSurveyChunksTask::Result to its own struct, and updates the four sibling call sites (cwSurveyExportManager, test_cwSurvex3DFileReader, test_cwSurvexportTask, FindUnconnectedSurveyChunksTaskTest).
  • Phase 2 is a refactor + a new UI page. The pipeline already runs off the UI thread. Phase 2 flattens the nested AsyncFuture::observe(), switches cwLinePlotTask::run() to return QFuture<Monad::Result<LinePlotResultData>>, and wires the new CavernOutputPage (§7.5). The page's text comes from cavern's .log file, captured by the new runCavern free function and threaded through the Result.
  • Worker-side const access is load-bearing. The Input snapshot is built from implicitly-shared Qt containers, not deep-copied. Any non-const access on the snapshot will detach a shared buffer the UI thread is mutating. Phase 1 reviewers should audit the worker for non-const access (at(i) only; no operator[]).

Most Concerning Risk

Cavern crashes take the whole CaveWhere process down (Q3). cavern_run is an in-process library call into cavernlib (not a subprocess — that's how survex runs on mobile), so a segfault or abort inside cavernlib aborts CaveWhere. No clean v1 mitigation: returning Monad::Result from the worker only covers cleanly-detected errors. Treat any reproducible cavernlib crash as a release-blocker for that submodule bump (#519 in the issue tracker). A long-term option is to wrap cavernlib in a child QProcess for fault-isolation, but that trades off the simplicity and mobile-friendliness we gained from the library port — not a v1 task.

Secondary risk: cancellation must propagate through every step (Q4) or a stale solve can hold the cavernlib mutex (Q1) in front of the user's next edit. Concrete fix lives in the Q4 section: thread a QPromise through each worker step and check isCanceled() at every progress point.
