Trace-rule unification

One traced terminal rule; fill is an optional property of the layer, not a separate rule. Folds the uncommitted commit-4.2 work (the Trace polygon rule / OutputKind::Polygons) into the existing Trace polyline path. Supersedes the two-rule split — see SYMBOLOGY_PALETTE_PLAN.html Decisions 10–12 and CEILING_CHANNEL_DESIGN.html.

Status of the tree. The 4.2 changes (filled stamped glyphs) are in the working tree but will not be committed as-is. This plan is the shape we actually want; the implementer can evolve the working tree or start from a clean checkout — the end state is identical either way. The pieces 4.2 added that this plan keeps are called out in "What 4.2 keeps" below.

The realization: polyline and polygon are one concept

Reading the code rather than the design comments, the two terminal rules are not two concepts. They emit byte-identical geometry:

// cwTracePolylineRule::tracePolylines  AND  cwTracePolygonRule::tracePolylines — same line:
return {QPolygonF(strokeWorld)};

And the layout never closes the path — QPainterPath::addPolygon appends an open subpath. So:

  • An unfilled polygon and a polyline render identically — the pen strokes the same authored points.
  • Closure is not a property of "polygon." It happens only at paint time, because the fill rule visually closes the path. A cartographer who wants a closed outline closes it in their own point list (exactly what the demo's kStreamTriangle does — it repeats the first vertex).
  • The only thing "Polygon" adds over "Polyline" is that it carries a fill (and a per-region pen).

So there is no second terminal. There is one operation — trace the stroke — and an optional fill. The user's mental model is the correct one: draw a line, then fill it or don't.

Target shape — four decisions

1 · One rule: cwTraceRule, schema key "Trace" recommended

Rename cwTracePolylineRule → cwTraceRule (file + class) and delete cwTracePolygonRule.{h,cpp}. Its tracePolylines() body is unchanged (verbatim trace; any lateral offset is still the earlier Offset stroke rule's job). Display name becomes "Trace".

The display name is the stable palette-schema key. Renaming "Trace polyline"/"Trace polygon" → "Trace" would normally be a migration, but the format is unreleased on this branch (see memory project_symbology_unreleased_no_protobump), so we just rewrite the seed and tests. No protoVersion() bump.

Naming alternatives considered: "Trace line" (reads well, but it fills too) and "Trace region" (implies always-filled). "Trace" is the most honest: it traces; fill is a layer property shown next to line colour in the editor. Bikeshed freely — it's one string in one place.

2 · OutputKind { None, Stamps, Trace }

// cwPlacementRule.h
enum class OutputKind { None, Stamps, Polylines, Polygons Trace };

Drop two values, add one. The -Wswitch guarantee still pays for itself across the three remaining switches (validator, cache, layout) — the compiler will walk you to each site. Keeping the enum (rather than the agent's "infer everything") preserves the rule's self-description and the validator's targeted message.

3 · Fill discriminator: infer from fillColorLight.isValid() chosen

A traced region is filled iff the layer carries a valid fill colour. No new state, and it is exactly "set a fill or don't." This mirrors how polygonPen() already treats an invalid pen colour as "no outline." This is the path the rest of the plan assumes — there is no separate filled flag and no new proto field for it.

Decided: A. The rejected alternative (B) was an explicit bool filled on the layer plus a proto field, to let the editor un-check "fill" without losing the chosen colour. We are not doing that at the model level — when the editor (commit 8/9) wants an off-toggle that remembers a colour, it can hold that in editor state and write an invalid fill* when off. Keeping the data model to "valid fill colour ⇒ filled" avoids a second source of truth.

4 · One output container: QVector<cwGlyphSubPath> traced

This is the structural payoff. Collapse the two Layer containers into one list of typed sub-paths; a traced region is a Stroke with no fill, or a Polygon when the layer has a fill.

// cwSketchDecorationLayout.h
struct cwBrushDecorationGeometry {
    struct Layer {
        enum Kind { Stamps, Trace };               // was { Stamps, Polylines, Polygons }
        Kind                     kind = Stamps;
        QVector<cwGlyphSubPath>  traced;          // kind == Trace; one per region (Stroke, or Polygon if filled)
        QVector<cwResolvedStamp> stamps;          // kind == Stamps
        QVector<QPolygonF>       offsetPolylines;
        QVector<cwGlyphSubPath>  polygons;
    };
    QVector<Layer> layers;
};

The two layout branches (lines ~154 and ~164 today) merge into one:

// cwSketchDecorationLayout::layout()
if (terminal.rule->outputKind() == cwPlacementRule::OutputKind::Trace) {
    out.kind = cwBrushDecorationGeometry::Layer::Trace;
    const QVector<QPolygonF> regions = terminal.rule->tracePolylines(layerStroke, terminalCtx);
    const bool filled = layer.fillColorLight.isValid();
    out.traced.reserve(regions.size());
    for (const QPolygonF &region : regions) {
        cwGlyphSubPath p;
        p.kind = filled ? cwGlyphSubPath::Polygon : cwGlyphSubPath::Stroke;
        p.path.addPolygon(region);
        p.penColorLight = layer.lineColorLight;
        p.penColorDark  = layer.lineColorDark;
        p.penWidthMm    = layer.lineWidthMm;
        if (filled) {
            p.fillColorLight = layer.fillColorLight;
            p.fillColorDark  = layer.fillColorDark;
        }
        out.traced.append(p);
    }
    geometry.layers.append(out);
    continue;
}

The eventual commit-5 painter then iterates a single list and asks each sub-path "filled?" — no branch on Layer kind, and the same cwGlyphSubPath drawing path serves traced lines, traced fills, and glyph stamps. A side benefit: each of a railroad's two rails is now its own cwGlyphSubPath and could carry an independent pen later, instead of all rails sharing one layer pen.

What 4.2 keeps

  • cwDecorationLayer::fillColorLight/Dark (the layer is the staging ground for brush appearance until the commit-9 param migration — same altitude as the existing line* pen fields).
  • Proto fields DecorationLayer.fillColorLight = 16 / fillColorDark = 17 and their cwSymbologyPaletteIO round-trip. No protoVersion() bump.
  • The cwGlyphSubPath value type and the typed cwGlyphTessellationCache::tessellate() return (now returning const& per the cleanup pass).
  • The cwResolvedStamp::subPaths shape and the per-sub-path stamp materialisation in the layout.

Out of scope / future

  • Per-rail / per-region independent pens. The unified traced list makes this possible (each region is its own sub-path) but this plan still copies the one layer pen to every region. Differentiated pens are a later enhancement.
  • Commit 5 — the RHI painter (cwSketchPainter) consumes traced + stamps; it inherits the single-list simplification. Not built here.
  • Commit 9 — param migration. line* and fill* move from the layer into the Trace rule's params together; until then both live on cwDecorationLayer.
  • Editor off-toggle that remembers a colour. Rejected at the model level (decision §3 = A). If the editor later wants this UX, it lives in editor state and writes an invalid fill* when off — no model/proto change.
