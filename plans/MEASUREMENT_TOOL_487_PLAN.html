3D Measurement Tool

A two-point measuring interaction in the 3D view: hover to highlight the station under the cursor, click two points (snapping to survey stations), and read back distance / azimuth / inclination / horizontal / vertical / deltas — with a copy-to-clipboard. Built on the pickable centerline and clamp-to-station seams delivered by #530.

Scope. GitHub #487, delivered as four commits: C1 — shared snap-pick core C2 — measurement math C3 — interaction C4 — QML view & UI. C1 is a behaviour-preserving refactor of cwCoordinatePicker that is worthwhile on its own (it also closes a #530 §11 follow-up); C2–C4 build the feature on top.

2. Commit breakdown & why multi-commit

One commit would braid a refactor of existing picker code together with a large new feature spanning C++ logic, pure math, a QML view and clipboard glue — hard to review, hard to bisect. The seams give four commits that each build and test on their own:

Why occlusion is free. The pick uses kinds=All and the BVH returns the nearest hit by depth. A station behind geometry yields a triangle hit, not a line hit, so qobject_cast<cwRenderLinePlot*>(hit.object()) is nullptr, the snap is refused, and hoverSnapped is false. An occluded station never highlights. Consequence: the indicator itself is a plain always-on-top 2D overlay — no depth-occluded billboards needed.

4. C1 — shared snap-pick core

Both the coordinate picker and the measuring tool need the same primitive: "the snapped world point under this screen position." Today that logic is inline in cwCoordinatePicker::pick(). Lift it into a reusable, header-only-friendly seam so neither tool duplicates query-building or the snap.

4.1 cwCamera::pickQuery(double pixelRadius) const new

Move the tolerance derivation from cwCoordinatePicker.cpp:113–128 verbatim onto the camera — it is pure camera math (ortho uses pixelsPerMeter(); perspective uses projectionMatrix()(1,1) and viewport height). Returns a cwPickQuery with kinds = All. This is the #530 §11 "consolidate into a cwCamera::pickTolerance() helper" follow-up, now paid for.

4.2 cwScenePick free function new

namespace cwScenePick {
    struct Result {
        QVector3D world;                 // hit point, or station position if snapped
        bool hit = false;
        bool snappedToStation = false;
        int  stationVertexIndex = -1;    // from-vertex index when snapped, else -1
    };

    // Ray-cast screenPoint against the scene's BVH (nearest by depth), then clamp
    // a centerline hit to the nearest survey station within pixelRadius. Pure w.r.t.
    // scene state: reads, never mutates.
    Result snappedPoint(QPointF screenPoint,
                        const cwCamera& camera,
                        cwGeometryItersecter& intersecter,
                        double pixelRadius);
}

Body = exactly what cwCoordinatePicker::pick() does between building the ray and setting m_scenePoint: rayFromQtViewport → camera.pickQuery(pixelRadius) → intersectsDetailed → qobject_cast<cwRenderLinePlot*> + segmentEndpoints + snapToStation. cwCoordinatePicker::pick() shrinks to a caller that maps the Result into its CRS/WGS84 outputs. No behaviour change — the existing picker and linePick tests are the guard.

4.3 cwInteraction::pixelsForMillimeters(double mm) new

The pixelRadius that snappedPoint/pickQuery consume must be a physical size, not a fixed pixel count: a logical pixel is not a constant physical size across macOS "scaled resolution" modes, so a hard-coded px radius drifts physically as the user changes display scaling (the DPR bug found in testing). The radius is therefore specified in millimeters and converted per-call:

// on cwInteraction (the QQuickItem base all pickers inherit)
double pixelsForMillimeters(double mm) const;                       // instance: reads window()->screen()
static double pixelsPerMillimeter(double geometryWidthPx,           // pure, unit-tested without QGuiApplication
                                  double physicalWidthMm);

pixelsForMillimeters derives logical-px/mm from screen()->geometry().width() / physicalSize().width() (width only — pick tolerance only needs to be isotropic), falling back to 96 DPI when off-screen / headless / no EDID physical size. Both cwCoordinatePicker (kPickRadiusMillimeters = 1.5) and cwBaseTurnTableInteraction (kPivotPickRadiusMillimeters = 4.0) feed it into camera.pickQuery(); the measurement tool's hover (C3) must do the same so its snap radius is DPI-correct for free. The pure pixelsPerMillimeter seam is what the [cwInteraction] unit test covers.

5. C2 — measurement math

A pure function of two scene points, unit-tested exactly like snapToStation. Scene axes are +X East, +Y North, +Z up (the local frame is the region's projected CRS translated to the origin).

namespace cwMeasurementMath {
    struct Measurement {
        double distance;      // |b - a|, 3D
        double horizontal;    // hypot(dx, dy)
        double vertical;      // dz  (signed, + up)
        double azimuth;       // atan2(dx, dy) normalized to [0,360)  — GRID north (see §8)
        double inclination;   // atan2(dz, horizontal), [-90,90], + up
        double deltaEast;     // dx
        double deltaNorth;    // dy
        double deltaUp;       // dz
    };
    Measurement between(QVector3D a, QVector3D b);
}
Degenerate cases to name in code. Coincident points (distance == 0): azimuth and inclination are undefined — return 0 for both and let the readout show —. Purely vertical shot (horizontal == 0, dz != 0): azimuth undefined → 0; inclination is ±90. Use std::atan2 (handles the zero-args branch without a divide).

Angles are computed in radians and converted to degrees with a k-prefixed constant. Display formatting (metres/feet, degrees/grads, precision) is the QML layer's job via CaveWhere's existing unit settings — the helper returns canonical SI + degrees only.

6. C3 — the interaction state machine

cwMeasurementInteraction : cwInteraction, shaped like cwCoordinatePicker. It holds the two points and the current mode, drives hover, and exposes the readout as properties.

State machine. A click in Complete resets and starts a new measurement. Esc exits the tool entirely (back to turn-table), per the locked decision.

6.1 Surface

class cwMeasurementInteraction : public cwInteraction {
    enum class Mode { Free, StationOnly };   // Q_ENUM
    // properties (Q_PROPERTY ... NOTIFY, plain members + setters — no bindable):
    //   cwCamera* camera; cwScene* scene;
    //   Mode mode;                      // Free | StationOnly
    //   QVector3D hoverPoint; bool hoverSnapped; bool hoverValid;
    //   QVector3D firstPoint, secondPoint; bool hasFirst, hasMeasurement;
    //   double distance, azimuth, inclination, horizontal, vertical;
    //   double deltaEast, deltaNorth, deltaUp;

    Q_INVOKABLE void hover(QPointF screenPoint);   // every mouse-move while active
    Q_INVOKABLE void place(QPointF screenPoint);   // a click
    Q_INVOKABLE void reset();                      // back to Idle, clear points
    Q_INVOKABLE void copyToClipboard() const;      // QGuiApplication::clipboard()
};

6.2 Behaviour

  • hover() calls cwScenePick::snappedPoint(pos, camera, intersecter, pixelsForMillimeters(kRadiusMm)) (the mm radius keeps the snap physically constant across display scaling — §4.3); sets hoverPoint/ hoverSnapped. hoverValid = hit && (mode==Free || snappedToStation) — i.e. would a click here place a point. In AwaitingSecond it also recomputes the live preview measurement A→hoverPoint so the readout tracks the cursor.
  • place() ignores the click when !hoverValid (Station-only over empty space is a no-op). Idle→store A→AwaitingSecond; AwaitingSecond→store B, compute via cwMeasurementMath::between→Complete; Complete→reset() then place A.
  • Mode is a plain settable property; the QML exposes a Free/Station toggle. Switching to Station-only mid-measure does not retroactively invalidate an already-placed free point — it only gates future placements (documented).
  • copyToClipboard() formats the active-unit readout as multi-line text and writes it via QGuiApplication::clipboard(). Kept in C++ because QML has no clipboard write.
  • Esc is handled in QML (a Shortcut/Keys calling deactivate()); the manager restores the turn-table.
Hover cost. One BVH ray-cast per mouse-move. The BVH pick is sub-millisecond and only runs while the tool is the active interaction, so no throttling in v1; revisit only if profiling on a large cave says otherwise.

7. C4 — QML view, overlay, readout, copy

MeasurementInteractionView.qml wraps the interaction, mirroring CoordinatePickerInteraction.qml:

  • Input. A HoverHandler (onPointChanged → interaction.hover(point.position)) and a TapHandler (onTapped → interaction.place(eventPoint.position)). A Shortcut on Esc → deactivate(). Reuse TurnTableForwardingHandlers so orbit/zoom still work while measuring (the clip tool and picker both do this).
  • Overlay markers via one TransformUpdater re-projecting world→viewport on camera change (the picker's pattern): the hover indicator (accent ring when hoverSnapped, a smaller free-point dot otherwise — and hidden / "snap to a station" hint when Station-only over empty space), and the placed A / B dots (always visible, so they stay readable when orbited behind geometry).
  • Connecting line: a QCanvasPainterItem drawing A↔B (and the rubber-band A↔hover in AwaitingSecond) in screen space, on top — never a 3D line (a 3D line would be occluded, wrong for an always-on annotation).
  • Readout panel (template: CoordinatePickerPopup.qml): distance / azimuth / inclination / horizontal / vertical / ΔE ΔN ΔZ, formatted in the user's units, a Free/Station toggle, and a Copy button calling copyToClipboard(). Use QC.Label/BodyText and Theme tokens (no bare Text, no hardcoded colours/sizes).
  • Toolbar. Add a measure IconButton beside the pick button (GLTerrainRenderer.qml:199), selected: activeInteraction === measurementId, toggling activate()/deactivate(); add measurementId to the InteractionManager.interactions list (:117).

8. The grid-azimuth caveat

Scene XY is the region's projected CRS translated to the origin, so atan2(dx, dy) yields the azimuth relative to grid north (scene +Y) — not true north and not magnetic. For a north-up projection near its central meridian this equals true north; away from it, grid convergence rotates grid north off true north by up to a couple degrees, and a genuinely rotated CRS makes the bearing wrong versus surveyor expectation.

v1 scope: report grid azimuth relative to scene +Y and label it as such (e.g. "Azimuth (grid)"). True-north correction — apply the CRS grid convergence at the measurement location via PROJ — is a documented follow-up, deliberately kept out of the pure math helper so between() stays a function of two points. (Consistent with the "ship the simple tier first" preference.)

Cheaper than it looks: the PROJ math already exists as the pure helper cwGridConvergence::computeAt() (cwGridConvergence.h:89), so the follow-up is mostly wiring, not new geodesy. Apply it in the C3 interaction layer (trueAzimuth = gridAzimuth + convergence) rather than in between() — that keeps CRS knowledge out of the pure-math helper and preserves the front/back split.

10. Out of scope / follow-ups

  • True-north / grid-convergence correction for azimuth (see §8) — PROJ-based, deferred, but cheaper than it looks: cwGridConvergence::computeAt() (cwGridConvergence.h:89) already supplies the convergence; the C3 layer just adds it to the grid azimuth.
  • Station names in the readout — needs the additive cwLinePlotGeometry::Result metadata flagged in #530 (vertex→station-name map was discarded). Snap currently yields position + vertex index only; "from STA12 to STA15" waits on that.
  • Multiple / persisted measurements — v1 keeps exactly one live measurement.
  • Unify azimuth/inclination math with LiDAR note georeferencing — cwMeasurementMath::between() shares byte-identical bearing (atan2(dx, dy) + [0,360) wrap) and clino (atan2(dz, horizontal), ±90 guard) math with cwNoteLiDARTransformation::calculateNorth() / calculateVerticalAngle() (cwNoteLiDARTransformation.cpp:102,123). Once C2 lands, have the LiDAR path delegate to between() after its up-quaternion rotation so the two stay in sync. Its own commit + tests; C2 ships as a dependency-free leaf first. (dewalls vector.cpp is not a merge target — vendored, unit-typed, applies instrument corrections.)
  • Explicit cwPickQuery in the other 3 BVH callers (cwLeadView, cwBaseTurnTableInteraction ×2) — the standing #530 §11 item; unrelated to this tool but in the same neighbourhood. Done alongside C1: cwLeadView occlusion now uses cwPickQuery::Solid (lines don't occlude a lead), and the turntable pivot uses pickQuery(pixelsForMillimeters(...)) with kinds = All so it can orbit a bare centerline when the solid geometry is hidden.
