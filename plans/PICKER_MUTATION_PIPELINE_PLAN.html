Picker Mutation Pipeline

cwGeometryItersecter · consolidates review items R1 + A1 + R3 from the issue #505 review, expanded with hidden-until-pick-ready (Phase 4) · no governing plan predates this · status: proposed · R3 decided = Option A (2026-07-19)

The issue #505 fix added a fourth way to keep the published pick BVH (m_bvh) consistent under a mutation. That crossed the threshold where the pattern is worth naming: the intersecter has no single model for "apply a mutation to the published BVH synchronously, then reconcile the heavy work asynchronously." Each mutation path improvises the sync/async split, one recompute helper is now duplicated verbatim, and one mutation (addObject) has no synchronous story at all.

The three review items, and why they are one refactor
  • A1 — the structure: unify all published-BVH mutations behind one applyPublishedDelta + async-recompute pipeline.
  • R1 — A1's first, free step: the world-box loop + buildTopLevel sequence is now duplicated between refreshPublishedModelMatrix and launchBuildJob. Factoring it is the shared core A1 is built around.
  • R3 — the acceptance question A1 must answer: addObject needs an O(vertices) sub-BVH build, so it is the one mutation that can't refresh synchronously. A uniform pipeline turns "new object invisible to picks until the worker lands" from a silent hole into a designed case.

Out of scope: R2 (migrate cwRenderGLTF's remaining Q_OBJECT_BINDABLE_PROPERTY + cwTracked dual-store to plain members) — that's a QML-object idiom cleanup, unrelated to intersecter architecture. Track it separately.

Current state — four ad-hoc consistency paths

Every mutation must leave m_bvh internally consistent (parallel arrays subBvhs / modelMatrices / inverseModelMatrices aligned, keyToSlot valid, topLevel leaves indexing live slots) while in-flight query snapshots keep reading the old immutable copy. Today each site does that its own way:

MutationSync effect on m_bvh todayAsync follow-up
removeObject / clear(id) Copy-on-write, null the slot via invalidatePublishedSlot scheduleObjectRebuild → worker drops the key sync
setModelMatrix Copy-on-write, update matrices + rebuild top-level via refreshPublishedModelMatrix (added for #505) scheduleTopLevelRebuild re-snapshots Nodes sync
addObject (triangles/lines/points) none — the object is unpickable until the worker installs scheduleObjectRebuild builds the sub-BVH (O(vertices)) gap (R3)
worker install callback Swaps in the freshly built BvhData, re-applying the dirty + live-Nodes filters (shouldPublish) — async

Target state — one delta pipeline

Collapse the four variants into a single main-thread primitive that applies a typed delta to m_bvh copy-on-write, plus the existing async recompute for the heavy sub-BVH work. The published-BVH invariants are then enforced in exactly one place.

// main thread only; copy-on-write, never edits a published BvhData in place
enum class PublishedDelta { RemoveSlot, SetMatrix, InsertCoarse /* R3 */ };
void applyPublishedDelta(const Key& key, PublishedDelta, /* payload */ ...);

// the shared core R1 extracts — used by applyPublishedDelta AND launchBuildJob
static QVector<BvhNode> BvhData::rebuildTopLevel(const BvhData&);   // world boxes ⇐ modelRootBox·matrix

Phases

Phase 1 · R1 · low risk, pure dedup

Extract BvhData::rebuildTopLevel()

Factor the world-box loop + buildTopLevel call — currently identical in refreshPublishedModelMatrix and the tail of launchBuildJob — into one function. World box for a slot = subBvhs[i]->modelRootBox.transformed(modelMatrices[i]), null slots contribute a null box (buildTopLevel already skips nulls and keys leaves by slot index).

  • Both call sites must produce byte-identical top-levels — that equivalence is the whole correctness argument for the synchronous refresh, so a shared function makes it true by construction rather than by comment.
  • Mind the thread boundary: launchBuildJob's tail runs on the worker thread building a fresh BvhData; refreshPublishedModelMatrix runs on the main thread over a CoW copy. The extracted helper must be a pure function of a BvhData (no this, no member state) so it's safe on both — like the existing static buildTopLevel.

Gate: no behavior change. Existing [cwGeometryItersecter][twoLevel] suite + the #505 regression (setModelMatrix updates picks before the async rebuild lands) stay green.

Phase 2 · A1 · the structure

Route removeObject / setModelMatrix through one applyPublishedDelta

Fold invalidatePublishedSlot and refreshPublishedModelMatrix into a single copy-on-write entry point that takes a typed delta, applies it, calls rebuildTopLevel where needed, and publishes. One place asserts the parallel-array / keyToSlot / slot-index invariants.

  • Keep it purely mechanical first: same two behaviors, one door. No semantic change to picks.
  • Leave the async recompute (scheduleTopLevelRebuild / scheduleObjectRebuild + the shouldPublish install filters) exactly as is — A1 is about the sync delta, not the worker reconciliation.
  • Preserves the documented "eventually correct + usually-immediate" guarantee: a stale competing build can still transiently re-install, and the last restarted build (re-snapshotting the updated Nodes) remains the source of truth.

Gate: full intersecter suite green; the removal-race test (a28ff96f) and stale-install test still pass; no new sync path for addObject yet.

Phase 3 · R3 · decided: Option A

Decide addObject's pre-build pick policy

Decision (2026-07-19): Option A — the not-ready contract. Picks ignore an object until its async build lands; the engine documents that, and Phase 4 (below) makes the window invisible to the user by keeping heavy objects render-hidden until they are pick-ready. Options B and C are kept here for the record:

A new object's sub-BVH is O(vertices) and can't be built synchronously the way an O(objects) transform refresh can, so between addObject and the worker install a pick can't test its primitives. The candidate policies were:

OptionBehaviorCost / caveat
A. Not-ready contract Document that picks ignore an object until its build lands; callers gate on readiness (e.g. NoteLiDARItem already checks gltf.status===Ready). Zero engine cost; pushes the burden to every caller. Simplest, honest.
B. Coarse-box broad phase applyPublishedDelta(InsertCoarse) inserts a top-level leaf with the object's world AABB and a null sub-BVH; a pick against it returns a broad-phase / bounding hit until the exact sub-BVH installs. The object's box is already computed synchronously in addTriangles. But a box hit isn't a triangle hit — only acceptable where approximate is better than nothing; must not report false-exact hits.
C. Sync sub-BVH for small objects Build the sub-BVH inline when vertex count is under a threshold (k-constant); fall back to async above it. Helps small notes, not large LiDAR scans — which are exactly the #505 case. Weakest payoff.

Why A won: C helps small notes but not large LiDAR scans — which are exactly the #505 case — and B's box-hit-isn't-a-triangle-hit caveat pushes complexity into every pick consumer. A is honest and zero-cost in the engine; Phase 4 then closes the UX hole A leaves open (a visible-but-unpickable object) by inverting it: nothing is visible until it is pickable. The InsertCoarse delta stays reserved as a follow-up if a real caller ever needs pre-build broad-phase picks.

Phase 4 · scope expansion · hidden-until-pick-ready

Keep heavy objects render-hidden until their BVH lands

Option A leaves a window where an object renders but can't be picked. For huge geometry — LAZ point clouds above all, plus large note GLBs — that window is long enough for the user to see the object and click it. Instead of teaching every caller about readiness, establish the invariant the other way around: rendered ⇒ pickable. An object registered with the intersecter stays render-hidden until its sub-BVH is published, then appears — already pickable on its first visible frame.

Mechanism

  • Per-key readiness query on the intersecter: bool isObjectPickReady(const Key&) const — true when the published m_bvh has the key in keyToSlot with a non-null sub-BVH slot. Cheap: both structures already exist. bvhReady() (emitted on every install) is the notification edge; owners re-check on it.
  • Owner-level gating, not store-level: owners publish effective visibility = userVisible && pickReady into the scene visibility store. The store and RHI backend stay dumb readers (preserves the #579 architecture — no store→intersecter coupling), and user hide/show toggles keep working unchanged.
  • Centralized in cwRenderObject, not per-type. Only three render types register with the intersecter (four addObject sites: cwRenderPointCloud.cpp:36; cwRenderTexturedItems.cpp:145, :214 — covers GLTF + scraps; cwRenderLinePlot.cpp:59). The base class already owns the store-publish path (setVisible / publishVisibility) and can reach the intersecter, so it carries the shared machinery once: the pending-key set, the single bvhReady() connection, the isObjectPickReady re-check, and the AND into the publish. Each subclass only reports "registered key X" / "removed key X" at its existing call sites; the publishVisibility() overrides (line plot, textured items) compose the gate into the sub-item state they already publish.
  • Verified precondition (no deadlock): the BVH build is visibility-independent — visibility filters queries via the snapshot at traversal time (cwGeometryItersecter.cpp:1173, visibleBoundingBox), never the build's input Nodes. A hidden object still gets its sub-BVH built, so the gate always releases.

Traps to design around

  • Silent addObject rejection must not mean permanently invisible. addObject drops geometry it can't index (layout reject in cwRenderTexturedItems::addItem; Type::None from cwGltfLoader for strip/fan GLBs). Gate only the span accepted-by-addObject → published; a rejected or never-registered object renders under plain userVisible as today. Concretely: the gate arms when addObject accepts and disarms on readiness — it is never armed by default.
  • Mutations that re-dirty an already-visible object must not re-hide it. setModelMatrix and visibility churn schedule rebuilds constantly; the gate is first-publish-only per key, not "hidden whenever a build is in flight" — otherwise every rotation blinks the note. Once ready, always ready (until the key is removed). This includes same-key geometry replacement: updateGeometry (cwRenderTexturedItems.cpp:214) and the line plot re-call addObject with the same key on every geometry update — re-arming there would blink the line plot on every survey edit. Old geometry stays rendered and pickable until the new build installs (the eventual-consistency model), so replacement needs no hiding. Arm only when the key is not already published.
  • Removal + re-add of the same key (LAZ reload, GLB path change) re-arms the gate — it's a fresh sub-BVH.
  • Prerequisite — fix readiness starvation first (Phase 4a). One restarter serves all build work, every mutation cancels the in-flight build, and a cancelled build discards everything: the worker returns at cwGeometryItersecter.cpp:2566 before banking its built sub-BVHs (:2570-2574), and cache promotion lives in the install callback, which early-returns on cancel. A multi-second LAZ sub-BVH build repeatedly cancelled by unrelated churn (a rotation drag restarts per mouse-move) restarts from scratch each time. Today that means "unpickable during interaction" — mild. Under the gate it means invisible during interaction — severe. Fix: bank completed sub-BVHs across cancels so each restart makes incremental progress (builtSlot already flows worker→install; deposit before the cancel check and promote it even when the full BvhData was abandoned — mind the worker-still-running race on the cancel path). With banking, readiness converges in a bounded number of restarts regardless of churn.

Gate: test-pinned invariant — after addObject accepts, the effective visibility published to the store stays false until bvhReady() installs the key, then flips true without further owner calls; a rotation of a ready object never un-publishes visibility; a layout-rejected item renders under userVisible alone. Existing [cwGeometryItersecter] and visibility-store suites stay green.

Performance

  • real Readiness starvation under churn — see the Phase 4a prerequisite above. The only build-path issue with teeth, because the gate escalates it from "unpickable while interacting" to "invisible while interacting".
  • trade Delayed first pixels for huge clouds — time-to-visible becomes max(GPU upload, sub-BVH build); for millions of points the build is the long pole (seconds). Deliberate (nothing visible-but-dead), but worth a busy indicator on the LAZ layer, and Phase 4a is what makes the wait bounded rather than open-ended.
  • fine Per-mutation CoW refresh (landed with #505, kept by Phases 1–2) — O(objects) copy + top-level rebuild per setModelMatrix; object count is hundreds at most → microseconds per mouse-move. Phase 2 consolidates paths without adding cost.
  • fine bvhReady() fan-out — owners connect only while they have pending keys and disconnect when drained; the re-check is a hash lookup per pending key per install.
  • fine Gate flips — ordinary visibility-store writes, identical in cost to a user hide/show toggle; the RHI backend stays a snapshot reader. Steady-state rebuild churn already reuses cached sub-BVHs (cwGeometryItersecter.cpp:2519) — starvation is exclusively a first-build-of-a-new-heavy-object problem.

Blast radius

  • Phase 1: cwGeometryItersecter.cpp/.h only — one new static/pure helper, two call sites. Contained.
  • Phase 2: cwGeometryItersecter mutators + the private helpers they call; no header API change visible to cwRenderObject subclasses. The worker install callback (shouldPublish) is read but not restructured.
  • Phase 3: Option A — docs + a comment on addObject; no code change beyond what Phase 4 delivers.
  • Phase 4: cwGeometryItersecter.h/.cpp (one const query + doc on bvhReady()); cwRenderObject (the shared gate machinery); a couple of lines each at the four addObject/removal sites in cwRenderPointCloud, cwRenderTexturedItems, cwRenderLinePlot, plus their publishVisibility() overrides; tests for the gate lifecycle. The visibility store, RHI backend, and QML surface are untouched — the gate rides the existing owner→store publish path.

Acceptance criteria

  • Zero behavior change through Phases 1–2: the entire [cwGeometryItersecter] suite, the #505 regression test, the removal-race test, and the stale-install test all stay green.
  • Exactly one function computes a published top-level from slots + matrices (R1 done).
  • Exactly one function mutates m_bvh synchronously (A1 done); invalidatePublishedSlot / refreshPublishedModelMatrix are gone or become thin wrappers.
  • The addObject pre-build pick behavior is documented and test-pinned (R3 done): Option A, the not-ready contract.
  • Rendered ⇒ pickable (Phase 4 done): an intersecter-registered heavy object is never visible before its sub-BVH is published; first-publish-only gating (no re-hide on rebuilds); rejected geometry falls back to plain userVisible.

Not in this plan

  • R2 — cwRenderGLTF bindable-property → plain-member migration (separate idiom cleanup).
  • The two #505 cwRenderGLTF::setGLTFFilePath follow-ups (empty-path deselect dispatches a benign spurious load; no same-path retry after a failed load) — those live with R2 in the GLTF-object cleanup, not here.
  • Any change to the async worker's split heuristics, BVH quality, or the visibility-snapshot pairing — orthogonal.
