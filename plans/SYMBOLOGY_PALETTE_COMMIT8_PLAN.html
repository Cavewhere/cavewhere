Commit 8.0 — Writable palette  +  Commit 8 — Glyph editor

Untracked planning doc, fall2026 branch. Drafted 2026-06-17. Brush editor (commit 9) deliberately out of scope here — planned separately once 8.0/8 land. This supersedes the stale commit-8 row in SYMBOLOGY_PALETTE_PLAN.html (see "What changed" below).

0. Why two commits

The glyph editor is not a hard prerequisite of the brush editor — the data dependency runs one way (cwDecorationLayer.glyphName → glyph), and the seed already ships three glyphs a brush editor could reference. But both editors share one prerequisite that does not exist anywhere in the code yet: a writable palette. The seed is hard read-only (:/palettes/cavewhere-default, setWritable(false)), cwSymbologyPalette mutates only wholesale via setData(), and user palettes are discovered by scanning AppDataLocation/palettes but there is no API to create or fork one. So we split:

  • 8.0 Writable-palette foundation — pure C++/model. Fork the seed (or any read-only palette) into a writable user palette, plus glyph write-through. Fully unit-testable, no UI.
  • 8 Glyph editor — cwSketchGlyphCanvas + GlyphEditorPage.qml on top of 8.0.

2. The crux: palette injection (canvas decoupling)

cwSketchCanvas feeds its path source from m_sketch->paletteSnapshot(), and cwSketch::resolvedPalette() walks parentTrip → region → settings → seed (cwSketch.h:95,143-157). The glyph editor has no trip and no region; it must draw using exactly the palette being edited — and that palette may be a fresh writable fork that isn't any region's default. So the editor needs to inject a palette snapshot directly, bypassing the resolver chain.

This is not a revival of the dropped per-sketch activePaletteId (which was about the persisted active-palette choice for normal sketches). This is a non-persisted authoring context override that lives on the new cwSketchGlyphCanvas, not on cwSketch. The production cwSketch API stays untouched.

3. Commit 8.0 — Writable palette foundation C++ / model only

3.1 New manager API (cwSymbologyPaletteManager)

All write to paletteDirectory() (default QStandardPaths::AppDataLocation/palettes/<slug>), then reload() so disk stays the source of truth (directory-as-truth).

// Fork an existing palette (seed or installed) into a new writable user palette.
// Deep-copies brushes + glyphs, assigns a fresh QUuid, writes a new subdir,
// reloads, returns the new (writable) palette. nullptr on failure (error pushed
// to errorModel()).
Q_INVOKABLE cwSymbologyPalette *duplicatePalette(cwSymbologyPalette *source,
                                                 const QString &newName);

// Write a single glyph into a writable palette's glyphs/ dir and reload it.
// No-op + error if the palette is read-only. Used by the glyph editor's Save.
Q_INVOKABLE bool saveGlyph(cwSymbologyPalette *palette,
                           const cwSymbologyGlyph &glyph);

// Remove a glyph file (rename in the editor = saveGlyph(new) + removeGlyph(old)).
Q_INVOKABLE bool removeGlyph(cwSymbologyPalette *palette, const QString &glyphName);
  • Slug: kebab-case of newName, de-duplicated against existing subdirs (name, name-2, …). The on-disk directory name is cosmetic; identity is the new QUuid inside palette.json (master plan line 1825).
  • Writability already derives from file version (cwRegionIOTask::isVersionSupported); a freshly-written fork is current-version, so it loads back writable = true for free. No new writability machinery.
  • No granular palette mutators. saveGlyph goes straight through cwSymbologyPaletteIO::saveGlyph (one .cwglyph file) and reloads — honoring the "membership = files present, mutate wholesale via setData" design. We do not add addGlyph/setGlyph to cwSymbologyPalette.

3.2 Tests own file

testcases/test_cwSymbologyPaletteWritable.cpp (tag [cwSymbologyPaletteWritable]), manager pointed at a QTemporaryDir (concurrency-safe):

  • duplicatePalette(seed, "My Palette") → a new subdir exists with a different UUID, same brushes/glyphs by content, and the returned palette is writable.
  • Fork name collision → second fork lands in my-palette-2, both discoverable after reload.
  • saveGlyph into the fork → a glyphs/<name>.cwglyph file appears; reload surfaces the glyph in glyphs(); round-trip equality (cwSymbologyGlyph::operator==).
  • saveGlyph into the read-only seed → returns false, no file written, error pushed to errorModel() (assert via the model, not a counter).
  • removeGlyph → file gone, absent after reload.

3.5 Commit 8.1 — Reconciling reload prerequisite for the editor

The review surfaced the keystone issue (§8.1): reload() calls clearInstalled(), which deletes every cwSymbologyPalette* and rebuilds the list, so runtime object identity is not stable across a reload even though a palette's identity is its UUID. refreshPalette (in-place glyph save) and duplicatePalette's pointer-invalidation hazard are both workarounds for this. The editor will hold a palette pointer across user edits, so this must be fixed first — otherwise every write→reload either dangles the editor's pointer or needs yet another bespoke in-place path.

Plan

  • Replace clearInstalled()-then-rebuild with a reconciling reload: scan the directory into a QHash<QUuid, scanned-data>, then diff against the live m_palettes keyed by id() —
    • survivor (id in both): keep the existing cwSymbologyPalette*, setData() it in place (only emits if content changed), refresh its directory()/writable flags.
    • added (id only in scan): new cwSymbologyPalette, append.
    • removed (id only in live set): delete that one object.
  • The default (seed) palette is always a survivor (stable id), so it never churns.
  • Keep the existing duplicate-id first-scanned-wins handling and the forward-version read-only gate — they move into the scan/diff, not away.
  • duplicatePalette then just calls the reconciling reload and returns paletteById(newId) — no more "source pointer dangles" hazard; drop that header caveat. refreshPalette collapses into the reconciling reload (a single-palette case of it) or stays as a thin shortcut; decide during implementation.
  • Change signal: keep palettesChanged() for list-shape changes (add/remove); per-palette content changes already emit cwSymbologyPalette::dataChanged(). Only add a finer-grained signal if a consumer actually needs add-vs-remove granularity.

Tests

  • After a reload that adds one palette, pointers to the pre-existing palettes (incl. the default) are unchanged (identity preserved) — the test that currently documents "reload invalidates pointers" inverts.
  • Editing a file on disk and reloading updates the surviving palette's data in place (same pointer, new content).
  • Removing a palette directory and reloading deletes only that object.
  • duplicatePalette no longer invalidates source (assert the passed-in pointer is still valid and unchanged afterward).

This is small and self-contained C++/model work — do it as its own green, reviewed commit before starting the editor (§4).

4.1 cwSketchGlyphCanvas (C++)

A thin subclass of cwSketchCanvas (reuses pen capture, RHI render, path source, undo) adding the editor seam and chrome data. It owns its own cwSketch internally (no trip/region). 

class cwSketchGlyphCanvas : public cwSketchCanvas {
    // Authoring palette, injected directly (bypasses the region resolver).
    // Setting it feeds the snapshot to the inherited path source.
    Q_PROPERTY(cwSymbologyPalette *palette READ palette WRITE setPalette NOTIFY paletteChanged)

    // Fixed paper-mm authoring extent (default 10 x 10 mm); drives the overlay.
    Q_PROPERTY(QSizeF paperExtentMm READ paperExtentMm WRITE setPaperExtentMm NOTIFY paperExtentMmChanged)

    // Load an existing glyph for editing (paper-mm -> canvas world units), or
    // start blank. Save serializes the current strokes back to a cwSymbologyGlyph.
    Q_INVOKABLE void loadGlyph(const cwSymbologyGlyph &glyph);
    Q_INVOKABLE cwSymbologyGlyph toGlyph(const QString &name, const QString &displayName) const;
    Q_INVOKABLE void clear();
};

Coordinate model

A glyph stores cwPenStroke points in glyph-local paper-mm; the canvas stores points in world-m. The editor canvas uses the app-default mapScale = 1:250 (same as a normal sketch, so zoom/feel match). toGlyph/loadGlyph convert through the existing cwGlyphTessellationCache::paperMmToWorldM and its inverse (world_m = paper_mm / (1000 · scaleRatio); at 1:250 that's a linear ×4 / ÷4 factor — lossless, no special-casing). Origin is canvas (0,0); +X is the tangent (master plan §9). View zoom/pan handle on-screen sizing independently.

4.2 GlyphEditorPage.qml

A StandardPage registered in MainContent.qml's onCompleted and reached via RootData.pageSelectionModel (the established page pattern). Layout:

  • Canvas: SketchGlyphCanvas filling the page, palette bound to the writable palette being edited.
  • Overlays (QML, drawn over the canvas): paper-mm extent rectangle, origin crosshair at world (0,0), tangent arrow along +X. Positioned via the canvas's world→item transform; QC.Label captions, Theme tokens — no hardcoded colors/sizes.
  • Brush picker: reuse BrushPicker, fed palette directly (not sketch.resolvedPalette).
  • Chrome: name / display-name fields, Save, Cancel, plus Undo/Redo/Clear (the canvas already has an undo stack).
  • Save → paletteManager.saveGlyph(palette, canvas.toGlyph(name, displayName)) → cache.invalidateGlyph(name) happens via reload; Cancel pops the page.

8. Review findings (qt-cpp-review + simplify, 2026-06-18)

Eight read-only review agents ran over the 8.0 diff. Consensus below.

Fixed in 8.0

  • Path-safety / layering (5-agent consensus). removeGlyph originally hand-built glyphs/<name>.cwglyph in the manager, bypassing the kebab-case cwSymbology::isValidName guard every other glyph file op enforces — a ../ name could escape the glyphs/ dir. Fixed: added cwSymbologyPaletteIO::removeGlyph(dir, name) (validates the name, reuses the IO path builder); the manager now delegates to it. New test "Manager rejects a path-unsafe glyph name on remove".
  • Warning parity. refreshPalette now forwards load() warnings to errorModel(), matching reload() (previously a glyph that introduced a rule-stack warning stayed silent until the next full reload).

Findings & dispositions (decided 2026-06-18)

  1. Keystone — reload() destroyed every palette object (no stable identity across reload). → DONE in Commit 8.1 (§3.5). A palette's identity is its UUID; the runtime cwSymbologyPalette* is now keyed on UUID across a reload (reconciling reload — survivors updated in place, only deltas created/deleted). clearInstalled() and refreshPalette() were removed; saveGlyph/removeGlyph/duplicatePalette all just call reload() and keep their pointers valid.
  2. m_directory assumes one palette ↔ one filesystem directory. → OK for prototyping; idea to work on. Fine as-is for now. The idea to develop: a project (the .cwproj directory) gets its own palettes/ subdirectory holding that project's custom palettes, so they travel with the project and Git sync carries them (ties into the Decision-4 "bundle project-used palettes for sync" revision in the main plan). This needs design work — how a project-local palettes/ dir is discovered, reconciled with the user's AppData palettes, and addressed by directory() (which will likely become a location handle rather than a bare path). Tracked as an idea to work on, not yet scheduled.
  3. Non-transactional write. → ADDRESS LATER. If a saved glyph makes the palette fail validation (e.g. a glyph→brush→glyph cycle), the file is written but the subsequent reload() drops the now-invalid palette, leaving a bad file on disk. The editor can pre-validate (it holds the glyph), and/or add manager-level rollback (delete the just-written file on post-write load failure). Explicitly deferred — a known gap we need to come back to, not fixed in 8.0/8.1. (8.1 made reload() itself non-transactional too — see the 8.1 follow-ups below.)
  4. Commit-9 / autosave extraction. → ADDED to the plan. The gate→IO-write→refreshPalette shape in saveGlyph/ removeGlyph will repeat for saveBrush/removeBrush (brush editor); extract a private mutatePalette(palette, op) when the third copy lands. And these synchronous main-thread writes should route through cwConcurrent::run + AsyncFuture::observe(...).context(...) once palette elements autosave (per the Decision-9 autosave revision in the main plan).

Deliberately not changed (judged correct as-is): the slugify loop, the m_directory member choice (vs. a manager-side id→dir map), and duplicatePalette's full reload (a cold, user-initiated path; cost is negligible at realistic palette counts). Minor optional cleanups left for later: relocating slugify next to cwSymbology::isValidName for testability, and hoisting the test makeGlyph helper into testlib/cwSketchTestHelpers.

Commit 8.1 review — follow-ups (revisit after the brush editor)

The 8.1 reconciling-reload diff got its own read-only review pass (ownership / lifecycle + correctness, test-quality, architectural; 2026-06-18). No in-scope defects — the reconcile is sound, tests are correct, and the one simplify fix (drop the redundant QSet + second pass in favor of QHash::take()) was applied and is green. The pass surfaced four out-of-scope items. These are intentionally deferred: revisit as a batch once the brush editor (commit 9) is landed, because the editor is the client that makes the right design obvious, and several of these are about a signal/refresh contract best shaped against a real editing client rather than speculatively.

  1. Targeted single-palette refresh for glyph writes. → REVISIT AFTER BRUSH EDITOR. saveGlyph/removeGlyph now route a one-glyph write through full reload() (re-reads the qrc default plus every on-disk subdir). The deleted refreshPalette() was exactly the one-directory refresh this wants; the building block (load(directory)) still exists. Each save currently fans out to a full cwSymbologyPaletteListModel reset and cwSketch::resolveSnapshot() for every open sketch — an editor-responsiveness cost that scales with palette count and open sketches. Out of scope now: 8.1's point was making the pointer survive reload so a targeted refresh is safe to design later; there is no QML caller of saveGlyph yet to tune against, and restoring it now re-introduces the dual-path (reload vs refresh) parity burden 8.1 just removed.
  2. Add/remove-palette signal granularity. → REVISIT AFTER BRUSH EDITOR. palettesChanged() is the manager's sole mutation signal and reload() emits it unconditionally (even on a zero-delta reload). The reconcile already computes the exact add/remove/survive sets and then discards them. Finer signals (or at least a "no deltas → don't emit" guard plus add/remove signals) would let the list model do beginInsertRows/beginRemoveRows instead of a full reset, and let consumers like cwSketch::resolveSnapshot skip work when their resolved id didn't change. Out of scope now: it's a manager API-surface change rippling into the list model and cwSketch, best designed once the glyph/brush editor (the client that benefits) exists.
  3. Architectural — reload() is non-transactional. → REVISIT AFTER BRUSH EDITOR. reload() clears the error model and mutates live objects in place (setData/setWritable/ setDirectory) and deletes dropped palettes before m_palettes = rebuilt. A mid-scan IO/parse failure leaves earlier survivors updated and later ones not, with no rollback — and the in-place mutation now reaches objects the editor holds. A "scan fully, then swap" guarantee (extend phase-1's already-buffered scan so no live mutation happens until the whole scan succeeds) would make a failed reload a no-op. Broader than the identity-preservation goal of 8.1; ties into the per-write non-transactional gap (item 3 above).
  4. Architectural — front/back split: list-model resets discard the identity the backend preserves. → REVISIT AFTER BRUSH EDITOR. The backend works to keep object identity stable, but cwSymbologyPaletteListModel::refresh() collapses any change into a full beginResetModel/endResetModel — dropping selection/scroll state the identity preservation was meant to protect. By contrast cwLineBrushListModel already binds to the per-palette cwSymbologyPalette::dataChanged (which 8.1's surviving pointer now keeps wired across reloads). Generalizable pattern: a model layer should react to the finest signal the backend offers, and the backend should offer one matching its mutation granularity. This is the architectural framing of item 2; blast radius is the two list models plus whatever the editor adds.
