Ceiling channel — lateral offset for stamp terminals

Design note / discussion aid. Provisional commit 4.6, sibling of 4.1 (bending stamp) and 4.5 (offset polyline rails). Nothing here is built yet — this document is to agree on the approach.

What we want

A ceiling channel is a recessed groove in the ceiling: two parallel "ceiling-step" lines that run alongside the passage centerline, each offset a fixed distance, with hatch ticks pointing inward toward the channel. It's two of the dashed-and-ticked ceiling-step symbols we just built, mirrored across the centerline.

Why the bending stamp is already 90% of the way there

Recall the warp at the heart of cwBendingStampRule (and cwJointedStampRule):

warp(x, y) = pointAtArcLength(S + scale·x)  +  scale·y · normal(S + scale·x)
                      └── advance along the curve ──┘     └── push sideways along the normal ──┘

The glyph's x walks along the centerline's arclength; the glyph's y is already a push along the local normal. So a point authored at y = +d lands d off the centerline and rides the curve. Geometrically, offset-following is the same operation the bending warp already performs — a parallel offset curve is just "every sample at the same y."

The catch: bending throws away the anchor's sideways position

So why can't we just seed the stamps off to the side? Because the bending terminal recovers its base arclength by projecting the anchor back onto the centerline:

const double S = strokePath.arcLengthNearPoint(position.anchorWorld); // collapses the anchor onto the centerline

Everything after that is rebuilt from S along the centerline. The anchor's lateral (sideways) component is discarded. A MutatePerLayer rule that nudges anchorWorld out along the normal would be silently ignored by the bending warp — it re-projects to the same S and offsets only by the glyph's own y.

The fix is small and lives in the shared bendWarp helper, so Bending and Jointed both gain it, and Rigid already honours a moved anchor:

// once per stampPath(), in the stamp terminal:
const double  S = strokePath.arcLengthNearPoint(anchorWorld);
const QPointF base = strokePath.pointAtArcLength(S);
const QPointF n    = strokePath.normalAtArcLength(S);
const double  anchorLateral = QPointF::dotProduct(anchorWorld - base, n);   // 0 when on the line

// bendWarp(x, y) = pointAtArcLength(S + scale·x) + (anchorLateral + scale·y)·normal(S + scale·x)
Superseded — see "Resolved direction" below. The per-stamp anchorLateral fix shown above is not the path we took. Offsetting the centerline the pipeline runs against (a stroke transform) subsumes it, needs no stamp-terminal change, and also covers traced lines and referenced sub-symbols. The figure stays because it explains why a naive anchor-nudge is ignored.

Two ways to supply the offset

A Bake it into the glyph's y

Author dash-+d and dash-−d glyph variants (the dash shifted up / down in glyph space). Two bending layers, zero engine code — it works against the build we have today.

  • + Instant; proves the channel visually right now.
  • − Glyph explosion — every symbol needs ±d twins.
  • − Channel width is frozen into glyph geometry, not a tunable parameter.
  • − Conflates "what the symbol looks like" with "where it sits."

B A parametric Offset along normal rule

A new MutatePerLayer rule pushes each anchor d·worldPerPaperMm along the local normal; the anchorLateral tweak above lets the bending warp honour it.

  • + One offset value, any glyph — no twins.
  • + Composes across Rigid / Jointed / Bending uniformly.
  • + Channel width becomes a real parameter (editor-tunable later).
  • − Needs a signed ±d per layer → the first feature that reads a rule param (we'd been deferring param interpretation to the editor, commit 8).

The two layers of a channel need opposite offsets (+d and −d). A single rule running from a hard-coded default can only produce one sign — so option B is genuinely the first place a rule parameter has to be read. That's the one piece of new plumbing it forces.

The stack, the way B composes it

Each of the channel's four layers is just a rule stack. The new box is highlighted:

Uniform spacing→ Offset along normal (+d)→ Bending stamp — dashed line, offset +d
Uniform spacing→ Offset along normal (+d)→ Align to tangent→ Rigid stamp — inward ticks on the +d line
Uniform spacing→ Offset along normal (−d)→ Bending stamp — dashed line, offset −d
Uniform spacing→ Offset along normal (−d)→ Align to tangent→ Rigid stamp — inward ticks on the −d line

Tick direction (inward on both sides) is the existing mirror concern — the cwStampPosition::side field / a future Mirror on side rule flips the tick glyph for the far side. Not new work for the offset itself.

One caveat either way: tight concave bends

This is the stamp twin of commit 4.5 (offset polyline rails). Same geometry, different terminal: 4.5 offsets a traced solid line, this offsets stamped glyphs. If a channel ever wants solid edges instead of dashes, that's 4.5; the dashed/ticked channel is this. Their offset semantics should stay aligned (sign = side, paper-mm, left-normal positive).

Resolved direction

Offset is a stroke transform, surfaced as its own rule; constant for now; and symbol reuse via a DAG is the destination it's built toward.

1 — Offset is a stroke transform, surfaced as a rule

The layout builds an offset cwStrokePath (the centerline pushed ±d·worldPerPaperMm along its left normal) and runs the layer's rule stack against that stroke. The offset rule is a new stage that runs before Generate; the layout pulls it out of the stack and applies it first, the same way it already pulls out the Terminal rule.

  • Bending / Jointed / Rigid stamps follow the offset for free — they sample the stroke. No bendWarp / stamp-terminal change.
  • A traced line would follow too — so this absorbs commit 4.5 (parallel rails); both share one offset-polyline helper (push along the left normal, miter join + bevel fallback).
  • It honours "offset is its own rule" (declarative, composable) while doing the right thing geometrically.

2 — Symbol reuse via a DAG (where this is heading)

A decoration layer gains a second shape: instead of glyph + rules, it can reference another brush plus a transform (offset, mirror). The layout renders the referenced brush's decorations against the transformed stroke — recursively. The graph must be acyclic (a DAG): a name/uuid reference, cycle detection, and a depth cap. This is symbol instancing — SVG <use>, CAD blocks.

ceiling-step is authored once; ceiling-channel = two references to it, offset +d and −d (the second mirrored). The stroke-transform offset built in step 1 is the exact primitive the reference reuses — so step 1 is the down payment on the DAG, not a detour.

3 — Constant for now

No param/editor plumbing yet. The offset's magnitude + sign comes from a temporary per-layer carrier (a channel needs +d on one side and −d on the other — a singleton rule with a hardcoded constant can't emit both). The carrier is set in code (seed/test), replaced later by rule params and ultimately by the DAG reference's transform.

Steps

  1. Now: offset-polyline helper + an Offset stroke rule + the layout's offset-stroke step; a ceiling-channel demo brush (constant ±d, ticks mirrored inward) + a PNG panel. (Supersedes the earlier anchorLateral step — not needed.)
  2. Later: parameterise — first via rule params, then fold the offset onto the DAG reference's transform and drop the temporary carrier.
  3. Later: the DAG itself — brush references + recursive, cycle-checked layout — turning ceiling-channel into two references of the one ceiling-step symbol.

Open sub-decision for step 1: how the constant ±d is carried — a signed lateralOffsetMm layer field (matches the existing offsetCurveOffsetMm precedent, trivial to delete later) vs. two constant rule classes vs. the glyph-y demo. Recommend the layer field.
