1. The vision — a recursive symbol system

A brush is a node in a DAG: brush → ordered decoration layers; each layer has a rule stack (pipeline) and references a glyph; glyphs reference sub-glyphs; brushes reference brushes (instancing). Editing a node has blast radius — a shared glyph edit changes every dependent. The editor is the safety rail and the power tool for that: live feedback on your own sketch, visible dependents, and a navigable map of the whole palette.

3. Architecture — the feedback loop

3.1 cwBrushEditor — working-copy controller (QObject, QML element)

Holds one editable cwLineBrush working copy + (when scoped) a glyph working copy. Plain members + explicit setters (no QBindable). Ops: loadBrushNamed, setRuleEnabled, layer/rule mutators, scalar setters, apply() (= palette->setBrush(working) [+ setGlyph]), discard(), dirty; emits brushChanged → preview refresh.

3.2 Preview-snapshot override — live feedback, no mutation

Non-persisted authoring override on cwSketch/canvas (setPreviewSnapshot/clearPreviewSnapshot) — the same trick commit 8 used to inject a palette into the glyph canvas. On each edit, push base snapshot with the edited brush/glyph swapped. cwPaletteSnapshot is cheap implicitly-shared, so it's a copy-with-substitution. Discard → clear; Apply → real setBrush (real re-skin + async persist), then clear.

3.3 Discard / Apply / Duplicate & Save (fork-on-edit)

Closing dirty prompts (Apply / Discard / Keep). Disk is never written until commit. The glyph editor converges on this working-copy model in Phase 4 (gives up autosave).

Fork on save. A fresh region resolves to the read-only shipped default (cwSketch::resolveSnapshot, region→settings→shipped), and palette mutation is writable-guarded. Rather than block entry, the editor opens and is fully editable on its working copy regardless — only the commit button adapts:

  • Writable palette: footer [Discard] [Apply] → palette.setBrush(working).
  • Read-only palette: a banner ("Editing a copy of 'X' (read-only)") + footer [Discard] [Duplicate & Save…], which paletteManager.duplicatePalette(resolved) (writable fork) → region.setDefaultPaletteId(fork.id) → fork.setBrush(working). The working copy the user was experimenting with carries straight into the fork.

This is feasible because the live preview already rides a non-resolved snapshot (§3.2), so the read-only palette is never mutated — the experiment lives entirely in the working copy until the fork materializes it.

3.4 Switching palettes

Two surfaces over the existing PaletteListModel, both setting region.defaultPaletteId() (switching while dirty fires the Apply/Discard/Keep prompt first): a Palette: [name ▾] combo in the editor's library rail (switch + Duplicate / New), and a fuller palette-management page (rename / delete / import / export). See §8.1.

5. Data — enabled flag (and a note on brush refs)

struct cwPlacementRuleData { QString name; QVariant parameters; bool enabled = true; ... };

Layout filters to the effective (enabled) stack before resolving terminal/transform; the validator validates the effective stack (a disabled terminal → the existing "no terminal" warning — exactly the feedback we want). Additive proto field, no version bump. Brush→brush references (Phase 6) add a reference construct to the data model (a layer/brush that instances another brush) — schema + engine compose; specced in its phase, not here.

6. Dependency system Phase 4

One shared index feeds the inline Used by affordance, the dependents gallery, and the Palette Graph.

  • cwSymbologyDependencyGraph (new): forward uses + reverse dependents, spanning brush→glyph, glyph→glyph, and brush→brush uniformly. Built from a palette snapshot.
  • Transitive dependents: editing floor-tick must reach floor-step and passage (which references it) — the blast-radius query is transitive, not direct-only.
  • Live cycle check spanning brushes + glyphs: setting a glyph or a brush reference that would cycle is rejected on edit (generalizes findGlyphDependencyCycle from load-time/glyph-only).
  • Orphan detection: a glyph with zero dependents (deletion candidate; the graph flags it).
  • Reused brushes / glyphs are not freely deletable: delete/rename must respect references (brush + glyph alike).

7. Palette Graph view Phase 5

An independent navigable DAG built on QuickQanava (Qan.GraphView): qan::Node per brush/glyph, qan::Edge per dependency (distinct styling for uses-glyph vs references-brush). Populated from cwSymbologyDependencyGraph. Pan/zoom/drag, select → detail (used-by / uses), Open in editor (routes to brush or glyph editor), Delete (orphans only), shared/orphan highlighting. Cross-editor navigation is shared with the inline Used by jump-list.

Phase-1 structure-tree STOPGAP (shipped — replace in phase 2)LowWhat shipped: BrushEditorPanel.qml renders the layer/rule tree with nested Repeaters bound to cwBrushEditor's invokable accessors (layerCount()/layerLabel()/ruleCount()/ruleName()/ruleEnabled()). Because those are function calls (not bindable properties), QML can't track them as dependencies, so the panel bumps a _structureRevision counter on structureChanged and smuggles it into each binding via the comma operator ((_structureRevision, editor.ruleEnabled(l, r))) to force re-evaluation. It works for the read-only-plus-toggle thin loop (no row insert/remove/reorder), but the comma-operator trick is obscure and a flat-accessor view doesn't scale to drag-reorder/add/remove. Phase 2 replacement: add a real QAbstractItemModel on cwBrushEditor (a 2-level layers→rules tree) and drive the panel with a TreeView + role-based delegates. That deletes _structureRevision and the comma-operator bindings, and gives incremental rowsInserted/rowsRemoved/moveRows (with animations) that stack editing needs anyway. The invokable accessors can then be retired (or kept only for point queries/tests). (A simpler interim — a notifying QVariantList layers snapshot property — was considered and rejected in favor of going straight to the real tree model in phase 2.)

16. Out of scope / deferred

  • Each phase is its own commit, paused for review (feedback_pause_between_phases).
  • #37 targeted single-palette refresh — working-copy/preview-override keeps editing off the full-reskin path; the cost is only at Apply. The editor is the client that motivates the targeted refresh.
  • Signal granularity / error-model aggregation (4.4 / commit-8 deferrals) — revisit with the editing client.
  • Category grouping in the graph (QuickQanava groups), node-layout persistence, cycle-prevention demo polish — graph follow-ups.
  • 2D spatial index for stroke hit-testing — cwSketch::brushNameNear() (the phase-1 right-click entry, B1) does an O(n) linear scan over strokes with a bbox prefilter, as do findContinuationTarget() and the eraser. Fine for a one-shot right-click on today's sketches, but a large sketch will eventually want a quadtree / 2D-BVH over stroke bounding boxes shared by all three call sites. Not scoped to commit 9; flagged in a code comment on brushNameNear so a future profiler finds it.
  • Snapshot-source seam — keep cwSketchCanvas pure DONE — the phase-1 preview (§3.2) bolted mutable authoring state (m_previewSnapshot / m_hasPreviewSnapshot + a branch in snapshotForPathModel()) onto cwSketchCanvas. There were three snapshot sources reached two ways: sketch-resolved (base), a direct palette (cwSketchGlyphCanvas override), and the injected preview (base-class state). The preview had to live on the base because it re-skins the live canvas — subclassing can't reach it. Fix shipped: a small observed provider cwPaletteSnapshotSource (abstract: snapshot() + snapshotChanged()). The canvas holds one optional source (setSnapshotSource / snapshotSource): currentSnapshot() returns the source's snapshot if set, else the sketch's resolved palette, and the canvas re-pulls on snapshotChanged() (and on the source's destroyed, falling back to the sketch). No virtual, no preview members. As built, the design refined two of the bullet's guesses: (1) the sketch-resolved case stays inline in the canvas as the null-source fallback — no separate default-source class, since sketch resolution is the canvas's own job, not authoring; (2) the brush editor is the preview source (cwBrushEditor : cwPaletteSnapshotSource, snapshot() = the copy-with-substitution), so no separate preview-source class — it installs itself via setSnapshotSource(this) and clears on discard/apply/dispose. The glyph editor gets a tiny cwPaletteBackedSnapshotSource (deletes the cwSketchGlyphCanvas::snapshotForPathModel override). The old preview equality short-circuit moved into cwDecoratedStrokePathSource::setSnapshot — the right altitude (every caller skips the redundant glyph-cache wipe + re-tessellation). New files: cwPaletteSnapshotSource.{h,cpp}, cwPaletteBackedSnapshotSource.{h,cpp}. Verified: C++ 184 assertions; BrushEditor QML 4/4; GlyphEditor QML 7/7.
  • Fork-on-save policy out of the controller — cwBrushEditor::apply() orchestrates project policy (cwSymbologyPaletteManager::instance() → duplicatePalette, then region->setDefaultPaletteId, then re-target). Move it to the consumer-reaches-provider direction — e.g. cwCavingRegion::adoptWritableCopyOf(palette) (region already depends on the manager; it owns its own default) — not a manager.forkForEdit(palette, region), which would invert the dependency by making the palette subsystem know about regions. Hold until a second editor (glyph/color) confirms the policy generalizes; today it's a single call site and bakes in "fork ⇒ becomes the region default," which is only true because the editor edits sketch.resolvedPalette.

17. Rejected alternatives

AlternativeWhy rejected
Gadget-ify the value structs — add Q_GADGET + Q_PROPERTY + QML_FOREIGN to cwLineBrush / cwSymbologyGlyph (/ cwDecorationLayer / cwPlacementRuleData) so QML reads/edits their fields directly, instead of the cwBrushEditor controller + list-model approach (Decision A). Wrong tool for an editor. (1) Gadget properties are value-copy in QML: brush.decorations[0].glyphName = "x" mutates a throwaway temporary, so the write never reaches the real brush — live-mutation, the whole point of commit 9, silently breaks. (2) Live preview needs change signals (brushChanged → rebuild preview snapshot); gadgets aren't QObjects and emit nothing, so the editable surface still has to be cwBrushEditor regardless. (3) Nested QVectors (brush→layers→rules) are read-only-awkward and uneditable as gadget collections; a model per level is idiomatic and is what the B-series already specs. (4) It couples a serialization/IO value type to the QML type system and invites the value-copy footgun. The only real win — reading a scalar like displayName — is already served by a getter (brushDisplayName/glyphDisplayName) or a model role. Raised by the Phase-0 architectural review; recorded here so it isn't re-litigated.
