Prep commit — default palette as a qrc resource

A small, self-contained refactor that lands before commit 6. Today the shipped default palette is built in C++ by cwSymbologyPaletteSeed::create() and wrapped, read-only, straight into the manager — a parallel construction path that never touches the file format. This commit makes the default palette a real data palette compiled into the binary as a Qt resource, loaded through the normal cwSymbologyPaletteIO path like any disk palette. Commit 6's resolver chain then bottoms out on a genuine palette id instead of a special-cased seedPalette().

Bottom line. Generate the default palette files once from the existing create() generator, commit the directory, embed it via qt_add_resources at :/palettes/cavewhere-default/. The manager loads that through cwSymbologyPaletteIO::load() instead of wrapping create(); seedPalette() becomes defaultPalette(). create() is not deleted — it's demoted from "the runtime seed" to "the generator + test fixture," and a drift-guard test asserts the committed qrc files still equal what create() produces. The qrc palette is read-only; cloning it to a writable disk palette is a UI action that rides with commit 6/8, but the architecture supports it now (stable id, enumerates alongside disk palettes). No proto/format-version change.

2. The qrc layout & how the files get generated

The default palette is the directory format from SYMBOLOGY_PALETTE_DIRECTORY_TRUTH_PLAN.md (palette.json + brushes/*.cwbrush + glyphs/*.cwglyph), committed under the source tree and embedded:

cavewherelib/palettes/cavewhere-default/
    palette.json
    brushes/{wall,feature,scrap-outline,floor-step}.cwbrush
    glyphs/floor-step-tick.cwglyph
cavewherelib/cwPalettes.qrc        # <qresource prefix="/palettes"> ... maps to :/palettes/cavewhere-default/

Wired through the existing block at cavewherelib/CMakeLists.txt:217 (qt_add_resources(QT_RESOURCES …)) — add cwPalettes.qrc next to cwResources.qrc.

Generation — programmatic, generate-then-commit

The files are produced from create() so nobody hand-edits serialized JSON, then committed and frozen (build-time generation is rejected — it needs a host tool that runs during the build, which fights cross-compilation and CI). The regeneration step is:

cwSymbologyPaletteIO::save(cwSymbologyPaletteSeed::create(), "<repo>/cavewherelib/palettes/cavewhere-default");

Exposed as a hidden, manually-run regenerator so it never writes the source tree during a normal test run — mirroring the repo's existing env-var-path convention (the decoration render PNG). A Catch2 case tagged [.][regenerateDefaultPalette] (the leading . hides it from default runs) writes to the path in a CW_DEFAULT_PALETTE_DIR env var; run it by hand after changing the generator, then git add the regenerated files. Alternative considered: a dedicated tool target. Heavier; the hidden-test path needs no new build target and reuses the test harness. Open question §7.A.

Drift guard — the committed files can't silently go stale

One new assertion keeps the frozen artifact honest against its generator: load the qrc palette and compare it to create().

auto loaded = cwSymbologyPaletteIO::load(QStringLiteral(":/palettes/cavewhere-default"));
REQUIRE_FALSE(loaded.hasError());
CHECK(loaded.value().palette == cwSymbologyPaletteSeed::create());   // value-equality on cwSymbologyPaletteData

If someone edits create() and forgets to regenerate, this fails loudly with a clear message. (It also exercises the qrc read path through real IO.) Needs operator== on cwSymbologyPaletteData — verify it exists / is deep; add if missing.

  • Rename m_seed → m_default, seedPalette() → defaultPalette(). Update the one comment ("The seed is always present and always wins its id." → default).
  • Keep "default is listed first, wins its id on duplicates" — unchanged semantics, new source.

4. Cloneability (architecture now, button later)

qrc resources are read-only at runtime, so "the user can modify it" means clone the qrc palette into the writable palette directory (the disk dir the manager already scans), with a fresh QUuid and writable = true. After cloning it's an ordinary disk palette the editor can change.

This commit does not add the clone button — that surfaces with the palette picker (commit 6) or the editor (commit 8). It only guarantees the preconditions: the default enumerates in palettes() alongside disk palettes, carries a stable id, and round-trips through the same IO::save() a clone would call. A natural home later is cwSymbologyPaletteManager::clonePalette(QUuid source) → QUuid; noted, not built here.

5. Naming — "seed" → "default"

The shipped palette is no longer a "seed" (a code generator's output); it's the default palette. Renames in scope:

  • cwSymbologyPaletteManager: seedPalette()→defaultPalette(), m_seed→m_default (+ test references in test_cwSymbologyPaletteManager.cpp).
  • The generator namespace cwSymbologyPaletteSeed and create() — keep the names. They now mean "the generator that seeds the default palette's data," which is accurate, and renaming the namespace touches ~15 files for little gain. Open question §7.B if you'd rather rename to cwDefaultPaletteSource.
  • The stable brush/glyph name + id accessors (wallBrushName() etc.) stay where they are — they are the cross-palette substitution contract, not seed-specific.

6. Why cwSketch stays on create() this commit

It would be tempting to also swap cwSketch's construction snapshot to manager.defaultPalette()->snapshot() here. But that introduces a cwSketch→manager singleton dependency that commit 6 introduces anyway (the resolver chain needs the manager for the region/settings/default lookups), and doing it now would couple standalone cwSketch unit tests to a configured manager singleton for no immediate benefit. So:

  • This commit: the shipped default palette (the manager's) is qrc data. The momentary second copy in cwSketch's ctor is identical to it (generated from the same create()) and the drift guard proves it.
  • Commit 6: cwSketch stops calling create() entirely and resolves through the chain → defaultPalette(). At that point create() is out of the runtime path completely (generator + tests only).

If you'd prefer to pull that swap forward into this commit (so create() is runtime-free one commit sooner), it's a one-line change plus a manager-init guard in the affected cwSketch tests — flagged as §7.C.
