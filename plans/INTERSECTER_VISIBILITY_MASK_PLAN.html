Intersecter visibility mask

Making cwGeometryItersecter visibility-aware at per-primitive granularity, so picking and bounding-box queries honor keyword-driven visibility from one source of truth.

Fixes #575 (hidden shots still pickable) and #549 (reset frames hidden geometry) through a single mechanism.

0 · Goal & the two bugs

Visibility in CaveWhere is keyed on keywords: the user filters by keyword (Cave, Trip, Year, Type…), a filter pipeline splits keyword items into an accepted (visible) and a rejected (hidden) set, and each item hides its geometry at whatever granularity it owns. That hiding is applied on the render side only. The intersecter — which answers picks and bounding-box queries — never learns about it, except at whole render-object granularity.

BugSymptomCause
#575 The Distance tool snaps to a hidden shot; turntable pivot can land on a hidden trip. Per-shot line-plot visibility never reaches the intersecter; a Kind::Lines pick hits hidden segments.
#575-class
(scrap flavor)		 Keyword-hidden scraps and glTF models are still pickable, and the rotation pivot (nearestGeometryPoint) anchors onto them. Manually confirmed. Per-id visibility (cwRenderTexturedItems::setVisible(id)) is render-side only; all scraps share one render object, so the whole-object gate never fires.
#549 Reset view frames hidden caves/trips/scraps/LiDAR — the model zooms out to include things you hid. boundingBox() unions every registered object with no visibility filter at all (its doc now says so explicitly).

Design goal. Make the intersecter return correct visible-only results, so consumers (reset-view, Distance tool, rotation pivot, lead occlusion) are written as if cwGeometryItersecter is simply correct — no visibility logic duplicated into each consumer. The failing tests that pinned the bugs (testcases/test_cwRenderLinePlotPickVisibility.cpp #575 per-shot; testcases/test_cwRenderTexturedItemsPickVisibility.cpp per-id pick + framing bounds) have been flipped to regression tests — all phases below are implemented (July 2026): Phase 1 + the framing-bounds seam, Phase 2a (per-Key flag: setVisible(Key, bool), live m_hiddenKeys that survives geometry replacement, per-slot slotVisible in the published snapshot), and Phase 2b (per-vertex mask: setRangeVisible(Key, start, count, bool), mask on the live Object, per-slot shared array in the snapshot; collapses to empty when fully re-shown). Both republish copy-on-write like invalidatePublishedSlot(), and the build-install callback re-stamps visibility from live state so a toggle can't race the worker.

Phase 0 · How it works today current

Three layers, wired end-to-end. Visibility flows left→right; the intersecter is the one box the flow skips.

Phase 0a · Visibility is keyed on keywords

  • KeywordTab.qml:17-20 — KeywordVisibility { visibleModel: pipeline.acceptedModel; hideModel: pipeline.rejectedModel }. The filter pipeline is the source of truth for what's shown.
  • cwKeywordVisibility.cpp:105 resolveVisibility() — hide wins: if an object is in the reject model, hide it; else show. Fires on rowsInserted/Removed/modelReset.
  • cwKeywordVisibility.cpp:17 invokeVisible() dispatches by type: QQuickItem, cwRenderObject, or a generic setVisible(bool) — the last is how the proxy objects below get called.
  • Each keyword item carries an ObjectRole pointing at the object that knows how to hide the corresponding geometry at the right granularity.

Key structural fact. Each textured item id is already its own intersecter Key — addItem registers Object({this, id}, …) (cwRenderTexturedItems.cpp:140). The intersecter already has per-scrap / per-glTF-part granularity structurally; only the visibility bit is missing. So textured items need a per-Key visible flag, not the per-primitive mask — the mask is needed only by the line plot, which is one Key with per-vertex visibility.

Phase 0b · How the renderer honors visibility

  • Whole object — cwRHIObject::m_isVisible, synced in cwRhiScene.cpp:71 and skipped at gather in cwRhiFrameRenderer.cpp:569 (if (!object->isVisible() …) continue;).
  • Per-vertex centerline — cwRenderLinePlot::m_visibility is a QVector<quint8> (one byte/vertex, kVisible=0xFF/kHidden=0x00). cwRHILinePlot.cpp:44 uploads a parallel m_visibilityBuffer (uint attribute, binding 1); cwRHILinePlot.cpp:102 updateVisibilityBuffer() re-uploads on change; LinePlot.vert/.frag discard hidden vertices. This is the only place per-shot visibility exists.
  • Per-scrap — cwRhiTexturedItems stores per-id visible and skips hidden items at draw.

Phase 0c · What the intersecter does — and doesn't

Refreshed July 2026 after the traversal rework (tube-fallback deletion, pivot anchoring, traverseBvh unification).

  • Geometry is registered once as the full vertex list in cwRenderLinePlot.cpp:46-59 (setGeometry → addObject). Visibility toggles (setRangeVisible) never call the intersecter. Likewise cwRenderTexturedItems::setVisible(id) (cwRenderTexturedItems.cpp:222) updates render state only.
  • Both ray entry points — intersectsDetailed (ExactPick) and the rotation-pivot anchor nearestGeometryPoint (AnchorPick) — now share one traversal, traverseBvh<Policy>. The only visibility gate is whole render-object: isPickable() = parent==nullptr || parent->isVisible(), hoisted once per sub-BVH descent inside traverseBvh. One hook point now covers every ray query.
  • boundingBox() unions all node boxes with no visibility check whatsoever — its doc now says "Counts hidden objects" explicitly.
  • isPickableEmpty() (added for the pivot fallback) already contains the Phase 1 loop: it unions boxes over isPickable-filtered live Nodes, then throws the box away and returns only its emptiness.
  • pointsInBox(QuerySnapshot, box, Key) is an off-thread, explicit-Key query (sink-scan broad phase). It is deliberately unfiltered: the caller already chose the object, so visibility policy belongs to the caller, not this query.
  • Key type is Key{cwRenderObject* parentObject, uint64_t id}; the published BVH keeps a keyToSlot reverse index, so per-Key state can be republished with a direct lookup.

Phase 1 · Whole-object visibleBoundingBox() interim implemented

Ship the reset-view API immediately, before the deeper work. The body already exists: isPickableEmpty() unions boxes over isPickable-filtered live Nodes and discards the box. Phase 1 is an extraction, not new logic.

  • Extract that loop into cwGeometryItersecter::visibleBoundingBox(); reimplement isPickableEmpty() as { const auto box = visibleBoundingBox(); return box.isNull() || !box.isFinite(); }. Main-thread only (computed against live Nodes), flat loop over cached per-Node boxes — O(#Keys), no BVH or vertex traversal.
  • Point cwBaseTurnTableInteraction::resetView() at visibleBoundingBox() (replacing boundingBox() at cwBaseTurnTableInteraction.cpp:424); keep the empty-box fallback.
Coverage is narrower than it looks. The only keyword items that carry a whole cwRenderObject are point clouds (cwLazLayersSceneNode.cpp:367). Scraps, glTF models, and centerline trips all hide through per-id / per-range proxies whose shared parent stays visible — so Phase 1 fixes framing for hidden clouds only. Phases 2a/2b upgrade correctness for the rest with no change to reset-view, which is written once against the final API.

Architectural note · put the seam in the right place

Today resetView() reads scene()->geometryItersecter()->boundingBox() directly (cwBaseTurnTableInteraction.cpp:331). "What bounds should the camera frame" is a scene / view-presentation concern, but cwGeometryItersecter::boundingBox() is an artifact of the pick-acceleration structure. Sourcing framing policy straight off the picking BVH leaks the picking subsystem's boundary into camera behavior.

Preferred seam: expose a scene-level framing-bounds accessor on cwScene / cwRegionSceneManager (e.g. framingBounds() / visibleFramingBounds()), and have resetView() (and any capture/export framing) call that. The visibleBoundingBox() work below then lives behind the scene seam rather than being one more method bolted onto the intersecter — the interaction depends on the scene for framing, not on the pick structure.

Blast radius: moderate — one new scene-level accessor, re-point resetView (and the Phase 1/2 work) at it; no data-model changes. Tracked as the architectural follow-up from the #549 review; can land before or alongside Phase 1.

Phase 2a · Per-Key visible flag scraps + glTF implemented

Textured items don't need the mask: each item id is already its own intersecter Key with its own Node and sub-BVH slot. They need one bit.

  • Add cwGeometryItersecter::setVisible(const Key&, bool visible) — a per-Key flag on the live Node, republished into the snapshot slot (the keyToSlot index makes this a direct lookup). No geometry change, no BVH rebuild.
  • traverseBvh consults it beside the existing isPickable() hoist — one boolean per descended sub-BVH, zero per-primitive cost. Because the traversal is shared, picking (ExactPick) and the rotation pivot (AnchorPick) are both fixed by the same check.
  • visibleBoundingBox() / isPickableEmpty() skip per-Key-hidden nodes — per-scrap and per-glTF framing become correct.
  • Forward from cwRenderTexturedItems::setVisible(id) (cwRenderTexturedItems.cpp:222): one line, intersecter->setVisible({this, id}, visible). Scraps (per-id proxies) and glTF groups (N per-id calls) are both covered — no manager changes.
  • Flip the two [!shouldfail] cases in test_cwRenderTexturedItemsPickVisibility.cpp; repoint its bounds test at visibleBoundingBox().
Snapshot subtlety. sub.object inside a published snapshot is a copy of the Object, so a plain member on the live Object won't be seen by held snapshots. The whole-object gate gets away with this because it dereferences the live parent->isVisible(); the per-Key flag instead follows the same discipline as the Phase 2b mask — per-slot shared state, republished on change. Ray queries run on the main thread, so a simpler live-read is also defensible; decide at implementation time, but keep it consistent with the mask.

Phase 2b · The per-primitive mask line plot implemented

Give the intersecter its own visibility channel, fed from the same setRangeVisible calls the renderer already makes. Both picking and bounding box consult it. One mechanism → both #575 and per-trip #549 fall out. Only the line plot needs this — it is one Key whose visibility varies per vertex.

2.1 · Mask on the Object

// cwGeometryItersecter.h — Object
QVector<quint8> m_visibility;   // empty = all visible (fast path)
const QVector<quint8>& visibility() const { return m_visibility; }
// helper: a Lines primitive at index i is visible iff both its
// endpoint vertices' bytes != kHidden (indices[i], indices[i+1]).

Empty mask ⇒ zero per-primitive cost for clouds, scraps, triangles, and any object that never partially hides.

2.2 · Mutation API (no BVH rebuild)

void cwGeometryItersecter::setRangeVisible(const Key& key, int start, int count, bool visible);
// Updates the mask on the live Node AND republishes the mask into the
// query snapshot (copy-on-write swap of a per-slot shared array).
// Geometry/BVH structure is unchanged, so this is O(range), not a rebuild.

2.3 · Render side pushes the mask

  • cwRenderLinePlot::setGeometry() — after addObject, seed an all-visible mask (or none).
  • cwRenderLinePlot::setRangeVisible() — additionally call geometryItersecter()->setRangeVisible(Key{this,0}, start, count, visible). The intersecter mask becomes the twin of the render m_visibility, fed by the same call.
  • Textured items do not push a mask — Phase 2a's per-Key flag covers them.

2.4 · Consumers honor the mask

  • Ray queries — in traverseBvh's leaf loop, check the mask just before policy.testPrimitive(ctx, prim). One check in the shared traversal covers both policies (ExactPick picking and AnchorPick pivot anchoring) with no per-policy duplication. Hoist the has-mask test to the sub-BVH descent (beside isPickable()) so unmasked objects — clouds, scraps, triangles — pay one pointer compare and nothing per primitive.
  • Bounding box — visibleBoundingBox() unions only visible primitives. Masked objects iterate their primitives (can't use the cached node box); unmasked objects keep the fast whole-box path. O(vertices), only on reset, only for the line plot.

2.5 · Thread-safety & the snapshot

The one real cost. Picks run off-thread against an immutable snapshot (SubBvhs). Bounding box runs on the main thread against live Nodes.
  • Bounding box: main-thread, reads the live mask directly — trivial.
  • Picking: the mask must live in the published snapshot so an in-flight pick sees a consistent view. Store the mask as a per-slot shared_ptr<const QVector<quint8>> alongside each SubBvh; setRangeVisible builds a new array and atomically swaps the pointer (keyToSlot finds the slot directly). BVH nodes are untouched, so toggling never triggers a rebuild and captured snapshots stay valid.
  • This mirrors the existing snapshot discipline (QuerySnapshot, slot invalidation) — the mask is just one more immutable-per-publish array.
  • Republish coalescing (watch, don't pre-build): each setRangeVisible copies the whole mask, and cwLinePlotTripVisibility fires once per trip — a keyword flip hiding N trips does N full-array copies. At 1 byte/vertex that's memcpy-speed (~1 MB per copy on a 1M-vertex plot), a few ms worst case, never per-frame. If profiling ever cares, coalesce like scheduleTopLevelRebuild: mutate a pending mask, publish once per event-loop tick.
  • pointsInBox reads the same snapshot but stays visibility-unfiltered on purpose — it is an explicit-Key query; the caller already chose the object.

Alternatives considered

  • Per-trip Objects — register each trip as its own Key and reuse whole-object isPickable. Rejected: restructures centerline registration, multiplies top-level BVH objects (perf), and changes the pick firstIndex contract (#530). The mask is purely additive — one object, one BVH.
  • Visibility callback into the render object — intersecter reads cwRenderLinePlot::m_visibility live. Rejected: couples the intersecter to render internals and breaks off-thread pick safety (the array mutates under the pick). The owned-copy-in-snapshot avoids both.
  • Fix only reset-view in the interaction (enumerate visible ranges, union in the interaction). Rejected: duplicates visibility logic per consumer and leaves #575 unfixed.

Follow-ups post-implementation review, July 2026

Surfaced by the pre-commit review of the implemented phases. None block the landed work; ordered by expected value.

1 · Masked visibleBoundingBox() walk runs at interaction rate, not reset-only

isPickableEmpty() delegates to visibleBoundingBox() and is called from cwBaseTurnTableInteraction's pivot fallback (cwBaseTurnTableInteraction.cpp:136) — reached on every pan mouse-down over background and on the zoom m_perspectiveIntersection re-evaluation. With a line plot partially hidden, each of those pays the O(vertices) visibleNodeBox walk (~1–3 ms at 500 k vertices) — the perf table's "reset only" row was wrong about the caller set. Fix per the table's own escape hatch: cache the visible box per Node, invalidate in setRangeVisible (mask changes at user rate), or give isPickableEmpty() an any-visible early exit. Headers already document the corrected caller set.

2 · Producer-published immutable mask (third option, replaces the twin)

The implemented twin keeps two byte-for-byte copies of the same mask — cwRenderLinePlot::m_visibility and Object::m_visibility — synchronized by convention ("both reset on setGeometry, same ranges applied") across two files, plus an O(vertices) copy per republishSlotVisibility. The plan's rejected alternative was a live read (races off-thread picks); the implementation surfaced a third option it never weighed: the render object publishes the mask as std::shared_ptr<const QVector<quint8>> copy-on-write, and the intersecter (live Object and snapshot slot) holds only the pointer. Kills the duplicate buffer, the parallel range-write/validation loops, the per-call copy, and the comment-only sync invariant — while keeping snapshot immutability. Out of scope for the landed commit because it changes m_visibility's type, which the RHI upload path (cwRHILinePlot) consumes.

3 · Stable identity in Key + a destruction lifetime hook

Key holds a raw cwRenderObject* as identity, and clear(parentObject) has zero production callers — nothing purges a destroyed render object's entries. m_hiddenKeys raises the stakes: it is deliberately long-lived identity state (survives geometry replacement, may be seeded pre-registration), so an address-reused new object could be born spuriously hidden — the same failure family as the issue #512 fix in cwScene.cpp. Mitigated in the landed commit by addItem's unconditional visibility seed, but the real fix is Key{cwRenderObjectId, uint64_t} (precedent: cwRenderObject::renderObjectId()) plus cwScene::removeItem → GeometryItersecter->clear(item). Wide but mechanical: every registration call site (cwRenderTexturedItems, cwRenderLinePlot, cwRenderPointCloud, cwRenderGLTF, tests). Pairs well with folding the whole-object isPickable() gate into the snapshot (prerequisite for off-thread picks — the sink-scan BVH broad-phase wants this).
