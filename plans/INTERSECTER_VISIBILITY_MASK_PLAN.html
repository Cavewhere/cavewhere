Intersecter visibility mask

Making cwGeometryItersecter visibility-aware at per-primitive granularity, so picking and bounding-box queries honor keyword-driven visibility from one source of truth.

Fixes #575 (hidden shots still pickable) and #549 (reset frames hidden geometry) through a single mechanism.

0 · Goal & the two bugs

Visibility in CaveWhere is keyed on keywords: the user filters by keyword (Cave, Trip, Year, Type…), a filter pipeline splits keyword items into an accepted (visible) and a rejected (hidden) set, and each item hides its geometry at whatever granularity it owns. That hiding is applied on the render side only. The intersecter — which answers picks and bounding-box queries — never learns about it, except at whole render-object granularity.

BugSymptomCause
#575 The Distance tool snaps to a hidden shot; turntable pivot can land on a hidden trip. Per-shot line-plot visibility never reaches the intersecter; a Kind::Lines pick hits hidden segments.
#575-class
(scrap flavor)		 Keyword-hidden scraps and glTF models are still pickable, and the rotation pivot (nearestGeometryPoint) anchors onto them. Manually confirmed. Per-id visibility (cwRenderTexturedItems::setVisible(id)) is render-side only; all scraps share one render object, so the whole-object gate never fires.
#549 Reset view frames hidden caves/trips/scraps/LiDAR — the model zooms out to include things you hid. boundingBox() unions every registered object with no visibility filter at all (its doc now says so explicitly).

Design goal. Make the intersecter return correct visible-only results, so consumers (reset-view, Distance tool, rotation pivot, lead occlusion) are written as if cwGeometryItersecter is simply correct — no visibility logic duplicated into each consumer. The failing tests that pinned the bugs (testcases/test_cwRenderLinePlotPickVisibility.cpp #575 per-shot; testcases/test_cwRenderTexturedItemsPickVisibility.cpp per-id pick + framing bounds) have been flipped to regression tests — all phases below are implemented (July 2026): Phase 1 + the framing-bounds seam, Phase 2a (per-Key flag: setVisible(Key, bool), live m_hiddenKeys that survives geometry replacement, per-slot slotVisible in the published snapshot), and Phase 2b (per-vertex mask: setVisibilityMask(Key, QVector<quint8>), mask on the live Object, per-slot array in the snapshot; empty = all-visible). Both republish copy-on-write like invalidatePublishedSlot(), and the build-install callback re-stamps visibility from live state so a toggle can't race the worker.

Two follow-ups from the review have since landed on top and supersede the Phase 2b design below — read Follow-ups first. In short: the mask's owner now publishes its own buffer (setVisibilityMask) instead of the intersecter deriving a twin from setRangeVisible(Key, start, count, bool), and the masked bounding-box walk is memoized per Key on the intersecter rather than re-run per query. The sections below are kept as the design record that got us here.

Phase 0 · How it works today current

Three layers, wired end-to-end. Visibility flows left→right; the intersecter is the one box the flow skips.

Phase 0a · Visibility is keyed on keywords

  • KeywordTab.qml:17-20 — KeywordVisibility { visibleModel: pipeline.acceptedModel; hideModel: pipeline.rejectedModel }. The filter pipeline is the source of truth for what's shown.
  • cwKeywordVisibility.cpp:105 resolveVisibility() — hide wins: if an object is in the reject model, hide it; else show. Fires on rowsInserted/Removed/modelReset.
  • cwKeywordVisibility.cpp:17 invokeVisible() dispatches by type: QQuickItem, cwRenderObject, or a generic setVisible(bool) — the last is how the proxy objects below get called.
  • Each keyword item carries an ObjectRole pointing at the object that knows how to hide the corresponding geometry at the right granularity.

Phase 0b · How the renderer honors visibility

  • Whole object — cwRHIObject::m_isVisible, synced in cwRhiScene.cpp:71 and skipped at gather in cwRhiFrameRenderer.cpp:569 (if (!object->isVisible() …) continue;).
  • Per-vertex centerline — cwRenderLinePlot::m_visibility is a QVector<quint8> (one byte/vertex, kVisible=0xFF/kHidden=0x00). cwRHILinePlot.cpp:44 uploads a parallel m_visibilityBuffer (uint attribute, binding 1); cwRHILinePlot.cpp:102 updateVisibilityBuffer() re-uploads on change; LinePlot.vert/.frag discard hidden vertices. This is the only place per-shot visibility exists.
  • Per-scrap — cwRhiTexturedItems stores per-id visible and skips hidden items at draw.

Phase 0c · What the intersecter does — and doesn't

Refreshed July 2026 after the traversal rework (tube-fallback deletion, pivot anchoring, traverseBvh unification).

  • Geometry is registered once as the full vertex list in cwRenderLinePlot.cpp:46-59 (setGeometry → addObject). Visibility toggles (setRangeVisible) never call the intersecter. Likewise cwRenderTexturedItems::setVisible(id) (cwRenderTexturedItems.cpp:222) updates render state only.
  • Both ray entry points — intersectsDetailed (ExactPick) and the rotation-pivot anchor nearestGeometryPoint (AnchorPick) — now share one traversal, traverseBvh<Policy>. The only visibility gate is whole render-object: isPickable() = parent==nullptr || parent->isVisible(), hoisted once per sub-BVH descent inside traverseBvh. One hook point now covers every ray query.
  • boundingBox() unions all node boxes with no visibility check whatsoever — its doc now says "Counts hidden objects" explicitly.
  • isPickableEmpty() (added for the pivot fallback) already contains the Phase 1 loop: it unions boxes over isPickable-filtered live Nodes, then throws the box away and returns only its emptiness.
  • pointsInBox(QuerySnapshot, box, Key) is an off-thread, explicit-Key query (sink-scan broad phase). It is deliberately unfiltered: the caller already chose the object, so visibility policy belongs to the caller, not this query.
  • Key type is Key{cwRenderObject* parentObject, uint64_t id}; the published BVH keeps a keyToSlot reverse index, so per-Key state can be republished with a direct lookup.

Phase 1 · Whole-object visibleBoundingBox() interim implemented

Ship the reset-view API immediately, before the deeper work. The body already exists: isPickableEmpty() unions boxes over isPickable-filtered live Nodes and discards the box. Phase 1 is an extraction, not new logic.

  • Extract that loop into cwGeometryItersecter::visibleBoundingBox(); reimplement isPickableEmpty() as { const auto box = visibleBoundingBox(); return box.isNull() || !box.isFinite(); }. Main-thread only (computed against live Nodes), flat loop over cached per-Node boxes — O(#Keys), no BVH or vertex traversal.
  • Point cwBaseTurnTableInteraction::resetView() at visibleBoundingBox() (replacing boundingBox() at cwBaseTurnTableInteraction.cpp:424); keep the empty-box fallback.
Coverage is narrower than it looks. The only keyword items that carry a whole cwRenderObject are point clouds (cwLazLayersSceneNode.cpp:367). Scraps, glTF models, and centerline trips all hide through per-id / per-range proxies whose shared parent stays visible — so Phase 1 fixes framing for hidden clouds only. Phases 2a/2b upgrade correctness for the rest with no change to reset-view, which is written once against the final API.

Architectural note · put the seam in the right place

Today resetView() reads scene()->geometryItersecter()->boundingBox() directly (cwBaseTurnTableInteraction.cpp:331). "What bounds should the camera frame" is a scene / view-presentation concern, but cwGeometryItersecter::boundingBox() is an artifact of the pick-acceleration structure. Sourcing framing policy straight off the picking BVH leaks the picking subsystem's boundary into camera behavior.

Preferred seam: expose a scene-level framing-bounds accessor on cwScene / cwRegionSceneManager (e.g. framingBounds() / visibleFramingBounds()), and have resetView() (and any capture/export framing) call that. The visibleBoundingBox() work below then lives behind the scene seam rather than being one more method bolted onto the intersecter — the interaction depends on the scene for framing, not on the pick structure.

Blast radius: moderate — one new scene-level accessor, re-point resetView (and the Phase 1/2 work) at it; no data-model changes. Tracked as the architectural follow-up from the #549 review; can land before or alongside Phase 1.

Phase 2a · Per-Key visible flag scraps + glTF implemented

Textured items don't need the mask: each item id is already its own intersecter Key with its own Node and sub-BVH slot. They need one bit.

  • Add cwGeometryItersecter::setVisible(const Key&, bool visible) — a per-Key flag on the live Node, republished into the snapshot slot (the keyToSlot index makes this a direct lookup). No geometry change, no BVH rebuild.
  • traverseBvh consults it beside the existing isPickable() hoist — one boolean per descended sub-BVH, zero per-primitive cost. Because the traversal is shared, picking (ExactPick) and the rotation pivot (AnchorPick) are both fixed by the same check.
  • visibleBoundingBox() / isPickableEmpty() skip per-Key-hidden nodes — per-scrap and per-glTF framing become correct.
  • Forward from cwRenderTexturedItems::setVisible(id) (cwRenderTexturedItems.cpp:222): one line, intersecter->setVisible({this, id}, visible). Scraps (per-id proxies) and glTF groups (N per-id calls) are both covered — no manager changes.
  • Flip the two [!shouldfail] cases in test_cwRenderTexturedItemsPickVisibility.cpp; repoint its bounds test at visibleBoundingBox().
Snapshot subtlety. sub.object inside a published snapshot is a copy of the Object, so a plain member on the live Object won't be seen by held snapshots. The whole-object gate gets away with this because it dereferences the live parent->isVisible(); the per-Key flag instead follows the same discipline as the Phase 2b mask — per-slot shared state, republished on change. Ray queries run on the main thread, so a simpler live-read is also defensible; decide at implementation time, but keep it consistent with the mask.

Phase 2b · The per-primitive mask line plot implemented

Give the intersecter its own visibility channel, fed from the same setRangeVisible calls the renderer already makes. Both picking and bounding box consult it. One mechanism → both #575 and per-trip #549 fall out. Only the line plot needs this — it is one Key whose visibility varies per vertex.

2.1 · Mask on the Object

// cwGeometryItersecter.h — Object
QVector<quint8> m_visibility;   // empty = all visible (fast path)
const QVector<quint8>& visibility() const { return m_visibility; }
// helper: a Lines primitive at index i is visible iff both its
// endpoint vertices' bytes != kHidden (indices[i], indices[i+1]).

Empty mask ⇒ zero per-primitive cost for clouds, scraps, triangles, and any object that never partially hides.

2.2 · Mutation API (no BVH rebuild)

void cwGeometryItersecter::setRangeVisible(const Key& key, int start, int count, bool visible);
// Updates the mask on the live Node AND republishes the mask into the
// query snapshot (copy-on-write swap of a per-slot shared array).
// Geometry/BVH structure is unchanged, so this is O(range), not a rebuild.

2.3 · Render side pushes the mask

  • cwRenderLinePlot::setGeometry() — after addObject, seed an all-visible mask (or none).
  • cwRenderLinePlot::setRangeVisible() — additionally call geometryItersecter()->setRangeVisible(Key{this,0}, start, count, visible). The intersecter mask becomes the twin of the render m_visibility, fed by the same call.
  • Textured items do not push a mask — Phase 2a's per-Key flag covers them.

2.4 · Consumers honor the mask

  • Ray queries — in traverseBvh's leaf loop, check the mask just before policy.testPrimitive(ctx, prim). One check in the shared traversal covers both policies (ExactPick picking and AnchorPick pivot anchoring) with no per-policy duplication. Hoist the has-mask test to the sub-BVH descent (beside isPickable()) so unmasked objects — clouds, scraps, triangles — pay one pointer compare and nothing per primitive.
  • Bounding box — visibleBoundingBox() unions only visible primitives. Masked objects iterate their primitives (can't use the cached node box); unmasked objects keep the fast whole-box path. O(vertices), only on reset, only for the line plot.

2.5 · Thread-safety & the snapshot

The one real cost. Picks run off-thread against an immutable snapshot (SubBvhs). Bounding box runs on the main thread against live Nodes.
  • Bounding box: main-thread, reads the live mask directly — trivial.
  • Picking: the mask must live in the published snapshot so an in-flight pick sees a consistent view. Store the mask as a per-slot shared_ptr<const QVector<quint8>> alongside each SubBvh; setRangeVisible builds a new array and atomically swaps the pointer (keyToSlot finds the slot directly). BVH nodes are untouched, so toggling never triggers a rebuild and captured snapshots stay valid.
  • This mirrors the existing snapshot discipline (QuerySnapshot, slot invalidation) — the mask is just one more immutable-per-publish array.
  • Republish coalescing (watch, don't pre-build): each setRangeVisible copies the whole mask, and cwLinePlotTripVisibility fires once per trip — a keyword flip hiding N trips does N full-array copies. At 1 byte/vertex that's memcpy-speed (~1 MB per copy on a 1M-vertex plot), a few ms worst case, never per-frame. If profiling ever cares, coalesce like scheduleTopLevelRebuild: mutate a pending mask, publish once per event-loop tick.
  • pointsInBox reads the same snapshot but stays visibility-unfiltered on purpose — it is an explicit-Key query; the caller already chose the object.

Follow-ups post-implementation review, July 2026

Surfaced by the pre-commit reviews of the implemented phases. 1 and 2 landed together (they touch the same mask path); neither is blocked by what follows.

Read 6 first. It is the only open item that is a live defect rather than a structural improvement — removing a LAZ layer leaves the intersecter dereferencing a deleted render object on every pick — and it is also the smallest. 3, 4 and 5 are architecture and belong with issue #579: 5 is its natural first phase, 4 is how the authoring/built split gets expressed (not a precursor to it), and 3 is the belt-and-braces once 6 supplies the belt.

1 · Masked visibleBoundingBox() walk runs at interaction rate, not reset-only implemented

isPickableEmpty() delegates to visibleBoundingBox() and is called from cwBaseTurnTableInteraction's pivot fallback (cwBaseTurnTableInteraction.cpp:136) — reached on every pan mouse-down over background and on the zoom m_perspectiveIntersection re-evaluation. With a line plot partially hidden, each of those pays the O(vertices) visibleNodeBox walk (~1–3 ms at 500 k vertices) — the perf table's "reset only" row was wrong about the caller set. Fix per the table's own escape hatch: cache the visible box per Node, invalidate in setRangeVisible (mask changes at user rate), or give isPickableEmpty() an any-visible early exit. Headers already document the corrected caller set.

Landed as m_maskedBoxCache, a mutable QHash<Key, QBox3D> filled on demand by visibleNodeBox(). The walk now runs once per mask change instead of once per query, so the interaction-rate callers pay only the O(#Keys) loop. Cached in model space, matching the sub-BVH's model-space discipline: a setModelMatrix() then costs a transform rather than a re-walk (pinned by "masked framing box follows the model matrix"). Chose the cache over the any-visible early exit because the early exit only rescues isPickableEmpty(), leaving reset-view's real box to re-walk. Staleness is regression-tested: disabling the invalidation fails "hidden shot does not inflate framing bounds".

Why the cache is not on Object, where the mask it derives from lives. That was the first cut — a lazy mutable std::optional<QBox3D> reset by setVisibilityMask(), which has the real advantage that invalidation is structurally impossible to forget. It is a data race. launchBuildJob snapshots the source nodes with nodesSnapshot = Nodes — implicitly shared, and visibleBoundingBox() walks Nodes const, which never detaches — so the worker's SubBvh copies each Object out of the same memory the main thread would be lazily writing the cache into. Reachable whenever a top-level rebuild is in flight (setModelMatrix, or removing an unrelated Key) while a partially hidden plot is framed or pivoted; no test would catch it. The intersecter itself is never handed to a worker, so a cache on it cannot race — at the cost of explicit invalidation sites instead of one structural drop, mirroring how m_subBvhs and m_hiddenKeys are already maintained.

Only setVisibilityMask's drop is load-bearing, which is worth stating because the review initially mis-read the others as correctness: the cache is consulted only for a non-empty mask, and a mask can only become non-empty through setVisibilityMask — while addObject always re-registers with a fresh empty-masked Object. So a stale entry can be leaked but never read, and the drops in eraseNodeIfPresent / clear bound growth rather than protect a result. Verified by deleting the eraseNodeIfPresent drop: the suite stays green, whereas deleting setVisibilityMask's fails "hidden shot does not inflate framing bounds".

2 · Producer-published immutable mask (third option, replaces the twin) implemented

The implemented twin keeps two byte-for-byte copies of the same mask — cwRenderLinePlot::m_visibility and Object::m_visibility — synchronized by convention ("both reset on setGeometry, same ranges applied") across two files, plus an O(vertices) copy per republishSlotVisibility. The plan's rejected alternative was a live read (races off-thread picks); the implementation surfaced a third option it never weighed: the render object publishes the mask as std::shared_ptr<const QVector<quint8>> copy-on-write, and the intersecter (live Object and snapshot slot) holds only the pointer. Kills the duplicate buffer, the parallel range-write/validation loops, the per-call copy, and the comment-only sync invariant — while keeping snapshot immutability. Out of scope for the landed commit because it changes m_visibility's type, which the RHI upload path (cwRHILinePlot) consumes.

Landed — but without the shared_ptr. CLAUDE.md forbids wrapping a QVector in a shared_ptr, and it is right: QVector is already an immutably-shareable value type, so cwGeometryItersecter::setVisibilityMask(Key, QVector<quint8>) taking it by value buys the same refcount-bump publication with no wrapper. The type of m_visibility never changes, so cwRHILinePlot was untouched and the change is far smaller than scoped. This removed the pre-existing QVector<std::shared_ptr<const QVector<quint8>>> from BvhData in favour of QVector<QVector<quint8>> (empty = all-visible, same signal null carried).

The intersecter no longer derives a mask: setRangeVisible(Key,start,count,bool) is gone, and the owner — which already maintained this exact buffer for its shader — publishes it. One buffer now backs the renderer, the live Object, and every snapshot slot. Per toggle that is one deep copy (the owner's own detach, unavoidable), down from two, plus one make_shared allocation eliminated. The byte encodings never needed to agree: every reader only asks is this byte nonzero, so the renderer's 0xFF and the slot flag's 1 coexist. Cost of the new seam: the published buffer is immutable by contract (owners write to a copy and replace), documented on both sides. Out-of-range guarding now lives only at the owner's setRangeVisible, and a short mask reads permissive — pinned by "mask shorter than the geometry hides only what it covers".

3 · Stable identity in Key (the destruction hook split out to 6)

Key holds a raw cwRenderObject* as identity, and clear(parentObject) has zero production callers — nothing purges a destroyed render object's entries (that half is now follow-up 6, which is a live bug and should land first; this item is the belt-and-braces for a purge that is missed rather than absent). m_hiddenKeys raises the stakes: it is deliberately long-lived identity state (survives geometry replacement, may be seeded pre-registration), so an address-reused new object could be born spuriously hidden — the same failure family as the issue #512 fix in cwScene.cpp. Mitigated in the landed commit by addItem's unconditional visibility seed, but the real fix is Key{cwRenderObjectId, uint64_t} (precedent: cwRenderObject::renderObjectId()) plus cwScene::removeItem → GeometryItersecter->clear(item). Wide but mechanical: every registration call site (cwRenderTexturedItems, cwRenderLinePlot, cwRenderPointCloud, cwRenderGLTF, tests). Pairs well with folding the whole-object isPickable() gate into the snapshot (prerequisite for off-thread picks — the sink-scan BVH broad-phase wants this).

4 · Five parallel per-Key side tables want to be one per-Key record — but only after the authoring/built split do it, not standalone

Nodes (cwGeometryItersecter.h:305), m_subBvhs, m_dirtyKeys, m_hiddenKeys, and now m_maskedBoxCache are five per-Key stores with five independent lifetime rules — and Nodes isn't even keyed: findNode() is a linear std::find_if. This commit is the evidence for the complaint: adding the fifth store meant hand-patching every lifetime site, one of them deliberately placed above an early return — exactly the ordering hazard a single QHash<Key, PerKeyRecord>::erase makes unrepresentable.

Worth doing — but not as its own commit, and not first. Three things temper it, and the review that proposed it also produced the counter-evidence:

  • The headline benefit is overstated. "Five invalidation rules" implies five chances to be wrong; in fact only setVisibilityMask's drop is load-bearing (see follow-up 1) — the rest bound cache growth and cannot produce a wrong answer. Verified by deleting them: the suite stays green. Consolidating buys tidiness and future-proofing, not a live bug class.
  • The findNode win is negligible. Its callers (setVisibilityMask, republishSlotVisibility, boundingBox(Key), setModelMatrix) all run at user-interaction rate over tens-to-hundreds of Keys, never per frame. O(1) here is not a measured cost.
  • Done naively it re-creates the race follow-up 1 just removed. If PerKeyRecord holds the Object and its derived caches, and launchBuildJob snapshots that hash (QHash is implicitly shared, so the header-copy trick does survive), then a main-thread write to a cached box through a const read aliases the record the worker is concurrently copying into SubBvh::object. Same hazard, new container.

So the per-Key record is not a precursor to the authoring/built split — it is how that split gets expressed. The split: give the build worker an immutable built payload (geometry + matrix + published mask, produced once at launch) and keep authoring state (Object, masks, caches, dirty marks) main-thread-only and never shared. Then derived caches can live beside the inputs they derive from, invalidation becomes structural rather than four remembered call sites, and "is this member safe to make mutable?" stops needing a paragraph of comment each time (cwGeometryItersecter.h:411-425). It also composes with follow-up 3 and with item 5 below: an immutable built payload is exactly where a captured isPickable() belongs, which is what unblocks off-thread picks for the sink-scan BVH broad-phase.

Blast radius: large but contained to cwGeometryItersecter.{h,cpp} — Node/Object/SubBvh/BvhData/launchBuildJob/buildSubBvh. Multi-day; needs its own plan. Sequence it with follow-up 3 (both re-key the class) rather than against it.

5 · Pull the pick-registration convention up into cwRenderObject

Every renderer hand-wires the same undocumented checklist: null-check geometryItersecter(), addObject, then remember that addObject reset this Key's visibility and re-publish it. See cwRenderLinePlot.cpp:46-59 and :110-116; cwRenderTexturedItems.cpp:139, :160, :189, :249, :309, :322; cwRenderPointCloud.cpp:36; cwRenderGLTF.cpp:155.

The "re-publish after registration" step is a convention held only by comments, and the comments themselves record that forgetting it has already bitten — cwRenderTexturedItems.cpp:157-162 and :205-209 exist because a hidden scrap resurrected through an empty→non-empty geometry cycle. This commit adds a third prose-only rule to the pile (cwRenderLinePlot.h:47-51: "immutable once published — write to a copy and replace, never mutate in place"). None of the three is enforced by anything but a reader's diligence.

The change: a base-class seam on cwRenderObject — publishPickGeometry(id, geometry, matrix) and publishPickVisibility(id, …) — that owns the null-check, the ordering, and the post-registration re-publish, so a renderer cannot register geometry without also restating its visibility. Then route cwRenderObject::setVisible (cwRenderObject.cpp:74) through it, which is the missing hook that lets refreshPublishedVisibility AND parent->isVisible() into slotVisible at publish time — retiring the last live cwRenderObject* deref inside traversal (cwGeometryItersecter.cpp:182).

Two things fall out. First, it is the seam the per-Key-flag/per-vertex-mask convergence needs: setVisible(Key,bool) keeps m_hiddenKeys alive across addObject as "identity state", but cwRenderTexturedItems already re-seeds unconditionally after both its registration paths (:161, :209) — so the survival is a second copy of a fact the owner re-supplies immediately, and a second copy can only ever be wrong. It is belt and braces where the braces have a known gap. Once registration guarantees the belt, the flag can adopt the mask's lifetime (drop on addObject, ignore unknown Keys) and m_hiddenKeys disappears, collapsing two mechanisms into one. Caveat: that removes one of follow-up 3's stated motivations (an address-reused object born spuriously hidden) but not follow-up 3 itself — nothing purges a destroyed render object (cwScene::removeItem at cwScene.cpp:84 drops render-side refs only; ~cwRenderObject() is empty; clear(parentObject) has zero production callers), so the leaked Node's isPickable() deref stays a use-after-free regardless. It shrinks follow-up 3's rationale, not its necessity.

Second, the front-end/back-end split stops being "each renderer remembers the protocol", and the intersecter stops needing a raw back-pointer to a QObject it dereferences mid-traversal.

Blast radius: cwRenderObject, the four renderers above, cwScene, and their tests. Out of scope for the follow-up commit — this is the natural Phase 1 of issue #579's "first-class scene visibility model", and doing it there would swamp it.

6 · Nothing purges the intersecter when a render object dies — a live use-after-free bug, land first

Three facts that only bite when combined:

  • cwScene::removeItem (cwScene.cpp:84) drops render-side references only — it never touches geometryItersecter().
  • cwRenderObject::~cwRenderObject() (cwRenderObject.cpp:32) is empty.
  • cwGeometryItersecter::clear(parentObject) has zero production callers — the very API this needs.

So a destroyed render object leaves its Node behind, and the Node's Key holds a dangling cwRenderObject*. Every subsequent pick dereferences it: traverseBvh hoists isPickable(sub.object) (cwGeometryItersecter.cpp:182-185, :1191) → parent->isVisible() on freed memory. visibleBoundingBox() does the same (:807), so reset-view and the pan/zoom pivot fallback hit it too.

This is reachable today, by a normal user action. cwLazLayersSceneNode::dematerialize (cwLazLayersSceneNode.cpp:296-300), reached from removeLayer — i.e. removing a LAZ layer — does:

cwRenderPointCloud* renderObject = it.value();
renderObject->setScene(nullptr);   // → cwScene::removeItem: render-side refs only, then m_scene = nullptr
delete renderObject;               // → default dtor: no purge

It never calls cwRenderPointCloud::clear() — the one place that does intersecter->removeObject(this, 0) (cwRenderPointCloud.cpp:59-60). cwLazLayersSceneNode::clear() (:203-204) repeats the same two lines. And the ordering is fatal to the obvious fix: setScene(nullptr) nulls m_scene before the delete, and geometryItersecter() is m_scene == nullptr ? nullptr : … (cwRenderObject.cpp:68-71) — so a purge written in ~cwRenderObject() would find no intersecter and silently do nothing. Any fix must purge before the scene is unset, or reach the intersecter another way.

Second consequence, arguably as bad: the leaked Node retains the full geometry, and m_subBvhs retains its sub-BVH. For a LAZ cloud that is potentially gigabytes held forever, per removed layer — the exact memory the sub-BVH cache is elsewhere carefully written to avoid double-holding (cwGeometryItersecter.h:454-459).

Fix: the cheap version is cwScene::removeItem → geometryItersecter()->clear(item), which is presumably what clear(parentObject) was written for and would give it its first caller. Better, fold it into follow-up 5's seam so registration and release are symmetric and a renderer cannot have one without the other. Note cwScene::removeItem already captures renderObjectId() "while item is still alive … without ever dereferencing a dangling pointer" for the RHI side (issue #491/#512) — the scene learned this lesson once already; the intersecter side was never done.

Test: materialize a cloud, dematerialize it, then pick — under the macOS Debug Address Sanitizer that should fail loudly today. Worth writing the failing test first to confirm the path before fixing.

Blast radius: small — one call site plus a regression test. Land ahead of 3, 4 and 5: it is the only item here that is a live defect rather than a structural improvement, and each of those three is wide. Splitting it out of follow-up 3 is what lets it go now, since 3's re-key is mechanical but touches every registration call site.
