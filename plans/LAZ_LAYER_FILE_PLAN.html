LAZ Layer Metadata Files (.cwlaz)

One protobuf .cwlaz file per LAZ file in GIS Layers/, mirroring how cwNote stores its metadata. Replaces the ProjectMetadata.lazLayerStates overlay so two collaborators editing different layers don’t conflict in .cwproj.

Context

The current Disabling LAZ Layers plan stores per-layer state as a repeated LazLayerState entry on ProjectMetadata inside the project’s .cwproj file. Every layer’s enabled bit lives in the same file. Two collaborators toggling different layers diff in the same place — guaranteed merge conflict as soon as the field has more than one entry.

CaveWhere already has a cleaner pattern for this: notes. Each cwNote serializes to its own .cwnote protobuf next to its image; loads discover them by directory glob; saves write only the affected note. Two collaborators editing different notes don’t conflict.

This plan applies that pattern to LAZ. Each GIS Layers/foo.laz gets a paired GIS Layers/foo.cwlaz protobuf file carrying that layer’s persistable state (today: id + enabled; tomorrow: pointSize, sourceCSOverride, …). The lazLayerStates field on ProjectMetadata is deleted outright in the same series — it’s never been shipped, so there’s nothing to migrate.

File layout

<project>.cwproj/
  <dataRoot>/
    GIS Layers/
      foo.laz        ← existing
      foo.cwlaz      ← new: protobuf metadata, paired with foo.laz by basename
      bar.las
      bar.cwlaz
    Caves/…

Pairing is by basename. foo.laz and bar.LAS (case differences are preserved by cwLazLayerModel::rescan’s case-aware glob) each get a .cwlaz metadata file whose stem matches the source file exactly.

Schema

New message in cavewherelib/src/cavewhere.proto:

// Per-LAZ-layer persistent state. Stored as <basename>.cwlaz next to the
// matching .laz/.las file in "GIS Layers/". Discovery is by folder scan;
// the project's .cwproj file no longer references LAZ layers.
message LazLayer {
    optional string id = 1;       // UUID, stable across reopen
    optional bool enabled = 2;    // default true; absence == true
    // reserved for future per-layer settings:
    //   optional double pointSize = 3;
    //   optional string sourceCSOverride = 4;
}

repeated LazLayerState lazLayerStates = 7 on ProjectMetadata is removed in commit 3 and the slot becomes reserved 7;. The message LazLayerState definition is deleted at the same time. No on-disk project carries the field, so no compatibility shim is needed.

Identity

Every LAZ layer gets a persistent UUID. Today cwLazLayer generates a fresh QUuid in its constructor and the value is regenerated on every reopen; nothing outside the in-memory model knows or cares. With the per-layer .cwlaz file, the UUID becomes long-lived and survives reopen — useful for future sync/merge work, scene-graph hash keys, and any cross-system reference.

cwLazLayerData gains QUuid id;. cwLazLayer::data() returns m_id; cwLazLayer::setData(const cwLazLayerData&) overwrites m_id from the loaded .cwlaz before the layer is registered with the model (so the scene node and keyword model see the final UUID on their first observation).

Wire encoding follows the pattern already used by cwNote, cwTrip, cwCave, … — optional string id = N;, uuidToProtoString(layer->id()) for writes, cwProtoUtils::toUuid(proto.id()) for reads.

Discovery + load

cwLazLayerModel::rescan already enumerates *.laz / *.las in GIS Layers/. Extended behavior: after constructing a layer and calling setSourcePath(path), the model checks for a paired <basename>.cwlaz:

  • Metadata file present: ask cwSaveLoad::loadLazLayer(metadataPath) to deserialize. Call layer->setData(...). This overwrites the auto-generated UUID with the persisted one and applies any divergent state.
  • Metadata file absent: leave the auto-generated UUID and default state. Schedule a cwSaveLoad::save(layer) immediately so the first flush stamps the new layer’s .cwlaz to disk (the UUID has to land on disk so future loads see the same identity).

Apply state before beginInsertRows — same invariant as the applyPendingStateTo hook this plan replaces. That keeps row-insertion observers (cwLazLayersSceneNode::addLayer, cwKeywordItemModel wiring, cwSaveLoad’s enabledChanged connect) from seeing the transient default-state moment.

Save

Replace today’s enabledChanged → saveMetadata (which flushes the whole .cwproj file) with enabledChanged → cwSaveLoad::save(layer) — writes just <basename>.cwlaz. One state change ⇒ one .cwlaz diff. Concurrency, ordering, and Git commit all go through the existing cwSaveLoadPrivate::Job queue exactly the way cwNote already does it. The LAZ branch of saveObject<T> resolves the target path through absolutePathFor(this, layer) — same machinery notes use, so renames are tracked in m_objectStates automatically.

Every layer always has a .cwlaz on disk (eager writes). The lazy “skip writing if state is default” optimization that earlier drafts of this plan considered is dropped because UUIDs need to be stable across reopen — a layer with no .cwlaz would generate a new UUID every load.

Per-layer save flow

cwLazLayer::setEnabled(false)
  -> emit enabledChanged
    -> cwSaveLoad::save(layer)
      -> build CavewhereProto::LazLayer from layer->data()
      -> enqueue cwSaveLoadPrivate::Job (Kind::File, Action::Write)
      -> flush -> saveProtoMessage to GIS Layers/<basename>.cwlaz
      -> Git commit (same path that already commits .laz adds)

Removal

cwLazLayerModel::removeAt already triggers a queued delete of the .laz via cwSaveLoad::connectTreeModel’s rowsAboutToBeRemoved handler. Extend that handler to enqueue a second delete job for the paired .cwlaz. Both deletes funnel through the same Git commit, so the working tree never sees a half-deleted pair.

No rescan-time orphan sweep. Verified against notes: cwSaveLoad does not sweep stray .cwnote files at load time either. The policy is: removal goes through the model; files that show up on disk by other means (Git checkout, user deleting .laz in Finder, etc.) are left alone. If a .laz reappears later with a matching orphaned .cwlaz, pairing resumes — usefully preserving the layer’s identity across moves outside CaveWhere.

Rename

Users will rename layers. Driving case: the LAZ clip tool currently writes clip_001.laz, which is not a name anyone wants to keep. The user gives it a friendly basename after the clip lands.

Identity stays UUID-keyed inside the .cwlaz; the basename is a mutable display attribute. Renaming a layer atomically moves both sibling files (<old>.laz + <old>.cwlaz → <new>.laz + <new>.cwlaz) through the existing job queue. The UUID inside the .cwlaz is untouched, so reopen recovers the same layer identity.

API surface on cwLazLayerModel: 

Q_INVOKABLE bool rename(int row, const QString& newBaseName);
QString lastRenameError() const;  // populated when rename returns false
Q_PROPERTY(QString lastRenameError READ lastRenameError NOTIFY lastRenameErrorChanged)

// extension (UI strips display extensions before calling).
// On success: snapshots the old .laz path, then calls layer->setSourcePath

Collision handling: if the user picks a name that collides with another loaded layer, return an error result (UI surfaces a warning). If the name collides with a stray file in GIS Layers/ that isn’t a loaded layer (orphaned .cwlaz, or a .laz that didn’t load for some reason), also reject — the user has to resolve the conflict by hand.

Integration points with cwSaveLoad

The LAZ I/O reuses the existing infrastructure rather than introducing a parallel pipeline. Each of the touchpoints below mirrors how cwNote is wired today.

1. saveObject<T> template dispatch

cwSaveLoad::save(const cwLazLayer*) is a one-line delegate to d->saveObject(this, layer) — the same shape as save(const cwNote*), save(const cwTrip*), etc. The dispatch lives in cwSaveLoadPrivate.h:452 saveObject<T>: add a new if constexpr (std::is_same_v<T, cwLazLayer>) branch that calls saveProtoMessage(context, cwSaveLoad::toProtoLazLayer(object), object). This is the single mechanism for routing all saves through the job queue; no LAZ-specific code lives outside the template.

2. Proto helpers

Two new static helpers on cwSaveLoad, named to match the convention (toProtoNote / noteDataFromProtoNote): 

static std::unique_ptr<CavewhereProto::LazLayer> toProtoLazLayer(const cwLazLayer* layer);
static cwLazLayerData lazLayerDataFromProtoLazLayer(const CavewhereProto::LazLayer& proto,
                                                    const QString& filename);

toProtoLazLayer writes uuidToProtoString(layer->id()) and enabled. lazLayerDataFromProtoLazLayer reads them via cwProtoUtils::toUuid and a defaulted enabled. The filename parameter is the source .cwlaz path — needed to set cwLazLayerData::fileName from QFileInfo(path).completeBaseName() + ".laz" (or matching .las extension), so the discovery pass can pair the loaded data back to the right LAZ source.

3. Path resolution + m_objectStates

LAZ layers participate in cwSaveLoadPrivate::m_objectStates the same way cwNote / cwTrip / cwCave do. Each cwLazLayer* is registered in the state map on insertion (via discovery during rescan or via seedStatePathFromLoaded when a project loads) and deregistered on removal. saveObject<cwLazLayer> resolves the target through absolutePathFor(context, layer), exactly like notes.

Two overloads are added to support this: 

// cwSaveLoad.cpp (or wherever dirPrivate overloads live)
QDir cwSaveLoad::dirPrivate(const cwLazLayer* layer) const;
// returns <dataRoot>/GIS Layers/ — constant across all LAZ layers (flat directory)

// absolutePathFor builds: <dirPrivate(layer)>/<basename>.cwlaz

The private helper from earlier drafts — lazLayerMetadataPath — is replaced by routing through dirPrivate + the layer’s current basename. Single source of truth, but routed through the shared state-tracking machinery.

4. Job::artifactTag — lifting the one-artifact assumption

The job-queue infrastructure currently assumes one on-disk artifact per object. That assumption is baked into compressPendingJobs: jobs are grouped by (objectId, Kind) and the three compression rules (dropRedundantWrites, dropWritesSupersededByRemove, collapseSequentialMoves) all assume any group of same-(objectId, Kind) jobs operates on the same file. For LAZ — one layer, two sibling files — that assumption breaks: two paired Move jobs would be incorrectly collapsed into one corrupted job (the first’s oldPath and the last’s path).

The fix is to add a tag field that distinguishes artifacts within an object, and refine the three compression rules to subgroup by tag. The empty tag is the default and represents “the primary artifact” (matches every current caller bit-for-bit). LAZ uses "source" for the .laz file and the default empty for the .cwlaz.

Schema change in cwSaveLoadPrivate.h

struct Job {
    const void* objectId;
    Kind kind;
    Action action;
    QString artifactTag;   // NEW: empty = primary artifact (default).
                           //      Same-(objectId, kind, artifactTag) jobs may collapse.
                           //      Different artifactTag jobs never collapse with each other.
    QString oldPath;
    QString path;
    // ... existing fields ...
};

Compression refinement

collapseSequentialMoves subgroups the per-object indices by tag and applies its existing collapse rule independently to each subgroup: 

void cwSaveLoadPrivate::collapseSequentialMoves(const QList<int>& indices,
                                                QSet<int>& indicesToDrop)
{
    for (Job::Kind kind : {Job::Kind::File, Job::Kind::Directory}) {
        QHash<QString, QList<int>> byTag;
        for (int idx : indices) {
            const Job& j = m_pendingJobs[idx];
            if (j.action == Job::Action::Move && j.kind == kind) {
                byTag[j.artifactTag].append(idx);
            }
        }
        for (const QList<int>& tagIndices : std::as_const(byTag)) {
            if (tagIndices.size() > 1) {
                m_pendingJobs[tagIndices.last()].oldPath =
                    m_pendingJobs[tagIndices.first()].oldPath;
                for (int i = 0; i < tagIndices.size() - 1; ++i) {
                    indicesToDrop.insert(tagIndices[i]);
                }
            }
        }
    }
}

dropRedundantWrites and dropWritesSupersededByRemove get the same subgrouping treatment for consistency — a Remove on one artifact must not supersede a Write on a different artifact of the same object. LAZ doesn’t currently exercise these (the .laz source is user-supplied, we never WriteFile it), but the symmetry keeps the rules coherent.

Rename wiring — enqueueRenameIfNeeded<cwLazLayer>

With the tagged-job infrastructure in place, rename is straightforward and uses real Action::Move jobs through the standard pipeline — no Action::Custom workaround. Users will rename layers (renaming clip_001.laz to main_chamber.laz is the driving use case). Identity stays in the UUID inside the .cwlaz; the basename becomes a mutable display attribute.

  • cwLazLayerModel::rename(int row, const QString& newBaseName) validates the new name and snapshots the old .laz path before calling layer->setSourcePath(newPath).
  • The existing cwLazLayer::sourcePathChanged signal (cwLazLayer.h:126 — verified present) is connected in cwSaveLoad::connectTreeModel to d->enqueueRenameIfNeeded(this, layer).
  • enqueueRenameIfNeeded<T> gets a if constexpr (std::is_same_v<T, cwLazLayer>) branch that enqueues two real Move jobs: 
    // Primary artifact: .cwlaz. Standard path derivation via addFileSystemJob,
// tag = "" (default). absolutePathFor returns the .cwlaz path through
// dirPrivate(layer); m_objectStates tracks its currentPath as today.
addFileSystemJob(Job{object, Job::Kind::File, Job::Action::Move}, context);

// Secondary artifact: the .laz source file. Paths derived directly from
// the old/new sourcePath; tag = "source" so the compressor groups it
// independently from the primary.
Job lazMove;
lazMove.objectId    = object;
lazMove.kind        = Job::Kind::File;
lazMove.action      = Job::Action::Move;
lazMove.artifactTag = QStringLiteral("source");
lazMove.oldPath     = oldLazPath;   // snapshot before setSourcePath
lazMove.path        = newLazPath;
addExplicitFileSystemJob(lazMove, context);

return;
Verified: phantom-rename guard is already free. enqueueRenameIfNeeded at cwSaveLoadPrivate.h:393–396 short-circuits when state.currentPath.isEmpty() — it seeds the state and returns without enqueueing a job. So the first setSourcePath during rescan does not trigger a spurious Move. No special connect ordering needed.
Verified: pre-tag, two paired Move jobs corrupt under collapseSequentialMoves. cwSaveLoadPrivate.cpp:1373–1391: the current compressor groups by (objectId, Kind) only and collapses by taking the first job’s oldPath and the last’s path. For LAZ, that produces a single broken job (e.g. old.laz → new.cwlaz) and drops the other. The artifactTag change directly addresses this — not a workaround but a generalization that names what was always true: objects can have multiple distinct artifacts.
Verified: addFileSystemJob only resolves one path per layer. cwSaveLoadPrivate.cpp:815–821: it derives oldPath from m_objectStates[objectId].currentPath and path from absolutePathFor(context, objectId). For LAZ that returns the .cwlaz path; the .laz source needs explicit paths and routes through addExplicitFileSystemJob instead.

Behavior under sequential renames

A→B then B→C in quick succession enqueues four Move jobs (two per rename, one per tag group): 

Pre-collapse:
  {layer, File, Move, tag="",       A.cwlaz -> B.cwlaz}
  {layer, File, Move, tag="source", A.laz   -> B.laz}
  {layer, File, Move, tag="",       B.cwlaz -> C.cwlaz}
  {layer, File, Move, tag="source", B.laz   -> C.laz}

Post-collapse (tag="" group and tag="source" group collapse independently):
  {layer, File, Move, tag="",       A.cwlaz -> C.cwlaz}
  {layer, File, Move, tag="source", A.laz   -> C.laz}

Single net move per artifact. Optimal I/O. Single-rename case is unaffected.

5. Discovery loop reuse

cwLazLayerModel::rescan does not call the loadObjectsFromNotesDir helper in cwSaveLoad directly — that lambda is scoped inside loadRegion and tailored to the trip’s notes/ directory. Instead, rescan keeps its existing *.laz / *.las glob and, per discovered LAZ, calls cwSaveLoad::loadLazLayer(metadataPath) to deserialize the paired .cwlaz. The discovery shape (folder glob → per-file proto load → convert → append) is identical; we just don’t share the lambda body.

After construction, rescan calls d->seedStatePathFromLoaded(layer, absolutePathFor(this, layer)) so subsequent renames have a baseline to compare against in enqueueRenameIfNeeded.

6. Removal goes through the existing handler

No public delete/removeLazLayer method on cwSaveLoad. The rowsAboutToBeRemoved handler in cwSaveLoad::connectTreeModel is the single removal entry point — it already enqueues an Action::Remove job for the .laz; commit 2 extends it to enqueue a second job for the paired .cwlaz resolved through absolutePathFor(this, layer). Same as how note deletion is wired today. The state entry in m_objectStates is dropped at the same time.

No migration path. ProjectMetadata.lazLayerStates was added on the lazClipping branch and has never shipped — there are no on-disk projects carrying this field. The load path, save path, ProjectMetadataData::lazLayerStates field, cwLazLayerModel::data(QList) / setData(QList) / m_pendingStates, and the proto field + LazLayerState message are all deleted outright in commit 3. No compatibility shim, no pendingLazLayerMigration stash, no deterministic-UUID derivation.

Commit 3 — Rescan dirty bit + post-Git re-discovery

Two small bug fixes surfaced by commit 2 end-to-end testing. Both close the loop between on-disk state changes and the model / project dirty bit, so save / discard affordances behave predictably for external edits to GIS Layers/.

Bug A — rescan-driven removal does not dirty the project.
