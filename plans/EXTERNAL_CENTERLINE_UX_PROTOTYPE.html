External Centerline — UX Prototype

Companion to plans/EXTERNAL_FILE_INTEGRATION_PLAN.html. Purpose: pressure-test the UX against three real user scenarios before locking the implementation plan.

1. Goal & guiding principles

External centerlines are a power-user feature. Native CaveWhere has to stay simple — every new affordance is hidden, disabled, or absent until the user signals they want external data.

Concretely:

  • Native users see no new buttons on TripPage or CavePage. The "Attach external centerline" affordances are tucked under existing menus (Add Trip dropdown, cave actions menu) and the region-level entry point on DataMainPage is collapsed by default.
  • External-data screens are opt-in — reached by navigating into an attached scope, not pushed at users who don't have one.
  • The user's "Native CaveWhere only" workflow gets zero regressions: SurveyEditor, Add Trip, Fix Stations, etc. behave exactly as today.

2. The three use cases

Case 3 (most common) — TopoDroid trip as source

User: "My cave is already in CaveWhere. I went on a survey trip, recorded shots in TopoDroid, exported the trip as trip-2025-06-01.svx. I want CaveWhere to use that file as the centerline for one new trip in this cave."

Mental model: very close to "Import Survex as a trip" (already exists), except the import is live — if I edit the file in TopoDroid and re-export, CaveWhere updates.

Scope of change: one new cwTrip with externalCenterline set; sibling native trips unaffected.

Case 2 — Single cave in external format

User: "I survey a friend's cave that they maintain in Compass. They send me the .dat/.mak. I want to add my own notes and scraps in CaveWhere without duplicating the centerline."

Mental model: the file is the cave; CaveWhere is a notes-and-maps layer on top.

Scope of change: one cwCave with externalCenterline set. The file's internal *begin structure may give multiple trips; the user declares cwTrip-scope records under the cave to do per-survey carpet mode.

Case 1 — Region-level external dataset

User: "My survey group's entire region lives in Survex. Multiple caves, some connected. I want CaveWhere to read the region as-is and let me add notes/scraps per survey trip. I'll also create new caves in CaveWhere as we explore new entrances."

Mental model: CaveWhere is a viewer + annotation layer on top of the canonical Survex world.

Scope of change: region-level attachment plus many cwCave/cwTrip scope declarations mapped onto subtrees of the resolved network. Some caves coexist as fully-native CaveWhere caves (user adds new ones, then merges into the external via equates later).

3. Conceptual model

To make all three cases work cleanly with the same plumbing, a cwCave or cwTrip exists in one of three states:

State cwCave/cwTrip carries Driver emission Notes/scraps anchor on
Native (today's default) Survey chunks (shot data) + metadata Native *data normal… lines inside *begin cave_<uuid> / anonymous *begin Stations defined by own chunks
Attached (centerline source) externalCenterline pointing at a file *include "…" inside *begin cave_<uuid> / *begin trip_<uuid> Stations resolved under this scope's wrapper prefix
Scope (NEW — needed for cases 1 & 2) stationPrefix (a string like "bigpassage.east") None — purely a consumer-side filter into the resolved network Stations whose resolved name matches the prefix

The Scope state is what makes cases 1 and 2 work: a cwCave/cwTrip can exist purely to own notes/scraps anchored to a subtree of someone else's centerline. It doesn't contribute shots, doesn't get a *begin in the driver, doesn't appear in the exporter at all — it's just an addressing convention into a larger network.

Data model implication: add an optional stationPrefix string to cwCave and cwTrip. State resolution is mutually exclusive: a single cave/trip is either Native (has chunks) or Attached (has externalCenterline) or Scope (has stationPrefix). Three flags / a small enum on the protobuf.

7. The Structure view [DEFERRED with Case 1]

When a region-level (or cave-level) external centerline is attached, the Structure view exposes the resolved network's *begin hierarchy as a tree the user can map into cwCave/cwTrip scopes.

Wireframe

│  ▼ big-system                             (root)                     │
│    ▼ main-cave                  712 stns   ⚓ "Main Cave" (cave)     │
│      ▶ entrance                 47 stns    ⚓ "Entrance" (trip)      │
│      ▼ big-passage              634 stns   ⚓ "Big Passage" (trip)   │
│        ▶ main                   189 stns                             │
│        ▼ east                   267 stns                             │
│          ▶ east-2025-06-01      89 stns                              │
│          ▶ east-2025-07-15      178 stns                             │
│        ▶ west                   178 stns                             │
│      ▶ deep                     31 stns                              │
│    ▶ side-cave                  248 stns   ⚓ "Side Cave" (cave)     │
│    ▼ surface                    87 stns    (no scope)                │
│      ▶ ridge-traverse           54 stns                              │
│      ▶ sinkholes                33 stns                              │
│    ▼ ‹unattached fixes›         12 stns                              │
│                                                                       │
│  ────────────────────────────────────────────────────────────── │
│  Selected: big-system.main-cave.big-passage.east                │
│  Stations: 267                                                  │
│  [Make this a cave]  [Make this a trip]  [Auto-fill children…] │
└─────────────────────────────────────────────────────────────────┘

Interaction model:

  • Tree expands/collapses by clicking the ▸ chevron. Each node shows station count.
  • Nodes with an ⚛ (anchor) icon already have a scope mapped to them — the label shows the cwCave/cwTrip display name and whether it's a cave or trip scope.
  • Selecting a node shows a footer with mapping actions:
    • Make this a cave — creates a cwCave-scope at this node. Only enabled when no ancestor is already a cave or trip scope.
    • Make this a trip — creates a cwTrip-scope under the nearest cave-scope ancestor. The ancestor cave must exist first.
    • Auto-fill children — opens the auto-create dialog scoped to this subtree.
  • The "‹unattached fixes›" pseudo-node groups stations from *fix directives that aren't inside any *begin block — common in real Survex projects. Read-only.

Why a tree view and not a list

The unstructured-into-cwTrip problem (the user's case-1 question) has no single right answer — only the user knows how their data should be carved. A tree puts the file's own structure in front of them and lets them decide:

  • Some users will map at depth 1 (broad caves, no trips).
  • Some will map at depth 2 (caves + major sections).
  • Some will map at depth 3+ (individual survey days as trips).
  • Most will use Auto-fill at the top level and adjust.

Auto-fill heuristic

The defaults the Auto-fill dialog suggests (per case 2 above):

  • Top-level *begin children of the root → cave scopes (depth 1)
  • Inside each cave, leaf *begin blocks (no further sub-*begin) → trip scopes
  • Skip blocks with fewer than 5 stations by default (user toggle)

This isn't perfect — real projects often have intermediate organizational *begin blocks that aren't either a cave or a trip. The Structure view lets the user fix those individually.

9. Cross-case concerns

9.1 Adding native CaveWhere data into an externally-attached region

The user wants to add a new cave in CaveWhere that extends into the external dataset (e.g., they found a new entrance and want to add the first survey to it natively before exporting back to Survex).

Path: on DataMainPage with region attached, click Add Cave → Native cave. This creates a normal cwCave with native trips and chunks. To connect it to the external network, the user adds an equate (Phase 5) at the region level: equate native-cave.A1 = big-system.main-cave.entrance.E1.

9.2 Mixed states inside one cave

A cave can hold:

  • Native trips (own shots)
  • Attached trips (each with its own external file)
  • Scope trips (declare a prefix from a parent attachment)

Native + attached coexist as in §8.6 of the main plan. Scope trips only make sense when an ancestor (cave or region) is attached — without that, there's no resolved network to declare a prefix into. The Structure view should refuse to make a scope when there's no attachment in scope.

9.3 What appears in the 3D view

Everything resolved — native shots, attached centerlines, scope-declared subtrees — flows through the same cwStationPositionLookup and the same 3D scene renderer. Scopes don't add stations; they're just consumer-side filters for notes/scraps.

9.4 The "what's this trip's stations?" question

Native trip → the chunks own them, easy. Attached trip → resolved stations under cave_X.trip_Y.*. Scope trip → resolved stations matching the declared prefix.

One unified cwTripStationListModel with three backends. This is the shared model already called out in the main plan §7.2; the addition is the third backend for the scope case.

10. Open questions surfaced by this prototype

Resolved

  1. Terminology. Resolved: drop "scope" from the user-facing vocabulary. One word, "trip"; one badge, 📎; the Attached/Scope distinction is internal data-model only.
  2. Auto-create threshold and dialog. Resolved: no dialog. After cave-level attach, auto-create a cwTrip for every *begin block with ≥1 station, silently. User manages the resulting list (rename, delete, add more) like any other trip.
  3. Live-link vs. import mode. Resolved: radio in the attach dialog. Default to live-link (watch source + project copy). Import mode skips source-watching; source path is forgotten after the copy.
  4. Case 1 / Structure view in v1. Resolved: deferred to post-v1. The data-model prep (Scope state on cwTrip) still lands in Phase 3 for case 2.
  5. Detaching a scope trip. Resolved: there is no "detach" for a scope trip — it doesn't own anything. The buttons are Change prefix… and Remove trip…. Detach only applies to Attached caves/trips.

Newly resolved

  1. Renaming a Scope trip. Resolved: display name and stationPrefix are independent. Renaming "east" to "East 2025-06-01" changes the label only; the prefix-in-file binding (big-passage.east) is untouched. Changing the binding requires opening the trip and using Change prefix….
  2. Cave-level Detach with auto-created Scope trips. Resolved: deletes them, and deletes the copied files in External Centerlines/ too. Scope trips were created from the attachment; without the parent's centerline they have no meaning. Notes anchored to them go with them. Git history in the .cwproj is the recovery path if the user wants the closure back. Use the existing RemoveAskBox.qml pattern (the same lightweight inline confirm used by trip / cave / fix-station removal): 
    ┌─ Detach external centerline from "Big Cave"? ─────────────────┐
│  Removes 8 auto-created trips and their notes/scraps,         │
│  and deletes the copied files in External Centerlines/.       │
│                              [Cancel]  [Detach]               │
└───────────────────────────────────────────────────────────────┘
    For trip-level detach, the wording is shorter (no Scope-trip count; just "Detach external centerline from <trip name>? Deletes the copied files in External Centerlines/."). The "Convert Scope trip to native" escape hatch is a v1.x follow-up for the user who wants to preserve specific notes/scraps before detaching.

Resolved (this revision)

  1. Trip "Date" and "Decl" fields are kept and remain editable for both Attached and Scope trips. Date is not purely informational: CaveWhere computes auto-declination from date via the IGRF model (cwTripCalibration::autoDeclination), and many TopoDroid-exported .svx/.dat files don't set *calibrate declination at all. So we need:
    • Date — editable. Default: parsed from the external file's *date directive (Survex) or equivalent (Compass: SURVEY DATE; Walls: #DATE) when present. If absent, blank (user enters). Driving cavern uses this date by feeding it into the driver block before *include.
    • Decl — same model as native trips: autoDeclination on by default (uses Date + cave location for IGRF), with a manual override field (declinationManual) when the user wants to pin it. If the external file carries an explicit declination (Survex *calibrate declination, Compass header CORRECTIONS, Walls #UNITS DECL), import it as the initial manual value with autoDeclination = false.
    Net effect: external trips look the same in the trip header as native trips. The 📎 badge tells the user where the centerline came from; Date/Decl are still CaveWhere-managed.
  2. Replace / re-link an attachment. Promoted from deferred to v1.x. Two scenarios the user hits in practice:
    • Re-link: source file moved on disk (laptop reshuffle, new TopoDroid export folder). Same content, new path. UI: if the watcher reports the source as missing, the panel shows a "Source not found" banner with a Re-link… button. Picker selects the new path, scanner confirms the file has the same prefixes/station set as the cached copy in External Centerlines/…/, and we just update the source path in local_settings.json. No data change.
    • Replace: user picks a different file (newer export, revised survey). Same scope (trip or cave), different content. UI: Replace centerline… on the ExternalCenterlinePanel opens a picker, then a diff dialog:
