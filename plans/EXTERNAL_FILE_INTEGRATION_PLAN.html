External Centerline Integration Plan

Umbrella issue: #511 · Children: #508 Compass, #509 Walls, #510 Survex · Companion doc: plans/EXTERNAL_FILE_PREFIXING_RESEARCH.md

1. Overview

Let CaveWhere reference Compass (.dat / .mak), Walls (.wpj / .srv), and Survex (.svx) centerline data directly instead of importing it. The external centerline files are user-owned in their tool of choice, but on attach we copy them into the project under an External Centerlines/ directory, so the .cwproj stays self-contained and git sync works without surprises.

Inside CaveWhere these files are read-only: the user edits them with their native tool (or any text editor with the project open); a QFileSystemWatcher notices the change and triggers a re-solve. CaveWhere never writes back to .svx / .dat / .wpj / .srv — that drops the format-preserving-writer problem entirely.

The cavern solve, the .3d read, and the station-network pipeline are unchanged. Two new ingredients:

  1. cwCave and cwTrip can carry an cwExternalCenterline — a relative path into External Centerlines/ plus a format hint — that the exporter translates to *include "…" instead of native shot data.
  2. A cwExternalCenterlineScanner module finds the transitive dependency closure of an attached entry file (e.g. a .svx that *includes siblings, a .wpj's book tree), drives the initial copy into the project, and feeds the watcher set on subsequent runs.
Phase 0 is already complete — see plans/EXTERNAL_FILE_PREFIXING_RESEARCH.md for the prefix model investigation across CaveWhere, Survex, Compass, and Walls, plus the empirical cavern runs that answer the case-mode question (do not set *case preserve at the driver root) and the *export-for-equates question (not required).
Companion plan: line-plot async port. plans/LINEPLOT_ASYNC_PORT_RESEARCH.html covers the port of cwLinePlotTask + sub-tasks to a fully AsyncFuture chain on a worker thread, and surveys the existing cwSaveLoad job queue that the external-centerline filesystem mutations need to go through. Read it alongside this plan — the worker-thread chain and the cwSaveLoad::Custom job hook are prerequisites for Phase 1 here.

2. Architecture

Existing pipeline (today, all native data):

cwCavingRegion
  → cwLinePlotManager::runSurvex
  → cwLinePlotTask::buildInput      (snapshot region, rename caves to int index)
  → cwSurvexExporterRegionTask      (write driver .svx)
  → cwCavernTask                    (cavern_run on driver)
  → cwSurvex3DFileReader            (read .3d via img.h)
  → splitLookupByCave + station/scrap update

After this plan:

cwCavingRegion (with cwExternalCenterline attachments)
  → cwLinePlotManager::runSurvex                   [drives the AsyncFuture::Restarter chain]
  → cwExternalCenterlineScanner::scan(entry)       [NEW] per attachment (worker thread)
  → cwExternalCenterlineSync::reconcile()          [NEW] copy / GC, queued through cwSaveLoad
  → cwLinePlotTask::buildInput                     (uses cwCave::id() / cwTrip::id() as *begin label)
  → cwSurvexExporterRegionTask                     (NameStyle::InternalUuid; emits *include for attached scopes)
  → cwCavernTask                                   (unchanged)
  → cwSurvex3DFileReader                           (unchanged)
  → splitLookupByCave                              [REGEX UPDATED] UUID prefix + verbatim tail

Concurrency contract: the whole chain runs on a single worker thread via AsyncFuture; results land back on the UI thread via .context(this, …). Every filesystem mutation inside External Centerlines/ is submitted as a cwSaveLoad job so it serialises against the other project-tree mutators (autosave, sync-on-close, etc.). See plans/LINEPLOT_ASYNC_PORT_RESEARCH.html for the current-state survey, the existing cwSaveLoad job API, and the staged port.

Cave identifiers move from "integer index into the region snapshot" to the existing persisted cwCave::id() / cwTrip::id() UUIDs. Forced by the external-centerline work (external files contain their own nested *begin structure, and the current single-integer-prefix regex in splitLookupByCave doesn't survive it). No new identity field — reuse what's already persisted.

3. Data model changes

3.1 cwTrip / cwCave gain two new states

A cwCave or cwTrip exists in one of three states:

StateCarriesDriver emissionUI badge
Native (today) Survey chunks *begin… + native shots none
Attached (new) externalCenterline pointing at a file *begin…*include "…"…*end 📎
Scope (new, for case 2 internal trips and case 1 region scopes) stationPrefix string none — purely a consumer-side filter into another scope's resolved network 📎

The states are mutually exclusive on a single entity. From the user's perspective there's only "trip" with one optional 📎 badge; the Attached / Scope distinction is internal data-model only.

3.2 cwExternalCenterline (Attached state's payload)

Lives alongside cwSurveyChunk in the cave/trip data, owned by the cwCave or cwTrip in the Attached state. Deliberately tiny — just enough to find the entry file in the project tree.

FieldTypePurpose
entryFile QString Path relative to External Centerlines/ inside the project. e.g. "trip_aabbccdd…/trip.svx". Always a project-internal path; the original source location lives in local_settings.json per §5.3.

Things deliberately not on this class:

  • Format — derived from the entry file's extension (.svx → Survex, .dat/.mak → Compass, .wpj/.srv → Walls). A free function cwExternalCenterlineScanner::formatFor(entryFile) dispatches the scanner. Cavern itself auto-detects, so we don't need to persist it.
  • Last-solved timestamp — pure runtime UI state. Lives on the runtime cavern-output model (populated when the solve completes, cleared when a re-solve is queued). Not persisted, never dirties the project.

3.3 stationPrefix (Scope state's payload)

A simple QString on cwTrip (Phase 3) and later cwCave (Phase 6). Examples: "big-passage.east", "entrance". The consumer (cwLinePlotTask::splitLookupByCave et al.) filters the resolved station network into per-scope subsets using this string.

3.4 Stable identifiers (already present)

cwCave and cwTrip already expose QUuid id() const (cwCave.h:65, cwTrip.h:74), persisted on the existing Id member. Use these directly as the *begin label in the driver and as the lookup key in splitLookupByCave. No new field, no migration, no protobuf addition for identity.

3.5 Protobuf additions (cavewhere.proto)

message ExternalCenterline {
  string entry_file = 1;        // relative to "External Centerlines/" inside .cwproj
}

// Inside Cave message (Phase 3):
//   ExternalCenterline external_centerline = ...;
// Inside Trip message (Phase 2 for Attached, Phase 3 for Scope):
//   ExternalCenterline external_centerline = ...;
//   string station_prefix = ...;        // populated iff Scope state
// Inside CavingRegion message (Phase 6, post-v1):
//   ExternalCenterline region_external_centerline = ...;

Source paths (per-user, per-machine) live in .cwproj/local_settings.json, gitignored. See §5.3.

4. Scanner & dependency discovery

Every external-centerline operation — initial copy, watcher set refresh, garbage-collecting stranded copies — needs the same answer: which files does this attach actually depend on? The scanner is the single source of that truth.

4.1 API shape

namespace cwExternalCenterlineScanner {

struct ScanResult {
    QList<QString> dependencies;   // absolute paths, includes the entry file
    QList<QString> warnings;       // per-file non-fatal issues
};

// Synchronous. Fast for typical project sizes (regex over text + dewalls
// WPJ parse). Fatal errors (entry file unreadable, total parse failure)
// come out as Err. Soft errors (an *include points at a missing file)
// become warnings INSIDE ScanResult — the rest of the dependency
// list is still returned.
Monad::Result<ScanResult> scan(const QString& entryFile);

// Per-format scanners exposed for tests
Monad::Result<ScanResult> scanSurvex(const QString& entryFile);
Monad::Result<ScanResult> scanCompass(const QString& entryFile);  // .mak or .dat
Monad::Result<ScanResult> scanWalls(const QString& entryFile);    // .wpj

}

Why Monad::Result: its built-in Warning state (Result<T>(value, warningMessage)) returns hasError() == false, which gives clean partial-success semantics. The wrapper is reserved for "scan totally failed" (e.g. entry file doesn't exist on disk); soft per-file issues live in ScanResult.warnings since the Result<T> warning slot only holds one message.

4.2 Per-format implementations

FormatApproachLoC est.
Survex .svx Line-by-line regex for *include "<path>" and *include <path>. Strip ; comments first. Auto-append .svx when the include path has no extension (matches cavern's resolution rule). Resolve relative to current file's directory. Recurse with cycle detection (QSet of already-visited absolute paths). Follows cross-format includes (a .svx can *include a .dat). ~80
Compass .mak Regex for #filename.dat; references; resolve relative to the .mak's directory. .dat files have no inter-file dependencies (form-feed-separated surveys within a single file), so a bare .dat entry returns just itself. ~40
Walls .wpj Use dewalls::WallsProjectParser::parseFile(). Walk the resulting WpjBook tree, collect every entry where isSurvey(), resolve each entry's Name relative to its parent book's accumulated Path. dewalls handles the Borland-syntax weirdness for us. ~60

Bare .srv attaches (without a parent .wpj) also dispatch to scanWalls which simply returns the single file — .srv has no inter-file include directive of its own (multi-file structure lives in WPJ).

4.3 Why not reuse cavern for Survex scanning

Cavern is built to solve a network, not enumerate dependencies. Its diagnostic output mentions included files in "In file included from X:N:" form, but that text is not a stable API — it changes between Survex versions. The *include directive itself is stable; a 50-line regex is more durable over time than scraping cavern stderr and far cheaper than running cavern just to learn what files exist.

4.4 Async boundary

Scanners are pure-sync (matches cwSurvex3DFileReader's pattern: static, sync, fast). The async wrap happens at the call site.

From the UI (main thread, e.g. attach button):

using cwExternalCenterlineScanner::ScanResult;

QFuture<Monad::Result<ScanResult>> future =
    QtConcurrent::run([entryFile]() {
        return cwExternalCenterlineScanner::scan(entryFile);
    });

AsyncFuture::observe(future)
    .context(this, [this](Monad::Result<ScanResult> result) {
        if (result.hasError()) {
            showAttachError(result.errorMessage());
            return;
        }
        // copyDependenciesIntoProject is the first call to
        // cwExternalCenterlineSync::reconcile (see §5.2) against
        // a fresh empty attachment folder.
        m_sync->reconcile(result.value(), attachmentDirFor(ownerId));
        surfaceWarnings(result.value().warnings);
    });

Inside the line-plot pipeline (already off the main thread), the scanner is called directly — no extra wrapping needed.

Per CLAUDE.md, AsyncFuture::waitForFinished is test-only (it spins a nested event loop and causes re-entrancy bugs). All production uses go through observe().context(this, …) for Qt's auto-disconnect on destruction.

5. In-project copy workflow

5.1 Layout

my-project.cwproj/
  <existing .cwproj contents>
  External Centerlines/
    cave_6ba7b8109dad11d180b400c04fd430c8/    — one folder per attachment
      system.svx                              — entry file (original name)
      entrance.svx
      mainPassage.svx
      ...
    cave_6ba7b8119dad11d180b400c04fd430c8/
      walls.wpj
      survey1.srv
      survey2.srv
    trip_aabbccddeeff00112233445566778899/    — trip-level attachment
      trip.svx

Folder name = owner's UUID (cave_<uuid> or trip_<uuid>, matching the *begin label in the driver). The owner's QUuid id() is already persisted on cwCave / cwTrip (§3.4) and is unique by construction, so:

  • Two trips named "trip 1" in different caves — no collision (each has its own folder name).
  • Two attachments of the same entry-file name — no collision (different owner UUIDs).
  • The folder name is independent of the entry file name, so the user can rename or replace the entry file without renaming the folder.

Inside the attachment folder, the entry file keeps its original name; *include directives inside the entry resolve relative to it the same way they would on the user's disk, so the in-project copy is byte-for-byte indistinguishable from the source for Survex's purposes. cwExternalCenterline.entryFile is the path relative to External Centerlines/ — e.g. "cave_6ba7b810…/system.svx" — so the protobuf doesn't bake any per-machine assumptions.

5.2 Reconcile operation

A new helper cwExternalCenterlineSync drives the copy + GC. It runs on a worker thread as part of the line-plot async chain (§7.4), and all of its filesystem mutations are submitted through the existing cwProject / cwSaveLoad job queue — the same serialised queue that load and save go through. This is the only safe way to mutate the project tree: multiple unrelated jobs (autosave, sync-on-close, the line-plot reconcile) can otherwise race on the same files. See plans/LINEPLOT_ASYNC_PORT_RESEARCH.html for the full pipeline port and a survey of the existing job types in cwSaveLoad. 

// Sketch: the actual implementation submits each FS operation as a
// cwSaveLoad job and returns an AsyncFuture that completes when the
// last job in the batch finishes.
QFuture<Monad::Result<void>>
cwExternalCenterlineSync::reconcile(const ScanResult& scan,
                                    const QString& attachmentDir)
{
    QSet<QString> expected;
    for (const QString& absSourcePath : scan.dependencies) {
        const QString rel  = relativeToEntry(absSourcePath);
        const QString dest = attachmentDir + "/" + rel;
        enqueueCopyIfNewer(absSourcePath, dest);   // -> cwSaveLoad job
        expected.insert(dest);
    }
    // GC: anything in attachmentDir not in expected was dropped from
    // the include closure (e.g. user removed an *include line).
    for (const QString& existing : enumerateFiles(attachmentDir)) {
        if (!expected.contains(existing)) {
            enqueueRemove(existing);                // -> cwSaveLoad job
        }
    }
    return chainOnLastJob();
}

copyIfNewer definition. Compares (mtime, size) on the source and destination; copies if the source is newer or the size differs. We deliberately do not SHA the file contents:

  • Survey files are small (most under 100 KB; biggest projects in the wild are a few MB), but the scanner runs after every file change, so even a cheap hash adds up.
  • The dependency closure may include dozens of files; hashing them all on every reconcile would dominate the scan cost.
  • The downside of mtime/size is a false-negative on identical-size touch-without-edit, which just causes an unnecessary cavern re-solve — not a correctness bug.

If the mtime heuristic turns out to be flaky in practice (e.g. some text editors don't update mtime predictably), we can add an opt-in SHA-256 verify for "live-link" mode behind a setting. Not in v1.

Reconcile is called in two contexts:

  1. Initial attach. User picks an entry file from outside the project. We scan the source, copy the closure into a fresh attachment folder, persist the relative entry path. Subsequent scans run against the in-project copy. (This is the copyDependenciesIntoProject step referenced from §4.4 — it's just the first call to reconcile with an empty destination directory.)
  2. Edit-in-place. Once the closure is in the project, the user edits any file inside External Centerlines/cave_<uuid>/ directly — with their text editor, Survex's editor, Compass, whatever — pointed at the project tree. The watcher fires, we re-scan the entry, reconcile (no copy needed because source == dest), and re-solve.

Garbage collection

Two kinds of stranded copies need cleaning up:

  • Per-attachment GC (above): files inside an attachment folder that aren't in the current dependency closure. Runs on every reconcile.
  • Per-owner GC: attachment folders for trips/caves that no longer exist (the user deleted the trip, or undo brought a deletion forward). After every region-data save, walk External Centerlines/ and remove any cave_<uuid> / trip_<uuid> folder whose owner UUID isn't in the current region. Submitted as a single cwSaveLoad "remove tree" job.

Watcher ownership: a single QFileSystemWatcher lives on cwLinePlotManager (the same object that owns the AsyncFuture chain). Reconcile reports the new watched set back; the manager diffs against its current set and adds/removes watches. The watcher is never touched from worker threads — only from updateWatchedSet() on the UI thread.

5.3 Live-link vs. import mode

At attach time the user picks one of two modes:

  • Watch for changes (default, "live link"). CaveWhere records the original source path on this machine and watches both the source and the in-project copy. Edits to the source auto-copy into the project on next solve.
  • Copy once ("import mode"). CaveWhere copies the file into the project once and forgets the source path. Only the in-project copy is watched from then on. To refresh, the user re-attaches.

Source path is per-user, per-machine: stored in .cwproj/local_settings.json (gitignored), not in the region protobuf. Collaborators who pull the project never see the original attacher's path; they get the in-project copy and watch that. If they later attach their own files, they choose modes independently.

Schema: defined as a protobuf message in cavewhere.proto and serialised with protobuf's JSON codec (google::protobuf::util::MessageToJsonString) so we get the same versioning / forward-compat story as the rest of the project format. The file isn't checked in, but the schema is. 

// In cavewhere.proto
message LocalSettings {
  message Attachment {
    string owner_id = 1;        // cwCave or cwTrip QUuid (no braces, no hyphens)
    string source_path = 2;     // absolute path on this machine; empty == import mode
  }
  repeated Attachment attachments = 1;
}

Serialised to .cwproj/local_settings.json as e.g.: 

{
  "attachments": [
    {
      "ownerId": "6ba7b8109dad11d180b400c04fd430c8",
      "sourcePath": "/Users/alice/topodroid/2025-06-01/trip.svx"
    }
  ]
}

Absent entry for an owner ⇒ import mode for that attachment on this machine. Present entry with a path that no longer exists ⇒ surface a one-time warning ("Original source moved or deleted; switched to import mode"), clear the entry, continue with just the in-project copy.

6. Driver-file shape

Generated per cavern run by cwSurvexExporterRegionTask into the existing scratch location (handled by cwLinePlotTask). Native and external-centerline caves coexist; *include paths point into the project's External Centerlines/ directory. 

; Driver root. NO *case directive — cavern's default *case tolower
; is what existing native projects rely on (see prefix research, Q3).
*begin
*cs out <region-output-cs>

; ---- Cave with native shots only (today's path, but with UUID prefix) ----
*begin cave_550e8400e29b41d4a716446655440000
  *fix <...>
  *begin
    *title "Entrance Trip"
    *date 2025.06.01
    *data normal from to tape compass clino
    A1 A2 10 0 0
    A2 A3 8 90 0
  *end
*end cave_550e8400e29b41d4a716446655440000

; ---- Cave backed by an external Survex file ----
;      attachment folder == cave's own UUID
*begin cave_6ba7b8109dad11d180b400c04fd430c8
  *include "../External Centerlines/cave_6ba7b8109dad11d180b400c04fd430c8/system.svx"
*end cave_6ba7b8109dad11d180b400c04fd430c8

; ---- Cave backed by an external Compass project ----
*begin cave_6ba7b8119dad11d180b400c04fd430c8
  *include "../External Centerlines/cave_6ba7b8119dad11d180b400c04fd430c8/system.mak"
*end cave_6ba7b8119dad11d180b400c04fd430c8

; ---- Cave with one native trip AND one external-centerline trip ----
;      trip-level attachment folder == trip's own UUID
*begin cave_6ba7b8129dad11d180b400c04fd430c8
  *begin trip_aabbccdd00000000000000000000native
    *data normal from to tape compass clino
    B1 B2 5 45 0
  *end trip_aabbccdd00000000000000000000native

  *begin trip_eeff001100000000000000000000ext
    *include "../External Centerlines/trip_eeff001100000000000000000000ext/sidepassage.svx"
  *end trip_eeff001100000000000000000000ext
*end cave_6ba7b8129dad11d180b400c04fd430c8

*end
UUIDs in *begin labels are internal-driver-only. Survex requires identifiers to start with a letter, so the label is cave_<uuid-without-hyphens> / trip_<uuid-without-hyphens>. The path inside *include uses the relative path from the driver scratch location to the project's External Centerlines/; Survex accepts spaces in paths when the path is quoted.
The user-facing SVX exporter must use friendly names, not UUIDs. The driver file above is the internal one consumed by cavern on every solve — it never leaves the scratch directory and is never seen by the user. When the user picks Export → Survex, a separate code path emits friendly identifiers derived from cave/trip display names (sanitised to valid Survex identifiers: spaces → underscores, illegal characters stripped, prefix with cave_ only if the name starts with a digit, suffix with a short hash if two siblings collide). Implementation: cwSurvexExporterRegionTask grows a NameStyle enum (InternalUuid / FriendlyDisplayName) and the user-facing exporter passes FriendlyDisplayName. The internal line-plot pipeline always passes InternalUuid so name changes don't dirty the driver or risk collisions across sessions.
Native vs. external trip wrapping is asymmetric — on purpose. Native trips stay in an anonymous *begin (today's behavior, preserved); their stations resolve at cave_<X>.<name>. External trips get a named *begin trip_<uuid> wrapper so the file's internal *begin structure stays isolated; their stations resolve at cave_<X>.trip_<Y>.<tail>. Consequence: in a mixed cave, a native A1 and an external A1 are two distinct stations in cavern's network. They don't fuse on a name collision. Phase 5 equates bridge them when intended.

Region-level driver shape is deferred with Phase 6 (post-v1) — see §9. v1 only needs the cave-level and trip-level wrappers shown above.

7. Consumer-side pipeline changes

7.1 cwLinePlotTask::splitLookupByCave

Today (cwLinePlotTask.cpp:298–360):

QRegularExpression regex(QString("(\\d+)\\.(%1)").arg(stationPattern));
//                        ^integer^   ^single station-name token, no dots^

After:

// id is cwCave::id().toString(QUuid::WithoutBraces).remove('-').
// The tail is kept verbatim: it may contain dots (external *begin
// nesting) or spaces (Walls empty-name quirk emits "name " with a
// trailing space).
QRegularExpression caveRegex("cave_([0-9a-f]{32})\\.(.+)");

Lookup table goes from QVector<cwStationPositionLookup> (indexed by cave integer) to QHash<QUuid, cwStationPositionLookup>. The same UUID is the key in RegionDataPtrs::Caves so we can still emit stationPositionInCavesChanged with the right cwCave*.

7.2 Station-name validator

cwStationValidator::validCharactersRegex() is used for user input fields and should NOT be broadened — native-only projects shouldn't suddenly allow dots in station names users type.

Introduce cwStationValidator::externalStationCharactersRegex() that accepts [a-zA-Z0-9_.\- ]+ and use it only when the context is a station reference inside an externally-backed cave or trip (scrap station-reference fields, note station-reference fields). cwStation::canonicalKey remains the matching function and continues to lowercase.

Carpet-mode station picker. The carpet sub-page (TripPage.qml:32–44) currently autocompletes from cwTrip::stations(). For externally-backed scopes that list is empty — the stations live in the post-solve cwStationPositionLookup keyed by their qualified names. The picker model needs a branch: native trip → currentTrip.stations; external trip → lookup entries whose qualified prefix matches cave_<X>.trip_<Y>.*. Same data source as the per-trip station list in §8.5's External Centerline panel, so a small shared model is appropriate.

7.3 Exporter

cwSurvexExporterRegionTask::writeRegion and cwSurvexExporterCaveTask::writeCave branch on whether the cave/trip has an externalCenterline:

  • No centerline → *begin cave_<uuid>, native shots, *end (existing path with UUID label).
  • Cave centerline → *begin cave_<uuid>, then *include "<rel-path-to-external-centerlines>", then *end. Skip trip iteration.
  • Trip centerline (cave has both native and external trips) → emit each trip in its own *begin trip_<uuid> block; trips with centerlines emit *include.

All *include paths are written relative to the driver file's location via QDir::relativeFilePath. The destination is always inside the project's External Centerlines/ tree.

7.4 File watching, scanner re-run, and re-solving

cwLinePlotManager already drives re-runs via AsyncFuture::Restarter for native data changes. Extend it to also re-run when files inside External Centerlines/ change. See plans/LINEPLOT_ASYNC_PORT_RESEARCH.html for the full port plan (current pipeline survey, the cwSaveLoad job-queue API, and the staged migration to a fully AsyncFuture chain on a worker thread).

  • One QFileSystemWatcher owned by cwLinePlotManager. Watch set = union of:
    • All ScanResult.dependencies across attached centerlines (the in-project copies).
    • All source paths from local_settings.json for attachments in live-link mode, plus their scanner-discovered dependencies on the source side.
  • On region change or attachment add/remove, recompute the watched set by re-running the scanner over each attachment's entry file (twice for live-link attachments — once on source, once on project copy). Source-side scan informs the next reconcile.
  • On fileChanged, hand the work to the existing AsyncFuture::Restarter on cwLinePlotManager — no manual debounce. The restarter already coalesces rapid edits by cancelling the in-flight chain when a new edit arrives. (Native-side edits and external-file changes converge on the same restarter so they coalesce against each other, not just within their own stream.) The chain itself: 
    using cwExternalCenterlineScanner::ScanResult;

AsyncFuture::observe(sourceScanFuture)            // QFuture<Monad::Result<ScanResult>>
    .context(this, [this](Monad::Result<ScanResult> sourceScan) {
        return m_sync->copyFromSource(sourceScan);
    })
    .context(this, [this](Monad::Result<void>) {
        return scanProjectCopy();                 // QFuture<Monad::Result<ScanResult>>
    })
    .context(this, [this](Monad::Result<ScanResult> projectScan) {
        return m_sync->reconcile(projectScan);
    })
    .context(this, [this](Monad::Result<void>) {
        return runCavern();                       // QFuture<Monad::Result<CavernResult>>
    })
    .context(this, [this](Monad::Result<CavernResult> cavern) {
        updateNetwork(cavern);
    });
    Notes on this chain:
    • Every lambda parameter is fully typed (no auto r) so the chain's types are checked by the compiler, not by trial-and-error at runtime.
    • Captures are [this], never [&]. & would dangle if cwLinePlotManager stack frames unwind before the future fires.
    • .context(this, …) uses AsyncFuture's auto-disconnect on this destruction, so we don't need to manually cancel from the destructor.
    • For import-mode attachments the first two steps are skipped — the in-project copy is the source of truth.
    • Each step propagates Monad::Result — first fatal failure short-circuits the rest and lands in CavernOutputPage.
  • If a live-link source path disappears (user moved/deleted the original file outside CaveWhere), record the change in CavernOutputPage, switch that attachment to import-mode silently (clear its entry in local_settings.json), and proceed with the in-project copy. No crash.
  • If a path inside the in-project copy disappears (user deleted from the project tree), the next scan reports it as a warning, reconcile GC's it, and cavern proceeds without it — possibly with errors that surface in CavernOutputPage.

8. UX model

8.1 Attach flow

Entry points (region-level is post-v1 — see §9):

  • Trip-level: "Add Trip ▾ → External centerline…" on CavePage (case 3, TopoDroid).

After the user clicks Attach:

  1. Scanner runs (async) on the source file.
  2. If scanner errors, show inline; do not persist anything.
  3. If scanner returns warnings, surface them but proceed.
  4. Copy the dependency closure into External Centerlines/<name>/.
  5. Persist the cwExternalCenterline with the project-relative entry path.
  6. If live-link selected: write the source path to local_settings.json.
  7. Trigger a re-solve via the existing pipeline.
  8. For cave-level attach: silently auto-create one cwTrip per *begin block in the resolved network with ≥ 1 station, each declaring the matching stationPrefix. No dialog, no confirmation — the user can rename or delete trips on CavePage afterward, like any other trip.

Drag-and-drop attach is deferred (see §12).

8.2 Read-only indication

When a cave/trip has an external centerline, its shot editor is disabled with the hint "Edit in your text editor / Survex / Compass. CaveWhere will reload automatically." LRUDs, notes, scraps, team, calibrations remain editable (those live in the .cwproj, not the centerline file).

8.3 CavernOutputPage

A new StandardPage-based QML page, CavernOutputPage.qml, registered with the existing page-selection model. Single instance for the whole region. Contents (v1):

  • Cavern stderr / stdout — verbatim output of the most recent run, file:line links where cavern produced them.
  • Attached centerlines — list of attachments with last-solved timestamp, dependency file count, and any scanner warnings.
  • Driver source — collapsible disclosure showing the generated .svx for debugging.

Phase 3 adds: a status badge on cave/trip nodes in the nav tree (red when cavern errored, yellow when warnings) linking to the relevant section of CavernOutputPage.

8.4 DataMainPage entry point

A persistent LinkText on DataMainPage.qml: "Cavern output (last run: 0.4 s, 1247 stations, 2 warnings)" — updates after each solve. Click navigates to CavernOutputPage.

Key differences from Attached:

  • No [Reload] — the parent's reload covers it.
  • No own file path.
  • [Change prefix…] in place of [Detach…]. "Detaching" a Scope trip is meaningless since it doesn't own anything; [Remove trip…] just deletes the cwTrip record.
  • Trip name and prefix are independent. Renaming "east" to "East 2025-06-01" changes the display label only; the prefix-in-file (big-passage.east) is what binds to the resolved network. The prefix is exposed as a small read-only line in the panel; [Change prefix…] is how the user rebinds it.
  • Date and declination are kept and remain editable for Attached and Scope trips alike. Date drives auto-declination via the IGRF model (cwTripCalibration::autoDeclination) so it is not purely informational; many TopoDroid exports omit declination entirely and rely on CaveWhere to compute it. On scan, the date is seeded from the external file's *date (Survex), SURVEY DATE (Compass), or #DATE (Walls) directive when present. If the file carries an explicit declination (*calibrate declination / CORRECTIONS / #UNITS DECL), import it as declinationManual with autoDeclination = false; otherwise default to autoDeclination = true. The chosen declination is then injected into the driver block before *include so cavern uses it.
Scope trip lifetime. Auto-created Scope trips are tied to the parent attachment. Detaching the parent deletes them (per §8.7). A user who wants to preserve a Scope trip's notes/scraps must convert it to a native trip first (v1.x escape hatch — not in initial v1).

Notes about the panel:

  • Stations list is the entries in cwStationPositionLookup whose qualified name starts with cave_<X>.trip_<uuid>.; the displayed name strips that wrapper prefix but keeps any deeper nesting from the file (e.g. sidepassage.B1).
  • Trip metadata. Date and declination are CaveWhere-owned and editable (see §8.5 bullet above); team is CaveWhere-side informational. The scanner seeds these from the external file on first attach (Survex *date / *team / *calibrate declination, equivalents for Compass/Walls); subsequent edits are user-controlled. Trip name remains a CaveWhere display label.
  • Other calibrations (tape/compass/clino corrections, units) are hidden for externally-backed trips — cavern handles them from the file directly. Only the declination is surfaced because CaveWhere can compute it from date+location.
  • Status colours. Green dot when last solve was clean; yellow on warnings; red on errors. The line below the icon shows count + a link to the relevant CavernOutputPage section.

8.8 Open UX questions (holes surfaced by the walkthroughs)

These came out of writing the wireframes. Most are small but they need answers before Phase 2 ships.

  1. Detach default for copies. Resolved — Detach deletes the copied files. No "keep" toggle. Rationale: git history in the .cwproj is the backup; if the user wants to recover the closure they roll back the commit. Keeping stale copies around just clutters External Centerlines/ and accumulates over time. Use RemoveAskBox.qml (or a very small variant) as the confirm: the same lightweight inline pattern already used by trip / cave / fix-station removal. For cave-level detach, the confirm text also lists the count of auto-created Scope trips that will be removed (per §8.7).
  2. Cave-level attach with existing native trips. Not a real issue. Users who want to add external survey data to a Native cave use trip-level attach — "Add Trip ▾ → External centerline…" — which produces a Mixed cave with both native and external trips. Cave-level external is reserved for the fresh-cave entry point on DataMainPage. No convert-in-place flow is needed in v1.
  3. Add Trip dropdown. Use a QQ.Menu attached to the existing Add Trip button (popup on click) with two items: Native trip and External centerline…. Don't introduce a new split-button widget; the menu is enough.
  4. Trip metadata for external trips (date, decl, team). Resolved: Date and Decl are CaveWhere-owned and editable; the scanner seeds them from the file's *date / *calibrate declination (Survex equivalents in Compass/Walls) on first attach. Date drives auto-declination via IGRF the same way it does for native trips. The driver block re-emits the chosen date/decl before *include so cavern uses the CaveWhere-managed values. Team stays editable as informational.
  5. Mid-solve UI state. Resolved — inherited from cwLinePlotManager's existing pipeline (last-good network stays visible while a re-solve is in flight). No new UI.
  6. Fix stations interaction. Fix Stations sub-page stays visible for native, external, and mixed caves. CaveWhere fix stations always get emitted as *fix in the driver, inside the cave-level *begin cave_<uuid> wrapper, so they apply to whichever station names the user references — native or external. Caveat: if the external file already pins the same station with its own *fix, cavern errors on the duplicate. We surface that as a CavernOutputPage error referencing both the CaveWhere fix entry and the source-file line; the user decides which one wins (delete the CaveWhere fix, or remove the one in the source file). Same caveat already exists for native surveys imported with fixes, so no new UX is needed.
  7. Auto-declination. Resolved — applies to external trips too. cwSurvexExporterUtils::makeDeclinationContext uses cave fixes + per-trip date today; the same path runs for external trips. The driver block emits the resolved value as *calibrate declination before *include, overriding any value the external file carries. On first scan, if the file has its own *calibrate declination / CORRECTIONS / #UNITS DECL, it seeds declinationManual with autoDeclination = false; otherwise autoDeclination stays on.
  8. CavePage stats when cave-level attached. Keep the existing CavePage stats line unchanged in v1 — no extra cavern-derived numbers (station count, shot count, loop count, connected-component count). The wireframes in §8 that show "247 stations · 89 shots · 3 loops · 1 connected component" are aspirational; they get added when there's a clear demand for them, not as part of the initial implementation.
  9. What stations does the TripPage panel show for trip-level attach? Proposed: the file's resolved stations under this trip's *begin trip_<uuid> scope, displayed with the wrapper prefix stripped (so cave_X.trip_Y.A1 shows as A1) but any deeper nesting from the file kept (cave_X.trip_Y.sidepassage.B1 shows as sidepassage.B1).
  10. Naming collisions in mixed mode. Native and external trips can have the same CaveWhere display name (it's just a label). Their internal UUIDs differ. No real collision in cavern (see §6 note). UI-wise, append " (external)" to the display when both exist? Recommendation: don't, the 📎 icon is sufficient.
  11. Trip-level attach when the file defines no *begin (a flat .svx with just shot data). Stations come back as cave_X.trip_Y.<name> — no inner prefix. Trip station list shows names verbatim. No special case.
  12. Sub-pages on externally-backed scopes. The Carpet sub-page still makes sense (it's about notes, not shots). Fix Stations sub-page is gated per question 6. Leads sub-page stays. New sub-page: CavernOutputPage, but per §8.3 it's a global page linked from DataMainPage, not a cave/trip sub-page.

  • QML test: attach a fixture .svx at cave level, verify the trip list reflects the resolved trips from the file (or a stand-in indicator) and is read-only.
  • QML test: introduce a cavern error, verify the badge appears on the affected cave node and clicking it navigates to the right section of CavernOutputPage.

Phase 4 — Equates editor

Goal: bridge namespaces when CaveWhere-native data needs to connect to external data, or two external files need to be glued.

Design notes

  • From Phase 0 research: cavern accepts fully-qualified *equate at the driver root without requiring *export in the external file. The equate editor compiles to *equate cave_<A>.x cave_<B>.y lines emitted just before the driver's closing *end.
  • Equates live at the cave level (or region level once Phase 6 ships).

Acceptance criteria

  • Add an equate between a native-CaveWhere station and an external-Survex station, verify both end up at the same coordinate after solve.
  • Edit an equate, verify re-solve fires.

Phase 5 — Orphan detection

Goal: surface notes/scraps that reference station names no longer present in the resolved network. Quality-of-life feature that works for native-only projects too.

Mechanism

  • After each solve, diff the set of station names referenced by notes/scraps against the resolved network (per cave).
  • Show orphans in a CavernOutputPage section with per-row actions: "remap to…" (autocomplete) or "delete reference".

Acceptance criteria

  • Create a scrap referencing station A1, rename A1 in the source file, re-solve, verify the orphan appears with remap target options.

Phase 5b — Replace / re-link attachment

Goal: let the user point an existing attachment at a different file (replace) or a moved file (re-link) without losing the trips, notes, and scraps they've already built on top of the old attachment. v1.x — lands after Case 2/3 ship and we have real user data.

Two flows, one dialog

  • Re-link — source file moved on disk; content unchanged. UI: when the watcher reports the source path missing, the ExternalCenterlinePanel shows a yellow "Source not found" banner with a Re-link… button. Picker selects the new path; the scanner confirms prefix/station set matches the cached in-project copy; we update local_settings.json's sourcePath and resume watching. No project-data change.
  • Replace — user picks a different file for the same scope (e.g. newer TopoDroid export, revised cave survey). UI: Replace centerline… action on the panel opens a picker, then a diff-and-confirm dialog (see §10 in the UX prototype).

Diff logic (replace flow)

  • Scan the new file. Build the resolved *begin tree and station set.
  • For each existing externally-backed cwTrip in this scope, look up its stationPrefix in the new tree:
    • Found & stations ≥ 1 → Kept. Report new station count.
    • Found but station count fell (or rose) → Kept with a delta note.
    • Not found → Orphaned. Keep the trip + its notes/scraps; mark with error badge; user rebinds via Change prefix… or deletes later.
  • For each new *begin block in the new file with ≥1 station and no matching trip, offer auto-create (default on).

UI

  • Re-link… banner button when watcher reports the source path missing.
  • Replace centerline… action in the ExternalCenterlinePanel actions menu.
  • Diff dialog (per §10 in the prototype): list of existing trips with Kept / Orphaned fate, list of new blocks with checkbox for auto-create, toggle to delete orphaned trips outright.

Files touched

  • cwExternalCenterlineSync — new replaceSource() / relinkSource() APIs.
  • cwExternalCenterlineScanner — add a "diff against existing scan" helper that emits the Kept/Orphaned/New classification.
  • ExternalCenterlinePanel.qml — add Re-link banner + Replace menu action.
  • ReplaceCenterlineDialog.qml — new.

Acceptance criteria

  • QML test: attach fixture A, manually rename the source on disk, verify Re-link banner appears; pick new path, verify watcher resumes.
  • QML test: attach fixture A with three Scope trips, run Replace targeting fixture B (one prefix removed, one added). Verify the orphaned trip keeps its notes/scraps with an error badge, the new block gets an auto-created trip, and untouched trips' notes/scraps are unchanged.

10. Testing strategy

10.1 C++ tests (Catch2)

Per-process safety: all tests use QTemporaryDir with PID embedded in the name (CLAUDE.md convention). All cavern runs use in-process cavern_run serialized through the existing static mutex in cwCavernTask.

  • [ExternalCenterline] — serialize/deserialize round-trip, equality, hash.
  • [Scanner][Survex] — nested *include chain; cycle detection; missing-include warning; cross-format .dat include.
  • [Scanner][Compass] — .mak with multiple .dat references; bare .dat returns just itself.
  • [Scanner][Walls] — .wpj tree walk via dewalls; bare .srv returns just itself.
  • [Sync][Reconcile] — copy on first run; no-op when source == dest with matching mtime; GC of stranded files.
  • [Attach][Trip] — build a region with one cave + one trip with an attached .svx, run the full pipeline, assert resolved stations are present in the lookup.
  • [Attach][Cave] — cave-level attach.
  • [Attach][Region] — region-level attach with two cwCaves declaring prefix subtrees.
  • [Attach][Watcher] — touch a file in External Centerlines/, assert rerunSurvex fires (signal spy).
  • [Attach][Error] — broken external file produces an error captured on the cave/trip without crash.
  • [Attach][Mixed] — cave with both native trips and external-centerline trips coexisting.
  • [Equate] (Phase 5) — native + external equate produces matching coordinates.
  • [Orphan] (Phase 6) — orphan diff is correct after rename in source.

Run with narrow tag filters per the CLAUDE.md feedback (don't run the full suite during dev).

10.2 QML tests

  • tst_ExternalCenterlineAttachTrip.qml — attach via the UI, verify shot editor disabled, verify CavernOutputPage shows expected content.
  • tst_CavernOutputPage.qml — renders error text, attached-centerlines list, driver disclosure.
  • tst_NavBadge.qml (Phase 3) — badge appears on cave node when its cavern run errored.

Use tryVerify/tryCompare for cavern-completion waits (per CLAUDE.md). The async re-solve makes fixed waits especially brittle.

11. Risks and open items

RiskMitigation
UUID change breaks consumers of splitLookupByCave outside cwLinePlotTask. Grep for caveIndex / splitLookupByCave consumers in Phase 1; migrate to UUID lookup. Most are in the same file.
QFileSystemWatcher misses changes (notably Windows network shares, macOS sandboxed environments). Manual "Reload" button on CavernOutputPage as fallback. Document the limitation.
Walls .wpj via dewalls hasn't been exercised at scale by CaveWhere — behavior on edge-case projects is unknown. Phase 1 acceptance gate. If dewalls misbehaves, fall back to ".srv direct attach only" for v1; full WPJ becomes a follow-up.
Large region attaches on mobile may make cavern runs slow at open time. Survex submodule is mobile-capable now (per updated CLAUDE.md). Profile in Phase 4; if needed, persist a cached .3d in the project for read-only mobile viewing.
External file uses *infer equates or other settings that could leak past the *include boundary. Phase 0 verified Survex pops *begin settings on *end, including via *include return. Driver discipline: wrap every *include in a dedicated *begin…*end block. Lint-test the generator does this.
User accidentally deletes a file from External Centerlines/<name>/. Reconcile GC is based on the latest scan result — a missing file becomes a scan warning, not a silent GC. Cavern then errors on the missing *include, which surfaces in CavernOutputPage. Re-attaching restores.
Two attach levels conflict (e.g., region attach + per-cave attach pointing at the same data). Phase 4: region-level attach disables per-cave attach controls; loading old projects with both warns and uses region.
Cavern error message text changes between Survex versions, breaking error-path UX. Render verbatim; don't parse for content beyond "exit code ≠ 0". File:line links use the standard path:line: prefix.
Scanner re-runs on every file change could thrash on a project with many rapid saves. Debounce ~200 ms in the watcher; AsyncFuture::Restarter collapses bursts to one run. Scanners are sub-millisecond per file, so this is a budget concern, not a correctness one.
Native and external station names within the same cave live at different prefix depths (cave_X.A1 vs. cave_X.trip_Y.A1). Users could be confused when copying a station name from a native trip into an external trip's scrap reference. The per-trip station picker (carpet mode + ExternalCenterlinePanel list) handles this implicitly — users don't type qualified names, they pick from a filtered list. Document the asymmetry in user-facing help text on the Trip page for externally-backed trips ("station names here come from your survey file").
"Hide-not-delete" for native trips on cave-level attach leaves stale data the user forgot about; project file grows. Always-visible disclosure on CavePage ("3 hidden native trips — detach to restore"). Detach restores them. Phase 6 orphan-detection bonus: surface hidden trips that haven't been touched in N solves.

12. Deferred (post-v1)

  • Region-level attach (Case 1) — Phase 6 above. Adds the Structure view, bulk note-folder import, region-level equates UI, "covers stations under prefix" on cwCave. Deferred because Cases 2 + 3 + equates cover the realistic v1 audience, and the Structure view is the highest-UX-risk piece of the whole feature — better designed once we have feedback from earlier phases.
  • Drag-and-drop attach — nice polish, not load-bearing.
  • External-file conflict resolution — if two users edit the project copy and the live-link source concurrently, no merge UX. Reload-on-change is the v1 answer.
  • Per-tool deep linking — click a station in CaveWhere, jump to that line in Compass/Walls/Survex's editor. Future polish via OS file-handler integration.
  • Bundled external snapshot for mobile — cached .3d shipped in the project for read-only mobile viewing without re-solving. Becomes relevant if Phase 4 mobile profiling reveals a need.
