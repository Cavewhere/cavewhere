1. Overview

Let CaveWhere reference Compass (.dat / .mak), Walls (.wpj / .srv), and Survex (.svx) centerline data directly instead of importing it. The external centerline files are user-owned in their tool of choice, but on attach we copy them into the project — collocated with the owner cave or trip inside the project's existing <dataRoot>/ tree, in a new external-centerline/ subdirectory next to the existing notes/ — so the .cwproj stays self-contained and git sync works without surprises. There is no top-level External Centerlines/ directory; see §5.1 for the exact layout and the rationale.

Inside CaveWhere these files are read-only: the user edits them with their native tool (or any text editor with the project open); a QFileSystemWatcher notices the change and triggers a re-solve. CaveWhere never writes back to .svx / .dat / .wpj / .srv — that drops the format-preserving-writer problem entirely.

The cavern solve, the .3d read, and the station-network pipeline are unchanged. Two new ingredients:

  1. cwCave and cwTrip can carry an cwExternalCenterline — a path of the entry file within the owner's external-centerline/ subdir (usually just the basename) — that the exporter translates to *include "…" instead of native shot data.
  2. A cwExternalCenterlineScanner module finds the transitive dependency closure of an attached entry file (e.g. a .svx that *includes siblings, a .wpj's book tree), drives the initial copy into the owner's external-centerline/ subdir, and feeds the watcher set on subsequent runs.
Phase 0 is already complete — see plans/EXTERNAL_FILE_PREFIXING_RESEARCH.md for the prefix model investigation across CaveWhere, Survex, Compass, and Walls, plus the empirical cavern runs that answer the case-mode question (do not set *case preserve at the driver root) and the *export-for-equates question (not required).
Companion plan: line-plot async port. plans/LINEPLOT_ASYNC_PORT_RESEARCH.html covers the port of cwLinePlotTask + sub-tasks to a fully AsyncFuture chain on a worker thread, and surveys the existing cwSaveLoad job queue that the external-centerline filesystem mutations need to go through. Read it alongside this plan — the worker-thread chain and the cwSaveLoad::Custom job hook are prerequisites for Phase 1 here.

Concurrency contract: the whole chain runs on a single worker thread via AsyncFuture; results land back on the UI thread via .context(this, …). Every filesystem mutation inside an external-centerline/ subdir is submitted as a cwSaveLoad job so it serialises against the other project-tree mutators (autosave, sync-on-close, the existing cave/trip rename/remove jobs, etc.). See plans/LINEPLOT_ASYNC_PORT_RESEARCH.html for the current-state survey, the existing cwSaveLoad job API, and the staged port.

Cave identifiers move from "integer index into the region snapshot" to the existing persisted cwCave::id() / cwTrip::id() UUIDs. Forced by the external-centerline work (external files contain their own nested *begin structure, and the current single-integer-prefix regex in splitLookupByCave doesn't survive it). No new identity field — reuse what's already persisted.

3. Data model changes

3.1 cwTrip / cwCave gain two new states

A cwCave or cwTrip exists in one of three states:

StateCarriesDriver emissionUI badge
Native (today) Survey chunks *begin… + native shots none
Attached (new) externalCenterline pointing at a file *begin…*include "…"…*end 📎
Scope (new, for case 2 internal trips and case 1 region scopes) stationPrefix string none — purely a consumer-side filter into another scope's resolved network 📎

The states are mutually exclusive on a single entity. From the user's perspective there's only "trip" with one optional 📎 badge; the Attached / Scope distinction is internal data-model only.

3.1.1 Attached vs. Scope — what's the difference?

Both states render with the same 📎 badge and both are "externally-backed" from the user's point of view, but the data model treats them very differently. The short version:

  • Attached = "this entity owns a file." There's an external-centerline/ subdir under this cave's or trip's directory in the project tree, the file lives in it, and the driver emits *begin…*include "…"…*end around it so cavern reads the file directly. Detaching deletes the subdir.
  • Scope = "this entity is a view into another Attached entity's resolved network." No file. No external-centerline/ subdir. The driver emits nothing for it — cavern never sees Scope trips. Instead, stationPrefix filters the parent's solved station lookup, so the trip surfaces a labeled subset of the parent's stations (and is the anchor for the user's notes / scraps on those stations). Removing the parent's attachment removes the Scope entity automatically — it has no meaning standalone.

Concrete walkthrough — cave-level attach of a Walls .wpj that internally declares two passages: 

; Big Cave's external-centerline/ holds big-cave.wpj, which after
; resolving expands to a tree like:
*begin big_cave
  *begin east_passage
    E1 E2 10 0 0
    ...
  *end east_passage
  *begin west_passage
    W1 W2 8 90 0
    ...
  *end west_passage
*end big_cave

After the user clicks "Add Cave → External cave…":

EntityStateCarriesDriver emissionWhere the data lives
"Big Cave" (cwCave) Attached externalCenterline.entryFile = "big-cave.wpj" *begin cave_<uuid> *include "…/Big Cave/external-centerline/big-cave.wpj" *end File on disk in the project tree.
"East Passage" (auto-created cwTrip) Scope stationPrefix = "east_passage" (none — cavern doesn't know this trip exists) Nothing on disk. Just a CaveWhere-side label that filters the cave's resolved station lookup to cave_<uuid>.east_passage.* at display time.
"West Passage" (auto-created cwTrip) Scope stationPrefix = "west_passage" (none) Same as above, filter = cave_<uuid>.west_passage.*.

The user then attaches notes and scraps to East Passage just like they would on a native trip. Those notes / scraps reference stations like E1 and E2 — resolved by joining East Passage's prefix to produce cave_<uuid>.east_passage.E1, which exists in the station lookup that cavern produced from the file.

3.1.2 When to use which state

Use caseEntry pointResulting stateWhy
Case 3 — TopoDroid trip import. Field surveyor has a single day's shots in a .svx file; wants to use it directly without retyping into SurveyEditor. "Add Trip → External centerline…" on CavePage. The new trip is Attached. The cave can stay Native (or be Mixed with other native trips). The trip owns the file. Detach removes the file, the trip's external-centerline/ subdir, and the trip itself.
Case 2 — whole-cave external survey. User has an entire cave's centerline in a Walls .wpj or multi-file Survex/Compass project; wants CaveWhere as the notes / scraps layer on top. "Add Cave → External cave…" on DataMainPage. The cave is Attached (it owns the file). One Scope cwTrip is auto-created per *begin block in the resolved network. Only the cave owns a file — the file's internal *begin structure isn't ours to mirror as separate files. The Scope trips give the user a per-passage anchor for notes / scraps without inventing fake "files" they didn't write.
Case 1 — region-level external survey (post-v1, Phase 6). One file covers multiple caves; the user wants to split it by *begin hierarchy across cwCaves. "More options → Attach region-level centerline…" on DataMainPage (Phase 6). The region is Attached. Per-cave entries become Scope on cwCave (with stationPrefix there); per-trip entries become Scope on cwTrip. Same logic as Case 2, one level up: only the region owns the file; caves and trips are labeled windows into the resolved network.

In other words: Attached marks ownership of a file on disk; Scope marks a labeled subset of a parent's resolved stations. Cave-level attach (Case 2) is the only case that produces both states in the same cave — the cave is Attached, its auto-created trips are Scope.

Why this matters for the driver. The driver only knows about Attached entities (they emit *include). Scope entities never produce a line of driver output — cavern resolves the parent's file once and the Scope filter runs entirely on CaveWhere's side, against the station lookup that comes back from cwSurvex3DFileReader. That's why splitLookupByCave's regex (§7.1) only cares about cave_<uuid>. prefixes — the trip-level scope filter is applied later, per-trip, by the per-scope station-list model introduced in Phase 2.

3.2 cwExternalCenterline (Attached state's payload)

Lives alongside cwSurveyChunk in the cave/trip data, owned by the cwCave or cwTrip in the Attached state. Deliberately tiny — just enough to find the entry file in the project tree.

FieldTypePurpose
entryFile QString Path of the entry file within the owner's external-centerline/ subdir. Usually a basename (e.g. "mycave.svx"); may carry a subdirectory when the source layout requires it (e.g. "subdir/main.svx" when the user's source places the entry inside a nested folder of its include closure). The owner cave/trip's directory is resolved via cwSaveLoad::externalCenterlineDirHelper (§5.1), so the protobuf payload doesn't carry an owner UUID or any machine-specific prefix. The original source location lives in the cwExternalSourceSettings store per §5.3.

Things deliberately not on this class:

  • Format — derived from the entry file's extension (.svx → Survex, .dat/.mak → Compass, .wpj/.srv → Walls). A free function cwExternalCenterlineScanner::formatFor(entryFile) dispatches the scanner. Cavern itself auto-detects, so we don't need to persist it.
  • Last-solved timestamp — pure runtime UI state. Lives on the runtime cavern-output model (populated when the solve completes, cleared when a re-solve is queued). Not persisted, never dirties the project.

3.3 stationPrefix (Scope state's payload)

A simple QString on cwTrip (Phase 3) and later cwCave (Phase 6). Examples: "big-passage.east", "entrance". The consumer (cwLinePlotTask::splitLookupByCave et al.) filters the resolved station network into per-scope subsets using this string.

3.4 Stable identifiers (already present)

cwCave and cwTrip already expose QUuid id() const (cwCave.h:65, cwTrip.h:74), persisted on the existing Id member. Use these directly as the *begin label in the driver and as the lookup key in splitLookupByCave. No new field, no migration, no protobuf addition for identity.

4. Scanner & dependency discovery

Every external-centerline operation — initial copy, watcher set refresh, garbage-collecting stranded copies — needs the same answer: which files does this attach actually depend on? The scanner is the single source of that truth.

4.1 API shape

namespace cwExternalCenterlineScanner {

struct ScanResult {
    QList<QString> dependencies;   // absolute paths, includes the entry file
    QList<QString> warnings;       // per-file non-fatal issues
};

// Synchronous. Fast for typical project sizes (regex over text + dewalls
// WPJ parse). Fatal errors (entry file unreadable, total parse failure)
// come out as Err. Soft errors (an *include points at a missing file)
// become warnings INSIDE ScanResult — the rest of the dependency
// list is still returned.
Monad::Result<ScanResult> scan(const QString& entryFile);

// Per-format scanners exposed for tests
Monad::Result<ScanResult> scanSurvex(const QString& entryFile);
Monad::Result<ScanResult> scanCompass(const QString& entryFile);  // .mak or .dat
Monad::Result<ScanResult> scanWalls(const QString& entryFile);    // .wpj

}

Why Monad::Result: its built-in Warning state (Result<T>(value, warningMessage)) returns hasError() == false, which gives clean partial-success semantics. The wrapper is reserved for "scan totally failed" (e.g. entry file doesn't exist on disk); soft per-file issues live in ScanResult.warnings since the Result<T> warning slot only holds one message.

4.2 Per-format implementations

FormatApproachLoC est.
Survex .svx Line-by-line regex for *include "<path>" and *include <path>. Strip ; comments first. Auto-append .svx when the include path has no extension (matches cavern's resolution rule). Resolve relative to current file's directory. Recurse with cycle detection (QSet of already-visited absolute paths). Follows cross-format includes (a .svx can *include a .dat). ~80
Compass .mak Regex for #filename.dat; references; resolve relative to the .mak's directory. .dat files have no inter-file dependencies (form-feed-separated surveys within a single file), so a bare .dat entry returns just itself. ~40
Walls .wpj Use dewalls::WallsProjectParser::parseFile(). Walk the resulting WpjBook tree, collect every entry where isSurvey(), resolve each entry's Name relative to its parent book's accumulated Path. dewalls handles the Borland-syntax weirdness for us. ~60

Bare .srv attaches (without a parent .wpj) also dispatch to scanWalls which simply returns the single file — .srv has no inter-file include directive of its own (multi-file structure lives in WPJ).

4.3 Why not reuse cavern for Survex scanning

Cavern is built to solve a network, not enumerate dependencies. Its diagnostic output mentions included files in "In file included from X:N:" form, but that text is not a stable API — it changes between Survex versions. The *include directive itself is stable; a 50-line regex is more durable over time than scraping cavern stderr and far cheaper than running cavern just to learn what files exist.

4.4 Async boundary

Scanners are pure-sync (matches cwSurvex3DFileReader's pattern: static, sync, fast). The async wrap happens at the call site.

From the UI (main thread, e.g. attach button):

using cwExternalCenterlineScanner::ScanResult;

QFuture<Monad::Result<ScanResult>> future =
    QtConcurrent::run([entryFile]() {
        return cwExternalCenterlineScanner::scan(entryFile);
    });

AsyncFuture::observe(future)
    .context(this, [this](Monad::Result<ScanResult> result) {
        if (result.hasError()) {
            showAttachError(result.errorMessage());
            return;
        }
        // copyDependenciesIntoProject is the first call to
        // cwExternalCenterlineSync::reconcile (see §5.2) against
        // a fresh empty attachment folder. The attachment dir is the
        // owner cave/trip's external-centerline/ subdir, resolved via
        // cwSaveLoad::externalCenterlineDirHelper.
        m_sync->reconcile(result.value(),
                          m_project->saveLoad()->externalCenterlineDir(owner));
        surfaceWarnings(result.value().warnings);
    });

Inside the line-plot pipeline (already off the main thread), the scanner is called directly — no extra wrapping needed.

Per CLAUDE.md, AsyncFuture::waitForFinished is test-only (it spins a nested event loop and causes re-entrancy bugs). All production uses go through observe().context(this, …) for Qt's auto-disconnect on destruction.

5. In-project copy workflow

5.1 Layout

External-centerline files live inside the owner cave's or trip's directory under <dataRoot>/, in a new external-centerline/ subdirectory. The layout mirrors how notes/ already sits inside trips/<TripName>/ today (see docs/cavewhere-format-spec.md §Directory Layout). There is no top-level External Centerlines/ directory — each attachment is collocated with the cave or trip that owns it. 

Big Cave.cwproj/                          ; project root
  Big Cave.cwproj                         ; descriptor (sanitized region name)
  .git/
  .gitattributes
  Big Cave/                               ; dataRoot
    Big Cave/                             ; caveDir (sanitized cave name)
      Big Cave.cwcave
      external-centerline/                ; NEW — cave-level attach
        mycave.svx                        ; entry file (original basename)
        entrance.svx                      ; sibling files from the include closure
        mainPassage.svx
      trips/
        Trip 1/                           ; native trip (today's layout)
          Trip 1.cwtrip
          notes/
            ...
        Trip 2/                           ; trip-level attach
          Trip 2.cwtrip
          external-centerline/            ; NEW — trip-level attach
            mytrip.svx
          notes/
            ...

Why nested under the owner's dir. The existing cwSaveLoad infrastructure already handles every operation the attachment subtree needs:

  • Friendly names — for free. Cave and trip directories are sanitized display names via cwSaveLoad::caveDirHelper / cwSaveLoad::tripDirHelper (cwSaveLoad.cpp:4308-4322); they're collision-free by virtue of nesting inside their parent's tree. No cave_<uuid> / trip_<uuid> folder names are needed, and a user browsing the project in Finder sees a folder layout that matches the in-app cave/trip tree.
  • Rename — for free. When the user renames a cave or trip, the existing rename job in cwSaveLoad (region rename at cwSaveLoad.cpp:3304-3344, with parallel cave/trip handlers) re-paths the entire owner directory, including any external-centerline/ subtree. No separate rename plumbing on external attachments.
  • Delete — for free. When a cave or trip is removed, removeDirectory (cwSaveLoad.cpp:3259) recursively deletes the owner directory, including any external-centerline/. The per-owner GC pass that an unscoped top-level layout would need (walking External Centerlines/ after every region save to find folders whose owners no longer exist) is no longer necessary — per-attachment GC inside the dir is still done by reconcile (§5.2).

Inside external-centerline/, the entry file keeps its original basename, and any sibling files pulled in by the include closure sit alongside it (preserving their relative layout in the source, including nested subdirectories if the source used them). *include directives in the entry file resolve relative to it the same way they would on the user's disk, so the in-project copy is byte-for-byte indistinguishable from the source for cavern's purposes.

Two new helpers next to the existing caveDirHelper / tripDirHelper / noteDirHelper: 

// in cwSaveLoad.h, alongside the existing dir helpers
static QDir externalCenterlineDirHelper(const QDir &caveDir);
static QDir externalCenterlineDirHelper(const QDir &tripDir);

// returns <caveDir or tripDir>/external-centerline/

Public lookups follow the same pattern as the existing cwSaveLoad::dirPrivate(cwCave*) / (cwTrip*) accessors: 

QDir cwSaveLoad::externalCenterlineDir(const cwCave* cave);
QDir cwSaveLoad::externalCenterlineDir(const cwTrip* trip);

The owner is implicit from the location, so the protobuf payload (§3.2) doesn't carry an owner UUID inside the centerline message itself. cwExternalCenterline.entryFile is the path of the entry file relative to external-centerline/ (usually just a basename), so the protobuf bakes in no per-machine assumptions.

Collisions are handled by the existing sanitization layer, not by us. Two attachments of the same entry-file name on different owners can't collide because they live in different parent dirs. Two trips with display name "Trip 1" in different caves can't collide because each lives under its own caveDir/trips/. The only edge case — two siblings with the same sanitized name — is already handled by cwNameUtils::sanitizeFileName and the existing cave/trip-add path; external attachments inherit that for free.

5.2 Reconcile operation

A new helper cwExternalCenterlineSync drives the copy + GC. It runs on a worker thread as part of the line-plot async chain (§7.4), and all of its filesystem mutations are submitted through the existing cwProject / cwSaveLoad job queue — the same serialised queue that load and save go through. This is the only safe way to mutate the project tree: multiple unrelated jobs (autosave, sync-on-close, the line-plot reconcile) can otherwise race on the same files. See plans/LINEPLOT_ASYNC_PORT_RESEARCH.html for the full pipeline port and a survey of the existing job types in cwSaveLoad. 

// Sketch: the actual implementation submits each FS operation as a
// cwSaveLoad job and returns an AsyncFuture that completes when the
// last job in the batch finishes.
QFuture<Monad::Result<void>>
cwExternalCenterlineSync::reconcile(const ScanResult& scan,
                                    const QString& attachmentDir)
{
    QSet<QString> expected;
    for (const QString& absSourcePath : scan.dependencies) {
        const QString rel  = relativeToEntry(absSourcePath);
        const QString dest = attachmentDir + "/" + rel;
        enqueueCopyIfNewer(absSourcePath, dest);   // -> cwSaveLoad job
        expected.insert(dest);
    }
    // GC: anything in attachmentDir not in expected was dropped from
    // the include closure (e.g. user removed an *include line).
    for (const QString& existing : enumerateFiles(attachmentDir)) {
        if (!expected.contains(existing)) {
            enqueueRemove(existing);                // -> cwSaveLoad job
        }
    }
    return chainOnLastJob();
}

copyIfNewer definition. Compares (mtime, size) on the source and destination; copies if the source is newer or the size differs. We deliberately do not SHA the file contents:

  • Survey files are small (most under 100 KB; biggest projects in the wild are a few MB), but the scanner runs after every file change, so even a cheap hash adds up.
  • The dependency closure may include dozens of files; hashing them all on every reconcile would dominate the scan cost.
  • The downside of mtime/size is a false-negative on identical-size touch-without-edit, which just causes an unnecessary cavern re-solve — not a correctness bug.

If the mtime heuristic turns out to be flaky in practice (e.g. some text editors don't update mtime predictably), we can add an opt-in SHA-256 verify for "live-link" mode behind a setting. Not in v1.

Reconcile is called in two contexts:

  1. Initial attach. User picks an entry file from outside the project. We scan the source, copy the closure into a fresh attachment folder, persist the relative entry path. Subsequent scans run against the in-project copy. (This is the copyDependenciesIntoProject step referenced from §4.4 — it's just the first call to reconcile with an empty destination directory.)
  2. Edit-in-place. Once the closure is in the project, the user edits any file inside the cave's or trip's external-centerline/ directly — with their text editor, Survex's editor, Compass, whatever — pointed at the project tree. The watcher fires, we re-scan the entry, reconcile (no copy needed because source == dest), and re-solve.

Garbage collection

Only one kind of GC pass is needed:

  • Per-attachment GC (above): files inside an attachment dir that aren't in the current dependency closure (e.g. the user removed an *include line and the previously- included file is no longer reachable). Runs on every reconcile.

The "per-owner GC" pass that the previous (top-level External Centerlines/) layout required — walking the attachments tree after every region save to find folders whose owner UUID is gone — is no longer needed. When a cave or trip is removed, the existing cwSaveLoad remove job (cwSaveLoad.cpp:3251-3300) recursively deletes the owner directory, including any external-centerline/ subtree it contained. Same for the cave/trip rename path: the existing rename job moves the whole directory atomically, so we don't have to chase orphaned UUID-named folders left behind by renames.

Watcher ownership: a single QFileSystemWatcher lives on cwLinePlotManager (the same object that owns the AsyncFuture chain). Reconcile reports the new watched set back; the manager diffs against its current set and adds/removes watches. The watcher is never touched from worker threads — only from updateWatchedSet() on the UI thread.

5.3 Live-link vs. import mode

Superseded (2026-07-16, decided during Phase 2 after commit 4 — see the "Direction change" section at the top of Phase 2). The two-mode dichotomy below is dropped; there is no auto-copy and no mode choice. The single model: the in-project copy is the source of truth; the source path is always remembered (per-machine, QSettings-backed cwExternalSourceSettings, exactly as specified below); the source-side watcher and an open-time probe only set a per-owner stale flag; the trip panel offers a user-triggered [Update] that runs scan + reconcile + re-solve. Rationale: auto-reconcile at arbitrary times forced mid-git-sync fold-in semantics, surprise dirty-bit flips and save prompts, a collaborator ping-pong corollary, and a load-bearing async scan pipeline — all for the value of saving one click. The storage design, watcher infrastructure, and reconcile primitive below are unchanged; only the action taken on a source change moved from automatic to user-triggered. Mentions of "live link" elsewhere in this master plan read as "remembered source"; "auto-copy on next solve" reads as "stale banner + [Update]".

At attach time the user picks one of two modes:

  • Watch for changes (default, "live link"). CaveWhere records the original source path on this machine and watches both the source and the in-project copy. Edits to the source auto-copy into the project on next solve.
  • Copy once ("import mode"). CaveWhere copies the file into the project once and forgets the source path. Only the in-project copy is watched from then on. To refresh, the user re-attaches.

Source path is per-user, per-machine: stored in QSettings keyed by owner UUID — not in the project tree and not in the region protobuf. Owner UUIDs (cwCave::id() / cwTrip::id()) are globally unique, so one store serves every project on the machine with no per-project namespace. Collaborators who pull the project never see the original attacher's path; they get the in-project copy and watch that. If they later attach their own files, they choose modes independently.

Copies of a project alias each other — intended v1 behavior. Every mechanism that copies a project (Git clone, Save As, file copy, restored backup) preserves cave/trip UUIDs, so all copies of a project on one machine share their live-link entries: setting, re-linking, or switching an attachment to import mode in one copy applies to every copy. That's the desired outcome for the common case — two checkouts of the same survey pointed at the same source file — and per-copy divergence is deliberately not supported in v1. Widening the key to (project identity, owner) was considered and rejected: a path key orphans entries on every project move/rename, and a stored project UUID is duplicated by the same copy mechanisms it would need to distinguish.

Why not a file inside the project. The original design stored this as .cwproj/local_settings.json (gitignored), which is what Phase 1 shipped. That breaks for bundled .cw projects: a .cw is a zipped .cwproj extracted to a fresh QTemporaryDir on every open, and cwSaveLoad::saveBundledArchive applies the project's git-exclude patterns when zipping — so an in-project local_settings.json either evaporates when the project closes (excluded from the zip) or, if it were shipped, would leak the attacher's machine paths to everyone who receives the file. A machine-global store survives .cw open/close cycles, works identically for .cwproj, survives a project being moved or renamed on disk, and removes the gitignore/zip-exclusion dependency entirely. Phase 2 reworks the Phase 1 storage location; no migration path is needed — this is unreleased dev code. Dead entries for deleted projects are harmless (a UUID lookup that never matches) and are not GC'd in v1.

Why QSettings and not a hand-rolled app-data JSON file. An earlier revision of this section specified a protobuf-JSON file at <AppDataLocation>/local_settings.json. Dropped for three reasons. (1) Concurrency: a whole-file load→modify→save cycle is last-writer-wins across two CaveWhere instances; QSettings serializes writers with a lock file and merges per-key on sync, so instances upserting different owners both persist. (2) Consistency: QSettings is where the rest of CaveWhere's per-machine state already lives (recent projects, job/font/PDF settings) — a third persistence mechanism needs a reason to exist, and this one didn't have it. (3) Roaming: AppDataLocation is the roaming AppData dir on Windows, which would sync machine-local absolute paths across machines. CaveWhere is not single-instance-enforced, so (1) is real.

Shape: cwExternalSourceSettings is a QObject store owned by cwRootData (exposed as the externalSourceSettings CONSTANT property). Every read/write goes straight to QSettings — one key per owner under the externalCenterlineSources group: 

externalCenterlineSources/<owner-uuid-without-braces> = /Users/alice/topodroid/2025-06-01/trip.svx
// absent key            == import mode (no entry)
// key with empty value  == import mode entry (attached without live-link)

Mutations emit externalCenterlineSourcesChanged on the mutating instance; cwLinePlotManager observes the rootData-owned store and recomputes its watch set on every change, so writers (the attach orchestrator, the MissingSourceBanner's switch-to-import action) just write. Future per-machine state should use QSettings likewise (its own domain-specific group), not a new sibling store.

Absent entry for an owner ⇒ import mode for that attachment on this machine. Present entry with a path that no longer exists ⇒ surface a one-time warning ("Original source moved or deleted; switched to import mode"), clear the entry, continue with just the in-project copy.

5.4 Bundled .cw projects

A bundled .cw is a zipped .cwproj (cwProject::BundledGitFileType) extracted to a QTemporaryDir on load and re-zipped on save, with the .git history inside the archive. Because the extraction is a full .cwproj directory tree, everything in this plan works on it unchanged: the external-centerline/ layout (§5.1), the reconcile jobs (§5.2), the QFileSystemWatcher, the rename/delete cascades, and the git-history-as-backup rationale for detach (§8.8 q1). Two .cw-specific requirements:

  • Live-link storage must live outside the project tree — handled by §5.3's QSettings store. The extraction dir is a fresh temp path every session, so nothing per-machine can persist inside it.
  • Reconcile mutations must dirty the project. When a live-link reconcile copies an updated source file into the extraction mid-session, cwSaveLoad must emit localMutationOccurred so cwProject::modified() flips and the save affordance appears — otherwise a .cw user who quits without saving silently loses the refresh (for .cwproj the copy is durable immediately, so this never mattered). Precedent: commit 50595738 wired exactly this for Git-driven GIS-layer changes. Phase 2 verifies whether the reconcile jobs already flow through the modified-bit plumbing and wires them if not.

The practical consequence for .cw users: the "edit the in-project copy" workflow (§5.2 case 2) is a poor fit — the copy lives at a hidden temp path that changes every session. Live-link is the primary mode for bundled projects; the attach dialog's default (§8.1) already reflects that.

UUIDs in *begin labels are internal-driver-only. Survex requires identifiers to start with a letter, so the label is cave_<uuid-without-hyphens> / trip_<uuid-without-hyphens>. The path inside *include is computed by QDir::relativeFilePath from the driver scratch location to the owner's external-centerline/ subdir inside <dataRoot>; Survex accepts spaces in paths when the path is quoted (which sanitized cave/trip dir names may contain).
The user-facing SVX exporter must use friendly names, not UUIDs. The driver file above is the internal one consumed by cavern on every solve — it never leaves the scratch directory and is never seen by the user. When the user picks Export → Survex, a separate code path emits friendly identifiers derived from cave/trip display names (sanitised to valid Survex identifiers: spaces → underscores, illegal characters stripped, prefix with cave_ only if the name starts with a digit, suffix with a short hash if two siblings collide). Implementation: cwSurvexExporterRegion::exportRegion (post-port free function) grows a NameStyle enum (InternalUuid / FriendlyDisplayName) and the user-facing exporter passes FriendlyDisplayName. The internal line-plot pipeline always passes InternalUuid so name changes don't dirty the driver or risk collisions across sessions.
Native vs. external trip wrapping is asymmetric — on purpose. Native trips stay in an anonymous *begin (today's behavior, preserved); their stations resolve at cave_<X>.<name>. External trips get a named *begin trip_<uuid> wrapper so the file's internal *begin structure stays isolated; their stations resolve at cave_<X>.trip_<Y>.<tail>. Consequence: in a mixed cave, a native A1 and an external A1 are two distinct stations in cavern's network. They don't fuse on a name collision. Phase 5 equates bridge them when intended.

Region-level driver shape is deferred with Phase 6 (post-v1) — see §9. v1 only needs the cave-level and trip-level wrappers shown above.

7.4 File watching, scanner re-run, and re-solving

cwLinePlotManager already drives re-runs via AsyncFuture::Restarter for native data changes. Extend it to also re-run when files inside any cave/trip's external-centerline/ change. See plans/LINEPLOT_ASYNC_PORT_RESEARCH.html for the full port plan (current pipeline survey, the cwSaveLoad job-queue API, and the staged migration to a fully AsyncFuture chain on a worker thread).

  • One QFileSystemWatcher owned by cwLinePlotManager. Watch set = union of:
    • All ScanResult.dependencies across attached centerlines (the in-project copies).
    • All source paths from the cwExternalSourceSettings store for attachments in live-link mode, plus their scanner-discovered dependencies on the source side.
  • On region change or attachment add/remove, recompute the watched set by re-running the scanner over each attachment's entry file (twice for live-link attachments — once on source, once on project copy). Source-side scan informs the next reconcile.
  • On fileChanged, hand the work to the existing AsyncFuture::Restarter on cwLinePlotManager — no manual debounce. The restarter already coalesces rapid edits by cancelling the in-flight chain when a new edit arrives. (Native-side edits and external-file changes converge on the same restarter so they coalesce against each other, not just within their own stream.) The chain itself (flat — one .context(this, …) per step, matching the simpler 3-step shape from plans/LINEPLOT_ASYNC_PORT_RESEARCH.html §5; the earlier "copyFromSource + scanProjectCopy" pre-pass was cut because the in-project copy is byte-identical to the source by construction): 
    using cwExternalCenterlineScanner::ScanResult;
using Monad::Result;

QFuture<Result<ScanResult>> scanFuture =
    QtConcurrent::run([entryPath]() { return scan(entryPath); });

AsyncFuture::observe(scanFuture)
    .context(this, [this](Result<ScanResult> scan) -> QFuture<Result<void>> {
        if (scan.hasError()) return AsyncFuture::completed(scan.error());
        return m_sync->reconcile(scan.value(), m_attachmentDir);    // -> cwSaveLoad jobs
    })
    .context(this, [this](Result<void> reconciled) -> QFuture<Result<cwCavernRunner::Result>> {
        if (reconciled.hasError()) return AsyncFuture::completed(reconciled.error());
        return cwConcurrent::run([this]() { return cwCavernRunner::run(m_svxPath, m_3dPath); });
    })
    .context(this, [this](Result<cwCavernRunner::Result> cavern) {
        updateNetwork(cavern);   // UI thread; publishes log + loop-closure stats too
    });
    Notes on this chain:
    • Every lambda parameter is fully typed (no auto r) so the chain's types are checked by the compiler, not by trial-and-error at runtime.
    • Captures are [this], never [&]. & would dangle if cwLinePlotManager stack frames unwind before the future fires.
    • .context(this, …) uses AsyncFuture's auto-disconnect on this destruction, so we don't need to manually cancel from the destructor.
    • Driver-side note. The lineplot port keeps SolveError on LinePlotResultData (not wrapped in Monad::Result<…>) so the rich cavern log + netskel loop-closure stats can ride along to the UI. The external-centerline chain above terminates in updateNetwork, which can mint a SolveError with Step::Cavern from the cwCavernRunner::Result and forward it to the same cwLinePlotManager properties (hasSolveError, cavernLog, loopClosureStats, solveErrorMessage) that CavernOutputPage already binds to.
    • For import-mode attachments the source-scan step uses the in-project copy as input — no separate "copy from source" pre-pass; reconcile is a no-op when source == dest.
    • Each step propagates Monad::Result — first fatal failure short-circuits the rest and lands in CavernOutputPage.
  • If a live-link source path disappears (user moved/deleted the original file outside CaveWhere), record the change in CavernOutputPage, switch that attachment to import-mode silently (clear its entry in the cwExternalSourceSettings store), and proceed with the in-project copy. No crash.
  • If a path inside the in-project copy disappears (user deleted from the project tree), the next scan reports it as a warning, reconcile GC's it, and cavern proceeds without it — possibly with errors that surface in CavernOutputPage.

7.5 User-facing export (File → Export)

v1 rule: when the region contains any cave or trip in the Attached or Scope state, both the Survex and Compass exporters refuse to run. The user-facing export entry points (File → Export → Survex… and … → Compass… on cwSurveyExportManager) are disabled with a tooltip explaining why.

Why disable rather than emit something

Every flavor of "support export with externals present" breaks the round-trip story:

  • Inline the contents of the external file into the output: lossy. Comments, *case, *calibrate, *infer directives, original formatting, and the file's own *begin structure all collapse into the parent's namespace. The user's source file was the truth; the exported copy is a derivation that drifts the moment they re-edit the original.
  • Re-emit the solved network as native shots: worse — drops file structure entirely, flattens passages, loses every directive cavern consumed.
  • Multi-file export (driver .svx + bundled copies of each external) is technically feasible for Survex (the format is the format with *include), but: (a) only works when every external is Survex — a .wpj can't be *include'd from a Survex export; (b) Compass / Walls have no analogue; (c) output becomes a folder, which the existing exporter UX doesn't do; (d) re-introduces the "which copy is canonical" problem on re-import. Deferred — see §12.
  • Format translation (e.g. translating a .wpj to Survex on export): we don't have a translator and don't want to build one. cavern reads all three, but re-emitting in a different format is a separate, much harder problem.

The export audience is "I have native CaveWhere data, give me a .svx / .dat to share with another tool." External-centerline projects already came from another tool — the user has the source files (in the project's external-centerline/ subdirs) and doesn't need an export to share them.

Implementation

cwSurveyExportManager gains: 

Q_PROPERTY(bool canExport READ canExport NOTIFY canExportChanged FINAL)
Q_PROPERTY(QString exportDisabledReason READ exportDisabledReason NOTIFY canExportChanged FINAL)

bool canExport() const;                // false if any cave/trip is Attached or Scope
QString exportDisabledReason() const;  // empty when canExport() is true

Plain member + setter + signal pattern, recomputed whenever the region's cave/trip set changes or any cave/trip's externalCenterline / stationPrefix changes (signal from the existing change-notifier in cwLinePlotManager's observed set is sufficient — the same triggers that drive the line-plot pipeline). The export buttons on FileMenu.qml / SurveyExportMenu.qml bind their enabled to canExport and their tooltip text to exportDisabledReason.

The export functions themselves (cwSurveyExportManager::exportToSurvex / ::exportToCompass) keep an internal early-return guard so a programmatic caller (e.g. a script or test) that bypasses the UI gets a clear "won't export with externals present" error rather than a silent half-baked file.

UX text

The disabled-export tooltip / inline message:

Cannot export — this project contains external centerline attachments. Use your original .svx / .dat / .mak / .wpj / .srv files (in each cave or trip's external-centerline/ subdir inside the project) to share.

Doesn't matter whether the offending entity is Attached or Scope — both kinds break round-trip. The message is the same and points the user at the project tree (where the source files actually live). Future versions may add a "Detach all and export" escape hatch as a destructive menu item; not in v1.

Key differences from Attached:

  • No [Reload] — the parent's reload covers it.
  • No own file path.
  • [Change prefix…] in place of [Detach…]. "Detaching" a Scope trip is meaningless since it doesn't own anything; [Remove trip…] just deletes the cwTrip record.
  • Trip name and prefix are independent. Renaming "east" to "East 2025-06-01" changes the display label only; the prefix-in-file (big-passage.east) is what binds to the resolved network. The prefix is exposed as a small read-only line in the panel; [Change prefix…] is how the user rebinds it.
  • Date is kept and remains editable for Attached and Scope trips alike; on scan it is seeded from the external file's *date (Survex), SURVEY DATE (Compass), or #DATE (Walls) directive when present. Declination ownership depends on the file (verified empirically against cavern — a directive emitted in the driver before *include is overridden by the file's own directive, not the other way around): if the file carries an explicit declination (*calibrate declination / Compass DECLINATION: / #UNITS DECL), the file owns it — the panel shows it read-only ("Declination 7.2° — set by your survey file") and the driver emits nothing (any injected value would be dead text). Only when the file has no declination of its own (the common TopoDroid case) is it CaveWhere-owned: editable, autoDeclination = true by default (IGRF from date + fix location), and the resolved value is injected into the driver block before *include, where it demonstrably applies.
Scope trip lifetime. Auto-created Scope trips are tied to the parent attachment. Detaching the parent deletes them (per §8.7). A user who wants to preserve a Scope trip's notes/scraps must convert it to a native trip first (v1.x escape hatch — not in initial v1).

Notes about the panel:

  • Stations list is the entries in cwStationPositionLookup whose qualified name starts with cave_<X>.trip_<uuid>.; the displayed name strips that wrapper prefix but keeps any deeper nesting from the file (e.g. sidepassage.B1).
  • Trip metadata. Date is CaveWhere-owned and editable; declination is file-owned (read-only display) when the file carries its own directive and CaveWhere-owned (editable, IGRF auto) otherwise — see the §8.5 metadata bullet and q7; team is CaveWhere-side informational. The scanner seeds these from the external file on first attach (Survex *date / *team / *calibrate declination / *declination, equivalents for Compass/Walls); subsequent edits are user-controlled. Trip name remains a CaveWhere display label.
  • Other calibrations (tape/compass/clino corrections, units) are hidden for externally-backed trips — cavern handles them from the file directly. Only the declination is surfaced because CaveWhere can compute it from date+location.
  • Status colours. Green dot when last solve was clean; yellow on warnings; red on errors. The line below the icon shows count + a link to the relevant CavernOutputPage section.

8.8 Open UX questions (holes surfaced by the walkthroughs)

These came out of writing the wireframes. Most are small but they need answers before Phase 2 ships.

  1. Detach default for copies. Resolved — Detach deletes the copied files. No "keep" toggle. Rationale: git history in the .cwproj is the backup; if the user wants to recover the closure they roll back the commit. Keeping stale copies around just clutters the cave/trip dir's external-centerline/ subtree and accumulates over time. Use RemoveAskBox.qml (or a very small variant) as the confirm: the same lightweight inline pattern already used by trip / cave / fix-station removal. For cave-level detach, the confirm text also lists the count of auto-created Scope trips that will be removed (per §8.7).
  2. Cave-level attach with existing native trips. Not a real issue. Users who want to add external survey data to a Native cave use trip-level attach — "Add Trip ▾ → External centerline…" — which produces a Mixed cave with both native and external trips. Cave-level external is reserved for the fresh-cave entry point on DataMainPage. No convert-in-place flow is needed in v1.
  3. Add Trip dropdown. Use a QQ.Menu attached to the existing Add Trip button (popup on click) with two items: Native trip and External centerline…. Don't introduce a new split-button widget; the menu is enough.
  4. Trip metadata for external trips (date, decl, team). Resolved: Date is CaveWhere-owned and editable, seeded from the file's *date (Survex equivalents in Compass/Walls) on first attach. Declination is file-owned when the file carries its own directive (shown read-only with a "set by your survey file" hint) and CaveWhere-owned otherwise (editable, IGRF auto-declination, injected into the driver before *include). See the §8.5 metadata bullet and q7 for the cavern-verified precedence. Team stays editable as informational.
  5. Mid-solve UI state. Resolved — inherited from cwLinePlotManager's existing pipeline (last-good network stays visible while a re-solve is in flight). No new UI.
  6. Fix stations interaction. Fix Stations sub-page stays visible for native, external, and mixed caves. CaveWhere fix stations always get emitted as *fix in the driver, inside the cave-level *begin cave_<uuid> wrapper, so they apply to whichever station names the user references — native or external. Caveat: if the external file already pins the same station with its own *fix, cavern errors on the duplicate. We surface that as a CavernOutputPage error referencing both the CaveWhere fix entry and the source-file line; the user decides which one wins (delete the CaveWhere fix, or remove the one in the source file). Same caveat already exists for native surveys imported with fixes, so no new UX is needed.
  7. Auto-declination. Resolved — applies to external trips only when the file carries no declination of its own. Empirically verified against cavern: a *calibrate declination emitted in the driver before *include takes effect when the included file has no declination directive, but is overridden by the file's own *calibrate declination the moment the file sets one (later directive wins within the scope) — the original "driver overrides file" claim here was backwards. Contract: scanner detects whether the file carries its own declination (*calibrate declination, the modern *declination command incl. auto / Compass DECLINATION: / #UNITS DECL). If yes → file-owned; panel shows the value read-only with a "set by your survey file" hint ("automatic (IGRF)" for *declination auto); autoDeclination = false; driver emits no declination for this trip. If no → CaveWhere-owned; autoDeclination = true by default (cwSurvexExporterUtils::makeDeclinationContext, same path as native trips); the driver emits the resolved value before *include. Note Compass and Walls files essentially always embed their own declination handling, so the editable field is in practice a Survex/TopoDroid affordance.
  8. CavePage stats when cave-level attached. Keep the existing CavePage stats line unchanged in v1 — no extra cavern-derived numbers (station count, shot count, loop count, connected-component count). The wireframes in §8 that show "247 stations · 89 shots · 3 loops · 1 connected component" are aspirational; they get added when there's a clear demand for them, not as part of the initial implementation.
  9. What stations does the TripPage panel show for trip-level attach? Proposed: the file's resolved stations under this trip's *begin trip_<uuid> scope, displayed with the wrapper prefix stripped (so cave_X.trip_Y.A1 shows as A1) but any deeper nesting from the file kept (cave_X.trip_Y.sidepassage.B1 shows as sidepassage.B1).
  10. Naming collisions in mixed mode. Native and external trips can have the same CaveWhere display name (it's just a label). Their internal UUIDs differ. No real collision in cavern (see §6 note). UI-wise, append " (external)" to the display when both exist? Recommendation: don't, the 📎 icon is sufficient.
  11. Trip-level attach when the file defines no *begin (a flat .svx with just shot data). Stations come back as cave_X.trip_Y.<name> — no inner prefix. Trip station list shows names verbatim. No special case.
  12. Sub-pages on externally-backed scopes. The Carpet sub-page still makes sense (it's about notes, not shots). Fix Stations sub-page is gated per question 6. Leads sub-page stays. New sub-page: CavernOutputPage, but per §8.3 it's a global page linked from DataMainPage, not a cave/trip sub-page.
  13. Disabled-export presentation. Resolved — keep the File → Export → Survex… / … → Compass… menu items visible but disabled when the region contains any Attached or Scope entity (per §7.5). Tooltip on hover carries the exportDisabledReason text. Don't hide them — hiding hurts discoverability and confuses returning users who remember the feature being there.
  14. Cancel during attach. Resolved — the attach dialog stays open and modal while the scanner runs, showing a progress strip; a Cancel button aborts the scan/copy and persists nothing. No provisional cave / trip appears in the tree until the scan + copy completes (the cavern solve that follows runs in the background and falls under the existing last-good-network-while-solving rule, question 5). Survey files are typically <1 MB, so the perceived dialog time should be sub-second in the common case; the Cancel exists for the pathological cases (huge include closures, slow disks).
  15. Collaborator opens a Git-shared project for the first time. Resolved — on a machine whose cwExternalSourceSettings store has no entry for an attachment, the ExternalCenterlinePanel shows "Source: in-project copy (this machine)" with a small [Switch to live-link…] link that opens a file picker. CaveWhere uses the in-project copy as the source of truth until the user opts in to live-link by picking their own local copy. No warning, no error — this is the expected state for everyone who didn't originally attach the file. Collaborator's notes and scraps work normally either way.
  16. Re-link discoverability across app restarts. Resolved — the missing-source check is part of the attachment's startup probe, not just the QFileSystemWatcher path. When cwLinePlotManager loads the project, any attachment whose cwExternalSourceSettings sourcePath no longer exists on disk gets a persistent "Source not found at <path>" banner on its ExternalCenterlinePanel with [Re-link…] and [Forget source] buttons. The watcher fileChanged path (Phase 5b) feeds into the same banner state, so the two routes converge on one UI surface. Without this, the watcher-only design quietly drops the moved-while-app-closed case — which is the common one.
  17. Scope trip auto-naming. Resolved — use the *begin label verbatim from the file (*begin east_passage → trip name "east_passage"). No underscore-to-space substitution, no title-casing. Survey files have intentional naming conventions that we shouldn't second-guess; the user renames the trip from the trip-detail page if they want a friendlier display label. (The internal stationPrefix stays verbatim regardless of the display name — per §8.5 they're independent.)
  18. Attach error display in the picker dialog. Resolved — a red inline message area below the file-path field shows the scanner's error text (e.g. "Could not read /Users/…/trip.svx: permission denied" or "trip.svx:42: unknown directive *foo"). The [Attach] button stays disabled until the user picks a different file or fixes the source on disk. Warnings (scanner-side, e.g. a missing *include target) show in the same area in yellow but do not disable [Attach]; the user can proceed and see the warnings on CavernOutputPage after the solve.

Smaller items documented but not blockers

  • Bulk attach (multiple TopoDroid .svx files in one go) — deferred to v1.x. v1 attaches one file at a time.
  • Trip ordering in mixed mode (§8.6's wireframe interleaves native and external trips by date) — use the existing trip sort; no special case for the externalCenterline state.
  • Orphan scraps pre-Phase 5. If the user renames a station inside an external file, scraps referencing the old name show a broken reference until Phase 5 ships orphan-detection UI. Documented limitation, not a v1 blocker.

Phase 5b — Replace / re-link attachment

Goal: let the user point an existing attachment at a different file (replace) or a moved file (re-link) without losing the trips, notes, and scraps they've already built on top of the old attachment. v1.x — lands after Case 2/3 ship and we have real user data.

Two flows, one dialog

  • Re-link — source file moved on disk; content unchanged. UI: when the watcher reports the source path missing, the ExternalCenterlinePanel shows a yellow "Source not found" banner with a Re-link… button. Picker selects the new path; the scanner confirms prefix/station set matches the cached in-project copy; we update the cwExternalSourceSettings sourcePath and resume watching. No project-data change.
  • Replace — user picks a different file for the same scope (e.g. newer TopoDroid export, revised cave survey). UI: Replace centerline… action on the panel opens a picker, then a diff-and-confirm dialog (see §10 in the UX prototype).

Diff logic (replace flow)

  • Scan the new file. Build the resolved *begin tree and station set.
  • For each existing externally-backed cwTrip in this scope, look up its stationPrefix in the new tree:
    • Found & stations ≥ 1 → Kept. Report new station count.
    • Found but station count fell (or rose) → Kept with a delta note.
    • Not found → Orphaned. Keep the trip + its notes/scraps; mark with error badge; user rebinds via Change prefix… or deletes later.
  • For each new *begin block in the new file with ≥1 station and no matching trip, offer auto-create (default on).

UI

  • Re-link… banner button when watcher reports the source path missing.
  • Replace centerline… action in the ExternalCenterlinePanel actions menu.
  • Diff dialog (per §10 in the prototype): list of existing trips with Kept / Orphaned fate, list of new blocks with checkbox for auto-create, toggle to delete orphaned trips outright.

Files touched

  • cwExternalCenterlineSync — new replaceSource() / relinkSource() APIs.
  • cwExternalCenterlineScanner — add a "diff against existing scan" helper that emits the Kept/Orphaned/New classification.
  • ExternalCenterlineTripPanel.qml + ExternalCenterlineMissingSourceBanner.qml — wire up the Re-link button to ReplaceCenterlineDialog; add a Replace menu action to the panel's overflow / actions row.
  • ReplaceCenterlineDialog.qml — new.

Acceptance criteria

  • QML test: attach fixture A, manually rename the source on disk, verify Re-link banner appears; pick new path, verify watcher resumes.
  • QML test: attach fixture A with three Scope trips, run Replace targeting fixture B (one prefix removed, one added). Verify the orphaned trip keeps its notes/scraps with an error badge, the new block gets an auto-created trip, and untouched trips' notes/scraps are unchanged.

10. Testing strategy

10.1 Cross-cutting rules

  • Per-process safety — all C++ tests use QTemporaryDir with PID embedded in the name (CLAUDE.md convention) so the nightly multi-process Catch2 run doesn't collide.
  • Cavern serialisation — in-process cavern_run serialised through the existing static mutex inside cwCavernRunner::run; tests don't need to coordinate separately.
  • Narrow tag filters during dev — never run the full cavewhere-test binary while iterating (CLAUDE.md / memory). Pick the smallest tag intersection that covers the commit's gate.
  • Existing tests always green — every commit in any phase must keep the pre-existing Catch2 + QML suites passing. This is the hidden gate on every per-commit gate in each phase file; not re-listed per commit.
  • QML waits use tryVerify/tryCompare (CLAUDE.md). Fixed waits are flaky and especially bad against the async re-solve.

10.2 Phase-specific per-commit gates & fixtures

Each phase file owns its commit sequence, per-commit Catch2 tag gates, and fixture file list. Today only Phase 1 is fully planned:

  • Phase 1 per-commit test gates — 11 commits, one Catch2 tag block each.
  • Phase 1 fixture files — nine fixtures mapped to the commits that need them.

Phases 2–5 grow their own equivalents inside EXTERNAL_FILE_PHASE<N>.html when each phase gets a full planning pass.

11. Risks and open items

RiskMitigation
UUID change breaks consumers of splitLookupByCave outside cwLinePlotTask. Grep for caveIndex / splitLookupByCave consumers in Phase 1; migrate to UUID lookup. Most are in the same file.
QFileSystemWatcher misses changes (notably Windows network shares, macOS sandboxed environments). Manual "Reload" button on CavernOutputPage as fallback. Document the limitation.
Walls .wpj via dewalls hasn't been exercised at scale by CaveWhere — behavior on edge-case projects is unknown. Phase 1 acceptance gate. If dewalls misbehaves, fall back to ".srv direct attach only" for v1; full WPJ becomes a follow-up.
Large region attaches on mobile may make cavern runs slow at open time. Survex submodule is mobile-capable now (per updated CLAUDE.md). Profile in Phase 4; if needed, persist a cached .3d in the project for read-only mobile viewing.
External file uses *infer equates or other settings that could leak past the *include boundary. Phase 0 verified Survex pops *begin settings on *end, including via *include return. Driver discipline: wrap every *include in a dedicated *begin…*end block. Lint-test the generator does this.
User accidentally deletes a file from a cave/trip's external-centerline/ subdir. Reconcile GC is based on the latest scan result — a missing file becomes a scan warning, not a silent GC. Cavern then errors on the missing *include, which surfaces in CavernOutputPage. Re-attaching restores.
Two attach levels conflict (e.g., region attach + per-cave attach pointing at the same data). Phase 4: region-level attach disables per-cave attach controls; loading old projects with both warns and uses region.
Cavern error message text changes between Survex versions, breaking error-path UX. Render verbatim; don't parse for content beyond "exit code ≠ 0". File:line links use the standard path:line: prefix.
Scanner re-runs on every file change could thrash on a project with many rapid saves. Debounce ~200 ms in the watcher; AsyncFuture::Restarter collapses bursts to one run. Scanners are sub-millisecond per file, so this is a budget concern, not a correctness one.
Native and external station names within the same cave live at different prefix depths (cave_X.A1 vs. cave_X.trip_Y.A1). Users could be confused when copying a station name from a native trip into an external trip's scrap reference. The per-trip station picker (carpet mode + ExternalCenterlinePanel list) handles this implicitly — users don't type qualified names, they pick from a filtered list. Document the asymmetry in user-facing help text on the Trip page for externally-backed trips ("station names here come from your survey file").
"Hide-not-delete" for native trips on cave-level attach leaves stale data the user forgot about; project file grows. Always-visible disclosure on CavePage ("3 hidden native trips — detach to restore"). Detach restores them. Phase 6 orphan-detection bonus: surface hidden trips that haven't been touched in N solves.
